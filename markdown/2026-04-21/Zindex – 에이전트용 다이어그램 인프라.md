## 핵심 개념

> [!abstract]
> AI 에이전트가 다이어그램을 지속적인 상태로 관리할 수 있게 하는 인프라로, Diagram Scene Protocol(DSP)을 통해 40개 이상의 검증 규칙과 다중 렌더 방식을 지원합니다.

## 상세 내용

- DSP 프로토콜로 에이전트가 의미론적으로 다이어그램을 선언하면 자동으로 레이아웃과 렌더링이 처리됩니다.
- 버전 관리, 증분 편집, 결정적 실행을 통해 다이어그램을 데이터베이스처럼 다룰 수 있습니다.
- SVG, PNG 등 여러 출력 형식과 아키텍처, BPMN, ER 다이어그램 등 도메인별 기능을 지원합니다.

> [!tip] 왜 중요한가
> AI 에이전트와 협업하여 복잡한 다이어그램을 자동으로 생성하고 관리해야 하는 개발 환경에서 필수적인 인프라입니다.

## 전문 번역

# 다이어그램을 위한 에이전트 인프라, Zindex

AI 에이전트가 다이어그램을 만들고 편집할 때, 보통 일회용 결과물로 취급합니다. 하지만 실제로는 더 튼튼한 구조가 필요합니다. Zindex는 다이어그램을 지속 가능한 상태로 관리하는 플랫폼입니다. 구조화된 연산, 불변 버전 관리, 40개 이상의 검증 규칙, 그리고 여러 렌더링 타겟을 지원합니다. 모두 Diagram Scene Protocol(DSP)을 통해서요.

## DSP: 다이어그램의 기계 인터페이스

DSP의 핵심 아이디어는 간단합니다. 에이전트는 **어떻게 그려야 하는지가 아니라, 무엇이 존재하는지를 선언**합니다.

```json
{
  "schemaVersion": "0.1",
  "scene": {
    "id": "payments-arch",
    "units": "px"
  },
  "elements": [
    {
      "id": "api",
      "kind": "node",
      "nodeType": "service",
      "shape": "roundedRect",
      "label": "API Gateway"
    },
    {
      "id": "queue",
      "kind": "node",
      "nodeType": "queue",
      "label": "Job Queue"
    },
    {
      "id": "e1",
      "kind": "edge",
      "from": { "elementId": "api" },
      "to": { "elementId": "queue" },
      "router": "orthogonal"
    }
  ]
}
```

좌표를 일일이 지정할 필요가 없습니다. 노드와 엣지, 그리고 그들 사이의 관계만 명시하면 되거든요.

## 의미 중심의 설계

다이어그램은 기하학적 배치가 아니라 의미 관계로 정의됩니다. 위치 계산은 자동화된 레이아웃 엔진이 담당합니다.

내부에 탑재된 **Sugiyama 스타일의 계층적 레이아웃 파이프라인**이 노드 위치, 엣지 경로, 라벨 배치를 자동으로 결정합니다. 에이전트가 그래프 구조를 제시하면, 엔진이 기하학을 처리하는 것이죠.

## 수정 가능한 구조

안정적인 ID 덕분에 **증분 업데이트**가 가능합니다. 노드 하나를 추가하거나 엣지를 옮길 때, 전체 다이어그램을 다시 생성할 필요가 없습니다. 필요한 부분만 패치하면 됩니다.

## 렌더러 독립적

하나의 canonical한 scene에서 여러 형식으로 출력합니다. SVG와 PNG에 4가지 테마(clean, dark, blueprint, sketch)를 지원합니다.

같은 입력이라면 같은 출력이 나옵니다. 검증 → 정규화 → 레이아웃 → 렌더링. 파이프라인의 모든 단계가 검사 가능합니다.

## Zindex가 필요한 이유

다이어그램은 단순한 마크업 생성을 넘어섭니다. 검증, 레이아웃, 버전 관리, 다중 포맷 렌더링이 필요합니다.

### 6가지 핵심 특징

**1. 증분 편집**
전체 재생성 대신 패치 기반 업데이트. 에이전트가 개별 요소를 수정해도 나머지는 보존됩니다.

**2. 결정론적 실행**
실행할 때마다 동일한 결과. 실패 원인을 추적할 수 있고, 파이프라인의 각 단계를 검사할 수 있습니다.

**3. 지속 가능한 상태**
버전 이력, diff, 안정적인 ID. 다이어그램은 일회용 출력이 아니라 버전 관리되는 산출물입니다.

**4. 다중 에이전트 준비**
여러 에이전트가 협력할 수 있는 공유 인프라. 일회성 생성기가 아닙니다.

**5. 도메인 특화**
아키텍처, BPMN 워크플로우, ER 다이어그램, 시퀀스 다이어그램, 조직도, 네트워크 토폴로지를 목표로 설계. 범용 벡터 도형이 아니죠.

**6. 프로덕션 레벨**
17가지 연산 타입, 40개 이상의 의미 검증 규칙, 인증, 레이트 제한, PostgreSQL 저장소를 갖춘 완성도 높은 시스템입니다.

## 아키텍처에서의 위치

```
에이전트 / LLM
    ↓ (무엇을 바꿀지 결정)
  Zindex (DSP 프로토콜로 변경 적용)
    ↓ (렌더링 결과 전달)
  렌더러 (화면에 표시)
```

Zindex는 에이전트의 추론과 시각적 출력 사이의 중간 계층입니다. 애플리케이션 상태를 관리하는 데이터베이스처럼, Zindex는 다이어그램을 지속 가능한 상태로 만들고 생성, 수정, 검증, 렌더링하는 에이전트 네이티브 런타임입니다.
