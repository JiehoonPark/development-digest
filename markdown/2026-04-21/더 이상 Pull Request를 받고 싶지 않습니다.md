## 핵심 개념

> [!abstract]
> 오픈소스 유지보수자가 LLM 시대에 외부 PR보다는 다른 형태의 기여가 더 가치 있다고 주장합니다. 코드 작성이 더 이상 병목이 아니며, 보안 위험과 동기화 오버헤드를 고려할 때 자신이 직접 LLM으로 구현하는 것이 더 효율적이라고 설명합니다.

## 상세 내용

- LLM의 발전으로 코드 작성 시간 단축되었으며, 이제 병목은 이해, 설계, 리뷰 단계임
- 외부 PR은 보안 위험, 코드 스타일 차이, 동기화 오버헤드 등의 문제 야기
- 대신 피드백, 아이디어 논의, 버그 리포트, 프로토타입 PR, 코드 리뷰, 포크 등의 기여 방식을 권장

> [!tip] 왜 중요한가
> 오픈소스 프로젝트에 효과적으로 기여하는 방식과 LLM 시대의 개발 패러다임 변화를 이해할 수 있습니다.

## 전문 번역

# PR 병합 요청, 더 이상 받지 않습니다

당신이 제 오픈소스 프로젝트를 즐겨주고 도와주려는 마음이 고마워요. 하지만 우리의 협업 방식을 다시 생각해봐야 할 것 같습니다. 최근 시간을 낭비하고 있다는 느낌이 자꾸만 듭니다.

## PR을 병합하고 싶지 않은 이유

먼저 솔직한 얘기를 하자면, 저는 당신을 잘 모르기 때문에 악의적인 코드가 섞여 있을 가능성을 항상 염두에 두고 검토해야 합니다. 그래서 PR을 병합하는 것보다 제가 직접 구현하는 게 더 안전합니다.

더 큰 문제는 코드 스타일이 매우 개인적이고 주관적이라는 거예요. 포맷팅, 네이밍, 구조, 의존성, 접근 방식 등에서 당신과 제 선호도가 다를 수밖에 없습니다. 그 다음엔 리뷰, CI 실행, 머지 충돌까지 맞춰가야 하고요. 기여자와 메인테이너 사이의 지루한 왕복 토론까지 더해지면 결국 시간만 자꾸 밀립니다.

과거라면 이 정도 번거로움도 품질 좋은 PR 하나가 나와주면 감수할 만했어요. 코드 작성이 병목이었거든요. 그런데 LLM이 정말 잘해지면서 상황이 완전히 바뀌었습니다.

LLM이 생성한 코드를 검토하는 건 여전히 필요하지만, 모르는 사람의 악의적 코드를 걱정할 필요는 없습니다. 제 코딩 스타일과 선호도를 이미 LLM에 입력해뒀거든요. 그리고 다른 시간대에 사는 사람과 맞춰가지 않고도 제 페이스대로 빠르게 반복 작업할 수 있습니다.

**결국 LLM의 도움을 받아서 제가 직접 코드를 작성하는 게 훨씬 수월합니다.**

## 소프트웨어 개발의 본질이 바뀌고 있습니다

예전부터 그랬지만, 요즘엔 더 명확해졌어요. "소스코드"는 이제 진짜 의미의 "소스"가 아니라, 개발자 머릿속 생각과 컴퓨터 명령 사이에 있는 중간 단계일 뿐입니다. 코드 자동 생성이 쉬워지면서 이 사실이 더 드러난 거고요.

LLM에 대해 다양한 의견이 있죠. 금지해야 한다는 주장부터 "코딩은 죽었다, 이제는 감정 코딩의 시대"라는 주장까지요. 저는 그 중간쯤에 있습니다. 디자인을 먼저 하고, AI 에이전트가 대부분의 코드를 작성하게 한 다음, 저는 검토하고 다듬는 방식이죠.

엄청난 양의 코드를 만들 수 있지만, 저는 다른 것들이 병목입니다.

- **이해**: 기존 코드를 읽고 분석하는 데 시간이 걸려요
- **설계**: 올바른 변경과 아키텍처를 구상하는 데도 시간이 필요합니다
- **검토**: 코드가 제 의도대로 동작하는지 확인하는 일

당신의 PR은 이 세 가지 중 어디에도 크게 도움이 되지 않습니다. **그러니 코드 변경을 병합 목적으로 제출하는 것은 정말 스킵해주세요.**

## 그럼 어떻게 도와주면 좋을까요?

"코드 작성"의 가치가 떨어진 만큼, 다른 형태의 기여가 상대적으로 훨씬 값어치 있어졌습니다.

### 피드백 주기

저는 새 기능을 구현하느라 바빠서, 실제로 사용해보고 개선 방안을 깊이 있게 연구할 시간이 부족합니다. 뭐가 잘 작동하는지, 뭘 개선하면 좋을지 사용자가 알려주는 건 정말 도움이 돼요.

### 아이디어 토론

저도 완벽하지 않습니다. 다양한 경험과 관점을 가진 사람들과 대화하면 제가 무엇을 만들어야 하고 어떻게 만들어야 하는지 더 잘 보입니다.

### 버그 보고 및 조사

좋은 버그 리포트는 이미 버그의 75%를 해결한 거나 다름없습니다. 문제를 발견했다면 자세히 설명해주고, 가능하면 재현하는 방법과 정확한 문제 위치까지 찾아줄 수 있으면 좋아요. 그리고 가능한 해결 방안을 함께 토론하는 거죠.

### 시안 프로토타입 공유

참고용 PR이나 사용한 프롬프트를 공유해주세요. 아까 "PR 병합은 안 된다"고 했는데, 이건 다릅니다. LLM을 쓰면 저도 독립적으로 변경사항을 만들 수 있지만, 실제 구현 코드를 보는 것도 도움이 돼요. 빠르게 개념을 이해하는 데 좋거든요.

**더 중요한 건, 프롬프트라는 "소스"를 공유하면 저도 그걸 재활용하고 다듬을 수 있다는 겁니다.** 시간을 정말 많이 절약할 수 있죠.

### 코드 리뷰 및 문제 지적

저는 리뷰에서 병목을 겪고 있습니다. 또 다른 시각으로 코드를 봐주고 문제점을 지적해주는 건 정말 유용합니다.

### 프로젝트 포크하고 자기 방식대로 해보기

프로젝트를 포크하고 자기 입맛대로 변경하는 걸 두려워하지 마세요. 여러 사용 사례를 지원하는 디자인을 고민하고, 합의를 이루고, 최적안을 놓고 논쟁하고, 타협점을 찾는 일은 엄청 시간이 걸립니다.

LLM 덕분에 소프트웨어 커스터마이징은 예전보다 훨씬 쉬워졌어요. 당신이 원하는 기능을 직접 추가해서, 당신 방식대로 개발하고, 당신 페이스에 맞춰 업스트림과 리베이스하면 됩니다.

**그냥 포크하세요. 당신의 사용 사례에 맞게 코드를 수정하고, 자신의 방식으로 진행하면 됩니다. 허락을 구할 필요도 없습니다.**
