## 핵심 개념

> [!abstract]
> Theseus는 정적 바이너리 번역(static binary translation) 기법을 사용하는 새로운 Windows/x86 에뮬레이터로, 프로그램을 정적으로 번역하여 기존 동적 에뮬레이션의 문제들을 해결합니다. x86 바이너리를 중간 표현(IR)으로 변환한 후 최적화 컴파일러를 통해 네이티브 코드를 생성합니다.

## 상세 내용

- JIT 작성의 복잡성을 피하고 기존 최적화 컴파일러를 활용한 정적 바이너리 번역 접근
- x86 명령어를 C 코드로 변환한 후 컴파일러로 네이티브 바이너리 생성
- 기존 retrowin32 프로젝트의 경험을 바탕으로 '무엇을 만들 것인가'에 대한 재고찰

> [!tip] 왜 중요한가
> 에뮬레이션과 바이너리 번역의 고전적 문제에 대한 새로운 접근법을 제시합니다.

## 전문 번역

# Theseus, 정적 Windows 에뮬레이터

**2026년 4월 19일**

이 글이 retrowin32 시리즈의 마지막이 될 것 같습니다. 소개할 내용은 프로그램을 정적으로 변환하는 새로운 Windows/x86 에뮬레이터 Theseus입니다. 여러 에뮬레이션 문제를 해결하지만, 당연히 새로운 문제들도 함께 가져오게 되겠네요.

## retrowin32는 어떻게 됐나요?

retrowin32라는 win32 에뮬레이터를 더 이상 개발하고 있지 않습니다. 여러 이유가 있는데, 개인적인 사정도 있고 프로젝트를 어떻게 진행해야 할지 확실하지 않기도 했거든요. 그러던 와중에 과거에 이 프로젝트에 기여했던 누군가가 retrotick이라는 웹 기반 Windows 에뮬레이터를 공개했는데, 제 몇 년의 작업보다 훨씬 더 좋아 보였습니다. 그 개발자는 해커뉴스에 "Claude와 함께 1시간이면 되더라"는 댓글을 남겼어요.

이건 AI에 관한 글은 아닙니다. 이미 관련 글이 너무 많은 데다, 저 자신도 아직 명확한 생각이 없거든요. 다만 한 가지 생각해볼 점이 있습니다. (1) AI가 주니어에서 시니어 엔지니어 수준으로 계속 올라가고 있고, (2) 시니어 엔지니어가 갖춰야 할 핵심 능력 중 하나는 "어떻게 만들 것인가"가 아니라 "무엇을 만들어야 하는가"를 더 잘 이해하는 것이라는 점입니다.

(혹시 이것도 혁신 기업의 딜레마에서 말하는 "위쪽으로 시장 이동하기"를 제 자신에게 적용한 건 아닐까요? 잘 모르겠습니다. 다행히 저는 이 일을 여행처럼, 순수한 호기심을 만족하기 위해 하고 있어서 기업처럼 존재적으로 위협받지는 않습니다. Benny Feldman이 말한 대로죠: "물질에 대한 집착을 몰래 버리면 카지노에서도 속을 수 없다!")

그렇다면 시니어 엔지니어라면 뭘 만들어야 할까요? 에뮬레이터로 풀려는 문제가 정확히 뭐고, 기존 접근 방식들이 그 문제를 잘 해결했나요? 저는 꽤 색다른 해결책에 도달했고, 이걸 공유하고 싶습니다!

## 에뮬레이터와 JIT

가장 간단한 CPU 에뮬레이터는 인터프리터와 비슷합니다. 파싱된 프로그램은 다음과 같은 x86 명령어들이 됩니다.

```
mov eax, 3
add eax, 4
call ... ; Windows API 호출
```

해석 방식 에뮬레이터는 명령어를 차례대로 실행하는 큰 루프입니다.

```
loop {
  let instr = next_instruction();
  match instr {
    // 예: `mov eax, 3`
    Mov => { set(argument_1(), argument_2()); }
    // 예: `add eax, 4`
    Add => { set(argument_1(), argument_1() + argument_2()); }
    ...
  }
}
```

인터프리터처럼 이 방식은 느립니다.

인터프리터가 느린 이유는 각 명령어마다 동적인 작업을 많이 하기 때문입니다. 같은 add 명령어가 반복되는 루프를 에뮬레이트한다고 생각해보세요. 위 에뮬레이터는 매 반복마다 "지금 뭔 명령어를 실행하지?", "인자는 뭐지?"라고 물어본 다음에야 결국 같은 덧셈을 합니다. x86의 메모리 참조는 더 복잡한데, x86 주소 지정 방식이 엄청 유연하기 때문이죠.

더욱이 x86의 add 명령어는 단순히 더하기만 하는 게 아니라, 결과의 패리티 플래그(결과에 1 비트가 짝수 개 있는지 여부) 같은 파생 값 6개를 계산합니다. 정확한 에뮬레이터는 이 모든 값을 계산하거나, 아니면 코드를 분석해서 효율적으로 실행할 방법을 찾아야 합니다.

에뮬레이터를 개선하는 여러 재미있는 기법들이 있습니다. 그런데 정말 빠르게 하려면 코드 분석과 네이티브 기계어 생성을 조합해야 합니다. 그게 바로 JIT인데요. JIT는 작성이 정말 어렵기로 유명합니다! 기본적으로 최적화 컴파일러와 같은데, 컴파일 자체가 런타임 성능에 영향을 미치기 때문입니다. 파이썬 JIT만 해도 15번 이상 시도됐다고 하네요.

## 정적 바이너리 변환

그런데 효율적인 기계어를 생성하되 JIT는 작성하고 싶지 않다면? 코드를 분석하고 효율적인 기계어를 생성하는 데 정말 뛰어난 게 뭘까요? 바로 컴파일러입니다!

여기가 핵심 아이디어입니다. 위의 x86 코드 스니펫 같은 것이 주어지면, 우린 이를 다음과 같은 소스 코드로 변환할 수 있습니다.

```
regs.eax = 3;
regs.eax = add(regs.eax, 4);
windows_api(); // 호출된 API의 네이티브 구현
```

그 다음 이 코드를 최적화 컴파일러에 넣으면, 현재 아키텍처의 네이티브 프로그램이 나옵니다. x86은 더 이상 필요 없죠.

다시 말해, .exe 파일을 JIT 컴파일하는 에뮬레이터에 직접 건네는 대신, .exe를 정적으로 변환해서(중간에 컴파일러를 거쳐서) 직접 "네이티브" 실행 파일로 만드는 거예요.

(여기서 "네이티브"를 따옴표로 쓴 이유는 결과 실행 파일이 네이티브 바이너리긴 하지만, x86 상태를 나타내는 내부 가상 머신(위 코드의 regs 구조체 같은)을 가지고 다니기 때문입니다. 이건 나중에 더 얘기할게요.)

이 기본 아이디어는 제가 무엇을 이루려고 하는지 열심히 생각해서 스스로 떠올린 것 같지만, 실제로는 정적 바이너리 변환(static binary translation)이라는 잘 알려진 접근법이었네요. 여러 좋은 특성도 있고, 큰 문제점들도 있습니다.
