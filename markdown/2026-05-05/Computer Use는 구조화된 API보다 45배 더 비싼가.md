## 핵심 개념

> [!abstract]
> Reflex는 동일한 관리자 패널을 제어하는 두 가지 AI 에이전트 방식(비전 에이전트 vs API 에이전트)을 벤치마크하여 비전 에이전트의 실제 비용을 측정했다. 결과적으로 API 에이전트는 평균 8회 호출로 19.7초(입력 토큰 12,151개)에 작업을 완료한 반면, 비전 에이전트는 명확한 UI 가이드 없이는 작업을 완료하지 못했으며, 상세한 14단계 프롬프트 지침을 통해 겨우 약 17분(입력 토큰 550,976개)이 소요되었다. 비전 에이전트는 페이지 매김 신호 부재로 화면에 보이지 않는 데이터를 놓쳤고, 실행 결과도 높은 분산을 보였다(749-1257초, 407k-751k 토큰).

## 상세 내용

- API 에이전트와 비전 에이전트의 성능 차이: API 에이전트는 Sonnet으로 8회 호출로 안정적으로 작업을 완료(±0 분산)했고, 비전 에이전트는 Haiku 기준 평균 53회 호출이 필요했으며 높은 분산(±13)을 보였다. 이는 구조화된 응답이 모델에 명확한 의도 신호를 제공함을 의미한다.
- 비전 에이전트의 근본적 한계: 비전 에이전트는 렌더링된 페이지만 보므로 페이지 1/4라는 정보를 읽을 수 없고, 화면 아래 있는 3개의 미승인 리뷰를 완전히 놓쳤다. 이는 모델의 추론 능력 문제가 아니라 비구조화된 인터페이스의 근본적 문제이다.
- 숨은 엔지니어링 비용: 비전 에이전트를 작동시키기 위해 명시적인 14단계 UI 가이드를 작성해야 했으며, 이는 토큰 계산에 나타나지 않는 실제 엔지니어링 비용이다. 각 단계에서 정확히 어느 사이드바 항목, 탭, 폼 필드를 사용할지 명시해야 하므로 유지보수 부담이 크다.
- 비용 차이 정량화: API 에이전트(Sonnet)는 약 19.7초에 12,151개 입력 토큰 소비, 비전 에이전트(Sonnet)는 약 1003초에 550,976개 입력 토큰 소비로, 대략 45배의 입력 토큰 비용 차이가 발생한다. Haiku 기반 API 에이전트는 7.7초에 9,478개 토큰만 필요하다.
- 실행의 재현성 차이: API 경로는 5회 시도에서 동일한 8회 호출을 유지했고 입력 토큰 분산이 ±27에 불과했으나, 비전 경로는 3회 시도에서 43-68 사이클 변동과 407k-751k 토큰 변동을 보였다. 이는 스크린샷-추론-클릭 루프의 비결정성을 의미한다.
- 테스트 시나리오의 현실성: 고객 'Smith' 찾기, 최근 미처리 주문 위치, 모든 미승인 리뷰 수락, 주문 배송 완료 표시라는 작업은 3개 리소스, 필터링, 페이지네이션, 교차 엔티티 조회, 읽기/쓰기 모두를 포함하는 전형적인 내부 도구 작업이다.

> [!tip] 왜 중요한가
> AI 에이전트를 내부 도구에 배포할 때 비전 에이전트의 겉보기 편의성이 실제로는 45배 높은 비용과 유지보수 복잡성을 초래하므로, API 인터페이스 설계의 중요성과 장기적 ROI를 재고하는 계기를 제공한다.

## 전문 번역

# 비전 AI 에이전트의 실제 비용을 측정해봤습니다

AI 에이전트가 관리자 패널을 조작하는 두 가지 방식을 직접 벤치마크했습니다. 화면을 보고 클릭하는 방식과 API를 직접 호출하는 방식인데요, 과연 비전 에이전트(browser-use, computer-use)의 가격이 얼마나 될까요? 측정 결과와 예상 밖의 발견들을 공유하려고 합니다.

## 왜 비전 에이전트를 측정해봤나요?

API를 제공하지 않는 웹 앱을 자동화할 때 비전 에이전트가 기본 선택지가 되는 이유는 간단합니다. 다른 방법인 MCP나 REST API를 만드는 건 정말 큰 프로젝트이기 때문입니다. 대부분의 팀이 20개 이상의 내부 도구를 관리하고 있는데, 각각에 대해 API 층을 구축하는 건 너무 비싼 일이거든요.

그래서 더 좋은 방법이 있어서가 아니라, 비용이 많이 들어서 비전 에이전트를 쓰는 경우가 대부분입니다. 우리는 그 비용이 정확히 얼마나 되는지 측정하고 싶었어요.

## 어떻게 측정했나요?

테스트 앱은 고객, 주문, 리뷰를 관리하는 관리자 패널입니다. react-admin의 Posters Galore 데모를 모델로 만들었는데요, 동일한 앱을 두 가지 방식으로 제어했습니다.

- **경로 A: 비전 에이전트** — Claude Sonnet이 browser-use를 통해 스크린샷을 보고 클릭으로 조작
- **경로 B: API 에이전트** — Claude Sonnet이 HTTP 엔드포인트를 직접 호출

같은 모델, 같은 데이터셋, 같은 작업. 오직 인터페이스만 다릅니다.

### 작업 내용

"Smith"라는 고객을 찾아서 주문이 가장 많은 사람을 찾고, 가장 최근의 미처리 주문을 찾은 뒤, 그 주문의 모든 미처리 리뷰를 승인하고 주문을 배송 완료로 표시하세요." 

이 작업은 세 개의 리소스를 다루며, 필터링, 페이지네이션, 교차 엔티티 조회, 그리고 읽기와 쓰기를 모두 포함합니다. 정말 전형적인 내부 도구 작업이죠.

## 결과: 비전 에이전트가 작업을 완료하지 못했습니다

처음에는 두 에이전트에게 같은 지시를 내렸는데, 완전히 다른 결과가 나왔습니다.

**API 에이전트**는 8번의 호출로 작업을 완료했습니다. 고객의 리뷰를 미처리 상태로 필터링하고, 각각을 승인하고, 주문을 배송 완료로 표시했어요.

**비전 에이전트**는 네 개의 미처리 리뷰 중 하나만 찾아서 승인하고 멈췄습니다. 페이지를 스크롤하지 않았거든요. 나머지 세 개의 리뷰는 화면에 보이지 않는 영역에 있었는데, 에이전트는 스크롤할 필요가 있다는 신호를 받지 못한 거죠.

이건 모델의 문제가 아닙니다. **문제는 정보의 가시성**입니다. 비전 에이전트는 렌더링된 페이지를 보고 있으니까, 화면에 모든 게 다 보이는지 알 수 없어요. 반면 API 에이전트는 같은 핸들러를 호출하지만, 렌더링된 행들만 아니라 핸들러가 반환한 전체 결과셋을 받습니다. "페이지 1/4, 페이지당 50개" 같은 정보를 픽셀로 해석할 필요 없이 직접 읽을 수 있는 거죠.

## 14단계 지시문으로는 성공했습니다

공정한 비교를 위해 비전 에이전트의 프롬프트를 다시 썼습니다. 사이드바 항목, 탭, 폼 필드를 명시적으로 지정하는 14개의 단계별 지시문으로요. 에이전트가 스스로 찾지 못한 네비게이션을 전부 명시한 거랍니다.

이렇게 하니까 비전 에이전트가 작업을 완료했습니다. 다만 14분이 걸렸고 약 50만 개의 입력 토큰을 소비했어요.

**이 14단계 지시문 자체가 중요한 발견**입니다. 토큰 수에는 나타나지 않지만, 각 단계는 엔지니어링 작업이고 실제 비용이 드는 일입니다. 비전 에이전트를 내부 도구에 배포하는 누구든지, 이 정도 수준의 구체적인 프롬프트를 작성하거나 에이전트가 조용히 일을 놓칠 위험을 감수해야 합니다.

## 측정 방법

API 경로는 5번, 비전 경로는 3번 실행했습니다. 비전 경로는 3번으로 제한했는데, 각 실행마다 14~22분이 걸리고 40만~75만 토큰을 소비하기 때문입니다.

비전 결과에서 가장 놀라웠던 건 **분산(variance)**입니다. 3번의 시행에서 소요 시간은 749초부터 1257초까지 움직였고, 입력 토큰은 407k부터 751k까지 달랐어요. 최단 실행은 43 사이클, 최장 실행은 68 사이클이 걸렸습니다. 스크린샷-추론-클릭 루프는 비결정성이 충분해서, 단 한 번의 실행으로 비용을 추정할 수 없다는 뜻입니다.

반대로 **API 경로는 분산이 전혀 없었습니다**. Sonnet은 모든 시행에서 정확히 동일한 8개의 도구 호출을 수행했고, 입력 토큰은 ±27 범위 내에서만 달랐어요. 구조화된 응답이 에이전트가 다른 경로를 따를 이유를 주지 않기 때문입니다.

## 전체 결과

| 지표 | 비전 에이전트 (Sonnet) | API (Sonnet) | API (Haiku) |
|------|------|------|------|
| 스텝 / 호출 | 53 ± 13 | 8 ± 0 | 8 ± 0 |
| 실행 시간 | 1003초 ± 254초 (~17분) | 19.7초 ± 2.8초 | 7.7초 ± 0.5초 |
| 입력 토큰 | 550,976 ± 178,849 | 12,151 ± 27 | 9,478 ± 80 |
| 출력 토큰 | 37,962 ± 10,850 | 934 ± 41 | 819 ± 52 |

*(비전 경로 n=3, API 경로 n=5. 평균 ± 표본 표준편차로 표시)*
