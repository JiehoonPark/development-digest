- 기존 인-프로세스 아키텍처와 함께 사용 가능하여 사용자가 필요에 따라 선택 가능

## 핵심 요약

DuckDB 인스턴스끼리 Quack 원격 프로토콜을 사용해 통신할 수 있게 됐습니다. 이제 여러 개의 동시 쓰기 작업을 지원하는 클라이언트-서버 구조로 DuckDB를 운영할 수 있거든요. DuckDB의 철학에 맞춰 Quack은 설정이 간단하고 HTTP 같은 검증된 기술 위에 구축됐습니다. 게다가 성능도 뛰어나서 대량 작업부터 소규모 트랜잭션까지 다양한 워크로드를 지원합니다.

## 배경: 데이터베이스 아키텍처

데이터베이스가 처음 등장했을 때만 해도 '클라이언트'와 '서버'의 개념이 없었습니다. 그저 하나의 컴퓨터에서 전체 데이터베이스가 실행됐을 뿐이었거든요. 1980년대 어느 시점에 Sybase가 처음으로 데이터베이스 '서버'와 '클라이언트'를 다른 컴퓨터에서 실행하는 개념을 도입했습니다. 그 이후로 모든 데이터베이스 시스템은 당연하게 클라이언트-서버 아키텍처와 이들 사이를 통신하는 프로토콜을 갖추게 됐습니다.

이 방식이 편리한 이유는 명확합니다. 변경 가능한 상태가 한 곳의 서버에 집중되어 관리되는데, 많은 수의 클라이언트가 동시에 데이터를 읽고 쓸 수 있기 때문입니다. 물론 단점도 있습니다. 무엇보다 프로토콜이 상당한 오버헤드를 발생시킬 수 있다는 점이죠. 더 자세히 알고 싶다면 우리가 예전에 작성한 데이터베이스 프로토콜 연구 논문을 읽어보시면 됩니다.

그런데 클라이언트-서버 아키텍처에 반발한 사람들도 항상 있었습니다. 가장 유명한 예로는 2000년의 SQLite, 그리고 2019년에 처음 출시된 DuckDB를 들 수 있죠. 우리는 인-프로세스 아키텍처의 장점을 크게 강조해왔습니다. 클라이언트-서버도 없고, 프로토콜도 없고, 단지 저수준 API 호출만 있는 방식 말입니다.

이 방식은 특히 데이터 과학 분야의 대화형 사용에 정말 잘 맞습니다. 예를 들어 분석가가 Python 노트북에서 데이터와 상호작용하면서 같은 프로세스 내에서 실행되는 DuckDB 인스턴스로 데이터를 관리하는 경우죠. 또한 DuckDB가 기존 애플리케이션에 '접착제'처럼 붙어 SQL 기능을 제공하는 많은 사용 사례에서도 정말 잘 작동했습니다.

하지만 인-프로세스 시스템은 여러 프로세스에서 동시에 같은 데이터베이스 파일을 수정하려는 경우에는 "덜 잘" 작동합니다. 이런 필요가 있는 사용 사례는 매우 많습니다. 예를 들어 원격 측정 데이터를 수집하는 여러 프로세스가 같은 데이터베이스에 데이터를 삽입하는 동시에 다른 프로세스가 같은 테이블을 쿼리해서 대시보드를 업데이트하는 상황을 생각해보세요.

우리가 이를 구현할 수 없었던 데는 좋은 기술적 이유가 있습니다. 무엇보다 DuckDB는 메인 메모리에 많은 상태를 유지하는데, 여러 프로세스가 동시에 변경을 가하면 그 상태를 동기화해야 하기 때문이죠.

물론 해결책들이 있었습니다. DuckDB 데이터베이스 인스턴스를 보유한 프로세스가 다른 프로세스들에게 쿼리와 데이터 삽입 서비스를 제공하는 커스텀 원격 프로시저 호출(RPC) 솔루션을 만들 수 있었으니까요. 또한 Arrow Flight SQL 프로토콜을 사용하는 것처럼 DuckDB에 클라이언트-서버 기능을 추가하는 여러 프로젝트들도 있습니다. MotherDuck은 자체 커스텀 클라이언트-서버 프로토콜을 만들었고요. 당연히 PostgreSQL 같은 더 전통적인 데이터베이스로 전환할 수도 있습니다. 그리고 나서 (놀랍게도) pg_duckdb 같은 확장을 사용해서 PostgreSQL 내에서 "EleDucken" — 즉 DuckDB를 실행할 수도 있죠.

DuckDB에 클라이언트-서버 솔루션을 덧붙이기 위해 사람들이 만든 수많은 해결책들을 보면 이것이 정말 필요한 기능이라는 걸 확실히 알 수 있습니다. 우리는 DuckDB를 보편적인 데이터 처리 도구로 봅니다. 인-프로세스 기능에 더해 클라이언트-서버 프로토콜을 갖춘다면? 좋습니다. 이것이 DuckDB를 사용할 수 있는 새로운 사용 사례를 많이 열어준다면? 더욱 좋습니다! 결국 우리가 가장 신경 쓰는 건 사용자 경험이고, 아키텍처에 대한 고집은 덜합니다. 그래서 결국 마음을 먹고 진행했고, 오늘 기쁜 마음으로 그 결과물을 발표합니다.

## Quack 프로토콜 소개

두 마리(또는 그 이상)의 오리가 서로 말하고 싶을 때 뭘 할까요? 울음을 냅니다! 그러니 두 개의 DuckDB 인스턴스가 서로 말하기 위해 사용할 프로토콜을 "Quack"이라고 부르는 게 너무나 자연스럽습니다! 

우리는 2026년에 레거시를 고려할 필요 없이 데이터베이스 프로토콜을 처음부터 설계할 수 있다는 호사를 누렸습니다. 최신 Arrow Flight SQL을 포함한 기존 프로토콜들로부터 배울 수 있었거든요. 

Quack이 내부적으로 어떻게 작동하는지 살펴보기 전에, 먼저 사용자 관점에서 어떻게 사용하는지 보겠습니다. 우선 두 개의 DuckDB 인스턴스가 필요합니다. 맞습니다, DuckDB가 클라이언트이면서 동시에 서버로도 작동합니다! 두 인스턴스는 지구 반대편(또는 우주)의 다른 컴퓨터에 있을 수도 있고, 그냥 당신의 노트북 터미널 두 개에 있을 수도 있습니다.

먼저 두 DuckDB 인스턴스 모두에 Quack 확장을 설치해야 합니다. 현재 Quack은 core_nightly 저장소에 있으며 DuckDB v1.5.2에서 사용 가능합니다.
