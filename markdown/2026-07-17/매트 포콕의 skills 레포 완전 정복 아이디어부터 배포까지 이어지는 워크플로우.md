## 핵심 개념

> [!abstract]
> 매트 포콕이 16만 스타를 받은 자신의 skills 레포에 대한 첫 정식 튜토리얼을 공개했다. npx skills add 명령으로 설치하는 과정부터, 이슈 트래커·트리아지 라벨·도메인 문서를 구성하는 초기 설정, 그리고 grill with docs로 아이디어를 다듬어 implement까지 이어지는 핵심 워크플로우를 실제 레포에서 시연한다. 스킬은 사용자 직접 호출 방식으로 설계되어 38개를 모두 설치해도 컨텍스트 소모가 660토큰에 불과하다는 점도 강조한다.

## 아티클

# 매트 포콕의 skills 레포 완전 정복: 아이디어부터 배포까지 이어지는 워크플로우

매트 포콕(Matt Pocock)이 만든 `mattpocock/skills` 레포는 현재 GitHub 스타 16만 2천 개, 다운로드 750만 회를 기록하고 있는 인기 프로젝트입니다. 하지만 정작 정식 튜토리얼은 한 번도 만든 적이 없다고 하는데요. "어떤 순서로 스킬을 써야 하나요", "어떻게 설치하고 설정하나요" 같은 질문이 계속 들어오자, 이번 영상에서 처음부터 끝까지 전체 흐름을 직접 보여주기로 했습니다. 고급 기능이나 신규 기능은 제외하고, 시작하는 데 꼭 필요한 핵심 플로우만 다룹니다. 실습은 자신이 강의용으로 쓰는 CLI 도구인 AI Hero CLI 레포에 처음으로 스킬을 설치하면서 진행합니다.

## 설치하기: `npx skills add`

새 프로젝트에 스킬을 설치하는 방법은 브라운필드든 그린필드든 동일합니다. Node.js가 설치되어 있어야 하며, 다음 명령어로 시작합니다.

```
npx skills@latest add mattpocock/skills
```

이 명령은 Node.js의 NPX를 통해 Vercel의 `skills.sh` CLI 설치 프로그램을 실행하고, `mattpocockskills`라는 GitHub 레포에서 스킬을 가져옵니다. 실행하면 필요한 패키지 설치 여부를 묻고, 이어서 설치 가능한 스킬 목록이 쭉 뜨는데요. 총 38개의 스킬이 두 그룹으로 나뉘어 있습니다.

- **mattpocockskills**: 매트가 공개해도 좋다고 검증한 정식 스킬
- **그 외 스킬들**: 아직 실험 중이라 나중에 삭제될 수도 있는 것들

목록 맨 위로 이동해서 스페이스바를 누르면 전체 선택이 됩니다(UI가 다소 부자연스럽다고 언급하며, 향후 자체 CLI를 만들 수도 있다고 밝힙니다). 이후 엔터를 누르면 공식 스킬 전체가 선택됩니다.

## 에이전트 선택과 설치 범위

스킬은 어떤 에이전트와도 함께 쓸 수 있도록 설정할 수 있습니다. 매트는 Claude Code를 사용하지만, Cursor, Codex, Claude 등 "유니버설" 카테고리는 기본으로 지원되고, Claude Code처럼 Claude Skills 방식을 쓰는 도구는 별도로 선택해서 설정해야 합니다.

다음으로 **설치 범위(installation scope)**를 정합니다. 현재 디렉터리에만 설치할지, 전역으로 설치할지를 고르는 항목인데요.

- 팀으로 작업한다면 **프로젝트 단위** 설치를 추천합니다. 모든 팀원이 같은 스킬 세트를 쓰고, 스킬 구성도 함께 논의하고 기여할 수 있기 때문입니다.
- 혼자 개발하는 경우라면 **전역(global)** 설치도 무방합니다.

매트는 홈 디렉터리에 전역으로 설치하고, 방식은 **심링크(symlink)**를 선택했습니다. 여기서 선택지는 `.claude` 폴더뿐 아니라 agents 폴더에도 복사할지 여부인데, 심링크가 훨씬 깔끔하고 간단한 방법이라 별다른 고민 없이 심링크를 고르면 됩니다.

설치 요약을 확인한 뒤 진행하면 스킬 설치가 완료됩니다.

## Claude Code에서 스킬 확인하기

설치 후 해당 디렉터리에서 Claude(또는 사용 중인 에이전트)를 실행하면, 슬래시(`/`)를 입력했을 때 `grill me`, `grilling`, `way finder`, `grill with docs` 등 다양한 스킬 목록을 볼 수 있습니다.

여기서 매트의 스킬 레포가 다른 스킬 레포들과 다른 지점이 나옵니다. 대부분의 스킬이 **사용자가 직접 호출(user invoked)**하는 방식으로 만들어져 있어서, 컨텍스트에 스며들듯 자동으로 끼어드는 스킬이 거의 없습니다. 설명(description)도 짧고 명확하게 작성되어 있는데요. 그 결과 38개 스킬을 전부 다운로드해도 컨텍스트에는 단 **660토큰**만 차지합니다. 컨텍스트 부하가 매우 가볍다는 점이 핵심 설계 원칙입니다.

## `set up mattpocockskills` 실행하기

스킬 설치가 끝나면, 레포 내부에서 `set up mattpocockskills`를 실행해야 합니다. 이 스킬은 레포 설정에 필요한 몇 가지 항목을 구성해줍니다.

**1. 이슈 트래커 선택**

스킬은 스펙과 티켓을 저장할 곳이 필요합니다. GitHub, 로컬 마크다운, 혹은 그 어떤 도구든 선택할 수 있습니다. 스킬은 로컬 설정을 참조해 동작하기 때문에 Jira나 Linear로도 설정할 수 있는데요. 방법은 간단합니다. 에이전트에게 "Jira로 설정해줘"라고 말만 하면 됩니다.

> "Jira, Beads, Linear랑 어떻게 연동하나요?"라는 질문을 정말 많이 받는데, 이미 지원됩니다. `set up mattpocock`을 실행하고 "Jira로 설정해줘"라고만 하면 됩니다.

매트는 이번 실습에서는 로컬 마크다운을 선택했습니다.

**2. 트리아지 라벨(triage labels)**

티켓 관련 정보를 전달하는 데 쓰이는 라벨 세트입니다. 크게 중요한 설정은 아니라서 기본값을 그대로 사용해도 무방하며, 자세한 내용은 트리아지 스킬 문서를 참고하면 됩니다.

**3. 도메인 문서화**

스킬은 레포 안에 `context.md` 파일과 ADR(Architecture Decision Record)을 두는 것을 선호합니다. 이때 단일 컨텍스트로 갈지, 멀티 컨텍스트로 갈지 물어보는데요. 멀티 컨텍스트는 여러 바운디드 컨텍스트가 있는 대형 모노레포에 필요한 방식이고, 대부분(99%)의 경우에는 단일 컨텍스트로 충분합니다.

설정이 끝나면 `CLAUDE.md` 파일에 이슈 트래커 문서, 트리아지 라벨, 도메인 문서로 연결되는 링크가 추가됩니다(`docs/agents/domain`, `issue-tracker` 등). 이렇게 이슈와 스펙은 `scratch` 파일 안에 저장되며, 레포 셋업이 완료됩니다.

## `ask matt` 스킬로 워크플로우 물어보기

본격적으로 시작하기 전에, 흥미로운 스킬 하나를 소개합니다. 바로 **`ask matt`** 스킬인데요, 매트 자신을 스킬화한 것으로, 스킬 레포 전반과 "무엇부터 해야 하는지"에 대한 지식을 담고 있습니다.

실습에서 "코드 변경을 하려고 하는데, 어떤 메인 플로우를 써야 하나요?"라고 물어보자 다음과 같은 답이 돌아왔습니다.

> "메인 플로우는 idea to ship입니다. 이미 코드베이스가 있으니 메인 플로우의 맨 위에서 시작해서, 하나의 끊기지 않은 컨텍스트 윈도우 안에서 쭉 내려가세요."

이 답변에서 매트는 세션 사용법에 대한 조언까지 포함되어 있다고 짚었습니다. 컨텍스트 윈도우와 토큰 사용량을 의식하는 것이 AI를 잘 활용하는 데 핵심이라는 게 그의 지론입니다.

이어지는 `ask matt`의 설명을 정리하면 다음과 같습니다.

- **`grill with docs`**로 시작합니다. 아이디어를 다듬기 위해 인터뷰 형식으로 질문을 던지고, 레포 안에서 상태를 유지하기 때문에 배운 내용을 `context.md`와 ADR에 기록합니다. "X를 바꾸고 싶다"는 생각을 명확하고 방어 가능한(defensible) 계획으로 바꾸는 단계입니다.
- 대화만으로 모든 열린 질문에 답할 수 있는지 확인합니다. 만약 실행 가능한 답이 필요한 질문이라면 **prototype** 스킬을 쓸 수 있습니다(관련 영상 별도 존재).
- 인터뷰(`grill with docs`)가 끝나면, 곧바로 **implement** 스킬로 넘어가거나, 여러 세션에 걸쳐 진행해야 할 정도로 크다면 **to spec**과 **to tickets**로 넘어갑니다.

정리하면 기본 플로우는 다음과 같습니다.

```
grill with docs → (필요시 prototype) → implement
                                     └→ (규모가 크면) to spec → to tickets → implement
```

## `grill with docs` 실전 사용

이제 실제로 `grill with docs`를 아이디어와 함께 실행합니다. 매트가 던진 아이디어는 이렇습니다.

> "이 CLI에 있는 내부 도구용 기능들을 대부분 제거하고, 완전히 공개용으로만 쓰이게 만들고 싶어요. 여기 잡다한 것들이 너무 많아서 좀 정리하고 싶습니다."

이렇게 모호하게 던져도 상관없습니다. `grill with docs`가 후속 질문들을 던지면서 무거운 작업을 대신 해주기 때문인데요. 실행되자마자 레포 전체 구조를 파악하고, 서브커맨드 11개를 가진 `internal` 네임스페이스를 찾아내는 식으로 코드를 탐색합니다.

이 실습에서는 Claude Code와 Opus 4.5(medium effort 수준)를 사용했지만, 스킬 자체는 여러 사람들이 각기 다른 하니스, 다른 모델, 다른 effort 레벨로 사용하고 있어 특정 설정에 얽매일 필요는 없습니다.

첫 질문이 나온 뒤, 질문에 답을 이어가다 보면 에이전트와 사용자 모두가 "공유된 이해(shared understanding)"에 도달했다고 느끼는 시점이 옵니다. 이번 실습에서는 6개의 질문 만에 끝났는데, 보통은 작업 규모에 따라 20개 정도까지도 이어진다고 합니다. 결과적으로 커맨드 파일 10개 삭제, 테스트 3개 삭제, 공유 모듈 재배선이라는 꽤 구체적인 계획이 나왔습니다.

여기서 중요한 점은, 이 과정에서 플랜 모드(plan mode)를 쓴 게 아니라 Claude Code의 기본 모드인 오토 모드(auto mode)를 그대로 사용했다는 것입니다.

## 갈림길: implement로 바로 갈까, spec/tickets를 거칠까

`grill with docs`가 끝나면 갈림길에 서게 됩니다. 작업 규모가 커서 여러 에이전트 세션이 필요할 것 같다면, 2번과 3번 단계(to spec, to tickets)를 건너뛰고 곧바로 implement로 갈 수 있습니다. 방법은 간단히 `/implement this`라고 입력하면 됩니다.

이번 케이스에서는 매트도 바로 implement로 넘어가는 게 맞다고 판단했는데요. 이유는 컨텍스트 윈도우 때문입니다. 그는 컨텍스트 윈도우가 약 14만 토큰(140k) 지점을 넘어가면 LLM이 눈에 띄게 "멍청해지기" 시작한다고 보고, 그 이전까지를 "스마트 존(smart zone)"으로 여긴다고 설명합니다. 즉, 작업 규모가 크지 않다면 굳이 spec과 tickets 단계를 거치지 않고 하나의 컨텍스트 윈도우 안에서 끝까지 처리하는 것이 효율적이라는 판단입니다.

## 정리

- **설치**: `npx skills@latest add mattpocock/skills` 한 줄로 시작하며, 에이전트 종류와 설치 범위(프로젝트 vs 전역), 배포 방식(심링크 권장)을 순서대로 선택합니다.
- **가벼운 컨텍스트 설계**: 38개 스킬을 모두 설치해도 사용자 직접 호출 방식 덕분에 컨텍스트 소모는 660토큰에 불과합니다.
- **초기 설정**: `set up mattpocockskills`로 이슈 트래커(로컬 마크다운/Jira/Linear 등), 트리아지 라벨, 도메인 문서(단일/멀티 컨텍스트)를 구성합니다. 특히 이슈 트래커는 자연어로 "Jira로 설정해줘"라고 말하는 것만으로 연동됩니다.
- **핵심 플로우**: `ask matt`로 방향을 물어본 뒤, `grill with docs`(인터뷰형 계획 수립) → 필요시 `prototype` → 규모가 크면 `to spec`/`to tickets` → `implement` 순으로 진행합니다.
- **컨텍스트 윈도우 관리**: 약 14만 토큰을 넘어서면 LLM 성능이 저하된다고 보고, 작업 규모가 크지 않다면 하나의 끊기지 않은 세션 안에서 처리하는 것을 권장합니다.

실무에 적용한다면, 새 프로젝트를 시작할 때 스킬 설치와 초기 설정(이슈 트래커, 트리아지 라벨, 도메인 문서)을 표준 온보딩 절차로 삼고, 코드 변경 작업은 무작정 구현부터 들어가기보다 `grill with docs`로 요구사항을 먼저 명확히 하는 습관을 들이는 것이 좋겠습니다.
