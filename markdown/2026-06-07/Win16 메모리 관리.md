## 핵심 개념

> [!abstract]
> 16비트 Windows의 메모리 관리 메커니즘을 설명하는 기술 문서로, 8086 실모드 시대부터 비롯된 세그먼트 기반 메모리 관리와 NE 실행 파일 형식의 역할을 다룬다. Windows가 메모리 부족을 해결하기 위해 오버레이 매니저로 기능했으며, 세그먼트 핸들을 통해 메모리의 실제 위치와 관계없이 프로그램이 작동할 수 있도록 설계되었다.

## 상세 내용

- Windows 메모리 관리는 세그먼트 기반이며, 각 세그먼트는 최대 64KB 크기의 연속 메모리 블록으로 핸들을 통해 식별된다.
- NE(New Executable) 형식은 MZ 형식과 달리 세그먼트 단위로 디스크에 저장되어 개별 세그먼트의 로드와 메모리 이동을 가능하게 했다.
- Windows 세그먼트 핸들은 Intel 286 보호 모드 셀렉터에서 영감을 받았으며, 메모리 위치와 무관하게 세그먼트를 식별한다.

## 전문 번역

# 16비트 Windows 메모리 관리

*2026년 6월 5일 Michal Necasek 작성*

이 글은 16비트 Windows 메모리 관리가 정확히 어떻게 동작하는지 이해하려는 과정에서 나온 일종의 지식 기반 문서입니다. 완전히 미문서화된 건 아니지만, 명확하게 설명된 자료도 많지 않거든요. Windows 3.0이 나오기 전부터 대부분의 애플리케이션 개발자가 고수준 언어를 사용할 것으로 가정했기 때문입니다. 결국 개발 도구가 낮은 수준의 세부사항을 처리해줄 거라고 본 거죠.

게다가 초보 Windows 개발자를 위한 자료들은 주로 윈도우, 아이콘, 메뉴 같은 눈에 띄는 부분에만 집중했습니다. 메모리 관리는 대충 넘어가곤 했는데, 사실 Hello World보다 조금만 복잡한 Windows 애플리케이션을 만들려면 메모리 관리가 정말 중요합니다.

> Windows 3.0 SDK HeapWalker 메모리 분석 도구

메모리 관리의 복잡함은 Windows 1.x와 2.x의 8086 리얼 모드 시대까지 거슬러 올라갑니다. Windows가 보호 모드에서만 실행되던 Windows 3.1 이후에도 이러한 복잡성이 상당 부분 그대로 남아있었어요.

특별한 언급이 없는 한, 이 글의 "Windows"는 Windows NT가 아닌 16비트 버전을 의미합니다.

## Windows 메모리 관리 기초

Windows 메모리 관리를 이해하려면 이 한 가지를 알아야 합니다. 초기부터 Windows는 기본적으로 고급스러운 오버레이 관리자였거든요. 한동안 Windows는 당시 PC들에 비해 너무 커서, 자주 사용되는 메모리 세그먼트만 물리 RAM에 두고 덜 자주 쓰이는 세그먼트는 필요할 때 버렸다가 다시 불러오는 방식이 필요했습니다. 페이징은 쓸 수 없었어요. 8086과 80286 시스템에서 지원하지 않았거든요. 그리고 Windows 3.0이 나올 때까지 시장의 거의 대부분이 이 칩들을 사용했습니다.

코드 세그먼트 하나와 데이터 세그먼트 하나를 가진 간단한 애플리케이션이라면, Windows 세그먼트의 이동 가능한 특성이 거의 투명하게 작동합니다. 애플리케이션이 실행되면 CS(코드) 세그먼트 레지스터가 코드 세그먼트를 가리키고, DS(데이터)와 SS(스택) 세그먼트 레지스터가 데이터 세그먼트를 가리킵니다. 애플리케이션이 코드 세그먼트 내에서만 near 호출/점프를 사용하고, 데이터/스택 세그먼트로 near 포인터를 쓴다면, CS/DS/SS 레지스터에 실제로 로드된 값이 뭔지는 신경 쓸 필요가 없어요. Windows가 세그먼트를 이곳저곳으로 옮겨다니면서도 모든 게 잘 작동합니다.

하지만 Hello World 예제를 따라가보는 초보 개발자도 곧 의심하기 시작합니다. 16비트 Windows 세상이 그렇게 단순하지만은 않다는 걸요. 윈도우 프로시저는 `FAR PASCAL`로 선언해야 하는데, 이건 Windows 호출 규칙을 따라야 하니까 이해할 수 있습니다. 그런데 애플리케이션의 실행 파일에서 내보내야(export) 합니다. 그렇지 않으면 프로그램이 제대로 작동하지 않거든요. 이건 비-Windows 개발자들에게는 완전히 낯선 개념입니다.

Windows는 메모리 관리 방식을 구현하기 위해 "New Executable"(NE) 형식을 채택하고 확장했습니다. 이 형식은 먼저 "DOS 4"(보통 Multitasking DOS 4.0으로 알려짐)에서 사용됐는데, PC DOS나 MS-DOS 4.0/4.01과는 꽤 달랐어요. DOS의 MZ 실행 파일 형식에서는 애플리케이션이 사실상 하나의 거대한 이진 덩어리인 반면, NE 형식은 세그먼트 지향적입니다. 각 세그먼트가 디스크에 별도로 저장되기 때문에 Windows는 개별 세그먼트를 불러오거나 다시 불러올 수 있고, 메모리 안에서 옮겨 다닐 수 있게 된 거죠.

NE 형식은 임포트(import)와 익스포트(export)도 지원합니다. 임포트는 애플리케이션이 OS처럼 외부 코드를 호출해야 할 때 쓰입니다. 익스포트는 외부에서 호출되는 애플리케이션 코드용입니다.

윈도우 프로시저가 바로 그런 외부 호출 코드입니다. 익스포트되어야 Windows가 마법을 부릴 수 있게 되거든요. 그 마법이란 윈도우 프로시저 프롤로그(진입 순서)를 수정해서 애플리케이션 자신의 데이터 세그먼트를 DS 레지스터에 로드하도록 만드는 것입니다.

## 메모리 이동

Windows 메모리 관리의 모든 것은 세그먼트를 중심으로 돌아갑니다. 세그먼트는 최대 64KB 크기의 연속된 메모리 블록입니다. 보통 8086 프로그래밍에서는 각 세그먼트를 세그먼트 주소로 식별하는데, 이것은 물리 메모리의 주소와 직접 일치합니다. 그런데 Windows의 대부분 세그먼트는 이동되거나 버려질 수 있으므로, 대신 핸들로 식별됩니다. 핸들은 16비트 값인데, 실제로는 어떤 테이블의 단순한 인덱스일 수도 있지만, 불투명한 값으로 취급해야 합니다.

x86 보호 모드에 익숙한 프로그래머라면 Windows 세그먼트 핸들이 보호 모드 셀렉터(selector)와 비슷하다는 걸 알 수 있습니다. 둘 다 16비트 값으로 메모리 세그먼트를 유일하게 식별하지만, 시스템 메모리에서의 세그먼트 위치와는 무관합니다. 이 유사성은 우연이 아닙니다. Windows 1.0 메모리 관리의 설계자인 Steve Wood는 Intel 286의 보호 모드를 Windows 메모리 관리자의 영감으로 삼았거든요. (286은 1982년에 출시되었고, Windows 개발은 1983년에 시작됐습니다.)

핸들은 메모리 내 세그먼트의 위치와 무관하게 메모리 세그먼트를 가리킵니다. 즉, 8086 세그먼트 주소가 뭐든 상관없다는 뜻이죠.
