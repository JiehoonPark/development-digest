## 핵심 개념

> [!abstract]
> Alex Self의 AI 보조 개발 프로세스는 자동화된 의심(automated doubt)을 통해 LLM 생성 코드의 신뢰성을 높입니다. 설계 단계에서 여러 전문 에이전트(subagent)가 다양한 관점에서 검토하고, 개발 단계에서 단일 Claude 인스턴스가 작성하는 구조입니다.

## 상세 내용

- 설계 단계: Pre-implementation Architect, Documentation Validator, Assumption Excavator 등 3개 에이전트로 초기 검토
- 중규모 이상: Gap Analyzer, Implied Completeness Detector, Ambiguity Mapper 추가 검토로 누락된 요소 발견
- 개발 단계: 단일 Claude Code 인스턴스가 작성하며, 신뢰 문제로 인해 subagent 쓰기는 최소화

> [!tip] 왜 중요한가
> LLM으로 코드 생성 시 다중 관점 검토를 자동화하면 버그와 설계 결함을 초기에 발견할 수 있습니다.

## 전문 번역

# AI와 함께 개발할 때 의심을 자동화하기

이 프로세스는 불신에서 출발했습니다. 초기에 LLM 파트너에게 너무 많은 일을 너무 빨리 맡겼고, 표준적인 엔지니어링 관행을 건너뛰었거든요. 신뢰를 되찾기 위해 할 수 있는 한 많은 의심을 자동화하기로 결심했습니다.

그렇다면 "의심을 수행한다"는 게 무슨 뜻일까요? 결과물을 비판적으로 검토하고, 그걸 반복하는 것입니다. 만약 AI를 이용해 코드나 명세서, 문서나 기타 결과물을 작성하고 있다면 이 글이 도움이 될 거예요.

저는 서브에이전트를 많이 씁니다. 이들이 전체 프로세스의 핵심을 담당하거든요. 표준적인 Claude 하나로는 다루지 못할 전문화된 관점들을 갖고 있습니다. 핵심 아이디어는 여러 관점에서 의심을 자동화하고, 검토를 미리 앞당기는 것입니다. AI 개발에서 시점이 많을수록 좋습니다. 마치 두 눈이 깊이감을 주는 것처럼, 각기 다른 각도에서 다른 결함들을 잡아낼 수 있거든요.

개발 프로세스는 대략 이런 흐름으로 진행됩니다:

## 1단계: 설계

먼저 구현하고 싶은 아이디어나 기능, 그리고 명세서가 필요합니다. 좋은 개발 관행처럼, 먼저 명세서나 PRD, 계획 같은 설계 문서로 시작하는 게 현명합니다. Claude에게 명세서를 작성하도록 요청하고, 저는 2~5분 정도 훑어보면서 핵심 구현 요소들이 제대로 반영되었는지 확인합니다. 여기서부터 반복 작업이 시작됩니다.

Claude Code의 슬래시 커맨드로 구현 전 워크플로우(Pre-implementation workflow)를 시작하는데, 여기에 세 개의 에이전트가 첫 번째 검증을 수행합니다: Pre-Implementation Architect, Documentation Validator, Assumption Excavator입니다. 이들은 설계 품질, 범위 평가, 완성도, 문서 누락, 명세서에 숨어있는 가정들을 검증합니다. 발견된 모든 관련 사항은 메인 터미널 에이전트가 명세서에 반영합니다. 보통 스코프에 따라 10~25개 정도 수정 사항이 생기죠.

**발견 사례:**

**Assumption Excavator:** "registry-sdk의 executionStatsSchema는 {totalCount, recentCount, windowMinutes}를 반환하는데, 명세서는 {avgScore, medianDurationMs, passRate, lastRunDate, lastRunScore}를 가정하고 있습니다. 새로운 API 엔드포인트 없이는 전체 히스토리 섹션을 구현할 수 없습니다."

**Pre-Implementation Architect:** "HarnessProfile이 경로 설정과 함께 mcp의 read/merge/remove/write 메서드를 임베드하고 있는데, McpConfigStrategy를 분리할 것을 고려해보세요. 그렇지 않으면 각 harness 파일이 80~120줄로 커질 겁니다."

스코프에 따라 반복 횟수가 결정됩니다. 필요하면 다음 에이전트들을 투입합니다: Gap Analyzer, Implied Completeness Detector, Ambiguity Mapper. 이 에이전트들은 특히 시스템의 누락된 측면들을 찾는 데 탁월합니다. 갭이 발견되면 명세서에 추가됩니다.

**발견 사례:**

**Gap Analyst:** "McpConfigStrategy는 read/merge/write를 정의하지만, 형식이 잘못된 입력, 권한 거부, 부분 쓰기 실패, 파일 잠금에 대한 동작을 명시하지 않습니다. 3가지 형식의 4개 harness에서 사용자 설정 파일에 대한 파괴적 작업이 일어날 수 있습니다."

**Implied Completeness Detector:** "Manifest는 루트에서만 버전을 기록하지만, 설치 상태는 harness마다 다릅니다. v0.3.0 사용자(Claude Code)가 --harness opencode로 v0.4.0을 실행할 때 동작이 정의되지 않습니다. Harness별 버전 관리나 업그레이드 조정이 다루어지지 않았습니다."

**실무 활용:**

- **소규모:** Pre-implementation만 진행
- **중규모:** Pre-implementation + Gap, Implied, Ambiguity
- **대규모:** 각 에이전트마다 여러 번 전체 검토, 필요시 다른 전문 에이전트 추가

이제 시간을 가져 명세서를 읽습니다. 15~60분 정도 소요되는데요. 모든 게 확인되고 명세서가 개발 준비 완료 상태면, Claude에게 함께 따라갈 체크리스트를 만들도록 요청합니다. 체크리스트는 별도 파일로 생성되며, 중간에 세션을 종료했다가 다시 돌아와야 할 때 유용합니다.

## 2단계: 개발

Claude가 명세서와 체크리스트를 펼치고 개발을 시작합니다. 새 세션에서 개발이 이미 진행 중인 상태로 명세서를 받는다면, 보통 Chain Tracer나 Deep Explore 서브에이전트를 보내 전체 그림을 파악한 뒤 다시 시작합니다.

제 개발 프로세스에서 특이한 점이 하나 있는데, 저는 **쓰기 작업에 서브에이전트를 사용하지 않습니다.** 이것도 신뢰 문제와 연결되어 있습니다. 과거에 쓰기 작업용 서브에이전트를 띄웠다가 오히려 더 큰 문제가 생긴 경험 때문입니다. 물론 "일시적으로"라고 했는데, 이건 분명히 바뀔 거라고 생각합니다. 적절한 swarm 오케스트레이션이나 worktree, 에이전트-명세서 기반 개발 같은 방법들이 있다는 걸 알고 있지만, 아직 그 수준의 신뢰도에 도달하지 못했거든요. 대량 업데이트가 필요할 때는 Claude 터미널 에이전트가 서브에이전트를 띄우기도 하지만, 대체로 단일 Claude Code 터미널 인스턴스로 프로젝트를 구축하는 걸 선호합니다.

명세서의 모든 단계를 완료할 때까지 진행합니다. 빌드가 작동하는지 검증한 후, 구현 후 검증 프로세스가 시작됩니다.
