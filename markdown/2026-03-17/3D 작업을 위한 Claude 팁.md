## 핵심 개념

> [!abstract]
> Claude Code를 활용하여 3D 웹 앱 개발 시 시각적 작업 검증의 어려움을 극복하는 방법을 소개합니다. 반복적 검증 루프를 구성하여 Claude가 카메라 조작, 디버그 마커 배치, 스크린샷 비교를 통해 자체 검증하도록 하는 워크플로우를 제시합니다.

## 상세 내용

- Claude는 CSS와 2D 이미지 분석에는 능하지만 3D 공간 분석에서는 실패하므로, 스크린샷과 좌표 로깅을 통한 수동 피드백이 필요했습니다.
- 자동화된 검증 루프를 구성하면 Claude가 STL 재생성, 다양한 각도에서의 렌더링 캡처, 위치 검증을 사용자 개입 없이 반복할 수 있습니다.
- 3D 프로젝트에서는 단순히 요청을 이해시키는 것보다 공유 언어로 작동하는 도구를 먼저 구축하는 것이 효과적입니다.

> [!tip] 왜 중요한가
> LLM 기반 개발 도구의 한계를 극복하고 복잡한 시각적 작업에서 자동화된 검증 루프를 구성하는 패턴은 다른 도메인 개발에도 적용 가능합니다.

## 전문 번역

# Claude와 함께하는 3D 웹 개발: 실전 워크플로우

2026년 3월 13일

Claude Code를 쓸 때마다 약간의 불안감이 있긴 하지만, 현재 진행 중인 프로젝트에서는 정말 필수적인 도구가 되어버렸습니다. CSS, 디자인, API 패턴 같은 핵심 아키텍처 부분은 여전히 NeoVim에서 직접 손으로 작성하지만, 기본 골격을 잡은 후의 대부분 코드 작업은 이제 Claude에 맡기고 있어요.

물론 Claude가 생성한 코드가 항상 완벽한 건 아닙니다. 생성 후 꽤 많은 정제 작업이 필요하긴 한데, 이마저도 Claude에게 제안하는 방식으로 처리합니다. 편집 피드를 실시간으로 지켜보고 있어서, Claude가 이상한 걸 할 때 빠르게 멈출 수 있을 정도의 속독 기술도 생겼네요.

## 웹 프로젝트에서는 잘 작동하지만...

이 방식은 대부분의 웹 프로젝트에서 정말 잘 작동합니다. 그런데 "눈으로 봐야 하는" 시각적 작업이 들어가면 Claude의 한계가 명확해집니다. CSS는 꽤 잘 추론하고, 디자인 언어나 색감 이론도 적절히 프롬프트하면 괜찮은 수준으로 이해합니다. 다만 3D 공간 분석처럼 복잡한 시각적 작업은 항상 실패하곤 합니다.

다행히 Claude는 2D 이미지를 읽는 도구가 훌륭합니다. 다양한 각도에서 본 3D 공간의 스크린샷 여러 개와 적절한 참조 언어를 함께 제시하면, 대부분의 경우 뭔가를 해낼 수 있거든요. Table Slayer와 Counter Slayer 같은 3D 웹 앱을 만들 때도 처음엔 수동으로 스크린샷을 찍어서 "여기 봐, 이건 틀렸어"라는 식으로 계속 피드백해야 했습니다. 대부분의 시간이 제가 스크린샷을 일일이 촬영해서 제공하는 데 쓰였어요.

## LLM과 일할 땐 "읽을 수 있는 결과물"이 중요

Claude와 협업하려면 빠르게 배워야 할 게 있습니다. 바로 "읽을 수 있는 결과물"을 기준으로 생각해야 한다는 점입니다. 대부분의 경우 이건 테스트 레이어를 의미합니다. 그런데 디자인을 탐색하는 초기 단계에선 보통 테스트가 없으니, 임시 로깅을 대량으로 생성하도록 요청하는 방식으로 보완합니다. 3D 프로젝트라면 위치나 좌표 정보 같은 걸 말하는 거죠. 이 방식이 어느 정도는 작동하지만, 상당히 번거롭고 보통 큰 실패에 부딪친 후에야 시작합니다.

CAD 시스템에서는 합집합과 차집합 연산이 많이 일어납니다. 박스가 거기 있다고 해서 눈에 보이는 건 아니거든요.

## STL 파일의 현실적인 한계

Claude는 STL 파일을 읽을 수 있다고 말합니다. 하지만 실제로는 바이너리 내용을 대충 꾸며내고 있을 뿐이에요. 결국 Python 라이브러리를 설치하는 방향으로 빠지게 되는데, STL을 읽는 데는 어느 정도 도움이 됩니다. 카메라를 회전하고 특정 위치로 줌인할 수도 있죠.

하지만 Three.js 웹 앱에선 문제가 다릅니다. 진짜 원인은 CAD 시스템 자체에서 3D를 구성하는 로직에 있거든요.

## 해결책: 반복 검증 루프

이 문제를 해결하려면 애플리케이션의 상태를 읽고, 카메라를 조작하고, 디버그 마커를 추가하면서 반복할 수 있는 실제 컨텍스트 루프가 필요합니다. 핵심은 Claude가 앱을 스스로 네비게이션하고, 카메라를 제어하며, 시각적 마커(예: 위치 맥락을 파악하기 위한 빨간 구)를 추가해서 자체 검증하는 것입니다. 저는 이런 "반복 검증" 루프가 Claude와 작업할 때 본질적으로 중요하다는 걸 깨달았습니다. 문제를 해결했다고 가정하기 전에 자신의 작업을 이중 검증할 수 있거든요.

Counter Slayer에서 사용한 루프를 예로 들어보겠습니다.

### 기하학적 구조 반복 워크플로우

기하학적 구조를 변경할 때는 다음 자체 포함 루프를 사용해 사용자 개입 없이 반복합니다:

**핵심 루프**

1. 기하학적 구조 코드 변경 (lid.ts, counterTray.ts, box.ts)
2. STL 재생성: `npx tsx scripts/generate-geometry.ts`
3. 렌더링으로 검증: `npx tsx scripts/capture-view.ts --angle iso`
4. project.json에서 위치와 레이아웃 데이터 읽기
5. 여러 각도에서 확인, 문제 영역 줌인
6. 문제가 있으면 1단계로 돌아가기
7. 만족하면 사용자에게 보고

더 자세한 내용은 전체 Claude.md 문서에 있지만, 이것이 핵심입니다. "루프"를 통해 Claude는 Playwright로 앱을 네비게이션하고, 트레이나 박스 같은 3D 객체를 선택하고, 카메라를 변경하고, 특정 위치에 빨간 구를 배치하고, 스크린샷을 확인해서 요청과 일치하는지 볼 수 있습니다. 가장 중요한 점은 제 승인을 받지 않고도 이 모든 걸 할 수 있다는 것입니다.

예를 들어, 앱을 네비게이션할 수 있는 도구는 이렇습니다:

```bash
# 미리 설정된 각도로 보기
npx tsx scripts/capture-view.ts --angle iso
npx tsx scripts/capture-view.ts --angle top
npx tsx scripts/capture-view.ts --angle front

# 줌인
npx tsx scripts/capture-view.ts --angle left --zoom 3

# 사용자 정의 카메라 위치 (Three.js Y-up 좌표)
npx tsx scripts/capture-view.ts --pos "100,80,150" --look-at "0,25,50"

# 특정 파일로 출력
npx tsx scripts/capture-view.ts --angle top --out mesh-analysis/view-top.png

# ID로 특정 항목 보기 (항목 선택 및 적절한 보기 모드 설정)
npx tsx scripts/capture-view.ts --trayId nrme206 --angle bottom --zoom 2
```

## 생각의 전환: 도구 먼저, 협업 두 번째

전반적으로 Claude와 일하는 방식에 대한 생각이 바뀌었습니다. Claude가 제 요청을 이해할 것으로 기대하기보다는, 거의 항상 먼저 도구를 만들어서 프로젝트에 대해 함께 이야기할 수 있는 공용 언어를 만드는 거죠.
