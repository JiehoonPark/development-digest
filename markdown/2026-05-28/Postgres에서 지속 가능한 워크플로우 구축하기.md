## 핵심 개념

> [!abstract]
> 지속 가능한 워크플로우는 프로그램이 실행되면서 정기적으로 데이터베이스에 진행 상황을 체크포인트하는 방식으로, 크래시 발생 시 마지막 체크포인트에서 복구할 수 있다. 일반적으로 Temporal, Airflow, AWS Step Functions 같은 외부 오케스트레이터를 통해 구현되지만, 저자는 중앙 오케스트레이터가 불필요하며 Postgres 자체를 오케스트레이터로 사용하는 것이 더 간단하고 효율적이라고 주장한다. Postgres 기반 접근 방식에서는 애플리케이션 서버가 직접 Postgres와 통신하여 워크플로우를 실행하고, 서버들이 잠금 절(locking clause)을 통해 협력해서 워크플로우를 처리한다. 이 설계는 확장성, 가용성, 관찰성, 보안 등의 문제를 잘 알려진 Postgres 고유 솔루션으로 해결할 수 있다.

## 상세 내용

- 지속 가능한 워크플로우는 프로그램의 진행 상황을 데이터베이스에 정기적으로 저장하여 크래시 발생 시 마지막 완료된 단계에서 복구할 수 있는 구조이다. 비디오 게임의 세이브 기능과 유사하게 작동한다.
- 외부 오케스트레이터 방식(Temporal, Airflow 등)은 중앙 오케스트레이터 서버가 워크플로우 기록을 유지하고 워커에게 단계를 할당하며, 각 단계 완료 시 결과를 체크포인트하는 구조이다. 이 방식은 기본적으로 과도하게 복잡하다.
- Postgres 기반 접근 방식에서는 애플리케이션 서버가 Postgres 워크플로우 테이블에 직접 접근하여 워크플로우를 폴링하고 실행한다. 서버들은 잠금 절을 사용해 정확히 하나의 워커만 각 워크플로우를 처리하도록 협력하며, 단계 결과를 Postgres에 직접 체크포인트한다. 중앙 오케스트레이터가 필요 없어진다.
- 확장성 측면에서 단일 Postgres 서버는 초당 수만 개의 워크플로우를 처리할 수 있으며, 분산 Postgres(예: CockroachDB) 또는 샤드된 Postgres를 사용해 추가 확장이 가능하다. 가용성은 Postgres 스트리밍 레플리케이션과 자동 장애 조치, 관리형 서비스의 다중 가용 영역 배포로 확보할 수 있다.
- 관찰성은 워크플로우와 단계가 Postgres 테이블에 체크포인트되므로 기본적으로 내장되어 있다. SQL 쿼리로 모든 워크플로우 관찰성 질문을 표현할 수 있으며, 예를 들어 지난 한 달간 오류가 발생한 모든 워크플로우를 찾는 쿼리를 간단히 작성할 수 있다. 이는 Postgres의 관계형 모델이 제공하는 강력한 표현력 덕분이다.

> [!tip] 왜 중요한가
> Postgres 기반 워크플로우는 별도의 오케스트레이션 인프라 구축 및 관리 복잡성을 제거하면서도 확장성과 관찰성을 제공하므로, 개발팀이 워크플로우 안정성에 집중할 수 있다.

## 전문 번역

# Postgres 기반 Durable Workflow: 오케스트레이터 없이 구축하기

Durable workflow는 신뢰할 수 있는 프로그램을 만드는 강력한 기법입니다. 프로그램이 실행되면서 주기적으로 진행 상황을 데이터베이스에 저장해두는 거죠. 이렇게 하면 프로그램이 충돌해도 마지막 저장 지점에서 복구할 수 있습니다. 마치 게임에서 정기적으로 세이브하는 것과 같은 원리입니다.

## 기존 방식: 외부 오케스트레이터

지금까지 Temporal, Airflow, AWS Step Functions 같은 시스템들은 외부 오케스트레이터를 통해 durable workflow를 구현해왔습니다.

이 패턴에서는 클라이언트가 워크플로우를 제출하면, 오케스트레이터가 데이터베이스에 레코드를 생성한 뒤 워커에게 전달합니다. 워커가 단계를 완료할 때마다 결과를 오케스트레이터에게 보내고, 오케스트레이터는 이를 저장한 후 다음 단계를 파견합니다. 워커가 실패하면 오케스트레이터가 마지막 저장 지점부터 다른 워커에서 실행을 재개합니다.

하지만 여기서 생각해볼 점이 있습니다.

## 더 간단한 방식: Postgres를 오케스트레이터로

Durable workflow의 핵심은 프로그램 상태를 데이터베이스에 저장하는 것입니다. 그렇다면 굳이 별도의 오케스트레이터 서버가 필요할까요?

데이터베이스 자체를 오케스트레이터로 삼으면 훨씬 단순해집니다. 특히 Postgres는 인기도, 확장성, 풍부한 생태계 덕분에 이 역할에 최적입니다.

Postgres 기반 durable workflow 시스템에서는 애플리케이션 서버들이 중앙 오케스트레이터를 거치지 않고 직접 Postgres와 통신합니다. 워크플로우를 실행할 때는 Postgres의 workflows 테이블에 항목을 생성하고, 애플리케이션 서버들이 이 테이블을 주기적으로 폴링해서 워크플로우를 가져가 실행합니다. 실행 중에는 각 단계의 결과를 Postgres에 저장합니다. 서버가 실패해도 다른 서버가 저장된 체크포인트에서 복구할 수 있습니다.

## 이 방식의 장점

**1. 조정이 자동으로 이루어진다**

오케스트레이터에 의존하지 않고, 서버들이 Postgres를 통해 자체적으로 조정합니다. LOCK 절 같은 메커니즘을 사용해 각 워크플로우가 정확히 하나의 워커에서만 처리되도록 보장합니다. 같은 워크플로우를 여러 워커가 동시에 실행하려고 해도, Postgres의 무결성 제약조건이 중복 작업을 감지해서 자동으로 처리됩니다.

**2. 근본적으로 더 간단하다**

중앙 오케스트레이터를 제거하면, 확장성, 가용성, 모니터링, 보안 같은 어려운 문제들을 Postgres의 검증된 솔루션으로 해결할 수 있습니다.

## 확장성과 가용성

데이터베이스 기반 workflow 시스템의 확장성과 가용성은 기본적으로 하위 데이터베이스의 성능에 달려있습니다. 워커 서버를 추가해서 수평 확장할 수 있고, 최대 용량은 데이터베이스가 처리할 수 있는 워크플로우의 양으로 결정됩니다. 워커는 서로 교체 가능하고 상태를 자유롭게 복구할 수 있으므로, 데이터베이스만 정상이면 시스템은 계속 작동합니다.

Postgres를 사용할 때의 이점은 더욱 명확합니다. 단일 Postgres 서버는 초당 수만 개의 워크플로우를 처리할 수 있고, CockroachDB 같은 분산 데이터베이스나 샤드된 Postgres를 사용하면 더 확장할 수 있습니다. 고가용성은 스트리밍 복제와 자동 페일오버로 지원되며, 관리형 서비스들은 다중 AZ 배포와 높은 SLA를 기본으로 제공합니다. 결국 수십 년간 Postgres를 운영하며 축적된 엔지니어링 경험이 그대로 durable workflow 시스템에 적용됩니다.

## 모니터링

Postgres 기반 durable workflow에서는 모든 워크플로우와 단계가 테이블에 저장됩니다. 즉, 모니터링이 기본으로 내장되어 있습니다. 저장된 체크포인트를 조회해서 워크플로우를 실시간으로 모니터링하고 시각화할 수 있습니다.

Postgres는 이 부분에서 특히 강력합니다. 거의 모든 workflow 관찰 쿼리를 SQL로 표현할 수 있거든요.

예를 들어, 지난 한 달간 오류가 발생한 모든 워크플로우를 찾는 쿼리는 이렇게 간단합니다:

```sql
SELECT * FROM workflows 
WHERE status = 'error' 
AND created_at > now() - interval '1 month'
```

이런 쿼리는 간단해 보이지만, 강력합니다. Postgres의 관계형 모델이 복잡한 쿼리를 쉽게 표현할 수 있게 해주기 때문입니다. 중앙 오케스트레이터는 이 정도의 유연한 쿼리 기능을 제공하기 어렵습니다.
