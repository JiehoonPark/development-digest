## 핵심 개념

> [!abstract]
> QBE 1.3은 새로운 IL 매칭 알고리즘, 성능 최적화, Windows ABI 지원, 위치 독립 코드 생성 등을 포함한 1.0 이후 가장 중요한 릴리스이다. CoreMark 벤치마크에서 gcc -O2의 63% 성능을 달성했으며, Hare 테스트 스위트에서는 33% 성능 향상을 이루었다.

## 상세 내용

- OCaml 기반 mgen 도구를 사용한 명령어 선택 최적화로 IL 패턴 매칭을 메타프로그래밍화
- GVN/GCM, 루프 최적화, if-제거 등 다양한 최적화 기법 도입으로 성능 개선
- Windows ABI 지원 추가 및 동적 상수(DYNCONST) 플래그를 통한 위치 독립 코드와 공유 객체 생성 지원

> [!tip] 왜 중요한가
> 저수준 언어 개발자에게 더 효율적인 컴파일 백엔드 선택지를 제공하고, 경량 컴파일러 구현의 성능 최적화 기법을 시연한다.

## 전문 번역

# QBE 1.3 릴리스: 의미 있는 성장의 순간

QBE 1.3이 나오기까지는 꽤 오랜 시간이 걸렸습니다. 하지만 1.0 이후 가장 중요한 버전이라고 자신 있게 말할 수 있어요. 약 7천 줄의 새로운 코드가 추가되었고 1천5백 줄이 삭제되었거든요.

일상적인 버그 수정은 물론이고, 이번 릴리스에는 완전히 새로운 IL 매칭 알고리즘이 들어갔습니다. Roland Paterson-Jones의 최적화 기법들도 추가되었고, Scott Graham은 Windows ABI 지원을 구현했으며, 저는 Michael Forney의 아이디어를 바탕으로 QBE가 위치 독립적인 코드(공유 라이브러리 같은)를 생성하도록 했습니다. QBE는 팀의 작업물이고, 이번 릴리스에 기여해주신 모든 분들께 감사드립니다.

## 성능 최적화: Coremark를 통한 목표 달성

QBE 팀에는 가끔 "날파리"가 무는 순간이 있는데, 이번엔 Roland가 그 대상이었어요! Roland가 제안한 coremark 벤치마크는 최적화할 수 있는 구체적이면서도 단순한 놀이터가 되어주었습니다.

QBE 1.2로 초기 측정을 해보니 우리는 "GCC -O2의 70% 수준"이라는 목표에서 한참 미치지 못하고 있었습니다. 40% 정도에 그치고 있었거든요. 1.3 버전에서 이 문제를 해결하기로 결정했습니다.

프로파일링 데이터를 살펴보니 성능 차이의 대부분이 두 함수로 귀결되더군요: `ee_isdigit`과 `crcu8`이었습니다. 흥미로운 점은 이 함수들이 순수한 C 문법은 아니라는 거예요. 예를 들어 `ee_isdigit`은 보통 인라인으로 처리되며, `&&` 대신 `&`을 사용하고 불필요한 삼항 연산자를 피합니다. CRC의 경우 미리 계산된 테이블로 구현하는 게 최적이죠.

이 관찰은 조금 실망스러웠습니다. QBE의 인라이닝 부재를 제외하고는, 다른 컴파일 작업에도 적용 가능한 일반적인 오버헤드의 원인을 지적해주지 못했거든요. 물론 CPU 집약적인 코드는 대부분의 시간을 작은 코드 섹션에서 보내는 게 자연스럽긴 합니다.

그래도 여러 최적화 기법을 시도해봤습니다. GVN/GCM, 루프 최적화, if 제거, CFG 단순화 등을 coremark와 Hare 테스트 스위트 같은 더 현실적인 사례에 적용해본 거죠. 결국 검증된 최적화 패스의 부분집합만 유지하기로 결정했고, 이제 상용 컴파일러의 63% 이상 성능을 coremark에서 달성하고 있습니다.

특히 인라이닝은 최적화 세트에서 빼기로 했는데요. QBE의 함수별 스트리밍 컴파일 모델과의 비호환성을 나중에 해결하기로 미뤘기 때문입니다. Coremark 벤치마크를 수정해서 `ee_isdigit` 함수를 인라인하고 `crcu8`을 더 간단한 분기 없는 구현으로 바꾸면, QBE는 70% 목표를 달성합니다. 새로운 최적화들은 Hare 사용자들에게도 도움이 될 거예요. 저는 Hare 테스트 스위트에서 QBE 1.2 대비 33% 성능 향상(2.6초에서 1.7초)을 측정했습니다.

## IL 패턴 매칭: 메타프로그래밍으로 우아함을 얻다

QBE가 초창기부터 사용해온 하향식 트리 번호 매김 알고리즘이 있습니다. Ken Thompson의 Plan9 C 컴파일러에서 영감받은 이 알고리즘은 꽤 범용적이지만, 산술 연산자의 결합성과 교환성을 우아하게 다루는 데 있어 미묘한 부분들이 있었어요. 저는 오래전부터 이 문제를 메타프로그래밍으로 해결하고 싶었고, QBE 1.3에서 그 목표를 이루었습니다.

새로운 OCaml 도구인 `mgen`이 이제 Lispy IL 패턴을 자연스러운 C 코드로 컴파일합니다. `mgen`은 IL 패턴을 담은 특수 주석 블록을 찾아서, 그 아래에 매칭되는 C 코드를 직접 삽입해줍니다. 생성된 C 코드는 QBE에서 자연스러워 보이도록 설계되었고, 1.3 이전의 손으로 작성한 로직과 비슷하게 작동해요. `isel.c` 파일에서 현재의 `mgen` 사용 사례를 볼 수 있습니다.

좀 더 자세히 설명하자면, 명령 DAG는 Ken Thompson의 컴파일러처럼 번호 매김 방식을 따라 매칭됩니다. 그런 다음 `mgen`은 각 번호를 현재 IL 노드(임시 변수)로 매칭되는 최상위 사용자 패턴을 나타내는 비트셋과 연결합니다. 가장 적합한 패턴을 선택하는 것은 손으로 작성한 로직으로 할 수 있죠.

패턴에는 매처 프로그램을 실행해서 수집할 수 있는 변수들이 포함될 수 있습니다. 이 프로그램들도 `mgen`이 간단한 바이트코드 언어로 생성하는데, `runmatch()` 함수가 이를 해석합니다.

앞으로 `mgen`을 더 많은 백엔드의 명령 선택을 단순화하는 데 사용하고, 나아가 비트 로테이션 같은 IL 패턴을 최적화 패스에서 인식하는 데도 활용할 수 있을 거라고 기대합니다.

## Windows ABI와 위치 독립적 코드 지원

QBE 1.3 릴리스에서는 Scott Graham도 기여했습니다. Scott는 fun 파생 프로젝트에서 원래 작성했던 Windows ABI 구현을 관대하게 업스트림했거든요. QBE가 생성하는 어셈블리는 여전히 AT&T 신택스이고 mingw 어셈블러로 컴파일하는 게 가장 좋습니다(제가 직접 Windows에서 시도해본 건 아니지만요). 이제 Windows를 위해 컴파일하는 것은 QBE에 `-t amd64_win` 플래그를 전달하는 것만큼 간단해졌습니다.

마지막으로, QBE는 위치 독립적 코드 지원을 개선했고, 이제 대부분의 대상에서 공유 라이브러리와 부드럽게 연동되고 심지어 공유 객체를 생성할 수도 있습니다. 지금까지의 주요 걸림돌은 간접 글로벌 접근(예: ELF의 Global Offset Table) 지원이 부족했다는 점이었어요. 이제 IL 레벨에서 새로운 extern "dynamic constant" 플래그(IL 스펙의 DYNCONST)를 지원함으로써 가능하게 되었습니다.

예를 들어 동적으로 링크된 라이브러리의 `dlvar` 변수에 접근하려면 다음과 같이 사용합니다:

```c
%1 =l loadd $dlvar
```

혹시 궁금하신 분들을 위해 설명하자면, 우리는 "dynamic constant"라는 역설적인 표현을 사용합니다. 이는 실행 중에 상수이지만 동적으로만 접근할 수 있는 주소 심볼을 의미하거든요.
