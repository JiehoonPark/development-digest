## 핵심 개념

> [!abstract]
> Mozilla는 Anthropic의 Mythos AI 모델을 사용하여 2개월간 Firefox에서 271개의 보안 취약점을 발견했다. 핵심은 LLM을 특정 작업에 안내하는 커스텀 '하네스'와 결과를 검증하는 두 번째 LLM을 결합하여 거의 거짓 양성 없이 대규모 취약점 탐지를 달성했다는 점이다.

## 상세 내용

- 에이전트 하네스: LLM을 프로젝트 특화 도구와 파이프라인에 연결하여 메모리 안전 문제 탐지를 자동화
- 이중 검증 시스템: 첫 번째 LLM이 취약점을 찾고 두 번째 LLM이 결과를 등급 매겨 신뢰도 확보
- 실제 결과: 공개된 12개 취약점 보고서가 Mozilla의 보안 표준을 충족하며 실무 개발자들의 검증을 받음

> [!tip] 왜 중요한가
> AI 기반 보안 분석의 신뢰성 문제(거짓 양성)를 해결한 실질적 방법론을 보여주므로, 보안 도구 개발 및 자동화에 관심 있는 개발자들에게 참고할 만한 엔지니어링 접근법을 제시한다.

## 전문 번역

# Mozilla가 AI로 Firefox 보안 취약점 271개를 찾아낸 방법

Mozilla의 CTO가 "AI 기반 취약점 탐지로 제로데이 공격의 시대가 끝났다"고 선언했을 때, 반응은 회의적이었습니다. 그도 그럴 것이 이런 식의 주장을 이미 여러 번 들었기 때문이죠. 인상적인 AI 성과만 골라서 발표하고, 불편한 세부사항은 빼먹으면서 과대 광고를 일삼는 패턴이 반복되어 왔거든요.

이런 의심을 의식한 Mozilla는 지난주 실제 작업 과정을 공개했습니다. Anthropic의 Mythos라는 AI 모델을 사용해 2개월에 걸쳐 Firefox의 보안 취약점 271개를 발견한 방법을 설명한 것입니다. Mozilla 엔지니어들이 밝힌 바에 따르면, 이번 성공의 핵심은 두 가지였습니다. 첫째, 모델 자체의 성능 개선이고, 둘째는 Firefox 소스코드를 분석하도록 Mythos를 지원하는 맞춤형 "하네스"를 개발한 것입니다.

## 과거의 실패, 그리고 배운 점

Mozilla가 AI 기반 취약점 탐지를 처음 시도했을 때는 형편없었습니다. 개발자들이 AI 모델에게 코드를 분석해달라고 요청하면, 그럴듯한 버그 리포트가 대량으로 쏟아져 나왔거든요. 문제는 대부분이 환각(hallucination)이었다는 점입니다. 인간 개발자가 자세히 살펴보면 거의 모든 내용이 사실이 아니었고, 결국 기존 방식대로 수동으로 하나씩 검증해야 했습니다.

## 게임체인저: 에이전트 하네스

Mozilla Distinguished Engineer인 Brian Grinstead는 인터뷰에서 이번 달라진 점이 바로 "에이전트 하네스"라고 설명했습니다. 하네스는 LLM을 감싸는 코드로, 모델을 특정 작업 단계로 안내하는 역할을 합니다. 다만 프로젝트별 특성에 맞게 커스터마이징하려면 상당한 자원이 필요합니다.

Grinstead가 설명한 하네스의 작동 방식은 이렇습니다.

> 우리가 만든 하네스는 LLM을 특정 목표 달성을 위해 제어하는 코드입니다. 모델에게 지시(예: '이 파일에서 버그를 찾아')를 주고, 도구(파일 읽고 쓰기, 테스트 케이스 실행 등)를 제공한 다음, 완료될 때까지 반복 실행합니다.

여기서 중요한 것은 하네스가 인간 개발자들이 사용하는 것과 동일한 도구와 파이프라인에 접근한다는 점입니다. Firefox 빌드 및 테스트 환경도 포함되어 있죠.

Grinstead는 다음과 같이 덧붙였습니다.

> 명확하고 결정론적인 성공 신호만 정의하면, 모델에게 계속 일하도록 지시할 수 있습니다. 우리의 경우 메모리 안전 문제를 찾고 있으니 Firefox 샤니타이저 빌드를 가리키면서 "이 파일에 문제가 있으니 찾아줘"라고 말합니다. 모델이 테스트 케이스를 작성하면, 우리의 기존 퍼징 시스템으로 실행하죠. 모델이 "이렇게 HTML을 작성하면 문제가 발생할 것 같은데"라고 제안하면, 도구에 보내져서 검증됩니다. 도구가 긍정 반응을 주면 추가 검증이 진행됩니다.

## 이중 검증 시스템

두 번째 LLM이 첫 번째 LLM의 결과를 평가하는 방식으로 추가 검증이 이루어집니다. 높은 점수를 받으면 개발자들은 기존 방식으로 발견한 버그와 동일한 수준의 신뢰도를 갖게 됩니다.

Grinstead는 강조했습니다. "최종적으로 나오는 버그들은 거의 거짓양성이 없습니다."

Mozilla는 지난주 271개 취약점 중 12개의 전체 Bugzilla 리포트를 공개했습니다. 각 리포트에는 메모리 안전 문제를 유발하는 HTML 코드 등의 테스트 케이스가 포함되어 있으며, Firefox 보안 취약점으로 인정받기 위한 Mozilla의 모든 기준을 충족합니다. 한 연구자는 목요일에 이 리포트들을 보고 "상당히 인상적이다"고 평가했습니다.

## 신뢰할 수 있는 발견

과거의 형편없는 결과들과 달리, 하네스가 안내한 Mythos의 분석 결과는 두 번째 LLM의 검증을 거쳐 최종 리포트에 포함됩니다. 이것이 Mozilla 팀이 이전에 갖지 못했던 신뢰 수준입니다.

Grinstead는 이렇게 정리했습니다.

> 이것이 우리가 현재 규모로 운영할 수 있게 해준 핵심입니다. 엔지니어들이 "네, 이 문제가 맞습니다"라고 확인할 수 있는 손잡이를 제공하고, 코드를 수정한 후 명확하게 고쳤는지 알 수 있으며, 최종적으로 테스트 케이스를 저장소에 추가해 퇴행을 방지할 수 있습니다.

## 회의적인 시각도 존재

다만 Mozilla의 주장에 대한 비판도 있습니다. 271개 취약점 중 CVE 번호를 받은 것이 없다는 점이 초기 비판의 포인트였습니다. 하지만 많은 개발사들처럼 Mozilla도 내부에서 발견한 보안 버그에 대해서는 CVE를 따로 신청하지 않습니다. 대신 한 번의 패치로 여러 취약점을 묶어서 배포하는 방식을 취합니다.
