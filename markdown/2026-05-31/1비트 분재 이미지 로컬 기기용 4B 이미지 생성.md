## 핵심 개념

> [!abstract]
> Prism ML이 개발한 Bonsai Image 4B는 가중치를 이진/삼진 형식으로 압축한 컴팩트 이미지 생성 모델로, iPhone부터 노트북까지 로컬 기기에서 직접 실행 가능합니다. FLUX.2 Klein 4B 대비 8.3배 용량 감소(이진)와 6.4배 감소(삼진)를 달성했습니다.

## 상세 내용

- 1비트 Bonsai Image 4B는 이진 가중치 압축으로 트랜스포머를 0.93GB까지 감소시켜 iPhone 17 Pro Max에서 512x512 이미지를 9.4초에 생성합니다.
- 삼진 변형은 1.21GB 크기로 원본 모델의 95% 정확도를 유지하여 품질과 용량의 균형을 제공합니다.
- Apple Silicon과 CUDA GPU 모두 지원하며, Mac M4 Pro에서는 원본 대비 5.6배 빠른 속도를 달성합니다.

> [!tip] 왜 중요한가
> 개발자는 이제 클라우드 API 없이도 로컬 기기에서 고품질의 이미지 생성을 구현할 수 있어 지연시간 감소, 비용 절감, 반복적인 이미지 생성 작업의 효율성을 얻을 수 있습니다.

## 전문 번역

# Bonsai Image 4B: 로컬에서 돌아가는 경량 이미지 생성 모델

오늘 우리는 Bonsai Image 4B를 공개합니다. 이것은 노트북부터 스마트폰까지 다양한 기기에서 고품질 이미지 생성을 가능하게 하는 경량 모델 계열입니다.

Bonsai Image 4B는 두 가지 변형으로 제공됩니다. 우리가 아는 한, Bonsai Image 4B는 같은 파라미터 규모의 이미지 모델 중 처음으로 iPhone에서 직접 실행되는 모델입니다. 이것은 이미지 생성을 위한 새로운 배포 방식을 열어줍니다. 강력한 성능을 유지하면서도 오픈 소스 가중치를 제공하고, 기존에는 불가능했던 기기들에서 실제로 작동하는 로컬 추론이 가능해진 거죠.

## 로컬 실행이라는 제약 조건

로컬에서 이미지를 생성할 때는 피할 수 없는 제약이 있습니다. 바로 모델이 기기의 메모리 예산 안에 맞아야 한다는 것입니다.

4B급 이미지 모델에서 가장 메모리를 많이 차지하는 부분은 diffusion transformer입니다. 이것도 생성 과정 중에 반복적으로 실행되는데요. 매번 노이즈 제거 단계마다 transformer가 다시 호출되므로, transformer의 크기가 메모리 사용량, 대역폭 요구사항, 그리고 로컬 추론 속도를 모두 좌우합니다.

## Bonsai Image 4B는 어떻게 만들어졌나

Bonsai Image 4B는 FLUX.2 Klein 4B를 기반으로 합니다. 아키텍처는 그대로 유지하되, transformer 가중치를 어떻게 표현하는지를 바꿨습니다. 가중치를 1비트(binary)와 3값(ternary) 형식으로 변환해서, 로컬 배포에서 가장 중요한 부분을 압축한 것입니다.

**1비트 변형:**
Binary 레이어는 full-precision transformer 가중치 대비 약 14배 크기를 줄여줍니다. 다만 정밀도가 중요한 일부 지원 텐서(약 5%, 투영 레이어라고 부릅니다)는 FP16으로 유지합니다. 그 결과 최종 1비트 Bonsai Image 4B transformer는 0.93GB로, 원본 FLUX.2 Klein 4B의 7.75GB에서 8.3배 축소됐습니다.

**3값 변형:**
3값 변형도 같은 구조를 따릅니다. Ternary 레이어는 약 10배 크기를 줄이며, 최종 3값 Bonsai Image 4B transformer는 1.21GB로 full-precision에서 6.4배 축소됩니다. 1비트 모델보다는 약간 크지만, 추가된 영점 상태 덕분에 시각적 품질과 프롬프트 정확도가 더 나아집니다.

## 실제 메모리 사용량

압축된 텍스트 인코더와 FP16 VAE를 포함하면, Apple Silicon 배포 페이로드는 1비트 버전에서 3.42GB, 3값 버전에서 3.88GB입니다. 원본 FLUX.2 Klein 4B는 15.97GB가 필요했으니까요.

런타임에 텍스트 인코더는 프롬프트 인코딩 후 언로드되므로, 실제 활성 메모리는 총 페이로드보다 작습니다.

**512×512 이미지 생성 시:**
- 1비트: 평균 1.5GB (7.8배 감소)
- 3값: 평균 1.96GB (6.0배 감소)
- 원본: 평균 11.74GB

**1024×1024 이미지 생성 시:**
- 1비트: 평균 1.95GB (7.4배 감소)
- 3값: 평균 2.38GB (6.0배 감소)
- 원본: 평균 14.39GB

이런 메모리 감소는 모델이 실행될 수 있는 기기의 범위를 확 넓혀줍니다. Apple Silicon iPhone, iPad, Mac과 CUDA GPU를 지원하도록 구성했거든요. Apple 하드웨어는 MLX 저비트 경로를, CUDA는 Gemlite 저비트 GEMM 커널을 사용합니다.

실제로 iPhone 17 Pro Max에서는 원본 FLUX.2 Klein 4B 파이프라인이 기기 메모리에 맞지 않지만, Bonsai Image의 두 변형 모두 온디바이스 실행이 가능합니다.

## 성능은?

iPhone 17 Pro Max에서 512×512 이미지를 약 9.4초에 생성하고, Mac M4 Pro에서는 약 6초에 생성합니다. Mac M4 Pro에서는 원본 full-precision FLUX 파이프라인보다 5.6배 빠릅니다.

## 품질은 어떻게 유지했나

압축은 모델이 여전히 유용할 때만 의미가 있습니다. 우리는 Bonsai Image 4B를 세 가지 평가 지표로 검증했습니다:

- **GenEval**: 객체 구성과 속성 결합
- **HPSv3**: 인간 선호도와 미적 품질
- **DPG-Bench**: 세밀한 프롬프트 추종과 의미적 충실도

**3값 변형(품질 중심):**
1.21GB에서 FLUX.2 Klein 4B의 정확도 95%를 유지하면서, diffusion transformer를 6.4배 축소했습니다.

**1비트 변형(크기 중심):**
diffusion transformer를 1GB 이하로 줄였습니다(8.3배 감소). 그러면서도 동일한 세 평가에서 FLUX.2 Klein 4B 정확도의 88%를 유지합니다.

이 두 변형은 품질과 크기의 균형을 새롭게 재정의합니다. Bonsai Image는 같은 급의 최신 4B 모델들과 경쟁할 수 있는 품질을 유지하면서도, transformer 메모리는 훨씬 적게 사용합니다. 동시에 비슷한 메모리 예산을 가진 더 작은 모델들보다 훨씬 우수합니다. 이것이 이전 Bonsai 언어 모델에서 봤던 것과 같은 성능-크기 경계의 이동입니다. Bonsai Image는 현대적인 diffusion transformer 성능을 이전에는 훨씬 작고 약한 모델들만 차지했던 메모리 영역으로 가져왔습니다.

## 이미지 생성은 단순한 모델 품질 문제가 아니다

클라우드 API는 여전히 많은 제품에서 올바른 선택입니다. 하지만 클라우드 전용 생성에는 제약이 따릅니다. 매번 프롬프트마다 원격 요청이 필요하고, 반복할 때마다 서빙 비용이 들며, 모든 상호작용에 왕복 지연이 발생합니다.

이것이 중요한 이유는 이미지 생성이 본질적으로 반복적이기 때문입니다. 사용자들은 한 이미지로 멈추지 않거든요.
