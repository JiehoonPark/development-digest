## 핵심 개념

> [!abstract]
> "보안 난독화는 나쁘다"는 일반적인 주장에 대해, 난독화를 추가 방어층으로 사용하는 것은 실제로 효과적이라는 주장을 제시한다. 저자는 WordPress 테이블 프리픽스 변경과 CSGO 바이너리 분석 등 실제 사례를 통해 난독화의 가치를 설명한다.

## 상세 내용

- "난독화만으로는 안 되지만, 추가 방어층으로서의 난독화는 유효하다" - Kerckhoffs의 원칙을 기반으로 함
- WordPress DB 테이블명 변경, 바이너리 난독화 등으로 공격 비용을 증가시켜 공격자의 포기 가능성을 높임
- 방어 심층화 전략의 일부로서 난독화는 공격 시간과 비용을 증가시키는 실질적인 가치를 제공

> [!tip] 왜 중요한가
> 보안 설계 시 난독화를 적절한 추가층으로 활용함으로써 실제 공격 저항성을 높일 수 있다.

## 전문 번역

# 보안 대담 - "난독화는 나쁘다"는 말에 대하여

최근 웹 개발 커뮤니티에서 흥미로운 논쟁을 목격했습니다. 한 사용자(여기서는 Mini라고 부르겠습니다)가 자신의 웹사이트에서 실행되는 스크립트에 JavaScript 난독화를 적용할 가치가 있는지 물어봤거든요. 목표는 데이터 수집 봇이 API 요청을 역공학하거나 복제하기 어렵게 만드는 것이었습니다.

그때였습니다. Echo라는 다른 사용자의 댓글이 올라온 겁니다. 마치 4000줄이 넘는 코드 리뷰에 달린 외로운 LGTM 같은 댓글 말이에요.

> 보안 난독화는 나쁩니다

놀랍게도 이 댓글에 많은 추천이 달렸습니다. 아마도 그 표현을 한 번 들었을 뿐이고, 남들이 하니까 따라서 반복하는 사람들이었을 겁니다.

저는 Echo의 댓글에 답글을 달기로 했습니다.

## 명확히 해야 할 것들

**보안 난독화는 나쁘지 않습니다.**

**난독화만으로는 보안이 나쁩니다(Kerckhoffs 원칙).**

**추가 방어층으로서의 난독화는 좋습니다!**

처음엔 Echo가 이 정도 뉘앙스를 의도했을 거라 생각했어요. 그런데 대화를 나누다 보니 Echo는 모든 형태의 난독화가 무의미하며 절대 사용하면 안 된다고 주장하더라고요. 심지어 현대에는 AI 때문에 난독화를 우회하는 게 사소한 일이라는 논리였습니다.

이 글에서는 왜 Echo가 틀렸는지, 그리고 난독화가 보안 전략에서 어떤 역할을 할 수 있는지 설명해보겠습니다.

## 당신의 과정을 보여주지 마세요

보안 난독화는 애플리케이션의 내부 동작이나 구현 세부사항을 공격자로부터 덜 드러내는 방식으로 노출을 줄이는 기법입니다. 수학과는 달리, 웹 개발에서는 당신의 풀이 과정을 보여줄 필요가 없거든요.

현실의 비유로 생각해보면, 문 앞 현관매트 아래에 여분의 열쇠를 숨기는 것과 자물쇠에 꽂혀있는 열쇠를 그냥 두는 것의 차이 정도입니다. 악의적인 행동자는 현관매트 아래까지 찾기가 귀찮아서 그냥 가버릴 수도 있죠. 축하합니다, 난독화가 당신을 구했습니다.

물론 열쇠를 찾을 수도 있습니다. 하지만 그 전에 근처 화분이나 우편함을 먼저 뒤져봐야 할 겁니다. 이 모든 과정에는 시간이 필요하고, 시간은 곧 비용입니다. 공격자 입장에서는 헛고생하는 시간이 길수록 더 빨리 포기하고 다른 목표로 넘어갈 가능성이 높아집니다.

당연히 진정한 보안은 문 앞에 여분의 열쇠를 두지 않는 것입니다. 신뢰할 수 있는 가족이나 친구에게 맡기는 방식이 맞죠. 난독화만으로만 보안을 지키려는 건 나쁩니다. 당신의 애플리케이션을 그에 맞는 수준으로 먼저 보호한 다음, 난독화라는 추가 층을 얹어서 공격 난이도를 높이세요. 이게 바로 심층 방어(defense-in-depth) 전략의 일부인 겁니다.

## 실제 사례들

구체적인 사례가 이 개념을 더 잘 설명해줄 테니, 제가 직접 경험한 몇 가지 예시를 들어보겠습니다.

### WordPress 데이터베이스 테이블 접두사

WordPress 기본 데이터베이스 테이블 접두사를 임의의 값으로 변경하는 것은 오랫동안 권장되어온 보안 관행입니다. 예를 들어 `wp_users`를 `wp_8df7b8_users`로 바꾸는 식이죠. 이런 방법은 '난독화일 뿐이라서 쓸모없다'며 무시받곤 합니다.

저도 과거 이 블로그를 WordPress로 운영했었습니다. 2015년쯤 제가 사용하던 플러그인에 SQL 인젝션 취약점이 있었거든요. 그걸 악용하면 데이터베이스를 통째로 빼낼 수 있었어요. 공격자들의 봇이 웹을 돌아다니며 취약한 WordPress를 찾고 있던 시기였습니다.

제 웹사이트도 취약했습니다. 그런데 다행히 공격을 당하지 않았고, 며칠 뒤 패치된 버전으로 업데이트했어요. 다른 사이트들은 데이터가 날아가고 망가졌지만, 저는 무사했습니다.

나중에 GitHub에서 이 취약점을 활용하는 PoC 스크립트를 찾아 제 사이트에 실행해봤는데, "Table 'wordpress.wp_users' doesn't exist" 같은 일반적인 에러만 나왔어요. 제가 여전히 취약했을 수도 있고, 다른 SQL 쿼리로는 공격당했을 수도 있지만, 대부분의 사용자를 노리는 표준 쿼리로는 먹혀들지 않았던 겁니다.

결국 그 추가 난독화 층이 제 사이트를 지켜준 것 같아요. 요즘 AI 도구라면 계속 다른 쿼리를 시도할 수 있겠죠. 악의적 행동자 입장에서는 좋은 결과가 나올 수도 있고요. 하지만 토큰에 비용이 들잖아요. 봇이 더 많은 시간과 비용을 쓸수록, 포기하고 다른 타겟으로 가버릴 확률이 높아집니다. 결국 버티는 싸움인 셈입니다.

### CS:GO 디버그 심볼 유출 사건

저는 오스트레일리아와 뉴질랜드 지역의 CS:GO 커뮤니티 서버인 Invex Gaming을 몇 년간 운영했었습니다. 서버 운영자로서 저는 독특한 커스텀 모드를 추가해서 제 서버를 구별하고 싶었거든요. SourceMod라는 훌륭한 플랫폼을 사용했는데, 이걸 쓰면 커스텀 플러그인과 확장 프로그램을 직접 작성할 수 있었습니다.

유용한 모드를 만들려면 게임 바이너리 안의 함수를 직접 찾아서 호출해야 하는 경우가 많았어요. CS:GO는 engine.dll, client.dll, server.dll 같은 바이너리를 포함하고 있는데, 여기 들어있는 게임 로직을 우리가 활용하고 싶었던 거죠.

예를 들어 제가 특정 플레이어의 체력을 프로그래밍 방식으로 설정하고 싶다면, CS:GO 서버에서 실행되는 스크립트가 게임 바이너리의 올바른 함수를 호출해야 합니다. 이 경우 `CBaseEntity::SetHealth`가 그런 함수인데, 이걸 프로그래밍 방식으로 호출하면 어떤 엔티티의 체력이든 설정할 수 있었어요.

함수 시그니처는 이런 모양입니다.

```cpp
class CBaseEntity {
public:
    virtual void SetHealth(int health);
};
```

문제는 이런 함수들의 메모리 주소를 알아야 호출할 수 있다는 것입니다. 일반적으로 애플리케이션의 메모리 레이아웃은 매번 다르거든요(ASLR 기술 덕분에요). 하지만 Valve가 게임과 함께 디버그 심볼을 배포했었던 시절이 있었습니다. 이 심볼 파일에는 정확한 함수 주소가 모두 들어있었죠.

악의적인 행동자가 이런 심볼에 접근하면, 어떤 함수든 정확한 주소로 찾을 수 있었고, 기존 보안 메커니즘을 우회할 수 있었어요. 이것이 난독화의 중요성을 보여주는 완벽한 예시입니다. 심볼을 노출시키는 것 자체가 바로 보안의 허점이 되었으니까요.

나중에 Valve는 이런 심볼을 배포하지 않기로 결정했습니다. 배포하는 대신, 플레이어들이 게임을 즐기는 데 필요한 공개 API만 제공했어요. 이것이 난독화를 통한 보안의 좋은 사례입니다.

---

**결론적으로, 난독화는 완벽한 보안은 아니지만 공격 비용을 높이는 효과가 있습니다.** 온라인 보안 위협에 대항하는 종합적인 전략의 일부로서 그 역할을 충분히 하고 있습니다.
