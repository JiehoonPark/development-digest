## 핵심 개념

> [!abstract]
> TUI(Terminal User Interfaces)가 다시 주목받고 있는 이유를 분석한 글입니다. 저자는 Windows와 Linux의 GUI 전략 실패, Electron 앱의 한계, 그리고 네이티브 애플리케이션 개발의 복잡성 증가가 TUI의 부활로 이어지고 있다고 설명합니다.

## 상세 내용

- Windows는 수십 년간 MFC, COM, WinForms, WPF, MAUI 등 일관성 있는 GUI 전략 없이 계속 API를 변경해왔고, Linux는 설계 차원에서 GTK와 Qt의 UI 불일치를 초래했습니다.
- Electron 앱은 메모리 소비, 시각적 불일치, 키보드 기반 워크플로우 부족 등의 문제를 안고 있어 네이티브 경험을 제공하지 못합니다.
- 복잡한 GUI 프레임워크 대신 TUI는 즉각적인 피드백, 낮은 리소스 사용, 일관된 사용자 경험을 제공합니다.

> [!tip] 왜 중요한가
> 개발자들이 GUI 복잡성을 피하고 TUI 기반 도구 개발을 고려할 때 유용한 배경 지식을 제공하며, 특히 CLI 도구와 개발자 경험 설계에 영향을 미칩니다.

## 전문 번역

# 터미널 UI가 다시 떠오르는 이유

## TUI의 귀환

터미널 사용자 인터페이스(TUI)가 다시 주목받고 있습니다. DHH의 Omarchy를 보면 세 가지 UI 방식이 공존하는데요. 즉각적인 피드백과 개발자들의 선호도를 고려한 TUI, 37signals(DHH의 회사)가 SAAS 웹 애플리케이션을 판매하기 때문에 필요한 웹 앱, 그리고 배포판의 스타일과는 맞지 않는 네이티브 애플리케이션이 그것입니다.

약 10년 전 코드 에디터 시장에서도 비슷한 패턴이 나타났어요. BBEdit, Textmate(DHH가 추천했던), Notepad++, Sublime 같은 네이티브 에디터에서 Atom, VSCode 같은 Electron 기반 앱으로 넘어갔거든요. 그사이 하드코어 개발자들은 vim이나 emacs로 갈아탔는데, 가파른 학습곡선을 감수하면서까지 즉각적인 피드백과 높은 사용성을 포기한 셈입니다.

## Windows: 일관성 없는 API의 악순환

교훈은 명확합니다. 네이티브 애플리케이션이 점점 밀려나고 있다는 거죠. Windows의 GUI 라이브러리 현황을 보면 이게 얼마나 심각한지 알 수 있습니다.

하나의 API가 성공하지 못하면 또 다른 API를 만들어내고, 그것도 수많은 경쟁 기술에 묻혀버리는 악순환이 반복되었거든요. MFC(1992)는 Win32를 C++로 감쌌지만, Win32가 이미 투박했다면 MFC는 그걸 택시 위에 또 다른 택시를 올려놓은 꼴이었습니다. 그 다음엔 OLE, COM, ActiveX가 나왔는데, 이들은 사실 GUI 프레임워크가 아니라 컴포넌트 아키텍처였음에도 불구하고 Windows 개발의 곳곳에 스며들어 극도의 인지 복잡성을 만들어냈습니다.

Microsoft 전직 부사장 Jeffrey Snover의 말처럼, "Microsoft는 Petzold 이후로 일관된 GUI 전략을 갖지 못했습니다."

그 이후 Winforms, WPF, Silverlight, WinUI, MAUI를 거쳐왔지만 모두 실패했어요. 지금도 많은 엔터프라이즈와 개인 데스크톱 애플리케이션이 Electron 앱에 의존하고 있으며, OS 전체의 시각적 일관성을 본 기억이라곤 Windows 98이나 2000이 마지막입니다.

문제는 이게 엄청난 작업량이라는 데 있습니다. 몇 년마다 OS와 UI API를 처음부터 만들어야 하니까요. 게다가 간헐적으로 샌드박싱을 시도하고 "너무 강력한" 기능들을 deprecated 시키다 보니, 새로운 레이어마다 이전 프레임워크에서 가능하던 것을 할 수 없는 갭이 생겨나갑니다.

## Linux: 의도된 불일관성

Linux의 UI 불일관성은 처음부터 설계된 것입니다. 각기 다른 팀들이 서로 다른 목표를 추구할 자유가 있었으니까요. GTK와 Qt가 두 대세 프레임워크로 자리 잡았는데, Qt는 물론이고 둘 다 크로스플랫폼 네이티브 개발을 지원하려 했습니다. (과거에 저도 gedit을 Windows에서 컴파일하는 데 성공해본 적 있는데, C 컴파일, makefile, 환경 변수에 대해 정말 많이 배웠어요.) 다만 실제로는 Linux에서만 널리 쓰일 뿐입니다.

다행히 서로 다른 툴킷으로 만든 애플리케이션들도 나란히 있어도 어느 정도 봐줄 만한 모양을 유지하는데, Windows의 여러 프레임워크는 이 정도도 못 합니다. Control Panel을 다시 만드는 데 몇 명의 엔지니어가 몇 시간을 쏟아야 할까요?

수많은 배포판, 데스크톱 환경, 하드웨어 조합을 테스트하는 것이 거의 불가능하다 보니, 대부분의 회사는 Linux용 네이티브 앱을 만들지 않습니다. Electron으로 때우거나(물론 이건 문제가 많습니다), 아니면 오픈소스 커뮤니티에 맡겨버리죠(공개 API가 있을 때만).

## macOS: 제 집을 헤치는 격

Apple은 과거 일관성의 상징이었어요. Apple의 Human Interface Guidelines은 세계 모든 UI 강의에서 인용되었습니다. Xerox PARC과 함께 Apple은 좋은 휴먼 인터페이스가 무엇인지 연구한 두 기관 중 하나였거든요.

그런데 지금은 어떨까요? Apple은 자신이 알려진 지침과 일관성을 철저히 부수고 있습니다. Fitts' law를 무시해서 윈도우 리사이징이 거의 불가능해졌고(수정하려고 했음에도), 모든 메뉴에 아이콘을 집어넣었어요. macOS는 더 이상 디자이너들이 평화롭게 일할 수 있는 안식처가 아닙니다.

## Electron: 못 쓸 짓은 또 못 씀

Electron 앱의 사용자 경험이 형편없다는 건 다들 압니다. 가장 흔한 지적은 메모리 소비인데, 지난 10년간 나아지긴 했죠. 하지만 저 입장에선 (64GB RAM MacBook Pro를 쓰는 입장에서) 시각적 일관성 부족과 키보드 기반 워크플로우의 결핍이 더 큰 문제입니다.

제 dock을 보면 네이티브 앱 8개(Textmate과 macOS 시스템 유틸리티)와 Electron 앱 6개(Slack, Discord, Mattermost, VSCode, Cursor, Plex)가 섞여 있어요. Electron 앱을 되도록 피하려는 사람으로서도 말이죠.

Cursor를 예로 들어볼까요(VSCode도 마찬가지겠지만). Agent 패널에서 다음 기능을 요청하고 있다면, 키보드만으로 옆의 agent 리스트로 옮길 수 있을까요? 아카이브할 수 있을까요? 이런 동작은 모든 macOS 애플리케이션에서 동일해야 하는데, 설령 단축키가 있어도 메뉴에 표시되지 않습니다.

지난 10년간 개발자들은 메뉴에 기능을 추가하는 걸 자꾸 깜빡했어요. 애플리케이션이 샌드박스 안의 HTML일 뿐이라는 이유로요. 공정하게 말하자면 Slack이 이 부분을 더 잘하는 편이지만, 여전히 완벽하진 않습니다.
