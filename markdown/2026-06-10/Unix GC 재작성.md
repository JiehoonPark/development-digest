## 핵심 개념

> [!abstract]
> AF_UNIX 가비지 컬렉터는 SCM_RIGHTS로 전송된 소켓이 사용자 공간에서 도달 불가능하면서도 커널에 의해 유지되는 상황을 처리합니다. 최근 그래프/강연결요소 모델 기반으로 재작성되었으며, Tarjan 알고리즘을 사용해 순환 참조를 감지하고 수집 가능한 객체를 효율적으로 식별합니다.

## 상세 내용

- inflight 카운터를 통해 SCM_RIGHTS 페이로드로 전송 중인 소켓을 추적하며, file_count == inflight일 때 사용자 공간에서 도달 불가능한 소켓을 식별합니다.
- 새로운 GC는 각 inflight 소켓을 정점으로, struct file 포인터를 간선으로 하는 그래프를 구성한 후 Tarjan 알고리즘으로 강연결요소를 분할합니다.
- SCC가 변경되지 않으면 빠른 경로를 사용해 캐시된 SCC를 재사용하므로 O(|V| + |E|) 복잡도로 효율적입니다.

> [!tip] 왜 중요한가
> 커널 메모리 누수를 방지하고 소켓 통신의 안정성을 보장하는 저수준 메커니즘을 이해하는 데 필요합니다.

## 전문 번역

# AF_UNIX 가비지 컬렉터와 그 재구성: 커널의 숨은 버그

AF_UNIX 가비지 컬렉터는 커널의 흥미로운 부분입니다. 소켓이 SCM_RIGHTS를 통해 전송될 수 있는데, 사용자 공간에서는 더 이상 접근할 수 없으면서도 커널에 의해 살아있는 상태로 유지될 수 있거든요. 이는 메모리 낭비이므로 가비지 컬렉터가 이런 소켓들을 정리해줍니다. 최근에 이 서브시스템이 그래프와 강연결 요소(Strongly-Connected-Components) 모델 위에서 완전히 재작성되었는데, 아직도 버그가 많이 발견되고 있습니다.

이 글에서는 그 재구성 과정을 처음부터 끝까지 살펴보고, Use-After-Free 버그에 대해 논의해보겠습니다.

## 기본 개념: 왜 가비지 컬렉션이 필요한가?

각 서브시스템의 가비지 컬렉터는 더 이상 사용자 공간 핸들을 통해 접근할 수 없는 커널 객체를 회수합니다. AF_UNIX의 경우 진입점은 `unix_gc()` 함수입니다.

가비지 컬렉션에서 핵심이 되는 필드는 `inflight`입니다. 소켓이 "in flight" 상태라는 것은 그것의 `struct file *`가 SCM_RIGHTS 페이로드로 실려 있다는 뜻이거든요. 프로세스 A가 보냈지만 프로세스 B가 아직 받지 않은 상태죠. 전송될 때마다 `inflight` 값이 증가하고, 수신될 때마다 감소합니다.

GC는 `file_count == inflight`인 소켓을 찾고 있습니다. 즉, 유일한 참조가 다른 소켓의 수신 큐에 갇혀있고, 사용자 공간에서는 절대 접근할 수 없는 상태인 소켓이 대상입니다.

## 구체적인 예시

A와 B라는 AF_UNIX 소켓이 있다고 가정해봅시다. A의 파일 디스크립터를 B로 보내고, B의 파일 디스크립터를 A로 보낸 후 두 소켓을 모두 닫는다고 하겠습니다.

처음에 각 소켓의 `struct file`은 1개의 참조를 가집니다. 파일 디스크립터를 교환한 후에는 각각 2개로 올라갑니다. 그 다음 `close()`를 호출하면 다시 1개씩 감소해 1이 됩니다.

이 시점부터 아무도 파일이나 소켓에 접근할 수 없습니다. 하지만 `struct file`은 여전히 참조 카운트가 1이므로 AF_UNIX 소켓의 `release()` 함수가 호출되지 않습니다. 바로 이 상황이 가비지 컬렉션이 필요한 이유입니다. 커널은 모든 in-flight AF_UNIX 소켓을 추적하고 주기적으로 가비지 컬렉션을 실행해야 합니다.

## 2024년 이전의 구현

커널은 전역 `unix_tot_inflight` 카운터를 유지하는데, 매번 in-flight 상태로 전환될 때마다 증가하고 받아질 때마다 감소합니다. (단, `UNIX_INFLIGHT_TRIGGER_GC == 16000`일 때 GC가 트리거됩니다.)

이전 수집기는 in-flight 그래프를 탐색하고 사이클을 표시한 다음 `inflight != refcount`를 확인하여 각 사이클이 수집 가능한지 판단했습니다. 간단히 말해, 각 in-flight 소켓의 수신 큐를 잠그고 다른 곳에서 여러 가지 복잡한 처리를 해야 했습니다.

## 재설계된 구현

새로운 GC는 각 소켓의 큐를 잠그지 않으면서도 작동합니다. 또한 순환 참조가 없거나 in-flight 파일 디스크립터 그래프의 형태가 변하지 않았다면 경량 방식으로 동작하려고 설계되었습니다.

### 그래프 표현

각 in-flight 소켓은 정점(vertex)이 되고, SCM_RIGHTS cmsg에 담긴 각 `struct file *`는 방향성 간선(directed edge)이 됩니다. 선행 소켓에서 후속 소켓으로의 방향을 가집니다.

예를 들어 A를 C로 보내고, C를 D로 보내고, B를 D로 보낸다면? 세 개의 in-flight 소켓(A, B, C — D는 아닙니다)이 있고, 그래프는 이렇게 형성됩니다:

```
A → C → D
B ↗
```

### Tarjan 알고리즘과 강연결 요소

이 그래프를 Tarjan 알고리즘으로 강연결 요소(SCC)로 분할합니다. 왜 SCC일까요? 세 개 이상의 정점으로 이루어진 SCC에는 반드시 최소 하나의 사이클이 존재하기 때문입니다.

사이클이 존재하는 것은 정점이 수집 가능하기 위한 필요조건이지만 충분조건은 아닙니다. 정점이 in-flight 상태이면서 사용자 공간에서 접근 불가능해야 합니다(`file_count == out_degree`). 사이클에 포함되지 않은 소켓은 상호 참조 가비지가 될 수 없으므로 건너뜁니다.

`unix_graph_maybe_cyclic` 플래그는 양쪽 끝이 모두 in-flight인 새로운 간선이 추가될 때마다 토글됩니다:

```c
if (both_endpoints_inflight) {
    unix_graph_maybe_cyclic = true;
}
```

또한 `unix_update_graph()`는 `unix_graph_grouped = false`를 리셋하는데, 이는 다음 GC 실행 시 처음부터 SCC를 다시 빌드하도록 강제합니다.

### 빠른 경로와 느린 경로

```c
if (unix_graph_grouped && !unix_graph_maybe_cyclic) {
    // 빠른 경로: 캐시된 SCC 재사용
} else {
    // 느린 경로: Tarjan 알고리즘 실행
}
```

### SCC 구축

느린 경로에서는 실제로 SCC가 구축됩니다. 인덱싱은 `UNIX_VERTEX_INDEX_START == 2`부터 시작합니다. 그래프는 처음에는 비순환이라고 가정하고, 실제로 사이클을 발견할 때만 순환으로 표시됩니다.

복잡도 측면에서 보면, 바깥쪽 while 루프는 그래프가 연결되지 않은 부분 그래프들의 집합일 때만 한 번 이상 반복됩니다. 약하게 연결된 모든 그래프 G(V, E)에 대해 한 번의 반복으로 모든 정점을 방문합니다. 전체 계산량은 O(|V| + |E|)입니다.

### Tarjan 알고리즘의 동작

Tarjan 알고리즘은 방향 그래프를 받아서 그 SCC를 생성합니다. 각 정점은 정확히 하나의 SCC에 속하며, 들어오거나 나가는 사이클이 없는 정점은 자기 자신만 포함하는 단순 SCC를 형성합니다.

알고리즘은 깊이 우선 탐색(DFS)으로 동작합니다. 모든 정점이 처음에는 `(index, scc_index) = (k, k)`로 레이블이 지어지고, 여기서 k는 단조증가합니다. 그러면서 이웃 정점들의 `scc_index` 값이 스택을 거슬러 전파되어, 같은 사이클 내의 모든 정점이 그 사이클의 최소 `scc_index`로 수렴하게 됩니다.

### 빠른 경로: 캐시된 SCC 재사용

그래프 형태가 마지막 GC 이후 변하지 않았다면(`unix_graph_grouped == true`), SCC는 그대로 재사용됩니다.

빠른 경로는 캐시된 각 SCC를 역순으로 순회하고, 각 정점을 방문 목록으로 이동시킨 다음, `unix_vertex_dead()`를 실행합니다. SCC의 모든 정점이 검사를 통과하면, 전체 SCC가 회수 목록에 추가됩니다.

이 재설계된 가비지 컬렉터는 커널의 메모리 관리를 훨씬 더 효율적으로 만들었지만, 여전히 복잡한 시스템인 만큼 주의 깊은 관찰과 테스트가 필요합니다.
