## 핵심 개념

> [!abstract]
> 공공 서비스 성격의 유틸리티 회사 웹사이트를 Astro를 사용한 HTML 중심 아키텍처로 재구축하여 사용자 만족도를 획기적으로 개선한 사례입니다. 이전의 React 기반 앱은 접근성 문제, 로컬스토리지 제한(5MB)으로 인한 이미지 업로드 실패 등으로 3일 만에 폐기되었습니다. 새로운 버전은 JavaScript 없이도 완벽하게 작동하는 HTML 기반 폼으로 설계되었으며, Web Components를 통한 점진적 개선만 추가했습니다. 이 접근 방식은 오래되고 성능이 낮은 브라우저, 불안정한 연결 환경에서도 데이터 손실 없이 작동하도록 했습니다.

## 상세 내용

- 이전 React 앱의 실패 원인: 로딩 스피너 남발, 전역 JavaScript 상태 관리 복잡성, 접근성 미충족, 로컬스토리지 5MB 제한으로 인한 이미지 업로드 불가능. 규제 대상 독점기업이므로 고객 만족도가 96% 이하로 떨어지면 수백만 파운드의 벌금이 부과될 수 있었습니다.
- HTML 중심 설계 철학: 공공 서비스는 모든 기기에서 작동해야 하고, 연결이 좋지 않은 환경에서도 작동해야 하며, 입력된 데이터는 절대 손실되면 안 된다는 원칙을 바탕으로 설계했습니다. Terence Eden의 주택 급여 혜택 사례(PSP 게임기의 형편없는 브라우저에서도 GOV.UK 페이지가 작동)를 인용하며 단순 HTML의 중요성을 강조했습니다.
- 기술적 구현 방식: 각 폼 단계가 독립적인 페이지로 구성되며, 사용자가 다음을 클릭하면 폼이 제출되고 API가 유효성을 검증한 후 다음 단계로 리다이렉트됩니다. 이는 Remix가 부활시킨 고전적 웹 애플리케이션 패턴으로, 클라이언트 측 JavaScript 의존성을 최소화합니다.
- 데이터 처리 전략: 각 폼 세션에 고유 ID를 할당하고, 폼 마법사의 매 단계에서 제출된 데이터(이미지 업로드 포함)를 백엔드에 저장하므로 JavaScript 없이도 폼 완성이 가능합니다. 오래되고 성능이 낮은 브라우저에서도 완성할 수 있도록 설계되었습니다.
- 접근성 및 검증 기준: WCAG AA 수준의 접근성을 충족하도록 구현했습니다. React 유효성 검사 라이브러리에 드는 비용(숙련 부족, 수개월의 유지보수 시간)을 지적하며, 모든 브라우저에 내장된 HTML 검증 시스템을 활용하는 것이 더 효율적임을 강조했습니다.
- Web Components를 통한 점진적 개선: HTML 폼을 감싸는 커스텀 엘리먼트인 Web Component를 제작했으며, Shadow DOM이나 JavaScript에서의 HTML 렌더링 없이 기존 HTML 검증을 활용하여 폼을 기능 강화했습니다. 이는 JavaScript가 없어도 작동하면서 있을 때 사용자 경험을 향상시키는 구조입니다.
- 성능 및 사용자 환경 고려: 20MB의 JavaScript를 전송하기 전에 폼을 렌더링하는 것은 비합리적이며, 특히 10년 된 저가 안드로이드 폰을 사용하는 사용자나 신규 주택 부지의 들판에서 신호가 약한 지역의 사용자를 고려했습니다. 클라이언트 측 애플리케이션 자체는 반대하지 않지만, 단순 폼에는 과도하다고 판단했습니다.

> [!tip] 왜 중요한가
> 공공 서비스나 핵심 비즈니스 기능을 제공하는 웹사이트는 JavaScript 의존성을 최소화하고 HTML 기반으로 설계해야 한다는 실증적 증거를 제시합니다. 이는 접근성, 성능, 신뢰성 측면에서 모던 개발 트렌드(대규모 SPA)에 대한 중요한 반론을 제기합니다.

## 전문 번역

# HTML 우선 설계로 하루 만에 사용자를 2배로 늘린 이야기

2026년 6월 10일

한 공기업의 서비스 신청 폼이 너무 불편해서 사용자가 크게 줄어들고 있었어요. 고객들은 낡은 ASP 폼을 사용하거나 번거로운 수동 절차를 따라야 했거든요. 당연히 회사 입장에서는 수동 절차가 훨씬 비용이 많이 들었고, 여기에 고객 만족도가 96% 이하로 떨어지면 수백만 파운드의 벌금을 물 수 있다는 규제까지 있었습니다.

이미 두 번이나 실패한 프로젝트가 있었는데, 가장 최근 시도는 해외 계약사가 React 앱을 개발한 것이었어요. 그런데 온라인 3일 만에 사용자 불만으로 인해 내려가야 했습니다. 코드를 보니 로딩 스피너가 난무하고, 전역 상태 관리가 엉망이었어요. 접근성도 전혀 고려되지 않았습니다. 게다가 폼의 핵심인 이미지 업로드 기능은 로컬스토리지(5MB 제한)에 모든 데이터를 저장하려고 했으니까요.

## 과감한 결정: HTML 우선 접근

저는 대담하게 결정을 내렸습니다. Astro를 사용해 HTML 우선 방식으로 사이트를 새로 만들기로 한 거예요. JavaScript는 웹 컴포넌트 형태로만 존재했고, 오직 HTML만으로도 완벽하게 작동하는 사이트를 점진적으로 개선하는 데만 사용했습니다.

제 논리는 간단했습니다.

- 이건 공공서비스다
- 모든 기기에서 작동해야 한다
- 연결 상태가 좋지 않아도 작동해야 한다
- 폼에 입력된 데이터는 절대 손실되면 안 된다

Terence Eden의 일화가 제 생각을 강하게 했어요.

> 몇 년 전 런던의 주택보조금 사무소에서 정책 연구를 하고 있었습니다. 그곳은 정말 삭막한 곳이었어요. 벽에는 가정폭력 피해자를 위한 도움말 포스터가 붙어 있고, 경비원들은 무심하게 사람들을 지나쳐보냅니다. 공기는 긴장감으로 가득하고, 아이들의 울음소리가 울려 퍼집니다.
>
> 한 젊은 여성이 딱딱한 플라스틱 의자에 앉아 있었어요. 그녀의 전 재산이 들어있는 캔버스 가방으로 둘러싸여 있었습니다. 손에 들려 있던 게임 콘솔(PSP)에 시선을 고정한 채로요.
>
> 처음엔 Candy Crush를 하고 있다고 생각했어요. 그런데 뒤에 서서 화면을 봤더니 달랐습니다. 공용 WiFi에 연결해서 주택보조금에 대한 GOV.UK 페이지를 보고 있던 거예요.
>
> PSP의 웹 브라우저는 형편없었습니다. 느렸고, 자주 메모리가 부족했고, 한 번에 3개 탭만 열 수 있었어요.
>
> 그런데 GOV.UK 페이지는 단순한 HTML로 만들어져 있었어요. 가볍게 설계되었고, 형편없는 브라우저에서도 작동했습니다. 왜냐하면 이건 모두를 위한 서비스이기 때문입니다.

## 요구사항 정의

이 경험으로부터 저는 몇 가지 요구사항을 도출했습니다.

- 각 폼 세션마다 고유 ID를 부여한다
- 폼 위저드의 모든 단계에서 제출된 데이터(업로드 포함)를 백엔드에 저장한다
- JavaScript 없이도 폼을 완성할 수 있어야 한다
- 구형 브라우저와 성능이 낮은 브라우저에서도 작동해야 한다
- WCAG AA 레벨 접근성을 만족한다
- JavaScript와 모던 CSS로 사용자 경험을 개선한다

## 설계 방식

폼 위저드의 각 단계를 별도의 페이지로 만들었어요. 사용자가 '다음' 버튼을 클릭하면 폼이 제출되고, API가 데이터 유효성을 확인한 후 다음 단계로 리다이렉트하는 방식입니다.

Remix 덕분에 다시 주목받고 있는 전통적인 웹 애플리케이션 패턴이죠. 동료들을 설득하는 데는 시간이 걸렸어요. 모두가 클라이언트 중심 웹 애플리케이션에 익숙해져 있었거든요. 저는 클라이언트 중심 앱을 반대하지 않습니다. 하지만 이건 단순한 폼일 뿐입니다. 실시간 데이터를 표시하는 것도 아니고요. 사용자가 10년 된 안드로이드 폰을 들고 시골에서 폼을 채워야 할 수도 있어요. 폼을 렌더링하기도 전에 20MB의 JavaScript를 보내는 건 말도 안 되지 않을까요?

## 폼 검증 문제 해결

제가 가장 싫어하는 것 중 하나가 폼 검증이에요. 제가 봐온 팀들은 React 검증 라이브러리 때문에 수개월을 낭비했습니다. 사실 모든 브라우저에는 이미 훌륭한 검증 시스템이 내장되어 있는데도 말이에요.

저는 HTML 웹 컴포넌트를 만들었습니다. 기존 HTML을 감싸서 기능을 추가하는 간단한 요소들이죠. Shadow DOM도 없고, JavaScript에서 HTML을 렌더링하지도 않습니다. 제 컴포넌트는 모든 HTML 폼을 감싸서 기본 검증을 활용하고, 사용자 경험을 개선했어요.
