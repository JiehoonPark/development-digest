## 핵심 개념

> [!abstract]
> Rust로 작성된 결정적 유전체학 엔진으로, 100MB 미만의 메모리로 전체 게놈 정렬, 변이 호출, 커스텀 바이오인포매틱스 분석을 수행합니다. O(√t) 메모리 복잡도로 병원, 클리닉, 현장 등 제약된 환경에서 작동합니다.

## 상세 내용

- O(√t) 메모리 복잡도를 통해 50-100GB 메모리가 필요한 기존 도구와 달리 100MB 이하로 작동합니다.
- 결정적 재실행, 스트리밍 표준 포맷(SAM/BAM/VCF) 출력, Python 바인딩을 통한 확장성을 제공합니다.

> [!tip] 왜 중요한가
> 병원 임상 진단, 감염병 현장 대응, 교육용 게놈 분석 등 자원이 제한된 환경에서 전문적 수준의 유전체학 분석을 가능하게 합니다.

## 전문 번역

# Rosalind: 초저메모리 유전체 분석 엔진

메모리 사용량을 극도로 낮춰서 전체 게놈 분석을 100MB 정도의 RAM으로 처리할 수 있는 Rust 기반 엔진입니다.

Rosalind는 게놈 정렬, 실시간 변이 호출, 맞춤형 생물정보학 분석을 수행하는 도구인데요. 일반 PC나 엣지 기기에서 동작합니다. 이 도구의 핵심은 O(√t)의 작업 메모리, 결정론적 재실행, 그리고 새로운 파이프라인을 유연하게 추가할 수 있다는 점입니다. Rust 플러그인이나 Python 바인딩으로 기능을 확장할 수 있거든요.

기존 파이프라인들은 보통 50~100GB 이상의 메모리, 잘 구축된 데이터센터, 안정적인 네트워크 연결을 가정합니다. 반면 Rosalind는 정반대를 목표로 설계됐어요. 병원 워크스테이션, 클리닉 노트북, 현장 키트, 강의실 같은 환경을 염두에 두고 만들었습니다.

## 핵심 문제와 해결책

**문제**

표준 도구들(BWA, GATK)이나 클라우드 중심 워크플로우는 50GB 이상의 메모리, 중간 결과물의 완전한 복사본, 고용량 스토리지를 필요로 합니다. 많은 병원, 공중보건 실험실, 교육기관에서는 이런 환경을 마련하기 어렵습니다.

**Rosalind의 접근법**

작업을 √t개 블록으로 나누고, 블록 간 경계선을 재활용하며, 높이를 압축한 트리 구조로 평가하는 방식입니다. 이렇게 하면 메모리는 L1/L2 캐시 범위 안에 머물면서도 결정론적인 결과를 보장합니다. 전체 파이프라인이 게놈 데이터라도 100MB 미만으로 들어갑니다.

## 어떻게 사용하나요?

CLI로 직접 실행하거나, Rust API를 코드에 임베드할 수 있습니다. 플러그인이나 Python 바인딩을 통해 맞춤형 생물정보학 워크플로우를 구축하는 것도 가능해요. 빠른 임상 진단, 질병 확산 모니터링, 학생들이 실제 데이터를 노트북에서 탐색할 수 있는 강의 같은 곳에 딱 맞습니다.

## 주요 특징

- **O(√t) 작업 메모리**: 게놈 전체를 ~100MB 이하로 처리, 정보 손실 없음
- **엔드-투-엔드 결정론적 실행**: 실행할 때마다, 데이터 분할 방식이 달라도 비트 단위로 동일한 결과
- **전체 이력 동등성**: 재계산을 통해 무제한 메모리 환경과 같은 결과 보장
- **스트리밍 출력**: SAM/BAM/VCF 표준 형식으로 거대한 중간 파일 없이 출력
- **엣지 환경 대응**: 8~16GB 노트북/데스크톱에서 실행, 민감한 환자 정보가 외부로 나가지 않음
- **확장 가능한 설계**: 플러그인과 Python 바인딩이 같은 메모리, 결정론적 보장을 상속받음

## 왜 Rosalind가 필요한가

**병원의 임상 게놈 분석**

대형 연구기관 밖의 많은 병원들은 8~16GB RAM의 공유 데스크톱이 가장 좋은 장비입니다. Rosalind를 쓰면 임상의들이 클라우드를 빌리거나 민감 데이터를 외부로 보내지 않고도 한 번의 근무 교대로 환자 게놈을 정렬하고 변이를 찾을 수 있습니다.

**현장 질병 대응**

에볼라, 지카, COVID-19 대응 현장에서는 휴대용 시퀀싱 장비와 노트북은 있지만, 고메모리 서버는 없는 경우가 많습니다. Rosalind는 읽기 데이터를 스트리밍으로 처리하고, 변이를 실시간 호출하며, 간헐적 연결을 견딜 수 있어서 현장에서 바로 의사결정을 내릴 수 있게 해줍니다.

**규모 있는 연구**

대학이나 소규모 실험실은 중간 정도 규모 클러스터는 있지만, 수천 개 표본의 코호트를 처리해야 합니다. Rosalind는 표본당 메모리를 일정하게 유지하므로 비용 효율적인 인스턴스에서 더 많은 게놈을 병렬 처리할 수 있습니다.

**교육과 보급**

학생들이 정렬, 변이 호출, 플러그인 개발을 개인 컴퓨터에서 직접 실험할 수 있어요. 계산 생물정보학 강좌가 훨씬 더 실습 중심이고 공평해집니다.

**맞춤형 분석**

개발자들이 범위 대시보드, 품질 지표, 단일세포 스타일 집계 같은 기능을 추가할 때도 동일한 공간 보장을 활용합니다. 프로토타입에서 프로덕션까지의 과정이 단순해집니다.

## 핵심 보장사항

**메모리 경계**: `space_used ≤ block_size + num_blocks + O(log num_blocks) = O(√t)`. 가장 최근의 블록 경계만 메모리에 유지됩니다.

**결정론적 재실행**: 모든 블록은 이전 경계에서 다시 시뮬레이션되어, 자원이 제한된 기기에서도 전체 이력과 같은 결과를 생성합니다.

**구성 가능한 설계**: 블록 프로세서, 플러그인, 바인딩이 모두 같은 압축 평가기를 사용하므로 새로운 분석들이 자동으로 보장사항을 상속받습니다. 맞춤형 품질 관리나 역학 대시보드에 완벽합니다.

**안전장치 포함**: 회귀 테스트(`tests/space_bounds.rs`)와 스크립트(`scripts/run_scale_test.sh`)가 O(√t) 또는 부선형 스케일링 특성이 저하되면 실패합니다.

**분할 불변성**: 블록 크기와 청킹 방식을 바꿔도 출력은 동일하며, 병합은 결정론적이고 순서 독립적입니다.

**전체 이력 동등성**: 결과는 무제한 이력 평가와 일치합니다. 공간 절감은 재계산에서 나오며, 정보 손실이 아닙니다.

## O(√t) 메모리가 의미하는 바

**t란?**

처리된 전체 염기쌍의 개수입니다. 30× 인간 전체 게놈 시퀀싱의 경우, 커버리지 C ≈ 30, 게놈 크기 G ≈ 3.1×10⁹이므로 t ≈ C × G ≈ 9.3×10¹⁰입니다.

**√t는?**

약 3.0×10⁵인데, 이게 블록 버퍼 크기를 정합니다. 높이 압축 병합 스택은 log₂(t) ≈ 36 레벨 정도인데, 블록에 비하면 무시할 수 있을 정도입니다.

**실제 작업 세트**

대략 (α + β) · √t + γ입니다. α ≈ 64~128 B (활성 위치당), β는 메타데이터입니다. 이렇게 계산하면 게놈 전체 실행이 30~80MB 정도이며, 보수적인 가정 하에서도 100MB 미만으로 유지됩니다.

이 경계가 성립하는 이유는 현재 블록, 롤링 경계, 컴팩트 병합 스택만 메모리에 있고, 오래된 상태는 필요할 때 다시 계산하기 때문입니다.

FM-인덱스와 참조 게놈은 메모리 맵 방식으로 로드되므로, O(√t) 주장은 Rosalind의 실제 작업 메모리에만 해당합니다.
