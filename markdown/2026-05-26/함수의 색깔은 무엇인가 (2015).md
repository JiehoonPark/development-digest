> [!abstract]
> 프로그래밍 언어 설계의 문제점을 비유적으로 설명한 글로, 함수에 색깔을 부여하고 호출 방식을 제한하는 가상의 언어를 통해 async/await, 언어 설계의 일관성 문제를 논의합니다.

## 상세 내용

- 단순한 기능 추가가 언어 전체의 복잡도를 기하급수적으로 증가시킬 수 있습니다.
- async/await 같은 기능이 코드에 색채(색상) 구분을 강제하면 상위 호출자까지 제약이 전파됩니다.

> [!tip] 왜 중요한가
> 언어 또는 라이브러리 설계 시 기능 추가의 파급 효과를 고려해야 하며, async/await 같은 기능의 인지적 부담을 이해할 수 있습니다.

## 전문 번역

# 함수의 색깔은 무엇인가

아침에 일어나서 프로그래밍 언어에 대한 신랄한 비판만큼 기분을 좋게 만드는 게 없는데요. 평범한 사람들이 쓰는 "형편한" 언어를 비웃으면서 스택오버플로우를 들락거리며 하루를 지나가는 모습을 보면 정말 혈기가 돋거든요.

물론 그런 글을 쓰다 보면 위험이 따릅니다. 제가 놀린 언어가 바로 당신이 좋아하는 언어일 수도 있다는 거죠. 모르는 사이에 이 블로그를 보는 독자들이 화염병을 들고 몰려올 수도 있는 겁니다!

그래서 저는 좀 다르게 접근해보겠습니다. 아예 제가 만들어낸 언어에 대해 비판하는 거죠. 속 다 채워질 제 동료상(?)을 만들어서 마음껏 깎아내려고 합니다.

지금은 뭔가 허망하게 느껴질 수 있겠지만, 끝까지 읽으면 우리 누구의 얼굴이 그 밀짚 인형에 칠해지게 될지 알 수 있을 겁니다.

## 새로운 언어

블로그 글 하나 때문에 완전히 새로운 언어를 배우기는 너무 많은 수고가 들어요. 그래서 우리가 이미 아는 언어와 비슷하다고 가정해봅시다. JavaScript처럼 중괄호와 세미콜론, if, while 같은 문법들을 갖춘 언어 말이에요.

```javascript
function thisIsAFunction() {
  return "It's awesome";
}
```

현대적인 언어답게 함수를 일급 객체로 다룰 수 있습니다. 이런 식으로 작성할 수 있다는 뜻이죠:

```javascript
// collection에서 predicate와 일치하는 모든 요소를 담은 배열을 반환합니다.
function filter(collection, predicate) {
  var result = [];
  for (var i = 0; i < collection.length; i++) {
    if (predicate(collection[i])) result.push(collection[i]);
  }
  return result;
}
```

이렇게 다른 함수를 받거나 반환하는 고차 함수(higher-order function)는 정말 유용하거든요. 보통은 컬렉션 다루기에 많이 사용하지만, 개념을 이해하고 나면 프로그램 곳곳에서 쓰게 됩니다.

테스트 프레임워크:

```javascript
describe("An apple", function() {
  it("ain't no orange", function() {
    expect("Apple").not.toBe("Orange");
  });
});
```

데이터 파싱:

```javascript
tokens.match(Token.LEFT_BRACKET, function(token) {
  // 리스트 리터럴을 파싱합니다...
  tokens.consume(Token.RIGHT_BRACKET);
});
```

이렇게 함수를 이리저리 넘기고, 호출하고, 반환하는 걸 반복하면서 재사용 가능한 라이브러리와 애플리케이션을 만들어냅니다. 정말 강력하죠.

## 함수의 색깔은 무엇인가?

그런데 여기서 우리의 언어가 이상해집니다. 이 언어에는 특이한 특징이 하나 있거든요:

### 1. 모든 함수는 색깔을 가진다

모든 함수(익명 콜백이든 이름이 있는 함수든)는 빨강 또는 파랑이어야 합니다. 단 하나의 `function` 키워드 대신 두 개가 있는 거죠:

```javascript
blue_function doSomethingAzure() {
  // 파란 함수입니다...
}

red_function doSomethingCarnelian() {
  // 빨간 함수입니다...
}
```

색깔 없는 함수는 이 언어에 존재하지 않습니다. 함수를 만들려면 반드시 색을 정해야 합니다.

### 2. 함수를 호출하는 방식도 색깔에 따라 달라진다

"파란 호출" 문법과 "빨간 호출" 문법이 따로 있다고 생각해보세요:

```javascript
doSomethingAzure()blue;
doSomethingCarnelian()red;
```

함수를 호출할 때는 반드시 그 함수의 색깔에 맞춰서 호출해야 합니다. 만약 실수해서 빨간 함수를 파란 호출로 부르거나 그 반대로 하면? 최악의 상황이 벌어집니다.

### 3. 빨간 함수는 빨간 함수 안에서만 호출할 수 있다

파란 함수는 빨간 함수에서 호출할 수 있습니다. 이건 문제없습니다:

```javascript
red_function doSomethingCarnelian() {
  doSomethingAzure()blue;
}
```

하지만 반대 방향은 안 되죠. 파란 함수에서 빨간 함수를 호출하려고 하면:

```javascript
blue_function doSomethingAzure() {
  doSomethingCarnelian()red;  // 이건 작동하지 않습니다
}
```

이제 앞의 `filter()` 같은 고차 함수를 만들기가 복잡해집니다. 함수의 색깔을 정해야 하는데, 그것이 이 함수에 전달할 수 있는 다른 함수들의 색깔에 영향을 미치기 때문이죠.

당연히 `filter()`를 빨간 함수로 만들면 파란 함수든 빨간 함수든 모두 받아서 호출할 수 있습니다. 그런데 여기서 또 다른 문제가 나타납니다:

### 4. 빨간 함수는 호출하기가 더 번거롭다

정확히 어떻게 번거로운지는 차치하고, 프로그래머가 빨간 함수를 호출할 때마다 짜증나는 추가 작업을 해야 한다고 생각해봅시다. 코드가 길어질 수도 있고, 특정 문맥에서는 호출할 수 없을 수도 있을 겁니다.
