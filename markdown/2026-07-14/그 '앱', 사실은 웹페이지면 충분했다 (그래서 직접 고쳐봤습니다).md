## 핵심 개념

> [!abstract]
> 여행사가 일정표와 숙박 정보를 확인하려면 무거운 'Travelbound' 앱을 설치하라고 요구하자, 저자는 이 앱이 사실상 텍스트·이미지·PDF 링크를 보여주는 것에 불과하다는 점에 분노해 직접 리버스 엔지니어링에 나섰습니다. Android 가상 기기 루팅과 HTTP Toolkit을 이용한 트래픽 분석으로 앱의 API와 JSON 데이터 구조를 파악한 뒤, 동일한 정보를 제공하는 0.05MB짜리 웹페이지를 만들어 43MB(최대 124MB) 앱을 대체했습니다. 이 과정에서 앱이 추가로 제공하던 것은 사용자 트래킹과 광고뿐이었다는 사실이 드러났습니다.

## 아티클

여름철 자녀의 공연 예술학교 일정으로 디즈니랜드 여행을 앞둔 한 개발자가, 여행사에서 일정표와 숙박 정보를 확인하려면 "Travelbound"라는 앱을 설치하라는 안내를 받았습니다. 문제는 이 앱이 사실상 텍스트, 이미지, PDF 링크로만 구성된 콘텐츠를 보여주는 게 전부라는 점이었는데요. 그저 단순한 정보 전달을 위해 무거운 앱을 강요받는 상황에 화가 난 저자는, 앱의 네트워크 트래픽을 직접 리버스 엔지니어링해 동일한 정보를 제공하는 훨씬 가벼운 웹페이지를 만들어버렸습니다. 이 글은 그 과정을 상세히 기록한 것입니다.

## 왜 굳이 "앱"이어야 하는가

여행 일정, 이동 수단, 숙박 정보를 확인하려면 "Travelbound" 앱을 설치해야 한다는 말을 들은 저자는 강한 거부감을 느꼈습니다. 이 앱이 하는 일이라고는 텍스트와 이미지, 그리고 PDF 파일로 연결되는 링크를 보여주는 것뿐이었기 때문입니다. 웹으로 충분히 전달 가능한 문서를, 왜 굳이 무겁고 번거로운 앱으로 강제하는 걸까요?

실제로 뜯어본 결과, 이 앱이 웹페이지에는 없는 "기능"으로 제공하는 것은 딱 두 가지였는데, 둘 다 사용자 입장에서는 반갑지 않은 것들이었습니다.

- Google 계정과 연동된 추적 데이터를 개발사로 전송
- 같은 여행사가 주관하는 다른 여행 상품 광고("영감(inspirations)"이라는 이름으로 포장된)를 노출

반면 웹페이지였다면 다음과 같은 것들이 기본으로 따라왔을 겁니다.

- 복사-붙여넣기 가능
- 인쇄 가능
- 저장 가능
- 북마크 가능
- 검색 가능
- 사실상 모든 기기에서 사용 가능
- (잠재적으로) 더 나은 접근성

이 정도로 화가 난 저자는 결국 이 앱을 "고쳐보기"로 마음먹습니다.

## 앱 트래픽 가로채기

오랜만에 Android 앱을 네트워크 트래픽 분석으로 리버스 엔지니어링하게 된 저자는, 다음과 같은 절차를 거쳤습니다.

1. Android Studio의 Virtual Device Manager에서 새 가상 기기를 생성
2. `adb shell`이 정상 동작하는지 확인한 뒤, rootAVD를 이용해 루팅 진행: `./rootAVD.sh system-images/android-33/google_apis_playstore/x86_64/ramdisk.img`
3. 콜드 부팅 후 Magisk를 실행하고, 앱이 su 권한을 요청하면 자동으로 승인하도록 설정 변경

여기서 흥미로운 점은, Magisk 설정을 바꾸지 않으면 HTTP Toolkit이 su 권한을 요청은 하지만 그 응답을 기다리지 않고 곧바로 권한 없는 모드로 실행되어버려, 사용자가 미처 권한을 부여할 틈도 없이 실패한다는 것이었습니다.

루팅이 끝나면 이제 Wireshark나 TCPdump를 쓰는 것과 다를 바 없이, 모든 트래픽을 들여다볼 수 있게 됩니다.

4. HTTP Toolkit을 실행해 AVD 트래픽을 가로채도록 설정. 이 과정에서 가짜 VPN 제공자가 설치되어 폰의 트래픽이 프록시를 거치도록 라우팅됩니다.
5. Play 스토어에서 Travelbound 앱 설치
6. HTTP Toolkit을 Travelbound 앱의 트래픽만 프록시하도록 설정 (노이즈를 줄이고 신호만 남기기 위함)

참고로 루팅이 필요한 이유는, Certificate Pinning을 사용하는 애플리케이션이 중간자(man-in-the-middle) 프록시 서버를 신뢰하도록 강제하기 위해서입니다. 루팅하지 않으면 일부 앱—이번에 분석하려던 앱을 포함해서—은 자체 서명된 TLS 인증서를 무효한 것으로 인식하고 통신을 거부해버립니다. 또한 Android 보안 정책 때문에 HTTP Toolkit이 설치해준 루트 CA 인증서를 수동으로 설치해줘야 했지만, 안내된 절차대로 하니 문제없이 진행됐습니다.

## 몇 분 만에 드러난 API의 실체

리버스 엔지니어링을 시작한 지 몇 분 지나지 않아, 저자는 이 앱이 사용자 이름과 비밀번호를 이어붙여 다음과 같은 형태의 URL로 요청을 보낸다는 사실을 발견했습니다.

```
https://travelbound.api.vamoos.com/api/itineraries/{username}-{password}
```

여기서 흥미로운 점은, 이 사용자 이름과 비밀번호가 여행 그룹 전체가 공유하는 값이라는 것입니다. 저자는 "만약 이 자격 증명이 유출되면 대응 계획이 있는 건가?"라고 의문을 던지면서, 혹은 이들이 보유한 데이터가 애초에 민감도가 낮아 유출돼도 상관없다고 여기는 것이라면, "그럼 애초에 왜 이걸 그냥 웹페이지로 만들지 않았느냐"는 원래의 질문으로 다시 돌아가게 된다고 지적합니다.

이 요청은 JSON 뭉치를 반환하는데, 조금만 들여다보면 앱이 보여주는 모든 콘텐츠가 여기에 담겨 있다는 걸 알 수 있었습니다. 구체적으로는 다음과 같은 구조였습니다.

- 일정의 각 구간(leg)을 담은 배열
- 사용자에게 보여줄 모든 "영감(inspiration)" 광고를 담은 배열
- 다른 섹션에서 참조하는 모든 파일(이미지 등)을 상호 참조하는 배열

이쯤 되니 저자는 다시 한번 묻습니다. "어차피 HTML 코드를 만들어내고 있는 거잖아요. 그런데 왜 이게 웹페이지가 아닌 거죠?"

추가로 실험해보니, S3 이미지 URL은 비교적 짧은 만료 시간을 가지고 발급되고 있어서, 콘텐츠가 바뀌지 않았더라도 JSON을 주기적으로 다시 받아와야 했습니다. 그렇지 않으면 이미지를 로컬에 캐싱해야 하는데, 이 앱은 바로 그 캐싱을 아주 비효율적인 방식으로 수행하고 있었던 것으로 보입니다.

## 앱을 웹페이지로 되돌려놓기

필요한 정보를 모두 확보한 저자는, Cron 스케줄로 실행되는 Ruby 스크립트를 작성해 최신 JSON을 주기적으로 가져와 HTML 페이지를 생성하도록 만들었습니다.

이 과정에서 다음과 같은 설계 선택을 했습니다.

- 데이터 스키마상 `overlayRows`로 불리는 "영감" 광고 섹션은 완전히 건너뛰기
- 일정 항목들과, 영감/일정 어디에도 참조되지 않은 나머지 파일들(PDF 다운로드 링크를 모으는 다소 게으른 방법)만 나열하기

이렇게 만든 페이지는 여행 그룹이 원래 부여받았던 것과 동일한 비밀번호로 보호해 호스팅했습니다. 그리고 사용한 원본 JSON은 `<details>` 요소 안에 넣어두었는데, 혹시 저자가 미처 파악하지 못한 스키마 부분이 나중에 등장하더라도 확인할 수 있도록 하기 위해서였습니다.

저자 스스로도 인정하듯, 이 웹페이지는 원본 앱만큼 "예쁘지" 않습니다. 하지만 원본 콘텐츠에서 정보를 "빌려온" 것치고는 용량이 훨씬 작고, 웹의 표준 기능들을 공짜로 누릴 수 있습니다. 게다가 이 웹페이지는 트래킹 코드를 제거하고 광고도 보여주지 않는, 사용자 친화적이면서 동시에 (여행사 입장에서는) 개발자 비우호적인 기능도 갖추고 있습니다.

## 앱 문화에 대한 근본적인 물음

저자는 어떤 사람들은 "앱"을 선호하고, 그건 그 나름대로 괜찮다고 인정합니다. 하지만 일부 앱은 애초에 웹페이지였어야 했다고 강조합니다. 특히 이번 사례처럼 콘텐츠가 이미 HTML로 작성되어 HTTP로 전달되고 있다면, 그건 그냥 웹페이지여야 마땅하다는 것이죠.

물론 "앱"이라는 매체가 절대적으로 옳은 선택인 작업들도 존재합니다. 다만 Travelbound는 그런 경우가 아니라는 게 저자의 판단입니다.

저자는 소프트웨어 회사들이 앱스토어에 배포하는(결코 무료가 아닌) 더 어렵고 비싼 길을 자처하면서, 결과적으로는 웹에 직접 게시했을 때보다 더 적은 사람들에게, 더 적은 기능으로 HTML 콘텐츠를 전달하고 있다는 사실을 도무지 이해할 수 없다고 토로합니다. 게다가 이런 앱들은 접근성 면에서도 뒤떨어지는 경우가 많은데, 이번 Travelbound 앱을 정식으로 접근성 감사한 것은 아니지만, 저자가 만든 단순한 웹 버전보다 보조 기술로 사용하기 더 어려울 것으로 보이는 정황들이 있었다고 덧붙입니다.

## 43MB 앱 vs 0.05MB 웹페이지

결과적으로 저자와 여행 그룹 구성원들은 이제 이 콘텐츠에 접근하는 방식을 스스로 선택할 수 있게 됐습니다.

- 43MB짜리 앱 (추가 콘텐츠 다운로드가 끝나면 124MB까지 불어남), 트래킹과 광고 포함
- 0.05MB짜리 웹페이지 (선택적으로 이미지 35MB 추가), 더 많은 기능을 제공하고 더 많은 기기에서 작동

저자는 자신이 어느 쪽을 쓸지는 이미 정해져 있다고 마무리합니다.

## 정리

- 이번 사례의 Travelbound 앱은 텍스트, 이미지, PDF 링크를 보여주는 것이 사실상 전부인, 본질적으로 "문서"에 가까운 콘텐츠를 앱으로 강제 배포하고 있었습니다.
- 앱이 웹페이지 대비 추가로 제공하는 기능은 사용자 트래킹과 광고 노출뿐이었고, 반대로 웹페이지가 기본 제공하는 복사, 인쇄, 저장, 북마크, 검색, 범용 기기 지원, 접근성 등의 이점은 모두 사라졌습니다.
- 저자는 Android 가상 기기 루팅, Magisk, HTTP Toolkit을 이용한 트래픽 가로채기로 앱의 API 엔드포인트(`travelbound.api.vamoos.com/api/itineraries/{username}-{password}`)와 JSON 스키마를 파악했고, 이를 바탕으로 동일한 정보를 제공하는 정적 HTML 페이지를 Cron으로 자동 생성하는 파이프라인을 구축했습니다.
- 결과물은 43MB(최대 124MB)짜리 앱을 0.05MB(선택적으로 이미지 35MB) 웹페이지로 대체하면서도, 광고와 트래킹을 제거하고 원본 JSON까지 투명하게 공개하는 형태였습니다.
- 이 사례는 이미 HTML/HTTP로 전달 가능한 콘텐츠를 굳이 네이티브 앱으로 감싸는 관행이 개발 비용, 접근성, 사용자 경험 모든 면에서 비효율적일 수 있음을 잘 보여줍니다. 프론트엔드 개발자라면 "이게 정말 앱이어야 하는가"라는 질문을 프로젝트 초기에 스스로에게 던져볼 필요가 있습니다.
