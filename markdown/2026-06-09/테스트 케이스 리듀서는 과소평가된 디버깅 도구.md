## 핵심 개념

> [!abstract]
> 테스트 케이스 리듀서는 프로그램 충돌을 유발하는 입력값을 자동으로 축소하여 디버깅을 용이하게 하는 도구이다. 단순한 구조임에도 불구하고 95-99%의 놀라운 축소율을 달성할 수 있으며, '흥미로움 테스트'라는 조건만 정의하면 사용할 수 있다.

## 상세 내용

- 테스트 케이스 리듀서는 자동으로 문제를 유발하는 최소 입력값을 찾아주므로 수동 디버깅보다 훨씬 효율적이다.
- 리듀서는 컴파일러 개발자들 사이에서는 널리 알려져 있지만, 일반 개발자들에게는 덜 알려진 도구이다.
- 오류 발생 빈도나 실행 명령어 수 등 추가 요소를 고려하도록 리듀서를 커스터마이징할 수 있어 실무 효과를 높일 수 있다.

> [!tip] 왜 중요한가
> 복잡한 입력으로 인한 버그 재현을 빠르게 최소화하여 디버깅 시간을 크게 단축할 수 있다.

## 전문 번역

# 테스트 케이스 리듀서의 숨겨진 활용법

테스트 케이스 리듀서(Test-case reducer)는 생각보다 잘 알려져 있지 않습니다. 알고 있는 개발자들도 이 도구를 활용하는 다양한 방법을 모르는 경우가 많죠. 이 글에서는 제가 테스트 케이스 리듀서를 사용하면서 배운 것들을 공유해보겠습니다.

기초부터 시작할 텐데, 개념이 너무 단순해서 오히려 잘 작동하는지 의심스러울 정도입니다. 그다음 좀 더 깊이 있는 내용으로 들어가겠습니다.

테스트 케이스 리듀서의 기본 목적은 입력의 길이를 줄이는 것이지만, 우리는 여기에 다른 요소들을 고려하도록 강제할 수 있습니다. 예를 들어 에러가 발생하는 빈도나 실행되는 명령어의 개수 같은 것 말이에요. 디버깅하려는 문제에 따라 이런 조정이 실제 효율성에 큰 영향을 미칠 수 있습니다.

전문가라면 이 글의 대부분이 새로울 것은 없겠지만, 제가 이것들을 어렵게 배웠기에 다른 분들도 참고하실 수 있기를 바랍니다.

## 문제 상황: 큰 입력에서의 크래시

프로그램이 어떤 큰 입력값으로 크래시하는데, 정확히 어느 부분이 문제인지 모르는 상황을 생각해봅시다. 이럴 때 어떻게 해야 할까요?

대부분의 개발자는 기본 디버깅 기법부터 시작합니다. printf 같은 간단한 방법에서 시작해서 디버거로 진행하고, 필요하면 sanitizer나 valgrind 같은 고급 도구까지 쓰게 되죠. 모든 개발자는 나름의 디버깅 도구와 기법을 가지고 있는데, 이 중 일부는 다른 것보다 훨씬 효과적입니다.

그런데 실제로는 덜 사용되지만 큰 도움이 될 수 있는 기법이 하나 있습니다. 바로 입력의 크기를 줄이는 것입니다. 일반적으로 입력이 작을수록 왜 문제가 발생하는지 파악하기 쉬워거든요.

## 수동 리덕션의 한계

입력을 줄이는 과정은 수동으로도 가능합니다. 텍스트 에디터에서 파일을 열어 일부를 삭제한 뒤, 이 축소된 입력이 여전히 크래시를 유발하는지 확인하는 식이죠.

하지만 수동으로 하면 여러 문제가 생깁니다. 우리 인간은 쉽게 지루해하고 시야가 제한적이라 많은 리덕션 기회를 놓치게 됩니다. 또한 입력의 일부를 삭제하면 원래 조사하던 방식의 크래시가 더 이상 발생하지 않을 수 있습니다. 프로그램이 정상적으로 종료되거나, 예상되는 다른 에러가 나타날 수도 있죠.

더 골치 아픈 경우도 있습니다. A 부분만 삭제해서는 원하는 효과가 없는데, A와 B를 동시에 삭제해야만 효과가 생기는 경우가 있다는 거예요. 삭제 조합의 가능성이 얼마나 많을까요? 끝없는 시지프스의 시련이 기다리고 있습니다.

## 자동화된 해결책: 테스트 케이스 리듀서

다행히 이 과정을 자동화하는 도구들이 있습니다. 바로 테스트 케이스 리듀서인데요. 이 도구는 프로그램, 입력값, 그리고 "흥미로움 테스트(interestingness test)"를 받습니다. 리듀서는 계속해서 더 짧은 버전의 입력을 시도하고, 흥미로움 테스트가 그 입력이 여전히 문제를 재현하는지를 판단해줍니다. 

테스트 케이스 리듀서는 놀라운 효율성을 자랑합니다. 95~99%의 입력 축소도 흔하죠. 이렇게 되면 디버깅이 훨씬 쉬워집니다.

## 신비로움의 정체

테스트 케이스 리듀서가 마법처럼 들릴 수 있습니다. 입력에서 어떤 부분을 제거해야 하는지 도구가 어떻게 알 수 있을까요? 설상가상으로, 이런 도구를 가장 활발히 사용하는 커뮤니티는 컴파일러 개발자들입니다. 많은 프로그래머들은 이들을 도달 불가능한 엘리트라고 생각하죠. 이 두 가지가 합쳐지면서, 많은 사람들이 이런 도구를 시도하는 데서 물러나게 된 것 같습니다.

좋은 소식은 테스트 케이스 리듀서는 마법이 아니라는 것입니다. 직접 하나를 만들어보면 금방 이해할 수 있습니다.

## 실전 예제

파일에서 단어를 읽는 프로그램을 작성했다고 가정해봅시다:

```c
#include <stdio.h>
#include <string.h>

int main() {
  char word[10];
  FILE *fp = fopen("/usr/share/dict/words", "r");
  while (fscanf(fp, "%s\n", word) == 1) {
    if (strlen(word) > 8)
      printf("Word too long\n");
  }
  fclose(fp);
}
```

제 머신에서 이걸 실행하면 이런 경고가 나타납니다:

```
Word too long
```

이 예제를 위해, 이 경고를 "에러"라고 가정하겠습니다. 기존 디버깅 기법으로는 원인을 파악하지 못했다고 해봅시다. 테스트 케이스 리듀서가 도움이 될까요?

## 흥미로움 테스트 정의하기

첫 번째 단계는 흥미로움 테스트를 정의하는 것입니다. 제가 아는 테스트 케이스 리듀서 관례에 따르면, 흥미로움 테스트는 다음 조건을 만족하는 프로그램입니다:

- 첫 번째 인자로 파일명을 받는다
- 해당 파일을 처리한다
- 특정 조건이 만족되면 0을 반환하고, 아니면 1을 반환한다

"Word too long"이 출력되는지 확인하는 간단한 셸 스크립트를 작성하겠습니다. 조건이 만족되면 0을, 아니면 1을 반환하죠.

## 리듀서 구현하기

이제 어려운 부분, 리듀서를 만들 차례입니다. 다행히 생각보다 어렵지는 않습니다:

```bash
#!/bin/bash

cur=($(cat "$1"))
i=0

while [ $i -lt ${#cur[@]} ]; do
  cnd=("${cur[@]}")
  unset 'cnd[$i]'
  cnd=("${cnd[@]}")
  
  echo "${cnd[@]}" > /tmp/test_input
  if ./program /tmp/test_input > /dev/null 2>&1; then
    cur=("${cnd[@]}")
  else
    ((i++))
  fi
done

echo "${cur[@]}"
```

기본적으로 이 리듀서는 텍스트 입력을 읽어서 라인 단위로 분할합니다(`cur`). 그다음 입력을 순회하면서, 각 반복마다 한 라인씩 제거한 후보 입력(`cnd`)을 만듭니다. 임시 파일에 후보를 기록한 뒤 흥미로움 테스트를 실행하죠. 테스트가 0을 반환하면 이 축소된 버전을 새로운 시작점(`cur=cnd`)으로 삼거나, 아니면 다음 라인(`i+=1`)을 시도합니다. 더 이상 축소할 수 없으면 축소된 입력을 표준 출력으로 출력합니다.

이 구현이 느리다는 건 말할 것도 없지만, 실행하면 결국 이런 결과를 얻을 수 있습니다:

```
Word too long
```
