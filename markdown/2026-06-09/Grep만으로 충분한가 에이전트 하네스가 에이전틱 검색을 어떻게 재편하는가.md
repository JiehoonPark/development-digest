## 핵심 개념

> [!abstract]
> LLM 에이전트 시스템에서 검색 전략(grep vs 벡터 검색)이 에이전트 아키텍처와 도구 호출 방식과 어떻게 상호작용하는지 실증 연구했다. 두 가지 실험을 통해 grep이 벡터 검색보다 일반적으로 더 높은 정확도를 보였지만, 최종 성능은 사용된 하네스 유형에 강하게 의존한다는 것을 발견했다.

## 상세 내용

- grep 검색이 벡터 검색보다 에이전틱 검색에서 더 높은 정확도를 달성했다.
- 도구 결과 표시 방식(인라인 vs 파일 기반)과 하네스 종류가 성능에 중대한 영향을 미친다.
- RAG 시스템 설계 시 검색 알고리즘만큼 에이전트 아키텍처 선택이 중요하다는 것을 시사한다.

> [!tip] 왜 중요한가
> LLM 에이전트를 활용한 검색 시스템 개발 시 검색 알고리즘 선택과 에이전트 구조의 최적 조합을 판단하는 데 실용적인 지침을 제공한다.

## 전문 번역

# Grep만으로 충분할까? 에이전트 하네스가 에이전틱 검색을 바꾸는 방식

**논문 정보**
- arXiv:2605.15184 (cs.CL)
- 저자: Sahil Sen, Akhil Kasturi, Elias Lumer, Anmol Gulati, Vamse Kumar Subbiah
- 제출일: 2026년 5월 14일

## 요약

최근 대규모 언어 모델(LLM) 에이전트의 발전으로 복잡한 에이전틱 워크플로우가 가능해졌어요. 모델이 자동으로 정보를 검색하고, 도구를 호출하며, 대규모 데이터를 분석해서 사용자 대신 작업을 완료하는 것이죠.

RAG(Retrieval-Augmented Generation) 기술이 에이전틱 검색 시스템에 널리 도입되고 있긴 한데, 한 가지 문제가 있습니다. 검색 전략의 선택이 에이전트 아키텍처 및 도구 호출 방식과 어떻게 상호작용하는지에 대한 체계적인 비교 연구가 부족한 상태거든요. 

특히 다음 같은 실무적 차원들이 충분히 탐구되지 않았습니다:
- 도구의 출력 결과를 모델에 어떻게 제시하는가
- 검색 결과 주변에 관련 없는 텍스트가 많아지면 성능이 어떻게 변하는가

## 연구 방법

이 논문은 두 가지 실험으로 구성되어 있습니다.

**실험 1: 검색 방식 비교**

LongMemEval에서 추출한 116개 질문 샘플을 대상으로 grep과 벡터 검색을 비교했어요. 테스트 환경으로는 커스텀 에이전트 하네스(Chronos)와 주요 AI 제공사의 네이티브 CLI 하네스(Claude Code, Codex, Gemini CLI)를 사용했습니다.

또한 두 가지 도구 결과 제시 방식을 모두 검토했습니다:
- 인라인 방식: 도구 결과를 직접 대화에 삽입
- 파일 기반 방식: 모델이 별도로 읽을 수 있도록 파일에 저장

**실험 2: 노이즈에 대한 견고성 테스트**

grep 전용 검색과 벡터 전용 검색을 비교하면서, 단계적으로 관련 없는 대화 히스토리를 추가했습니다. 이렇게 하면 각 쿼리가 중요한 문서들 사이에 더 많은 방해 요소가 섞이는 상황을 시뮬레이션할 수 있거든요.

## 주요 발견

Chronos와 각 제공사 CLI 환경 모두에서 **grep이 벡터 검색보다 높은 정확도**를 보여줬습니다.

그런데 흥미로운 점이 있어요. 전체 성능은 사용된 하네스와 도구 호출 방식에 따라 여전히 크게 달라진다는 것입니다. 같은 대화 데이터를 사용하더라도 말이에요.

이는 에이전트 시스템을 구축할 때 검색 알고리즘 선택만큼이나 **에이전트 하네스 설계**가 중요하다는 점을 시사합니다. 단순한 grep이 복잡한 벡터 검색보다 실제로는 더 효과적일 수 있다는 결론인데, 이는 기존의 RAG 중심 접근 방식에 대한 흥미로운 질문을 던져줍니다.
