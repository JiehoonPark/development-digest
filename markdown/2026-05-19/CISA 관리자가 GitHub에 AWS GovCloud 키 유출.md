## 핵심 개념

> [!abstract]
> 미국 정부 기관 CISA 소속 계약업체 직원이 GitHub 공개 저장소에 AWS GovCloud 관리자 자격증명, 평문 비밀번호, 내부 시스템 정보 등을 노출시켰다. 이는 최근 정부 데이터 유출 사건 중 가장 심각한 사례로, 깃허브 보안 기능을 비활성화한 상태에서 발생했다.

## 상세 내용

- GitHub 시크릿 검출 기능을 명시적으로 비활성화하고 평문 비밀번호와 CSV에 AWS 토큰을 저장하는 등 기본적인 보안 관행 미준수
- 노출된 자격증명으로 3개의 AWS GovCloud 계정과 CISA 내부 아티팩토리(소프트웨어 패키지 저장소) 접근 가능하며, 공급망 공격에 악용될 수 있는 심각한 위험

> [!tip] 왜 중요한가
> 개발자와 기관 차원에서 시크릿 관리, CI/CD 보안, 저장소 접근 제어의 중요성을 강조하는 사례로, 보안 감사와 자동화된 시크릿 탐지 도구의 필수성을 입증한다.

## 전문 번역

# CISA 정부기관의 AWS 자격증명 대량 유출 사건

미국 사이버보안기반시설보안청(CISA) 소속 계약업체가 운영하던 GitHub 공개 저장소에서 AWS GovCloud의 고권한 계정 자격증명과 CISA 내부 시스템에 대한 접근 정보가 노출됐습니다. 보안 전문가들은 이를 최근 몇 년간 발생한 정부 데이터 유출 사건 중 가장 심각한 수준이라고 평가하고 있습니다. 유출된 저장소에는 CISA가 소프트웨어를 어떻게 구축, 테스트, 배포하는지에 관한 상세 정보가 담겨 있었거든요.

## 발견 경위

5월 15일, 보안 기업 GitGuardian의 연구원 기욤 발라동(Guillaume Valadon)이 이 문제를 보도했습니다. GitGuardian은 GitHub와 다른 코드 저장소를 지속적으로 모니터링하면서 유출된 시크릿을 자동으로 감지해 해당 계정 소유자에게 알립니다. 발라동은 저장소 소유자가 연락에 응하지 않고 노출된 정보의 민감도가 매우 높아서 직접 신고하게 됐다고 설명했습니다.

## 무엇이 노출됐나

문제가 된 저장소는 "Private-CISA"라는 이름으로, CISA와 국토안보부(DHS)의 광범위한 내부 자격증명과 파일을 포함하고 있었습니다.

- 클라우드 키
- 토큰
- 평문 형태의 비밀번호
- 로그
- 기타 민감한 CISA 자산

발라동은 이를 "교과서적인 수준의 보안 위생 실패"라고 표현했습니다. 해당 GitHub 계정의 커밋 로그를 보면 CISA 관리자가 SSH 키나 기타 시크릿을 공개 저장소에 올리는 것을 차단하는 GitHub의 기본 설정을 일부러 비활성화한 흔적이 있었거든요.

"CSV 파일에 평문으로 저장된 비밀번호, Git에 백업된 파일, GitHub 시크릿 감지 기능을 명시적으로 비활성화하는 명령어들... 처음엔 이게 다 가짜라고 생각했을 정도입니다. 제 경력에서 본 것 중 최악의 유출 사건입니다. 물론 개인의 실수지만, 이것이 내부 운영 관행에 대해 뭔가 드러낸다고 생각합니다."

## 노출된 자격증명의 위험성

"importantAWStokens"라는 파일에는 Amazon AWS GovCloud 서버 3개에 대한 관리자 자격증명이 포함돼 있었습니다. "AWS-Workspace-Firefox-Passwords.csv"라는 또 다른 파일에는 CISA 내부 시스템 수십 개의 평문 사용자명과 비밀번호가 나열돼 있었는데, 여기엔 "LZ-DSO"(Landing Zone DevSecOps로 추정)라는 CISA의 보안 코드 개발 환경도 포함됐습니다.

보안 컨설팅 회사 Seralys의 창립자 필립 카투렐리(Philippe Caturegli)는 노출된 AWS 키를 검증해봤습니다. 그 결과 이 자격증명들이 여전히 유효했으며, AWS GovCloud 계정 3개에 높은 권한으로 접근할 수 있다는 것을 확인했습니다.

카투렐리는 또한 CISA의 내부 "artifactory"(소프트웨어 빌드에 사용되는 모든 코드 패키지의 저장소)에 대한 평문 자격증명도 노출됐다고 지적했습니다. 이는 공격자 입장에서 CISA 시스템에 은폐된 백도어를 심어두기 위한 매우 매력적인 목표입니다.

"그곳은 네트워크 내에서 횡으로 이동하기에 최고의 위치입니다"라고 그는 설명했습니다. "소프트웨어 패키지에 백도어를 심어두면, CISA가 새로운 소프트웨어를 빌드할 때마다 그 백도어가 배포되는 거죠."

## 저장소 사용 패턴

저장소의 커밋 로그를 분석한 결과, 이 저장소는 정식 프로젝트보다는 개인이 작업용으로 사용하는 스크래치패드나 동기화 메커니즘에 가까웠습니다.

"CISA 이메일과 개인 이메일 주소가 모두 사용된 점을 보면, 서로 다르게 구성된 여러 환경에서 이 저장소가 사용됐을 가능성이 높습니다"라고 카투렐리는 분석했습니다. "Git 메타데이터만으로는 어느 기기나 엔드포인트에서 작업했는지까지는 파악하기 어렵습니다."

## CISA의 입장

CISA 대변인은 이 노출 사건을 인지했으며 조사를 진행 중이라고 밝혔습니다.

"현재까지 이번 사건으로 인해 민감한 데이터가 실제로 침해됐다는 증거는 없습니다. 우리는 팀원들에게 최고 수준의 성실성과 운영 인식을 요구하고 있으며, 향후 유사 사건을 방지하기 위해 추가 보안 조치를 시행하기 위해 노력 중입니다."

## 계약업체 정보

문제가 된 저장소는 버지니아주 덜레스에 소재한 정부 계약업체 Nightwing의 직원이 운영하던 GitHub 계정에서 발견됐습니다. Nightwing은 논평을 거절하고 문의사항을 CISA에 넘겼습니다.

CISA는 데이터 노출 기간이 얼마나 됐는지에 대해 답변하지 않았지만, 카투렐리는 "Private CISA" 저장소가 2025년 11월 13일 생성됐다고 확인했습니다. 그 계약업체의 GitHub 계정은 2018년 9월에 만들어진 것으로 나타났습니다.

KrebsOnSecurity와 Seralys가 CISA에 이 노출 사실을 알린 직후, 해당 GitHub 계정은 오프라인 상태로 전환됐습니다.
