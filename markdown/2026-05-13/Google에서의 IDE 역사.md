## 핵심 개념

> [!abstract]
> Google의 13년 개발자 도구 경험을 바탕으로 IDE 생태계의 분화에서 Cider라는 클라우드 IDE를 거쳐 VSCode 기반 솔루션으로 진화한 과정을 기술한다. 대규모 모노레포에서 언어 서버 프로토콜 기반의 백엔드 코드 인텔리전스가 핵심이었다.

## 상세 내용

- 모노레포 규모에서 전통적 로컬 IDE의 한계 극복을 위해 클라우드 기반 언어 그래프 인덱싱 도입
- Cider의 웹 기반 경량 클라이언트가 기술 문서 작성에서 시작해 코드 완성 기능으로 확대되며 인기 증대

> [!tip] 왜 중요한가
> 대규모 코드베이스 관리 시 개발 도구 아키텍처 설계와 클라우드 기반 솔루션의 중요성을 보여준다.

## 전문 번역

# Google의 IDE 전략 변화: Cider에서 배우는 개발 생산성

저는 2011년부터 2024년까지 Google에서 일했는데, 그 기간 동안 개발자 도구팀에서 근무하면서 흥미로운 변화를 목격했습니다. 오늘은 Google의 IDE 전략이 어떻게 진화했는지 얘기해보겠습니다.

## 편집기 선택의 자유는 과연 좋은가?

Google은 수년간 개발자들이 자신이 원하는 IDE를 자유롭게 선택하도록 허용했습니다. 2011년 당시 선임 엔지니어들에게 "모든 직원이 공통으로 사용할 수 있는 통합 IDE가 있을까?"라는 질문이 던져졌는데, Jeff Dean을 포함한 대다수가 부정적인 답변을 했거든요.

> 모든 개발자가 같은 에디터를 쓰도록 설득하는 건 불행의 지름길입니다. 사람마다 중요하게 여기는 부분이 다르고, IDE의 장단점을 평가하는 방식도 제각각이니까요.

이것이 오랫동안 Google의 기조였습니다. 동료가 어떤 IDE를 쓰든 코드 품질만 좋으면 된다는 생각이었죠.

그런데 회사 차원의 생산성을 생각해보면 얘기가 달라집니다. 각 엔지니어가 자신의 에디터를 설정하는 데 시간을 낭비할 수 없으니까요. 실제로 개발자들이 서로 다른 IDE를 사용하다 보니, Bazel 지원, Starlark 도구, 코드 포매터, 코드 검색 통합 등 유용한 기능들을 모든 IDE에서 반복적으로 구현해야 했습니다.

Google의 내부 문화는 이 문제를 어느 정도 해결했어요. 엔지니어들이 자발적으로 도구 프로젝트를 시작하면, 공유 저장소를 통해 다른 사람들이 발견하고 기여하는 식으로 진행되었거든요. 이런 활동은 20% 시간제나 동료 보너스 같은 제도로 장려되었습니다. 결국 중요한 프로젝트들은 공식 지원을 받게 되는데, IntelliJ 통합팀이 2015년 경 출범한 것도 같은 맥락입니다.

## 클라우드 IDE, Cider의 등장

전환점은 2013년경에 찾아왔습니다. 누군가 웹 기반 에디터를 만들기 시작한 것이죠. 이름은 "Cloud IDE"의 약자에 'r'을 붙인 **Cider**였습니다.

Google의 업무 문화를 생각해보면 이건 당연한 흐름이었어요. Google 직원들은 대부분 브라우저에서 코드 리뷰를 하고, Code Search로 저장소를 탐색하고, Chromebook을 사용합니다. 그렇다면 브라우저에서 직접 파일을 편집할 수 있는 도구가 있으면 훨씬 편할 겁니다.

처음엔 기술 문서 작성자들이 주로 사용했습니다. 마크다운 파일을 수정할 때 버전 관리 시스템을 건드릴 필요 없이, 버튼 한 번으로 풀 리퀘스트를 보내고 승인되면 자동으로 병합되는 워크플로우가 정말 효율적이었거든요. (지금 GitHub에도 비슷한 기능이 있지만, 당시엔 혁신적이었습니다.)

시간이 지나면서 팀은 개발자 중심의 기능들을 계속 추가했습니다. 정말 중요한 전환점은 **언어 서버 프로토콜(Language Server Protocol)을 통한 코드 자동완성**을 지원하기 시작했을 때였습니다.

## 웹 기반의 강점

Cider는 기존 IDE보다 훨씬 빠르게 로드되는 가벼운 클라이언트였습니다. 실제 작업은 전체 저장소를 인덱싱하는 백엔드에서 일어났으니까요. 누군가 페이지를 열 때 필요한 데이터가 이미 준비되어 있었던 겁니다.

코드 인텔리전스를 제공하려면 각 식별자를 그것의 타입과 참조들과 연결해야 합니다. 이건 엄청난 규모의 언어 그래프를 만드는데, 매번 커밋할 때마다 업데이트되어야 합니다. Google의 저장소는 초당 수많은 커밋을 받으니까요.

여기서 또 다른 도전이 있었는데, IDE도 과거 데이터에 접근해야 한다는 점입니다. 만약 제가 어떤 프로젝트를 작업 중인데 동료가 코드를 병합하면, 저는 그 변경사항을 즉시 반영하고 싶지 않을 수 있거든요. 그래서 제 에디터는 제가 마지막으로 동기화한 시점의 그래프를 사용해야 하고, 여기에 제 로컬 변경사항을 더해야 합니다.

이런 기능들이 추가되면서 Cider의 인기는 특정 개발자 그룹 사이에서 빠르게 올랐습니다. 예를 들어 Go 개발자들을 Cider로 옮기는 게 Java 개발자들을 설득하는 것보다 훨씬 쉬웠어요. (Java 개발자들은 더 고급 기능의 IDE를 기대했으니까요.) 하지만 10억 개의 파일 전체에서 검색하고 상호 참조를 볼 수 있다는 건 정말 매력적이었습니다.

## 다음 단계: VSCode 기반의 Cider V

백엔드에 대한 투자는 정당했습니다. Google 특화 문제들을 해결하고 있었고 다른 좋은 대안이 없었거든요. 하지만 프론트엔드는 여전히 제한적이었어요. 빠른 수정에는 좋았지만, 실제 IDE와 경쟁하기에는 부족했습니다.

2020년 저는 기술 리드 중 한 명으로 이 팀에 합류했고, 방향을 전환하게 됩니다...
