## 핵심 개념

> [!abstract]
> SBCL의 내부 기능을 활용해 스택 기반 가상 머신(VM) 설계를 탐구하는 글로, 제한된 스택 슬롯(8개)을 효율적으로 활용하기 위한 최적화 기법을 다룹니다.

## 상세 내용

- x87과 F18을 참고해 푸시/팝 연산 시 실제 데이터 이동 없이 모듈러 카운터로 스택 포인터를 관리하는 기법을 소개합니다.
- 프리미티브 연산을 8가지 스택 상태별로 특화된 버전으로 유지해 성능을 극대화합니다.
- SBCL의 어셈블러를 활용해 반복적인 머신 코드 생성을 효율적으로 수행하는 방법을 시연합니다.

> [!tip] 왜 중요한가
> 저수준 VM 설계와 레지스터 최적화 기법을 이해하면 고성능 인터프리터나 가상 머신 구현에 도움이 됩니다.

## 전문 번역

# 스택 기반 VM에서 레지스터 활용하기

최근에 Chuck Moore의 F18을 살펴보다 흥미로운 부분을 발견했습니다. 스택이 정말 작거든요—겨우 10개 슬롯인데, 오버플로우나 언더플로우 트랩 같은 복잡한 메커니즘이 없습니다. 철학은 간단합니다. 더 많은 슬롯이 필요하면 설계를 다시 생각해봐야 한다는 거죠. 저도 HP-41C와 x87 작업을 하면서 이 접근이 맞다고 느껴봤습니다.

이런 생각이 들었어요. 스택 기반 VM을 설계할 때 스택을 정확히 8개 슬롯으로 제한한다면 어떤 구현 기법들이 가능할까요?

이상적으로는 모든 값을 레지스터에만 보관하고 싶습니다. 그런데 단순하게 접근하면 push와 pop이 복잡해져버립니다. 그래서 대부분의 Forth 엔진은 스택 최상단의 1~2개 요소만 캐싱하는 방식을 씁니다.

저는 x87과 F18을 모방하기로 결정했습니다. push와 pop 작업이 데이터를 이동시키지 않는 방식이죠. 대신 스택 포인터라는 모듈로 카운터를 증가/감소시킵니다. 이 카운터가 스택의 최상단을 가리키는 거예요. 소프트웨어로 구현하면 느리겠지만, 여기가 핵심입니다—카운터가 취할 수 있는 값이 매우 제한적입니다. 8개 슬롯이 있다면 카운터는 8가지 값만 가질 수 있거든요.

스택 VM은 성능상의 이유로 이미 기본 연산(primop)을 여러 주소에 복제해둡니다. 따라서 스택 카운터가 취할 수 있는 8가지 값 각각에 대해 primop을 특화시키는 것도 합리적입니다.

## 직접 스레드 방식의 NEXT 명령어

일반적인 직접 스레드 VM에서 primop은 다음 명령어로 점프하는 NEXT라는 코드 시퀀스로 끝납니다.

```nasm
add rsi, 8          ; 가상 IP 증가
jmp [rsi-8]         ; RSI가 이전에 가리키던 주소로 점프
```

여기서 rsi는 가상 명령어 포인터고, VM 명령어는 단순히 관련 primop의 머신 코드 주소입니다.

저는 두 가지를 개선하기로 했습니다. 첫째, 바이트코드에 주소를 하드코딩하는 것이 싫었습니다. 둘째, 가상 명령어당 64비트는 낭비가 심합니다. 대신 primop 코드 블록으로부터의 오프셋으로 인코딩합니다.

```nasm
mov eax, [rsi]      ; 다음 primop의 오프셋 읽기
add rsi, 4          ; 가상 IP 4바이트 증가
add rax, rdi        ; primop 베이스 주소 더하기
jmp rax             ; primop으로 점프
```

여기서 rdi는 primop들의 베이스 주소입니다.

## 스택 카운터 기반 디스패칭

새로운 스택 카운터 값에 따라 디스패치해야 합니다. 각 primop의 변형들을 일정 간격(예: 한 페이지)으로 배치하는 방식을 선택했습니다. 정확히는 64 × 67 = 4288바이트 간격으로 두어 별칭 충돌을 줄였습니다.

```nasm
mov eax, [rsi]                           ; 오프셋 읽기
add rsi, 4                               ; IP 증가
lea rax, [rax + rdi + variant_offset]    ; 올바른 변형으로 이동
jmp rax
```

여기서 중요한 트릭은 `variant_offset = 4288 * stack_counter`라는 점입니다. 스택 카운터는 보통 primop을 컴파일할 때 알려져 있죠. 스택을 그대로 두면 카운터도 그대로고, 값을 push하면 카운터가 감소하고, pop하면 증가합니다.

이제 실제로 구현해볼까요.

## 구현 준비

대량의 반복적인 머신 코드를 생성할 문제를 탐구하려고 합니다. SLIME의 REPL과 SBCL의 어셈블러가 이에 완벽합니다!

VM의 기본 설계는 다음과 같습니다:

- **r8~r15**: 스택 슬롯(32비트)
- **rsi**: 머신 코드 primop들의 베이스 주소
- **rdi**: 가상 명령어 포인터(다음 명령어를 가리킴)
- **rax, rbx, rcx, rdx**: 임시 레지스터
- **rsp**: (가상) 반환 스택 포인터

```lisp
(import '(sb-assem:inst sb-vm::make-ea))

;; 스택의 백킹 저장소
(defvar *stack* (make-array 8 :initial-contents (list sb-vm::r8d-tn
                                                       sb-vm::r9d-tn
                                                       sb-vm::r10d-tn
                                                       sb-vm::r11d-tn
                                                       sb-vm::r12d-tn
                                                       sb-vm::r13d-tn
                                                       sb-vm::r14d-tn
                                                       sb-vm::r15d-tn)))

;; primop 생성 시간의 스택 포인터
(defvar *stack-pointer*)

;; (@ 0)은 현재 TOS 레지스터를 반환하고,
;; (@ 1)은 그 아래 레지스터를 반환합니다.
(defun @ (i)
  (aref *stack* (mod (+ i *stack-pointer*) (length *stack*))))

(defvar *code-base* sb-vm::rsi-tn)
```
