## 핵심 개념

> [!abstract]
> AI가 생성한 코드의 신뢰성을 보장하기 위해 프롬프트 기반 행동 제약 대신 컴파일러, 타입 체커 같은 구조적 게이트를 사용해야 한다는 주장으로, Shen-Backpressure 도구를 통해 다중 테넌트 인증 같은 보안 불변식을 구조적으로 강제하는 방법을 제시한다.

## 상세 내용

- 행동 게이트(프롬프트 제약)는 모델이 규칙을 기억하고 적용하는 데 의존하므로 불안정하며, 구조적 게이트(타입 체커, 증명 체커)가 더 신뢰할 수 있는 검증을 제공한다.
- Shen이라는 타입 언어를 사용해 보안 불변식을 코드 생성기를 통해 대상 언어(Go, TypeScript)의 타입과 가드로 변환하면 AI가 이를 위반하기 어렵게 만들 수 있다.

> [!tip] 왜 중요한가
> AI 코드 생성 도구가 보편화되면서 프롬프트만으로는 보안 취약점(예: 접근 제어)을 보장할 수 없으므로, 개발자는 형식 검증과 타입 시스템을 통한 강제 메커니즘 도입이 필수적이다.

## 전문 번역

# AI 코드 생성 시대, 보안을 어떻게 보장할까?

소프트웨어 버그 중 가장 심각한 것들이 역설적으로 가장 따분하기도 합니다. 한 사용자가 다른 사용자의 데이터를 읽으면 안 된다는 건 누구나 동의하죠. 설계 리뷰에서 "Alice가 Bob의 기록을 봐야 한다"고 주장할 사람은 없습니다. 그런데도 접근 제어 오류는 OWASP Top 10에서 계속 1위를 차지하고 있어요.

이런 버그들이 자꾸 프로덕션에 올라가는 이유는 간단합니다. 규칙을 시스템의 잘못된 곳에 둔 거거든요. 프롬프트에, 리뷰 체크리스트에, 그리고 "모든 엔지니어가 이 규칙을 기억하고 정확히 적용하겠지"라는 일종의 신뢰에 규칙을 맡겨둔 거예요.

이런 가정은 원래도 약했는데, AI가 대부분의 코드를 생성하는 시대에는 완전히 무너집니다. CLAUDE.md에 규칙을 써놓고, 시스템 프롬프트를 정성스럽게 작성하고, 에이전트 지시사항에 "authorization IS VERY IMPORTANT"라고 써넣을 수도 있죠. 그리고 당연히 그렇게 해야 합니다. 하지만 모델이 1만6천 줄의 코드를 생성했을 때 남는 질문은 이거예요: **과연 그 코드가 원하는 대로 작동하는지 어떻게 알 수 있을까요?**

테스트가 도움이 되긴 하지만, 테스트는 본질적으로 경험적입니다. 당신과 모델이 기억해서 작성한 경우들만 검증할 뿐, 다음주에 누군가 추가할 핸들러에 대해서는 아무것도 보장할 수 없거든요.

## 문제를 다른 방식으로 풀어보기

이 글에서 제시하고 싶은 가설은 이겁니다: **구조적 제약(structural backpressure)이 모델 성능 개선보다 훨씬 효과적이다.**

현재의 AI 모델들은 이미 당신 코드의 거의 대부분을 작성할 수 있습니다. 진짜 병목은 "모델이 우리가 원하는 걸 했는지 어떻게 알지?"라는 부분이에요. 그리고 그 답은 더 똑똑한 모델을 기다린다고 해서 나오는 게 아니라, 모델이 작동하는 **기반 구조(substrate)**에서 나온다고 봅니다.

Shen-Backpressure는 이 가설을 실제로 탐구하기 위해 만들었습니다. 동작 원리를 데모로 보여드리고, 이를 당신의 프로젝트에 어떻게 적용할 수 있을지 설명하겠습니다.

## 행동적 제약 vs 구조적 제약

프롬프트 레벨의 제약 대부분은 **행동적 제약(behavioral gates)**입니다. 모델에게 "인증을 건너뛰지 마", "입력을 검증해", "공유 헬퍼를 써"라고 지시하는 거죠. 모델들은 이런 지시를 충분히 자주 따르지만, 충분히 자주 놓치기도 합니다. 행동적 제약은 모델이 규칙을 기억하고, 어디에 적용되는지 인식하고, 로컬 문맥의 유혹에 저항할 수 있기를 바랍니다. 그리고 인간 리뷰어가 코드베이스 전체에서 같은 규칙을 유지하기를 기대하죠.

**구조적 제약(structural gates)**은 다릅니다. 컴파일러, 타입 체커, 테스트 러너, 린터, 증명 체커 같은 것들이요. 각각은 당신 앞의 코드에 대해 명확한 답변을 제시합니다. 그 답변이 완벽하지는 않지만, 현실적이고, 자신의 범위 내에서는 코드가 틀렸을 때 거부합니다.

이 "거부"가 핵심입니다. 이걸 통해 모델의 지시 공간 밖으로 일을 옮기고, 모델이 구축하는 기반 구조로 옮길 수 있거든요. 모델한테 몇백 토큰을 써서 규칙을 기억하라고 애원하는 대신, 코드 구조 자체를 이렇게 짜는 거예요:

1. 가장 중요한 속성을 고르고
2. 기계가 검증할 수 있는 형태로 표현하고
3. 구현에 투영한 다음
4. 그 검증을 통과할 때까지 루프를 돌립니다

이게 "backpressure"가 강력한 이유입니다. Geoff Huntley의 Ralph나 "Don't Waste Your Backpressure" 에세이에서 말하는 그 개념 말이에요. 이전 오류가 다음 반복으로 흘러들어올 때, 결정론적 제약은 루프에 "느낌"보다 훨씬 단단한 뭔가를 줍니다. 이제 이런 루프는 더 이상 틈새 아이디어가 아닙니다. OpenAI의 Codex CLI가 `/goal` 기능을 탑재했으니까요. OpenAI 자신이 만든 Ralph 루프인데, 목표를 여러 턴에 걸쳐 유지하고 만족할 때까지 멈추지 않는 방식이죠.

## 기반 구조의 변경

지켜야 할 규칙들은 보통 정확하게 표현하기 쉽습니다. "사용자는 인증되었고, 테넌트의 멤버이며, 리소스가 그 테넌트에 속할 때만 접근할 수 있다." 이건 완전하고 경계가 명확한 규칙입니다. 그런데 이걸 영어로 집행하려고 하면 안 돼요.

Shen-Backpressure는 **Shen**이라는 정적 타입의 작은 Lisp 방언을 사용합니다. 시퀀트 계산법(sequent calculus) 타입 시스템이 있죠. 이걸로 모델이 작성해야 하는 대상 언어의 타입, 생성자, 게이트 명령으로 투영할 수 있는 형태의 규칙을 쓸 수 있어요. 스펙을 한 번만 정의하면, 코드 생성기(shengen)가 이를 타겟 언어의 가드 타입으로 낮춥니다. Go나 TypeScript를 작성하는 모델은 Shen이 존재한다는 걸 알 필요가 없어요. 코드만 컴파일 되고 게이트만 통과하면 되니까요.

## 멀티테넌트 API의 증명 체인

다음은 멀티테넌트 API 데모의 핵심인데, `specs/core.shen`에서 발췌한 것입니다:

```shen
(datatype jwt-token 
  X : string; 
  (not (= X "")) : verified; 
  ============================ 
  X : jwt-token;)

(datatype tenant-access 
  Principal : authenticated-principal; 
  Tenant : tenant-id; 
  IsMember : boolean; 
  (= IsMember true) : verified; 
  ================================ 
  [Principal Tenant IsMember] : tenant-access;)

(datatype resource-access 
  Access : tenant-access; 
  Resource : resource-id; 
  IsOwned : boolean; 
  (= IsOwned true) : verified; 
  ================================ 
  [Access Resource IsOwned] : resource-access;)
```

가운데 그어진 선이 작동의 핵심입니다. 선 위의 모든 전제가 만족되어야만 선 아래의 결론이 성립하거든요.
