## 핵심 개념

> [!abstract]
> 개발자들이 OpenAI, Anthropic 같은 클라우드 AI API에 과도하게 의존하는 현실을 비판하고, 기기 내 로컬 AI 모델 활용의 중요성을 주장합니다. Apple의 온디바이스 AI API와 구조화된 출력 방식을 활용한 신뢰할 수 있는 소프트웨어 개발을 예시로 제시합니다.

## 상세 내용

- 클라우드 AI 의존은 프라이버시, 네트워크 안정성, 비용 문제를 야기하며 현재 기기의 성능이 이를 충분히 커버할 수 있다
- 로컬 AI는 사용자 데이터 변환 작업에 최적화되며 2000단어 규모의 개인정보 정책보다 신뢰 구축에 효과적이다
- Swift struct 기반의 타입 정의로 구조화된 AI 출력을 얻으면 UI 렌더링이 안정적이고 일관성 있어진다

> [!tip] 왜 중요한가
> 개발자가 클라우드 AI 의존성을 줄이고 로컬 모델을 활용하면 더 빠르고, 비용 효율적이며, 사용자 신뢰도 높은 애플리케이션을 구축할 수 있습니다.

## 전문 번역

# 클라우드 AI는 선택지일 뿐, 필수가 아니다

요즘 소프트웨어 개발 트렌드를 보면 OpenAI나 Anthropic의 API를 마구 가져다 쓰는 게 대세인 것 같습니다. 이런 기능들이 사용자에게 실질적인 가치를 주는지는 논쟁의 여지가 있지만, 제가 정말 이야기하고 싶은 건 클라우드 기반 AI 모델에 의존하는 게 얼마나 위험한지에 관한 거예요.

## 편의성이 만드는 취약한 소프트웨어

이런 식의 개발 관성이 만들어내는 건 매우 취약하고, 개인정보를 침해하며, 근본적으로 깨진 소프트웨어들입니다. 서버가 다운되거나 신용카드 결제가 실패하는 순간 작동을 멈추는 애플리케이션들 말이죠.

우리는 로컬 기기가 일을 처리하는 소프트웨어를 만드는 방식으로 돌아가야 합니다. 주머니 속의 칩셋이 10년 전의 어떤 것보다도 엄청나게 빠른데, 버지니아의 서버팜에서 JSON 응답을 기다리는 동안 Neural Engine은 거의 유휴 상태로 놀고만 있거든요. 이게 말이 되나요?

의도가 아무리 순수하더라도, 사용자 콘텐츠를 제3자 AI 서비스로 스트리밍하는 순간 제품의 성격 자체가 변합니다. 이제 데이터 보관 기간 문제가 생기고, 그에 따라 동의, 감시, 보안 침해, 정부 요청, 학습 데이터 활용 등 온갖 복잡한 것들이 따라옵니다.

그뿐만 아니라 스택까지 엄청 복잡해져요. 네트워크 상태, 외부 공급자의 가동률, 속도 제한, 계정 결제, 자신의 백엔드 상태 등 수많은 변수에 기능이 의존하게 됩니다. 결국 간단한 UX 기능을 분산 시스템으로 만들어서 돈까지 지불하게 되는 거죠.

**"AI 어디든지"가 목표가 아닙니다. 유용한 소프트웨어가 목표예요.**

## 현실의 사례: 로컬 AI로 구현한 뉴스 앱

몇 년 전 저는 1990년대 스타일 웹에서 영감을 얻은 뉴스 수집 서비스 "The Brutalist Report"를 만들었어요. 최근에 이 서비스를 위한 네이티브 iOS 클라이언트를 개발했는데, 높은 밀도의 뉴스 읽기 경험을 목표로 설계했습니다. 헤드라인을 깔끔한 목록으로 보여주고, 웹의 온갖 악의적인 요소들을 제거한 리더 모드를 제공하며, 선택적으로 기사를 요약해주는 "인텔리전스" 뷰를 포함했죠.

여기서 중요한 건데요: **이 요약은 Apple의 로컬 모델 API를 사용해서 기기에서 생성됩니다.** 서버로의 우회 경로가 없고, 프롬프트나 사용자 로그를 남기지 않으며, 벤더 계정도 필요 없고, "우리는 당신의 콘텐츠를 30일 동안 보관합니다"라는 각주도 없습니다.

AI 기능이라고 하면 모두 서버에서 처리되는 게 당연하다고 생각하는 분위기가 지배적인데, 업계 전체가 이걸 바꿔야 해요.

## 언제 클라우드가 필요한가

물론 세상의 모든 AI 사용 사례가 로컬 처리만으로 충분한 건 아닙니다. 클라우드 기반 모델이 제공할 수 있는 수준의 지능이 정말 필요한 경우도 있거든요. 하지만 여러분이 풀려고 하는 모든 문제가 그런 건 아닐 겁니다. 우리는 이 점을 진지하게 생각해야 합니다.

## Apple 생태계의 로컬 AI 도구들

Apple 생태계에서 사용 가능한 도구에 대해서만 말씀드릴 수 있는데, 지난해 Apple은 개발자들이 기기 내 AI 모델을 쉽게 사용할 수 있도록 적극 투자했습니다.

기본적인 흐름은 대략 이런 식입니다:

```
사용자가 기사 읽음
↓
기기 내 로컬 모델로 요약 생성
↓
구조화된 결과 반환
```

긴 콘텐츠의 경우, 순수 텍스트를 청크 단위로 나누고(청크당 약 10,000자), 각 청크별로 "사실만" 정리한 노트를 생성한 뒤, 두 번째 패스에서 이들을 합쳐 최종 요약을 만듭니다.

이게 바로 로컬 모델이 빛나는 작업입니다. 입력 데이터는 이미 기기에 있고(사용자가 방금 읽었으니까), 출력은 가볍고, 빠르며, 개인정보를 지킵니다. 박사 수준의 초지능일 필요도 없어요. 방금 로드한 페이지를 요약하는 거지, 세상의 모든 지식을 발명하는 게 아니니까요.

**로컬 AI는 사용자 소유의 데이터를 변환하는 게 역할일 때 진가를 발휘합니다.** 우주 모든 것을 검색하는 엔진이 아니라는 뜻이죠.

## 신뢰를 다시 정의하기

사람들이 원하면서도 신뢰하지 못하는 AI 기능들이 많아요. 이메일 요약, 메모에서 액션 아이템 추출, 문서 분류 등등 말이에요.

클라우드 방식으로 하면 이 모든 게 신뢰 게임이 됩니다. "우리 서버로 데이터를 보내주세요. 약속할게요, 잘 다루겠다고요."

로컬 AI는 이를 바꿉니다. 기기가 이미 데이터를 가지고 있으니까요. 여기서 일을 처리하면 됩니다.

사용자 신뢰를 쌓는 방법은 2,000단어짜리 개인정보처리방침을 쓰는 게 아닙니다. **처음부터 그런 정책이 필요 없게 만드는 거예요.**

## 구조화된 출력의 힘

Apple의 최고의 움직임 중 하나는 "AI 출력"을 비구조화된 텍스트 덩어리에서 타입이 정의된 데이터로 옮긴 겁니다.

예전 방식이 "모델한테 JSON을 달라고 하고 기도하기"였다면, 새로운 패턴은 원하는 것을 나타내는 Swift 구조체를 정의합니다. 각 필드마다 자연어로 모델을 안내하고, 그 타입의 인스턴스를 생성하도록 요청하는 거죠.

개념적으로는 이렇습니다:

```swift
struct ArticleSummary: Codable {
    let keyPoints: [String]
    let sentiment: String
    let actionItems: [String]
}

// 모델에게 가이드 제공
let summary = try await model.generate(ArticleSummary.self, 
    prompt: "이 기사를 요약해주고...")
```

이제 UI에서 Markdown에서 불릿 포인트를 긁어내거나 모델이 JSON 스키마를 기억하길 기도할 필요가 없어요. 실제 타입의 실제 필드를 얻고, 일관되게 렌더링할 수 있습니다. 앱이 실제로 사용할 수 있는 구조화된 출력을 얻는 거고, 이 모든 게 로컬에서 실행됩니다!

이건 단순한 사용성 개선이 아니라 **엔지니어링 차원의 향상**입니다. 로컬 우선 앱을 만들고 있다면, 이게 바로 "AI는 유행일 뿐"과 "AI는 핵심 기능"의 차이를 만듭니다.
