## 핵심 개념

> [!abstract]
> Let's Encrypt는 Merkle Tree Certificates(MTC)를 통해 포스트 양자 암호화에 안전한 Web PKI 구축 계획을 발표했습니다. MTC는 배치 서명으로 여러 인증서를 커버하여 현재의 Web PKI보다 작은 핸드셰이크로 포스트 양자 보안을 제공하며, Certificate Transparency를 기본으로 내장합니다.

## 상세 내용

- ML-DSA 등 포스트 양자 서명은 크기가 크므로 (2,420 바이트) 기존 방식으로 도입하면 TLS 핸드셰이크가 10KB를 초과하여 실제 네트워크에서 연결 실패 위험 증가
- Merkle Tree Certificates는 배치 서명으로 여러 인증서를 한 번에 커버하여 핸드셰이크 크기를 줄이면서도 포스트 양자 보안 제공
- Certificate Transparency가 발급 자체에 내장되어 기존의 별도 로깅 메커니즘 불필요

> [!tip] 왜 중요한가
> 2030년 이후 양자 컴퓨터 위협에 대비하면서 웹의 성능을 유지하는 것이 중요한 과제이며, MTC는 실제 배포 가능한 해결책입니다.

## 전문 번역

# Let's Encrypt의 양자 내성 미래

**Andrew Gabbitas | 2026년 6월 3일**

Let's Encrypt는 양자 컴퓨터에 안전한 웹 PKI를 구축하겠다는 약속을 지키고 있습니다. 우리가 추진하는 방법은 Merkle Tree Certificates(이하 "MTC")인데요. 이건 TLS를 보편화한 속도와 신뢰성을 포기하지 않으면서도 양자 컴퓨터 시대에 대비한 인증을 더하는 새로운 접근방식입니다.

이 글에서는 MTC 도입 계획과, 왜 우리가 이것이 양자 내성 미래의 핵심이라고 믿는지 설명하겠습니다.

## 갈수록 급해지는 문제

지난 몇 년간 양자 암호화 논의의 중심은 주로 "암호화"였습니다. 논리는 단순했거든요. 오늘날 누군가가 암호화된 통신을 기록했다가, 나중에 양자 컴퓨터가 그 암호를 깰 때까지 기다렸다가 과거 데이터를 복호화할 수 있다는 우려였죠.

반면 TLS의 인증 부분(서버가 자신이 맞다는 것을 증명하는 부분)은 덜 긴급한 문제로 취급받았습니다. 양자 컴퓨터가 서명을 위조하려면 실시간으로 해야 하지, 과거의 서명을 소급해서 위조할 필요는 없거든요. 따라서 인증에 대한 위협은 암호학적으로 관련 있는 양자 컴퓨터(CRQC)가 실제로 등장했을 때만 현실화됩니다.

그런데 이 안도감이 서서히 사라지고 있습니다. 미국의 NSA는 2022년부터 CNSA 2.0 스위트를 통해 국가 보안 시스템들이 2030~2035년 사이에 양자 내성 알고리즘으로 전환하도록 지시했고, NIST의 권고안에서는 2030년 이후 RSA-2048과 P-256을 단계적으로 폐기할 예정입니다. EU도 2030년 말까지 고위험 시스템부터 시작해서, 2035년까지 전반적인 전환을 목표로 하고 있습니다.

이런 지침들이 웹 PKI를 직접 구속하지는 않지만, 웹 PKI가 의존하는 벤더들, 라이브러리들, 표준화 기구들이 이미 움직이고 있는 10년 단위의 타임라인을 만들어냈습니다.

올해 들어 이 타임라인이 더 앞당겨졌습니다. Google이 2029년까지 자사 서비스를 양자 내성 알고리즘으로 전환하겠다고 발표했고, CRQC 출현 시기 추정이 점점 앞당겨지고 있다는 이유를 들었습니다. Cloudflare도 같은 약속을 따랐죠. 더불어 Go 1.27이 NIST 표준 양자 내성 서명 방식인 ML-DSA를 표준 라이브러리에 추가한 것도, 양자 내성 서명이 이제 실용적인 인프라 수준에 도달했다는 신호입니다.

웹 PKI 생태계가 더 이상 양자 내성 인증을 미룰 수 없다는 게 분명해졌습니다. 루트 인증 기관이나 코드 서명 키, 신원 인증 시스템처럼 오래 유지되는 키들은 특히 좋은 공격 대상이거든요. 새로운 기술이 광범위하게 채택되려면 시간이 오래 걸리니까, 지금부터 준비해야 합니다.

## 웹 PKI가 마주한 독특한 문제

웹 PKI에 양자 내성 서명을 도입하기는 정말 까다롭습니다. 바로 "크기" 때문입니다.

NIST 표준 양자 내성 서명 방식 중 상대적으로 작은 ML-DSA-44만 해도 서명 크기가 약 2,420바이트입니다. 현재 웹 PKI에서 쓰는 알고리즘들과는 비교가 안 되죠. RSA-2048 서명은 256바이트, ECDSA-P256 서명은 64바이트에 불과합니다. 공개 키도 마찬가지인데요. ML-DSA-44는 1,312바이트인 반면, RSA-2048은 256바이트, ECDSA-P256은 64바이트입니다.

일반적인 웹 PKI 핸드셰이크는 5개의 서명과 2개의 공개 키를 담습니다. 만약 이것들을 모두 ML-DSA로 바꾼다면, 단 하나의 TLS 핸드셰이크가 10킬로바이트를 훨씬 넘어갈 겁니다. Cloudflare의 연구에 따르면 이 정도 크기면 실제 네트워크에서 상당한 비율의 TLS 연결이 실패하고, 나머지도 훨씬 느려집니다.

더 큰 핸드셰이크는 실패하는 연결뿐 아니라 모든 TLS 연결에 영향을 미칩니다. 제한된 대역폭, 느린 연결 속도, 사용자 경험 악화로 이어지는데, 이 모든 대가를 치르면서 얻는 건 아직 나타나지도 않은 위협으로부터의 보안입니다. 이건 기본 설정으로 활성화하기에는 너무 큰 대가입니다. 그런데 웹 규모의 보안은 결국 이런 기본 설정에 의해 결정되거든요.

## Merkle Tree Certificates

지난 1년간 새로운 설계 방식이 떠올랐는데, 바로 **Merkle Tree Certificates(MTC)**입니다. 우리는 이것이 양자 내성 시대의 웹 PKI를 위한 강력한 해결책이라고 믿습니다.

MTC는 인증서를 하나씩 발급하고 각각에 서명하는 방식 대신, 인증 기관이 인증서를 배치로 발급하고 그 배치 전체를 하나의 서명으로 커버합니다. 브라우저는 TLS 핸드셰이크와 분리되어, 이 배치 서명(일명 "랜드마크")을 따로 업데이트하고 관리합니다.

일반적인 경우, MTC 핸드셰이크의 전체 인증 경로는 서명 1개, 공개 키 1개, 포함 증명(inclusion proof) 1개로 이루어집니다. 양자 내성 알고리즘을 사용함에도 불구하고, 오늘날의 웹 PKI 핸드셰이크보다 더 작습니다. 또 다른 형태인 "독립형"은 클라이언트의 랜드마크가 만료되었을 때 폴백으로 작동하며, 조금 더 큰 핸드셰이크를 사용합니다.

MTC의 장점은 크기 최적화뿐만이 아닙니다. 모든 인증서가 공개된 Merkle 트리의 일부이기 때문에, 투명성이 발급 과정 자체에 내장됩니다. 현재의 Certificate Transparency 생태계는 사후에 덧붙여지는 형태거든요. CA가 인증서를 발급하고, 나중에 별도로 로그에 기록하며, TLS 핸드셰이크에 그 로깅을 증명하는 추가 서명을 싣는 식입니다. MTC에서는 인증서가 Merkle 트리 밖에 존재할 수 없습니다. Certificate Transparency가 기본으로 탑재되는 겁니다.

이건 Let's Encrypt에겐 새로운 영역이 아닙니다. 우리는 2019년부터 Certificate Transparency 로그를 운영해왔으니까요.
