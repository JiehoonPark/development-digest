## 핵심 개념

> [!abstract]
> AI 앱 빌더를 위한 오픈소스 백엔드로, Docker와 Go로 구현된 다중 테넌트 샌드박스 인프라를 제공합니다. 한 대의 머신에서 격리된 개발 환경, AI 코딩 에이전트, 라이브 미리보기 URL을 지원합니다.

## 상세 내용

- Kubernetes 없이 Docker 기반으로 다중 테넌트 샌드박스 환경 운영 가능
- 유휴 환경 자동 정지로 메모리 절약하고 요청 시 자동 재시작 지원
- SQLite + Traefik + Go로 구성된 가벼운 아키텍처로 오후 동안 전체 코드 이해 가능

> [!tip] 왜 중요한가
> Lovable, Bolt, v0 같은 AI 앱 빌더를 만들려는 개발자가 인프라 레이어를 직접 구축할 필요 없이 바로 서비스 구현에 집중할 수 있습니다.

## 전문 번역

# sandboxed: AI 앱 빌더를 위한 오픈소스 백엔드

## sandboxed가 뭐죠?

"투두 앱을 만들어줄래?"라고 입력하면 몇 초 뒤 완성된 웹사이트가 나타나는 서비스들 말이에요. Lovable, Bolt, v0, Replit 같은 거 봤나요? sandboxed는 바로 그런 서비스들을 가능하게 하는 오픈소스 백엔드입니다. 자신의 서버에서 직접 운영할 수 있어요.

간단히 말하면, HTTP 요청 하나를 보내면 sandboxed가 이런 일들을 합니다:

**샌드박스 생성** — 독립된 Linux 컨테이너를 띄워요. 각 사용자는 자신의 파일 시스템과 메모리만 쓰고, 다른 사용자와는 완전히 격리됩니다.

**AI 코딩 에이전트 실행** — 샌드박스 안에서 AI가 코드를 작성합니다. OpenCode나 Claude Code CLI가 미리 설치돼 있어요.

**라이브 URL 제공** — 샌드박스 안의 개발 서버가 즉시 공유 가능한 미리보기 링크에서 접근 가능합니다.

```
POST /sandbox → 격리된 컨테이너 시작
POST .../tasks → AI가 앱을 작성
http://<id>.preview... → 완성된 앱이 라이브
```

비용 효율도 좋아요. 사용하지 않는 샌드박스는 자동으로 멈춰서 메모리를 해제하고, 누군가 링크를 열면 바로 깨어나니까요. 파일은 내내 디스크에 저장되어 있으니 데이터도 안전합니다. 덕분에 한 대의 일반 서버로도 많은 사용자의 샌드박스를 동시에 운영할 수 있어요.

## 구조는 얼마나 간단한가?

Go 프로그램 하나가 Docker를 제어하고, Traefik이 URL 라우팅을, SQLite가 데이터베이스 역할을 합니다. Kubernetes도 없고, 별도의 데이터베이스 서버도 없고, 메시지 큐도 없어요. 오후 하나면 코드 전체를 읽을 수 있을 정도로 작습니다.

```
┌──────────────── 호스트 (Docker만 필요) ────────────────┐
브라우저 ──▶│ Traefik ──▶ 샌드박스 (코딩 에이전트 + dev 서버 :3000) │
│ ▲ ▲ ▲ │
API/CLI ──▶│ sandboxd ─────────┘ └─ 작업 디렉토리 (유지됨) │
│ │ SQLite (신원) · 유휴→정지 · 요청→시작 │
└─────┴────────────────────────────────────────────────────┘
```

## 누가 써야 할까?

**적합한 경우:**
- AI 앱 빌더 (사용자의 설명을 받아 앱을 만들어줌)
- 에이전트 플랫폼
- 코딩 플레이그라운드
- 사용자별 또는 브랜치별 미리보기 환경
- 팀을 위한 멀티앱 호스팅

**불필요한 경우:**
- 자신용 컨테이너 1~2개만 필요하면? 쉘 스크립트나 `docker run`, lxd 쓰는 게 낫습니다.

## 왜 sandboxed인가?

AI 앱 빌더나 에이전트 플랫폼을 만들 때 진짜 어려운 부분은 프롬프트가 아니라 그 아래의 인프라거든요:

- 한 사용자의 코드가 다른 사용자의 것을 건드리지 않도록 완전히 격리하기
- 각 사용자마다 고유한 미리보기 URL과 자동 라우팅, TLS 관리
- 비용 제어 — 유휴 환경이 메모리를 해제해야 요금이 터지지 않음
- 에이전트 오케스트레이션 — 코딩 에이전트를 작업 디렉토리에서 실행하고 진행 상황을 스트리밍, 결과 캡처
- 지속성, 온디맨드 시작, 충돌이나 재부팅 후 복구

이런 게 몇 개월의 플랫폼 작업이에요. sandboxed는 바로 그런 플랫폼을 단 하나의 명령어로 줄여놨습니다.

**⚡ 한 명령어로 설치** — `./install.sh` 하나면 API와 미리보기가 준비됩니다.

**🧠 에이전트 포함** — 모든 샌드박스에 OpenCode와 Claude Code CLI가 들어가 있어요.

**💸 밀도 높은 설계** — 유휴 정지와 온디맨드 시작 덕분에 서너 개의 샌드박스가 한 서버에서 돌아갑니다. $2,000 클러스터가 필요할 일이 $20 서버로 충분해져요.

**🔓 당신의 것** — 셀프호스팅, MIT 라이선스, 공급사 종속성 없음. 데이터, 마진, 로드맵 모두 당신 것입니다.

**🪶 의도적으로 단순함** — SQLite + Docker CLI + Traefik. 재조정자가 부팅할 때마다 Docker 상태를 데이터베이스로 맞춥니다. 제어 평면 전체를 오후에 읽을 수 있어요.

## "그냥 쉘 스크립트는 왜 안 되나?"

좋은 질문입니다. 솔직히 말하면, 자신용 장기 컨테이너 1~2개면 쉘 스크립트나 `docker run`, lxd가 더 간단해요. 진짜 그렇습니다. sandboxed는 그런 경우엔 오버킬입니다.

그런데 많은 사용자를 위해 많은 샌드박스를 운영하게 되면? 팀이거나 서비스라면? 그때가 되면 깔끔한 작은 스크립트가 조용히 거대해집니다:

**URL이 필요함** — 각 샌드박스마다 깔끔한 미리보기 URL, 자동 라우팅, TLS. 포트 관리도 없고 충돌도 없어야 함.

**스스로 잠들고 깬다** — 유휴 샌드박스가 멈춰서 RAM을 해제하고, 다음 요청에 투명하게 재시작 (워밍업 페이지, readiness probe, 요청 대기). 이것만 해도 100줄 훨씬 넘음. 그리고 이게 저가형 서버 하나와 항상 켜져 있는 VM 랙의 차이를 만듭니다.

**재부팅을 견딘다** — SQLite가 신원이고, 재조정자가 부팅 시 Docker를 다시 맞춥니다. 스크립트는 호스트가 재시작되면 모든 걸 잊어버려요.

**API인지, CLI가 아니다** — create, exec, stop, destroy, write-files, run-agent-task가 진짜 HTTP 엔드포인트로 인증 지원. 앱 백엔드에서 사용자별로 대규모 호출합니다.

**한 사용자가 나머지를 망칠 수 없다** — 샌드박스별 메모리/PID 제한과 호스트 메모리 부하 해제 메커니즘.
