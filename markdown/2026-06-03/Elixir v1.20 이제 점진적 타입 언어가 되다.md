## 핵심 개념

> [!abstract]
> Elixir v1.20에서 타입 추론과 점진적 타입 검사 기능을 도입하여, 개발자의 타입 어노테이션 작성 없이도 데드 코드와 검증된 버그를 자동으로 발견합니다. 집합론 기반의 타입 시스템을 통해 거짓 양성을 극히 낮추면서도 정적 타입 시스템과 동적 타입의 이점을 모두 제공합니다. 'If T' 벤치마크에서 13개 카테고리 중 12개를 통과하여 동적 프로그래밍 언어에서 정확한 타입 정보를 복구할 수 있음을 입증했습니다.

## 상세 내용

- 타입 어노테이션 없이 자동 타입 추론 수행: Elixir v1.20은 개발자가 명시적인 타입 선언을 하지 않아도 타입 추론과 타입 검사를 수행하며, 이를 통해 데드 코드와 검증된 버그(런타임 실행 시 반드시 실패하는 타입 위반)를 발견합니다.
- dynamic() 타입의 혁신적 설계: 기존 any() 타입과 달리 dynamic() 타입은 호환성(compatibility)과 좁혀나가기(narrowing) 두 가지 특성을 가지며, 정적 타입과 동적 타입의 경계에서 거짓 양성을 최소화합니다.
- 호환성 속성으로 검증된 버그만 보고: dynamic(integer() | binary()) 타입의 값이 정수를 기대하는 / 연산자에 전달될 때는 교집합이 존재하므로 위반으로 보지 않지만, map만 받는 Map.fetch!에 전달될 때는 교집합이 없어 위반으로 보고합니다.
- 타입 좁혀나가기로 정확한 타입 복구: Elixir의 동적 타입은 제어 흐름(if/case 등)을 통해 좁혀나갈 수 있으며, 이는 조건문 내에서 가능한 타입을 정확히 추론하게 하여 검증된 버그를 효과적으로 발견합니다.
- 'If T' 벤치마크에서 높은 성능: Elixir는 13개 타입 좁혀나가기 벤치마크 카테고리 중 12개를 통과하여 동적 타입 언어에서도 정적 타입 시스템 수준의 정밀한 타입 정보를 복구할 수 있음을 증명합니다.
- 집합론 기반의 개발자 친화적 설계: 합집합(union), 교집합(intersection), 부정(negation) 같은 기본 집합 연산으로 타입을 표현하며, 명확한 에러 메시지를 제공하여 학습 곡선을 낮춥니다.
- 점진적 도입으로 기존 코드베이스 보호: 첫 번째 마일스톤에서 타입 어노테이션 도입을 미루고 타입 추론만 활성화하여 기존 Elixir 코드의 신뢰성을 잃지 않으면서 타입 시스템의 이점을 제공합니다.

> [!tip] 왜 중요한가
> 동적 언어인 Elixir에 정적 타입 검사의 이점을 점진적으로 도입하여 기존 코드 호환성을 유지하면서 런타임 에러를 사전에 방지할 수 있게 되어, 대규모 프로젝트의 안정성과 유지보수성이 크게 향상됩니다.

## 전문 번역

# Elixir v1.20 출시: 점진적 타입 언어로 진화하다

2022년에 Elixir에 집합론적 타입을 추가하려는 계획을 발표했습니다. 이듬해 2023년 6월에는 Elixir 타입 시스템 설계에 관한 수상작 논문을 발표했고, 그때부터 연구 단계에서 개발 단계로 전환했습니다.

Elixir v1.20에서는 첫 번째 개발 마일스톤을 완료했습니다. 타입 주석 없이도 모든 Elixir 프로그램의 타입을 추론하고 점진적으로 타입을 검사할 수 있게 된 것이죠. 이제 Elixir는 데드 코드와 검증된 버그(실행 시 반드시 실패할 타입 위반)를 점점 더 정확하게 찾아냅니다. 개발자의 부담을 늘리지 않으면서도 매우 낮은 오탐(false positive) 비율로 기존 프로그램의 버그를 효율적으로 발견할 수 있다는 뜻입니다.

이번 발표에서는 타입 시스템의 목표, Elixir의 `dynamic()` 타입이 무엇인지, 그리고 검증된 버그를 어떻게 찾아내는지 설명하겠습니다. 특히 우리의 구현은 "If T: Benchmark for Type Narrowing" 벤치마크에서 13개 카테고리 중 12개를 통과하며 우수한 성능을 보여줍니다. 이는 Elixir가 일반적인 Elixir 코드에서 정확한 타입 정보를 복구할 수 있으며, 이를 통해 동적 타입 프로그램에서 검증된 버그를 찾을 수 있음을 보여줍니다.

이 타입 시스템은 CNRS와 Remote 간의 파트너십으로 가능했습니다. 현재 개발 작업은 Fresha와 Tidewave가 후원하고 있습니다.

## Elixir에도 타입이 있다고?

우리가 목표로 삼은 타입 시스템은 다음과 같은 특징을 갖춰야 합니다:

**건전성(Sound)** - 타입 시스템이 추론하고 할당하는 타입이 프로그램의 실제 동작과 일치합니다.

**점진성(Gradual)** - Elixir의 타입 시스템에는 `dynamic()` 타입이 포함되어 있습니다. 변수나 표현식의 타입을 런타임에 검사해야 할 때 이를 사용할 수 있습니다. `dynamic()`이 없으면 Elixir의 타입 시스템은 정적 타입처럼 동작합니다.

**개발자 친화적(Developer Friendly)** - 타입을 기본 집합 연산(합집합, 교집합, 부정)을 통해 설명하고 구현하며 조합합니다(따라서 집합론적 타입 시스템입니다). 또한 명확한 에러 메시지를 제공합니다.

기존 언어에 타입 시스템을 도입하는 것은 복잡한 변화입니다. 그래서 첫 번째 마일스톤은 타입 주석을 도입하지 않으면서도 개발자에게 가치를 제공하는 것이었습니다. 데드 코드와 검증된 버그를 찾아내는 방식으로요. 이것이 바로 Elixir의 `dynamic()` 타입이 하는 역할인데, 다른 점진적 타입 언어와는 상당히 다릅니다. 자세히 살펴보겠습니다.

## dynamic() 타입

많은 점진적 타입 시스템에서는 `any()` 타입을 가지고 있습니다. 이 타입은 타입 시스템 관점에서 보면 "뭐든 괜찮다"는 의미이며, 타입 위반이 보고되지 않습니다. 반면 Elixir의 점진적 타입은 `dynamic()`이라고 부르며, 두 가지 중요한 특성이 있습니다: 호환성과 좁혀내기입니다.

정적 타입 시스템에서는 `integer()` 또는 `binary()` 타입을 가진 값으로 함수를 호출할 때, 그 함수는 두 타입을 모두 받아들일 수 있어야 합니다. 하지만 타입 시스템이 모든 프로그램의 의도를 완벽하게 포착할 수 없기 때문에, 이는 오탐을 유발할 수 있습니다. 다음의 간단한 코드를 살펴보세요:

```elixir
def percentage_or_error(value) when is_integer(value) do
  value_or_error =
    if value > 1 do
      value
    else
      "not well"
    end
  # ... more code ...
  if value > 1 do
    value_or_error / 100
  else
    String.upcase(value_or_error)
  end
end
```

`value_or_error`는 `integer()` 또는 `binary()` 타입을 가지고 있습니다. `/` 연산자는 숫자만 받고, `String.upcase`는 바이너리/문자열만 받습니다. 위 프로그램은 런타임에 예외를 발생시키지 않으므로 유효한 코드입니다. 하지만 타입 시스템은 여전히 두 군데서 위반을 보고할 겁니다. `/`과 `String.upcase`에 제공된 타입이 수용 가능한 타입의 서브타입이 아니기 때문이죠.

위 프로그램은 더 나은 방식으로 작성하면 타입 위반이 없게 할 수 있지만, 타입 시스템은 항상 유효한 프로그램을 거부하기도 합니다. 만약 Elixir가 기존 코드베이스에서 너무 많은 오탐을 일으킨다면, 타입 시스템에 대한 신뢰가 빠르게 무너질 것입니다. 따라서 Elixir의 점진적 타입 시스템은 위의 `value_or_error` 변수를 `dynamic(integer() or binary())` 타입으로 태깅합니다. 이는 런타임에 `integer()` 또는 `binary()` 중 하나라는 뜻입니다.

`dynamic()` 타입으로 함수를 호출할 때, Elixir는 제공된 타입과 수용 가능한 타입이 교집합이 없을 때만 타입 위반을 보고합니다. 위 프로그램에서 `/`는 숫자만 기대하지만, `dynamic(integer() or binary())`는 `integer()`일 수 있고, 수용 가능한 타입과 제공된 타입이 서로소(disjoint)가 아니므로 타입 위반이 없습니다. 하지만 프로그램을 이렇게 바꾸면 어떻게 될까요:

```elixir
value_or_error =
  if value > 1 do
    value
  else
    "not well"
  end
Map.fetch!(value_or_error, :some_key)
```

`Map.fetch!`는 맵 데이터 구조를 기대하는데, `value_or_error`는 런타임에 정수 또는 바이너리일 수만 있습니다. 수용 가능한 타입과 제공된 타입이 서로소이므로, 이제 위반이 발생합니다. 이것을 호환성(compatibility) 속성이라고 부르며, Elixir가 검증된 버그만 보고하는 방식을 설명합니다.

그런데 검증된 버그만 보고하는 것은 실제로 많은 버그를 찾을 수 없다면 별로 유용하지 않습니다. 우리는 Elixir의 `dynamic()` 타입을 좁혀낼 수 있게 만들어 이 문제를 해결했습니다. 다음 코드를 보세요:
