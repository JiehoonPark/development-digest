## 핵심 개념

> [!abstract]
> CPU 캐시 아키텍처를 이해하고 데이터 구조 배치 최적화를 통해 같은 알고리즘 복잡도에서도 30배의 성능 향상을 달성할 수 있습니다. Array of Structs(AoS)와 Struct of Arrays(SoA) 패턴의 차이가 메모리 캐시 효율성에 미치는 영향을 분석합니다.

## 상세 내용

- 캐시 라인(64바이트) 크기와 시간적, 공간적 지역성을 고려한 데이터 배치가 성능에 큰 영향
- SoA 패턴으로 재구성하면 관련 필드들을 같은 캐시 라인에 집중시켜 캐시 미스 감소
- 하드웨어 특성 이해가 고성능 최적화의 핵심 (예: 1KiB 구조체에서 30배 개선 가능)

> [!tip] 왜 중요한가
> 알고리즘 복잡도뿐 아니라 데이터 레이아웃이 실제 성능에 결정적 영향을 미치므로, 메모리 계층 구조를 고려한 개발이 필수입니다.

## 전문 번역

# 캐시 라인을 이해하면 코드가 30배 빨라진다

저는 커리어의 대부분을 Java로 개발했습니다. 그러다 보니 자연스럽게 거대한 클래스를 만드는 습관이 생겼거든요. 새로운 기능이 필요하면 그냥 클래스에 메서드와 필드를 추가하는 식으로요.

그런데 각 필드가 성능에 미치는 영향을 거의 고려하지 않았습니다. 성능 최적화라고 하면 보통 알고리즘의 점근 복잡도(Big O)를 분석하는 데만 집중했거든요.

하지만 흥미로운 사실을 발견했습니다. O(N) 같은 단순한 반복문이라도, 하드웨어를 조금 더 깊이 이해하면 실행 시간이 극적으로 달라질 수 있다는 겁니다.

## 우리의 하드웨어를 알아보자

먼저 현재 머신의 캐시 라인과 페이지 크기를 확인해봅시다.

```
$ lscpu | grep -i cache
L1d cache: 352 KiB (10 instances)
L1i cache: 640 KiB (10 instances)
L2 cache: 10 MiB (5 instances)
L3 cache: 12 MiB (1 instance)

$ getconf LEVEL1_DCACHE_LINESIZE
64
```

여기서 instances 숫자는 캐시가 CPU들 사이에서 어떻게 공유되는지 보여줍니다. 만약 10개의 CPU가 있다면, 각 CPU는 자신만의 L1d 캐시를 가지지만, L2 캐시는 2개씩 공유하게 되는 식이죠.

중요한 건 **캐시 라인 크기가 64바이트**라는 것입니다.

```
┌─────────────────────────────────────────────┐
│ 64 bytes                                    │
│ byte 0 byte 1 byte 2 ... byte 63            │
└─────────────────────────────────────────────┘
```

메모리에서 단 1바이트를 읽더라도, CPU는 주변 64바이트를 모두 캐시 라인에 가져옵니다. 왜냐하면 현실의 데이터는 시간상으로나 공간상으로나 서로 가까이 있을 확률이 높기 때문입니다.

## 메모리 계층 구조

각 레벨의 접근 속도를 시각화하면 이렇습니다.

```
┌──────────────────────────────────────────────────────────────┐
│ CPU Core                                                     │
│ ┌───────────┐                                                │
│ │ Registers │ < 1 ns                                         │
│ └─────┬─────┘                                                │
│       ▼                                                       │
│ ┌───────────┐                                                │
│ │ L1d Cache │ ~35 KiB/core, ~1-2 ns, ~560 cache lines       │
│ └─────┬─────┘                                                │
│       ▼                                                       │
│ ┌───────────┐                                                │
│ │ L2 Cache  │ ~2 MiB/core-pair, ~4-5 ns, ~32,000 lines      │
│ └─────┬─────┘                                                │
│       ▼                                                       │
│ ┌───────────┐                                                │
│ │ L3 Cache  │ 12 MiB shared, ~10-15 ns, ~196,000 lines      │
│ └─────┬─────┘                                                │
│       ▼                                                       │
│ ┌───────────┐                                                │
│ │ DRAM      │ ~60-100 ns                                     │
│ └───────────┘                                                │
└──────────────────────────────────────────────────────────────┘
```

각 캐시의 크기는 `lscpu`의 결과를 코어 수로 나눠서 구합니다. 예를 들어 L1d는 352 KiB ÷ 10 instances = 약 35 KiB죠. 캐시 라인 개수는 이를 다시 64로 나누면 됩니다. 35 KiB ÷ 64 bytes = 560 cache lines.

## 실제로는 어떤 영향이 있을까?

구체적인 예를 들어봅시다. Monster 배열을 순회하면서 `is_alive` 필드만 뽑아내려고 합니다.

```c
struct Monster {
    uint32_t id;        // 4 bytes
    float x, y, z;      // 12 bytes
    float vx, vy, vz;   // 12 bytes
    int32_t hp;         // 4 bytes
    int32_t attack;     // 4 bytes
    int32_t defense;    // 4 bytes
    uint8_t is_alive;   // 1 byte
    uint8_t team;       // 1 byte
    char name[22];      // 22 bytes
};  // total: 64 bytes
```

배열을 순회할 때 각 캐시 라인에는 몬스터 1마리 전체가 들어갑니다. 하지만 우리가 필요한 건 `is_alive` 한 바이트뿐입니다.

```
cache line 0              cache line 1
┌──────────────────────┐  ┌──────────────────────┐
│ id0 x0 y0 z0...     │  │ id1 x1 y1 z1...     │
│ ...alive0 team0...  │  │ ...alive1 team1...  │
│         ▲           │  │         ▲           │
└────────┼────────────┘  └────────┼────────────┘
         │                        │
      필요함                    필요함
```

이런 방식을 **Array of Structs(AoS)**라고 부릅니다. 엄청 비효율적이죠.

## 구조를 역으로 생각해보자

그 대신 각 필드를 별도의 배열로 분리하면 어떨까요?

```
cache line 0
┌───────────────────────────────────────────────────────────────┐
│ alive0 alive1 alive2 alive3 ... alive62 alive63              │
│   ▲     ▲     ▲     ▲              ▲      ▲                 │
└───┼─────┼─────┼─────┼──────────────┼──────┼─────────────────┘
    │     │     │     │              │      │
    └─────┴─────┴─────┴──────────────┴──────┘
    한 번의 캐시 로드로 64개 모두 획득!
```

```c
// Struct of Arrays (SoA) 레이아웃
struct Monsters {
    uint32_t *ids;
    float *xs, *ys, *zs;
    float *vxs, *vys, *vzs;
    int32_t *hps;
    int32_t *attacks;
    int32_t *defenses;
    uint8_t *is_alives;  // 연속으로 배치됨
    uint8_t *teams;
    char (*names)[22];
};
```

이 방식을 **Struct of Arrays(SoA)**라고 부릅니다.

## 성능 개선 효과

정말로 얼마나 빨라질까요?

Monster 구조체가 1KiB 크기일 때, **30배의 성능 개선**을 관찰할 수 있습니다! 🤯

물론 구조체가 작으면 여러 개가 한 캐시 라인에 들어가서 차이가 덜하긴 합니다. 그래도 이건 매우 자주 실행되는 코드이고, CPU의 프리페처가 순차 접근을 감지해서 미리 다음 캐시 라인을 가져오기 때문에 실제로는 메모리 대기 시간이 거의 없게 됩니다.

## 무작위 접근은 어떨까?

하지만 모든 데이터 접근이 순차적인 건 아닙니다. 해시맵, 트리, 그래프 순회, 포인터 기반 자료구조 같은 경우 예측 불가능한 위치로 점프합니다. 이런 상황에서야말로 캐시 최적화가 진정한 가치를 발휘합니다.
