## 핵심 개념

> [!abstract]
> Anthropic의 Bun 인수 이후 Claude Code의 품질 저하와 제한 정책이 증가하고 있으며, 개발자는 Bun도 같은 경로를 따를 수 있다는 우려를 제기합니다. Bun 자체는 우수하지만 Anthropic의 정책 영향으로 인한 '품질 저하(enshittification)' 위험성을 경고하는 글입니다.

## 상세 내용

- Claude Code가 제한, 청구 이상, 예측 불가능한 동작 등으로 악화되고 있음
- Bun이 Claude Code에 내장되어 있어 Anthropic의 정책 변화에 영향을 받을 가능성
- VC 펀딩의 지속 가능성 문제 해결을 위한 인수였으나 제품 관리 측면에서 우려 증가

> [!tip] 왜 중요한가
> JavaScript 개발자들이 의존하는 Bun의 미래 방향과 정책 변화가 개발 환경에 직접 영향을 미칠 수 있습니다.

## 전문 번역

# Bun의 미래가 걱정되는 이유

Bun은 정말 좋은 소프트웨어입니다. 저는 거의 매일 사용하고 있어요. 빠르고 실용적이며, 개발팀이 정말 자주 업데이트를 배포합니다. TypeScript를 작은 스크립트, 앱, 테스트, 도구 작성에서 즐겁게 쓸 수 있게 해주죠. 그래서 요즘 상황이 더 답답한 겁니다.

저는 Bun이 성공하길 원합니다. Node.js의 진정한 대안이 나타나길 원해요. 더 빠른 설치, 더 빠른 테스트, 더 나은 번들링, 그리고 도구 생태계의 비대함 감소를 기대했거든요. 하지만 지금 Bun의 미래가 걱정됩니다.

## Anthropic이 Bun을 인수했습니다

2025년 12월, Anthropic이 Bun을 인수했습니다. 공식 발표 내용은 제가 듣고 싶던 말들로 가득했어요. Bun은 오픈소스로 남고, MIT 라이선스를 유지하며, 같은 팀이 계속 개발하고, 로드맵은 여전히 고성능 JavaScript 도구와 Node.js 호환성에 집중한다고 했습니다.

그리고 이렇게도 덧붙였습니다:

> Claude Code는 Bun 실행 파일로 수백만 사용자에게 배포됩니다. Bun이 망가지면 Claude Code도 망가집니다. Anthropic은 Bun을 훌륭하게 유지할 직접적인 인센티브를 갖게 됩니다.

당시에는 이 말이 정말 안심이 됐어요. Anthropic이 Bun 위에 대규모 제품을 구축했다는 건, 즉 Bun을 빠르고 안정적으로 유지할 이유가 생겼다는 뜻이니까요. 저는 지금도 이 논리가 타당하다고 생각합니다. 하지만 이제 균열이 보이고 있습니다.

Bun은 여전히 훌륭한 JavaScript 런타임입니다. 문제는 그것을 관리하는 회사가 자신들의 소프트웨어에 전혀 신경 쓰지 않는 것처럼 보인다는 거예요.

## Claude Code는 썩어가고 있습니다

이건 "Anthropic은 나쁜 회사다"라는 글이 아닙니다. 음, 완전히 그렇지만은 않다고 해야겠네요. 저는 여전히 Anthropic의 모델이 훌륭하다고 생각합니다. Claude Opus(아마 4.6 정도?)는 여전히 코딩, 글쓰기, 추론, 일반 개발 작업에 있어 가장 뛰어난 모델 중 하나입니다. 모델의 품질이 제 걱정사항은 아니에요. 제가 우려하는 건 모델 주변의 제품 레이어입니다. 오늘날 Claude Code는 정직하게 말해서 써먹기 힘들어졌습니다.

### Claude Code는 과거에 정말 좋았어요

1년 전만 해도 Claude Code는 정말 놀라웠습니다. 개발자 워크플로우가 주로 자동완성에서 에이전트 기반으로 변할 거라고 확신하게 해준 첫 AI 코딩 도구 중 하나였거든요. 프로젝트를 읽고, 정확한 편집을 하고, 명령어를 실행하고, 실수를 고치고, 계속 진행할 수 있었습니다. 개발자가 실제로 어떻게 일하는지 이해하는 사람들이 만든 도구처럼 느껴졌어요. Anthropic의 모델(최근까지는 GPT-5.5보다 좋았어요)과 함께하면, Claude Code는 무적처럼 느껴졌습니다.

12월에 Claude Code가 이미 악화되고 있었지만, 여전히 좋은 상태였고 그 때문에 Bun 인수가 의미 있다고 생각했어요. Anthropic이 코딩 도구의 미래를 만들고 있고, Bun이 그 도구의 런타임이라면, Bun이 최고의 보금자리를 찾은 건 아닐까 싶었거든요. 저는 항상 VC 자금을 받은 Bun이 어떻게 지속 가능한 비즈니스가 될 수 있을지 약간 걱정했어요. 그래서 이번 인수는 합리적으로 느껴졌고, 저는 낙관적이었습니다.

### 지금의 Claude Code는 정말 나쁩니다

지금 정말 훌륭한 코딩 에이전트가 많이 있어요. Cursor, Augment, Codex, OpenCode, T3 Code, Pi, 그리고 아마 더 많을 겁니다. 한동안 저는 Cursor를 주로 썼는데, Claude Code는 계속 악화되는데 Cursor(CLI)는 Anthropic 모델을 정말 잘 활용했거든요. 최근에 특별한 이유로 Cursor를 쓸 수 없게 됐는데, 한 2개월 동안 Claude Code를 안 쓰다가 다시 띄워봤어요. 그 결과에 정말 충격을 받았습니다.

2026년 4월, 개발자들은 Claude Code의 품질, 제한 동작, 서드파티 통합 제한, 혼란스러운 청구, 그리고 느린 소통에 대해 불평하기 시작했습니다.

Anthropic은 엔지니어링 포스트모템을 발표했어요. 그들은 기본 추론 노력 감소, 오래된 세션 버그, 그리고 코딩 품질을 해친 프롬프트 변경 등 제품 레이어 문제들을 지적했습니다. 저는 포스트모템을 높이 평가합니다. 아무 일도 없었다고 하는 것보다는 낫거든요. 솔직하게 말하면, 이게 아마 Anthropic이 스스로 뭔가 잘못했다고 언급한 첫 번째 경우였던 것 같습니다.

그 다음에 OpenClaw 사건이 터졌어요. TechCrunch 보도에 따르면 Anthropic이 Claude Code 구독자에게 OpenClaw 같은 서드파티 통합에 추가 요금을 내라고 했대요. 이미 충분히 나쁜 일이죠. 그런데 이후 일어난 일은 더 이상했습니다.

Gigazine의 보도에 따르면 git 히스토리에 단순히 OpenClaw가 있기만 해도 Claude Code가 요청을 거부하거나 추가 요금을 청구할 수 있다고 합니다. 그 기사에서 Theo는 JSON blob에 OpenClaw를 언급한 최근 커밋만으로도, 심지어 빈 저장소에서 `claude -p "hi"`를 직접 호출할 때도 이런 동작이 발생할 수 있다고 말했어요. 관심 있으신 분은 그 클립을 보시면 정말 놀랄 겁니다.

Theo의 해석, 그리고 제가 그럴듯하다고 생각하는 것은, 이건 마치 아무도 실제 코드 레벨의 경험을 신중하게 테스트해보지 않고 변경사항을 배포하는 제품처럼 보인다는 거예요. 물론 이건 부당할 수도 있어요. 저는 Anthropic에서 실제로 뭐가 일어나는지 모르니까요. 하지만 외부에서 보면, Claude Code는 잘못된 방향으로 움직이는 도구처럼 보여요. 더 많은 제한, 혼란스러운 청구, 커밋 텍스트를 기반으로 한 예상 밖의 동작.

이것이 바로 전형적인 **품질 악화(enshittification)**입니다.

## 그래서 Bun이 걱정되는 겁니다

Bun은 Claude Code에 내장되어 있어요. 그리고 Claude Code는 품질이 악화되고 있습니다. 따라서 이제 Bun도 같은 길을 갈 수 있을까 봐 걱정됩니다.

Bun이 나쁠 리가 없어요. Bun은 훌륭합니다. Bun 팀이 신경 쓰기를 멈췄을 거라고도 생각하지 않습니다. 하지만 문제가 있어요.

Bun과 그 팀이 Anthropic에 더 깊숙이 통합될수록, Anthropic의 정책도 함께 적용되기 마련입니다. Claude Code를 망친 그 정책들 말이에요. 그럼 Bun에서도 문제가 나타나기 시작할까요? 그런 식의 변화가 일어날까요?
