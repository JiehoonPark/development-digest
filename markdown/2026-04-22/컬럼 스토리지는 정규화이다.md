## 핵심 개념

> [!abstract]
> 컬럼 지향 데이터 저장소를 행 지향 데이터에 대한 극단적 정규화로 해석할 수 있으며, 이를 통해 쿼리 처리와 데이터 포맷 조작을 통일적으로 이해할 수 있습니다.

## 상세 내용

- 컬럼 스토리지는 각 컬럼을 기본 키(ordinal position)와 단일 속성을 가진 정규화된 테이블로 생각할 수 있습니다.
- 행 재구성은 실제로 컬럼 간 조인 연산이며, 이 관점은 전통적 쿼리 처리를 데이터 포맷 선택과 통합합니다.

> [!tip] 왜 중요한가
> 데이터베이스 설계자와 최적화 담당자가 컬럼 스토리지의 성능 특성을 더 깊이 있게 이해할 수 있습니다.

## 전문 번역

# 칼럼 지향 저장소는 정규화다

처음엔 잘 이해 못 했던 부분인데, 행 지향 데이터를 칼럼 지향으로 변환하는 과정이 데이터베이스 세계에서 완전히 새로운 개념은 아니라는 거예요. 사실 관계형 데이터베이스의 추상화 개념 범주에 여전히 속한다고 볼 수 있습니다.

예를 들어 이런 데이터가 있다고 해봅시다.

```javascript
data = [
{ "name": "Smudge", "colour": "black" },
{ "name": "Sissel", "colour": "grey" },
{ "name": "Hamlet", "colour": "black" }
]
```

이것은 관계형 데이터베이스의 테이블을 나타내고 있어요. 이 데이터에 접근하려면 여러 번 디스크를 읽어야 한다고 가정하겠습니다.

## 행 지향 저장소의 장점

이 표현 방식은 몇 가지 좋은 특성이 있습니다.

새로운 행을 추가하기 간단해요. 새로운 행을 만들어서:

```javascript
{ "name": "Petee", "colour": "black" }
```

기존 리스트 끝에 붙이기만 하면 됩니다. 디스크상에서도 몇 페이지만 건드리면 되거든요. 만약 칼럼이 엄청 많아서 행이 정말 넓다 하더라도, 이 좋은 특성은 변하지 않습니다.

특정 행을 조회하는 것도 마찬가지예요. 한 행의 모든 칼럼이 메모리상 서로 옆에 있으니까, 어디에 저장되어 있든 그 행을 매우 빠르게 가져올 수 있습니다.

그런데 반대로 애완동물 색상의 히스토그램을 만든다면? 그럼 실제로 필요 없는 데이터까지 엄청 많이 읽어야 합니다.

## 칼럼 지향 저장소

칼럼 지향 표현은 이렇게 생겼어요:

```javascript
data = {
"name": [
    "Smudge",
    "Sissel",
    "Hamlet"
  ],
  "colour": [
    "black",
    "grey",
    "black"
  ],
}
```

이건 행 지향 설계와 정반대의 트레이드오프를 가집니다. 색상만 필요하다면 그 칼럼 데이터만 매우 효율적으로 읽을 수 있어요. 이름을 읽을 필요가 전혀 없거든요.

하지만 데이터를 수정하거나 특정 행을 읽는 건 훨씬 어려워집니다. 양쪽 다 여러 곳을 돌아다니며 접근해야 하니까요. 두 번째 행을 얻으려면 각 칼럼의 두 번째 인덱스에 가서 원래 행을 재구성해야 합니다.

## 데이터 인코딩 vs 정규화

이런 데이터 구조 변환을 보는 방법은 두 가지가 있습니다.

첫 번째는 **인코딩 레벨**로 보는 거예요. 즉, 데이터 모델보다 훨씬 아래 계층의 추상화 수준에 있다고 보는 겁니다. SQL 엔진은 이 둘을 쿼리의 성능 특성 외에는 구분할 수 없거든요.

두 번째는 **극단적인 정규화의 형태**로 보는 겁니다.

여러 벡터로 표현된 넓은 하나의 테이블 대신, 칼럼 지향 데이터를 기본키와 추가 속성 하나씩 가진 여러 테이블의 집합으로 생각해보세요.

비정규화된 테이블:
```
+----+------+-----+
| id | name | age |
+----+------+-----+
| 12 | Bob  | 30  |
| 93 | Tom  | 35  |
| 27 | Kim  | 28  |
+----+------+-----+
```

정규화된 테이블:

Name 테이블:
```
+----+------+
| id | name |
+----+------+
| 12 | Bob  |
| 93 | Tom  |
| 27 | Kim  |
+----+------+
```

Age 테이블:
```
+----+-----+
| id | age |
+----+-----+
| 12 | 30  |
| 93 | 35  |
| 27 | 28  |
+----+-----+
```

id 칼럼에서 조인하면 원래 테이블을 쉽게 복원할 수 있어요.

## 칼럼 저장소의 관점에서

칼럼 저장소의 맥락에서, 기본키를 데이터의 순서 위치(ordinal position)로 생각할 수 있습니다.

우리의 원래 데이터:

```javascript
data = {
"name": [
    "Smudge",
    "Sissel",
    "Hamlet"
  ],
  "colour": [
    "black",
    "grey",
    "black"
  ],
}
```

이렇게 표현하면:

```
+----+--------+
| id | name   |
+----+--------+
| 0  | Smudge |
| 1  | Sissel |
| 2  | Hamlet |
+----+--------+
```

```
+----+--------+
| id | colour |
+----+--------+
| 0  | black  |
| 1  | grey   |
| 2  | black  |
+----+--------+
```

그런데 id 칼럼은 배열의 위치로 암묵적으로 표현되니까, 실제로는 이렇게 돼요:

```
+--------+
| name   |
+--------+
| Smudge |
| Sissel |
| Hamlet |
+--------+
```

```
+--------+
| colour |
+--------+
| black  |
| grey   |
| black  |
+--------+
```

## 이 관점의 가치

이렇게 생각하는 게 중요한 이유는, 프로젝션(projection)이나 조인 같은 전통적인 쿼리 처리 방식과 데이터 포맷 조작을 연결해주기 때문입니다.

대부분의 경우 데이터 포맷은 쿼리가 논리적으로 인식하지 못하는 구현 세부사항이라고 생각해야 해요. 하지만 유용한 정신 모델로서, **"칼럼 저장소에서 행을 복원한다"는 것이 조인을 수행하는 것처럼 보일 뿐만 아니라, 실제로 조인**이라는 걸 알고 있으면 좋습니다.
