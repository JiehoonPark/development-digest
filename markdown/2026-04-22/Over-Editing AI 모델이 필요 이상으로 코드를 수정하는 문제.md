## 핵심 개념

> [!abstract]
> AI 코딩 도구들이 간단한 버그 수정 요청을 받을 때 전체 함수를 불필요하게 재작성하는 'Over-Editing' 문제를 분석한 연구이다. 연구팀은 BigCodeBench의 400개 문제를 체계적으로 손상시켜 최소한의 수정이 필요한 데이터셋을 구성했고, 토큰 레벨 Levenshtein 거리 등의 메트릭으로 모델의 과도한 수정 경향을 측정했다.

## 상세 내용

- Over-Editing은 기능적으로는 올바르지만 필요 이상으로 코드 구조를 변경하는 문제로, 코드 리뷰 병목 현상을 악화시킨다.
- 단순한 off-by-one 에러 하나를 수정할 때도 모델이 입력 검증, 헬퍼 함수, 변수명 변경 등을 추가로 수행하는 사례가 많다.
- Pass@1 같은 기존 정확도 지표로는 Over-Editing을 감지할 수 없으므로, 변경량을 측정하는 새로운 메트릭이 필요하다.

> [!tip] 왜 중요한가
> AI 코딩 도구의 실무 적용성을 높이려면 정확성뿐 아니라 최소한의 변경 원칙을 지켜야 하며, 이는 갈수록 중요해지는 개발자의 코드 리뷰 부담을 줄이는 데 핵심적이다.

## 전문 번역

# AI 코딩 도구의 '과도한 수정' 문제를 파헤쳐보자

이 글의 코드는 [여기](link)에서 확인할 수 있습니다.

요즘 Cursor, GitHub Copilot, Claude Code 같은 AI 코딩 도구를 쓰는 개발자라면 누구나 경험해봤을 겁니다. 간단한 버그 하나—예를 들어 off-by-one 에러나 잘못된 연산자—를 고쳐달라고 하면, AI가 그 부분은 수정하지만 함수의 절반이 새로 작성되어 돌아옵니다. 뜬금없는 헬퍼 함수가 추가되고, 멀쩡한 변수명이 바뀌고, 입력값 검증 코드가 생겨납니다. 결국 변경사항의 diff는 어마어마하게 커집니다.

이걸 저는 **'과도한 수정(Over-Editing)' 문제**라고 부릅니다. AI 모델들이 필요 없는 부분까지 코드를 다시 쓰는 경향이 있거든요.

왜 이게 중요할까요? 코드 리뷰는 이미 병목인데, 리뷰어는 뭐가 바뀌었는지, 왜 바뀌었는지, 안전한 변경인지를 모두 파악해야 합니다. AI가 함수 전체를 새로 작성해버리면(비록 정확하더라도) 리뷰 난이도가 급격히 올라갑니다. 코드가 완전히 달라 보이니까요.

이번 글에서는 이 문제를 직접 살펴보겠습니다. 현재의 LLM들이 정말 과도하게 수정하는 경향이 있는지, 그리고 더 신중한 수정을 하도록 학습시킬 수 있는지 말이죠.

## 과도한 수정, 구체적으로 뭘까?

**그림 1**을 보세요. GPT-5.4(고급 모드)가 함수 전체를 다시 쓰고 있습니다. 그런데 실제 버그는 단 한 줄이에요. `range(len(x) - 1)`을 `range(len(x))`로 바꾸면 끝입니다.

과도한 수정이란 **모델이 문제를 해결하는 데 필요한 것 이상으로 코드를 수정하는 것**입니다. 좀 더 정확히 말하면, 출력은 기능적으로 정확하지만 원본 코드와의 구조적 차이가 최소 수정선을 초과하는 경우죠.

그림 1의 예제가 이를 잘 보여줍니다. 버그는 `range()` 호출의 off-by-one 에러 하나인데, GPT-5.4는 어떻게 하나요?

- None 검사를 명시적으로 추가
- `np.asarray` 변환과 dtype 지정
- 유한값 마스킹 추가
- 배열 크기 검증
- `curve_fit` 호출 시그니처 변경
- 플로팅 로직 완전 교체

분명 테스트는 통과하고 기능적으로 맞습니다. 하지만 diff는 엄청나고, 요청하지도 않은 수정들이 잔뜩 들어가 있네요.

## 왜 이게 문제일까?

소프트웨어 개발은 크게 두 가지 모드로 나뉩니다.

**그린필드(Green-field)**: 새로운 프로젝트를 처음부터 만드는 작업
**브라운필드(Brown-field)**: 기존 코드베이스 안에서 작업하는 것

브라운필드에서는 기존 코드가 이미 팀에 의해 이해되었고, 의도를 가지고 그렇게 작성되었습니다. AI의 역할은 버그만 고치는 것이어야 해요.

"더 많은 테스트를 작성하세요. 테스트가 통과하면 괜찮습니다"라는 조언을 많이 들을 겁니다. 근데 과도한 수정은 **테스트로 감지할 수 없는 브라운필드 문제**입니다. 모델이 더 많은 코드를 생성할수록 리뷰어가 봐야 할 양은 늘어나고, 과도한 수정은 이를 훨씬 어렵게 만듭니다. 복잡한 로직이 늘어나고, 읽을 코드 양이 많아지고, 조용히 코드 품질이 떨어질 확률이 높아지는 거죠.

## 과도한 수정을 어떻게 측정할까?

먼저 "이게 최소한의 수정이다"라는 게 명확한 코드 수정 데이터셋이 필요합니다. 다른 LLM으로 버그를 만드는 방식(기존 벤치마크들이 주로 사용) 대신, 우리는 BigCodeBench의 400개 문제에 프로그래매틱 방식으로 버그를 주입했습니다. 이렇게 하면 훨씬 더 세밀한 제어가 가능해요.

비교 연산자 뒤집기(`<` → `<=`), 더하기를 빼기로 바꾸기(`+` → `-`), 불린값 반전(`True` → `False`) 같은 식으로 말이죠. 각 손상된 샘플은 문법적으로 유효하면서 대응하는 테스트를 깨뜨립니다. 덕분에 원본 수정은 정확히 손상의 역순이 되고, 이게 구조적으로 최소 수정이 됩니다.

이제 모델이 버그를 고쳤는지뿐만 아니라, **그 과정에서 얼마나 다른 것까지 바꿨는지**를 평가할 수 있습니다.

## 측정 지표

일반적인 코딩 벤치마크는 Pass@1 같은 정확성 지표를 씁니다. 그런데 Pass@1만으로는 충분하지 않습니다. 모델이 Pass@1에서 완벽한 점수를 받으면서 동시에 건드리는 모든 함수를 완전히 다시 쓸 수도 있거든요. 이 실험에서는 필요 이상으로 얼마나 바뀌었는지를 잡아낼 지표들이 필요합니다.

### 토큰 레벨 레벤슈타인 거리(Token-level Levenshtein Distance)

일반적인 레벤슈타인은 문자 단위로 삽입, 삭제, 치환의 최소 개수를 센다면, 우리는 **Python 토큰 단위**로 합니다.

코드를 먼저 Python 토크나이저에 통과시키면, `def`, `add`, `(`, `a`, `,`, `b`, `)`, `:`, `return`, `a`, `+`, `b` 같은 원자적 구문 단위로 쪼개집니다. 그 다음 이 토큰 시퀀스를 기준으로 레벤슈타인 거리를 계산합니다.

예를 들어보겠습니다:

```python
def add(a, b):
    return a + b
```

와

```python
def someotherfunctionname(a, b):
    return a + b
```

이 둘을 비교하면, **문자 레벨** 레벤슈타인은 19입니다. 하지만 **토큰 레벨**은 단 1입니다. 함수명 토큰이 하나만 달라졌거든요.
