## 핵심 개념

> [!abstract]
> 인프라 관리의 파편화 문제를 해결하기 위해, 프로그래밍 언어의 프로토콜/인터페이스 개념을 네트워크 경계에 적용하여 서로 다른 서비스들이 공통 형태를 따르도록 하는 OpenBindings 접근법을 제시합니다.

## 상세 내용

- 각 클라우드 제공자의 서로 다른 API 형태가 파편화의 근본 원인이며, 공통 인터페이스 정의로 해결 가능
- RFC 9727의 .well-known/api-catalog와 함께 서비스 자체가 공통 형태를 선언하는 방식 제안

> [!tip] 왜 중요한가
> 다중 클라우드 환경에서 벤더 락인을 줄이고 인프라 관리 복잡도를 낮출 수 있는 기반을 제시합니다.

## 전문 번역

# 인프라 관리의 단편화 문제, 어떻게 풀 것인가

이달 초 개발자 Dax Raad가 SNS에 올린 글이 큰 반향을 일으켰습니다.

> "요즘 누가 인프라를 관리하는 방법을 아는 사람이 있나요? 모든 서비스가 자기 식대로 CLI나 설정 파일을 만들어놓고 Terraform은 잘 지원 안 하더라고요. 어차피 한 개 제공사만 써야 하는 것도 아니고, 다들 이런 것들을 막 붙여 가면서 쓰는 건가요?"

24시간이 채 안 되어 5만 명이 이 글을 봤습니다. 댓글도 쏟아졌어요. SST, Pulumi, Ansible 같은 도구들이 언급되었고, "그냥 AWS만 써", "Python 스크립트로 REST 호출", "일자리 보장", "요즘 인프라는 덕트테이프가 대시보드를 입고 있는 거다" 같은 답변들이 나왔습니다.

문제는 모두가 인식했습니다. 그런데 제시된 해법들은 죄다 도구일 뿐, 근본적인 기초가 없었어요. 이 문제를 풀기 위해 뭔가를 만들고 있던 중에 5만 명의 개발자가 같은 답답함을 느낀다는 걸 확인했습니다. 결국 이건 제 생각만이 아니었던 거네요. **종속성은 증상일 뿐, 실제 병은 단편화입니다. 그런데 프로그래밍 언어는 이 문제를 수십 년 전에 이미 풀었거든요.**

## 종속성의 악순환

원래 문제로 거슬러 올라가보면, 한 제공사 위에 뭔가를 만들다가 그들이 가격을 올리거나 API를 없애거나, 아니면 더 이상 필요한 도구가 아니게 되면 마이그레이션이 정말 고통스럽습니다. 개념 자체가 어려워서는 아니에요. 문제는 모든 제공사가 다른 언어를 쓴다는 것입니다.

당연히 추상화 계층을 만들려고 합니다. 그 위에 뭔가 덮으면 되지 않을까? Terraform도, SST도, 그 외 수많은 도구들도 다 이걸 시도했습니다.

그런데 추상화 계층은 문제를 해결하지 못합니다. 문제를 옮길 뿐입니다. 여전히 누군가가 모든 제공사를 따라가길 기다려야 하고, 플러그인이 만들어지길 기다려야 하고, 라이선싱이 바뀌면 처음부터 다시입니다.

## 근본 원인

개발자들의 댓글을 보면, 핵심을 간파한 것들이 있습니다.

"추상화가 무너지고 있다. Terraform은 제공사들이 예측 가능할 때는 잘 작동했는데, 요즘은 모든 서비스가 자기 취향을 드러내는 계층을 만들어서 배포한다."

"Terraform 지원을 안 하는 건 인프라가 발전했기 때문이 아니라 벤더들이 종속을 최적화하려는 의도를 더 잘 보여준다."

"GCP, AWS, Azure, OCI 사이에 프로비저닝 호환성이 없으니까, 그 모두를 아우르는 단일 구현이 나올 가능성이 없다."

맞습니다. 호환성이 없어요. 그런데 근본 원인을 다르게 표현하자면, **서비스가 자신을 설명하는 표준 방식이 없다는 것입니다.**

## 프로그래밍 언어는 이미 풀었습니다

그때 깨달았어요. 이건 소프트웨어 내부에서는 이미 해결된 문제라고요.

Swift에는 Protocol이 있고, Go에는 Interface가 있고, Haskell에는 Typeclass가 있습니다. 이런 걸 쓰면 이렇게 코드를 짤 수 있어요.

"난 이게 정확히 뭔지 상관 없어. 이런 일들을 할 수 있기만 하면 돼."

모양을 보고 코딩하는 거죠. 구현 자체는 상관없습니다. 내부 타입을 바꿔도 코드는 여전히 컴파일됩니다.

오리처럼 걷고 오리처럼 울면, 그냥 오리처럼 다루면 된다는 거예요. **핵심은 의존성을 뒤집는 것입니다.** 당신의 코드가 특정 구현에 의존하는 게 아니라, 구현들이 공유된 모양을 만족해야 합니다. 새로운 제공사가 나타나도 위의 코드는 건들 필요가 없어요. 프로그래밍 언어는 수십 년 동안 이렇게 해왔는데, 웹은 네트워크 경계에서 널리 채택된 이 방식을 아직 갖지 못했습니다.

모든 서비스는 사전 지식이 필요합니다. 그냥 "파일 저장할 수 있어?"라고 물어서 호환하는 서비스가 답해주는 방식이 없어요.

만약 가능하다면?

## index.html의 문제

비슷하면서도 다른 게 계속 신경 쓰였어요. 모든 웹사이트는 index.html을 가지고 있습니다. 현관문이자, 발견을 위한 공유 관례인데, 사이트가 어떻게 구성됐는지 미리 알아야 할 필요가 없어요. 루트에 가면 시작점을 찾을 수 있거든요.

API는 이런 게 없습니다. 문서가 어디 있는지, 어떤 포맷을 쓰는지(OpenAPI? AsyncAPI? gRPC? 자기들만의 형식?), 어떻게 인증하는지, 어떤 작업들이 있는지 다 알아야 합니다. 그것도 찾아낸 후에, "S3에 파일 올리기"와 "R2에 파일 올리기"가 다른 프로토콜로 표현된 같은 작업이라는 걸 말해줄 방법이 없습니다.

작년에 IETF가 RFC 9727을 발표했는데, 잘 알려진 URI인 `/.well-known/api-catalog`로 API 존재 여부를 발견할 수 있게 했어요. 이건 어떤 API가 있는지 알려주지만, 그 API가 뭘 하는지, 어떤 게 동등한지는 안 알려줍니다. 이 둘은 보완적이에요. api-catalog 항목이 OBI 같은 걸 링크할 수 있죠. 하지만 발견만으로는 부족합니다. 여전히 미리 알고 있어야 쓸 수 있어요.

그래서 제공사들을 통합하려는 모든 도구는 억지로 합니다. 모든 연동을 하드코딩하고, 모든 매핑을 관리하고, 모든 API 변화를 계속 쫓아갑니다. 영원히요.

## AI는 문제를 더 숨기고 있어요

AI는 이 위에 덮개를 씌우는 일에 정말 좋아졌습니다. 그리고 그게 오히려 근본 문제를 덜 보이게 한다고 생각합니다.

LLM은 문서를 읽고, 인증을 파악하고, API 호출을 생성할 수 있습니다. MCP는 AI 에이전트가 서비스와 상호작용하는 깔끔한 방식을 제공합니다. 그런데 여전히 억지일 뿐입니다. 좀 더 빠른 억지일 뿐이에요. LLM은 선언될 수 있는 구조를 추측하고, 기계가 읽을 수 있는 문서를 파싱하고, 검증될 수 있는 동등성을 추론합니다.

**실제 문제는 단편화입니다.** 종속은 증상일 뿐입니다.
