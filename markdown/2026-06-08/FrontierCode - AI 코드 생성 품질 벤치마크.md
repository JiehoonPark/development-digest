## 핵심 개념

> [!abstract]
> Cognition이 공개한 FrontierCode는 AI 모델의 코드 품질을 측정하는 새로운 벤치마크로, 단순 정확성을 넘어 실제 프로덕션 코드의 유지보수성과 병합 가능성을 평가합니다. 오픈소스 메인테이너 20명 이상이 참여해 150개의 현실적이고 다양한 작업을 구성했으며, 기존 벤치마크 대비 81% 낮은 오류율을 달성합니다.

## 상세 내용

- "유지보수자가 실제로 병합할 코드인가"를 평가하는 첫 번째 벤치마크로, 코드 정확성뿐 아니라 테스트 품질, 스타일, 코드베이스 표준 준수 등을 종합 평가
- Claude Opus 4.8이 가장 높은 성능을 보이나 Diamond 난이도에서도 13.4%의 점수만 달성하며, 현재 최고 수준의 모델도 고품질 코드 작성에 어려움을 겪는 것으로 나타남

> [!tip] 왜 중요한가
> AI 코드 생성 도구의 실제 프로덕션 적용 가능성을 정확히 평가할 수 있는 신뢰할 수 있는 기준을 제공합니다.

## 전문 번역

# 정확함을 넘어 품질로: FrontierCode 벤치마크 소개

## 코드의 정확함만으로는 부족하다

요즘 AI 코드 생성 모델들은 이미 "정확한 코드를 작성한다"는 걸 증명했습니다. 그런데 AI가 생성한 코드가 프로덕션의 주요 경로가 되면서, 정확함은 이제 기본 요건일 뿐입니다. 진짜 우리가 물어야 할 질문은 이거거든요: "모델이 정말 좋은 코드를 작성할 수 있는가?"

이런 질문에 답하기 위해 저희는 FrontierCode라는 새로운 벤치마크를 만들었습니다. 이것은 모델이 실제 프로덕션 코드베이스의 수준을 얼마나 충족하는지를 측정하는 벤치마크입니다.

## FrontierCode의 차별점

**실제 머지 가능한 코드인가?**

저희가 처음으로 도입한 기준은 "코드 머지 가능성"입니다. PR을 받은 유지보수자가 실제로 머지하겠다고 생각할 만한 수준인지를 평가하는 거죠. 이를 위해 정확성, 테스트 품질, 범위 규율, 코드 스타일, 코드베이스 표준 준수 등 엔드-투-엔드 코드 품질을 종합적으로 판단합니다. 단위 테스트, 평가 기준표, 새로운 검증 방식들을 조합한 평가 기법을 사용하고 있어요.

**오픈소스 유지보수자들이 직접 만들었다**

20명 이상의 세계적 수준의 오픈소스 개발자들이 자신들이 관리하는 저장소에서 직접 작업들을 추려냈습니다. 작업당 40시간 이상을 투입했죠. 이들이 각자의 저장소에서 "머지 가능"이 뭔지를 정의했으니, 벤치마크의 현실성이 다를 수밖에 없습니다.

**품질 관리의 철저함**

평가 기준표 채점은 주관적일 수밖에 없으니까, 저희는 광범위한 QC 파이프라인을 구축했습니다. 적대적 테스팅, 보정, 그리고 여러 단계의 리뷰를 거치고, Cognition 연구원들이 모든 작업을 수동으로 검토합니다. 덕분에 SWE-Bench Pro 대비 거짓 양성 오류를 81% 줄였습니다.

## 벤치마크 구성과 결과

저희는 난이도에 따라 세 단계의 서브셋을 제공합니다:
- **Extended**: 150개 작업 전체
- **Main**: 상위 100개 작업
- **Diamond**: 가장 어려운 50개 작업

각 모델은 사용 가능한 모든 추론 단계에서 5회씩 실행되며, 각 시도의 점수를 평균낸 후 최고 성과의 추론 수준에서의 점수를 보고합니다.

**평가 지표**

- **통과(Pass)**: 모든 필수 기준을 충족하면 통과합니다. 필수 기준은 유지보수자가 코드 리뷰 중에 반드시 지적할 것들이죠.
- **점수(Score)**: 평가 항목의 가중 합계입니다. 필수 기준을 충족하지 못하면 0점입니다.

## 모델별 성과

**Diamond (가장 어려운 50개)**

현재 최고 성능 모델인 Claude Opus 4.8도 겨우 13.4%의 점수만 기록했습니다. 이는 Diamond 세트가 아직 완전히 포화되지 않았다는 뜻이에요. 다른 모델들의 성과는 더 낮습니다:

- GPT-5.5: 6.3%
- Gemini 3.1 Pro: 4.7%
- 기타 모델들: 더 낮음

다만 GPT 5.5는 Opus 4.8 대비 최대 4배 적은 토큰을 사용하면서도 비용 대비 성능 면에서 더 좋은 효율을 보여줍니다.

**Main과 Extended**

Opus 4.8이 여전히 선두를 유지하고 있습니다:
- Main: 34.3%
- Extended: 51.8%

오픈소스 모델과 최고 수준 모델 간의 격차는 상당합니다. 최고 성능 오픈소스 모델인 Kimi K2.6는:
- Diamond: 3.8%
- Main: 16%
- Extended: 37%

## 기존 벤치마크의 문제점

기존 SWE-Bench Verified와 Pro 같은 벤치마크들은 당시보다 능력이 떨어지는 모델들을 염두에 두고 설계되었습니다. 현실성과 견고함 면에서 부족한 부분이 많죠.

**기능적 정확성만 테스트한다**

가장 근본적인 문제는 이 벤치마크들이 기능 정확성만 테스트한다는 거입니다. 실제 코드 품질은 측정하지 않아요.

**분류 오류가 많다**

METR의 실험에 따르면, 이런 벤치마크에서 높은 점수를 받은 모델의 패치가 실제 인간 유지보수자들에게 거절당하는 경우가 많습니다. 

분류 오류는 두 가지 유형으로 나뉩니다:

- **거짓 양성(False Positive)**: 틀린 솔루션을 올바른 것으로 잘못 판정하는 경우. 테스트 커버리지가 불완전해서 모델이 틀린 해법을 제출해도 통과하는 거죠.
- **거짓 음성(False Negative)**: 올바른 솔루션을 틀린 것으로 잘못 판정하는 경우. 테스트가 너무 구체적이라서 정확한 에러 메시지나 함수명을 정확히 맞춰야 하거나, 테스트 자체가 해결 불가능한 경우입니다.

저희 분석 결과 FrontierCode는 기타 주요 벤치마크 대비 분류 오류를 81% 줄였습니다. 즉, FrontierCode 점수가 현재 가장 정확한 순위를 제공합니다.

**다양성 부족**

기존 벤치마크들은 여러 측면에서 다양성이 떨어집니다. 단일 PR에서 프로그래매틱하게 이슈를 추출한 반면, FrontierCode는 저장소 유지보수자들이 여러 PR에 걸친 이력과 자유로운 요청들에서 직접 선택한 것입니다.

또한 지원하는 프로그래밍 언어 수도 3배 늘렸습니다.

**너무 자세한 지시사항**

기존 벤치마크는 과하게 상세한 프롬프트로 모델을 과도하게 안내하는 경향이 있었습니다. 요즘 최고 수준의 모델들은 이렇게까지 세세한 지시가 필요하지 않거든요. FrontierCode는 에이전트가 유지보수자의 의도를 스스로 추론하도록 기대합니다.
