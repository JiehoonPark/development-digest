## 핵심 개념

> [!abstract]
> 상태차트는 상태 머신을 확장한 개념으로, 복잡한 시스템의 행동을 시각적으로 표현하고 상태 폭발(state explosion) 문제를 해결합니다. W3C에서 10년 이상 표준화한 SCXML(상태차트 XML) 형식이 있으며, 다양한 프로그래밍 언어의 라이브러리로 구현 가능합니다. 상태차트는 단순한 다이어그램이 아니라 런타임에서 실제 동작하는 코드의 진실의 단일 원천(single source of truth) 역할을 할 수 있습니다.

## 상세 내용

- 상태차트는 기본 상태 머신의 한계를 극복합니다. 상태 수가 증가할 때 상태 폭발로 인한 복잡성 증가를 계층적 구조와 중첩 상태로 효과적으로 관리할 수 있습니다.
- W3C 표준 SCXML(2005-2015년 표준화)은 엣지 케이스 처리와 진입/종료 액션의 정확한 순서 보장을 명시합니다. SCXML 기반 라이브러리를 사용하면 복잡한 상태 관리 로직을 신뢰할 수 있게 구현할 수 있습니다.
- 상태차트는 이미 코드에 숨겨진 상태 머신을 명시적으로 드러냅니다. 대부분의 소프트웨어는 상태 머신을 구현하지만 그 구조를 명확히 표현하지 않아 유지보수 어려움이 발생합니다.
- 코드 내 상태차트 정의에서 자동으로 시각적 다이어그램을 생성할 수 있습니다. JSON이나 XML 파일의 상태차트 정의를 단일 진실 공급원으로 사용하면, 런타임 동작을 구동하면서 동시에 정확한 시각화를 자동 생성할 수 있습니다.
- 상태차트 도입의 장단점이 있습니다. 초기 학습 비용과 도구 의존성이 있지만, 복잡도 감소, 버그 감소, 문서화 개선, 테스트 용이성 증가 등의 이점이 있습니다.
- 상태차트에 대한 공통 반박 주장들도 있지만, 전반적으로는 순이득입니다. Gitter와 GitHub Discussions에서 활발한 커뮤니티가 상태차트 학습과 적용을 지원합니다.

> [!tip] 왜 중요한가
> 상태차트는 복잡한 애플리케이션 상태 관리를 명확하고 유지보수 가능하게 만들어, 버그 감소와 개발 생산성 향상을 동시에 달성할 수 있는 검증된 패러다임입니다.

## 전문 번역

# 상태 차트(Statechart)란 무엇인가

상태 차트는 여러 방식으로 설명할 수 있지만, 본질적으로는 하나의 그림입니다. 그런데 소프트웨어 엔지니어 입장에서는 이 그림만으로는 실제 이점을 활용하기 어렵거든요. 그래서 상태 차트가 정확히 무엇인지 다른 방식으로 설명해보겠습니다.

상태 차트를 처음 정의한 논문(Harel, 1987)에서는 이를 "복잡한 시스템을 위한 시각적 형식(A visual formalism for complex systems)"이라고 부르고 있습니다.

## 상태 머신의 업그레이드 버전

간단히 말해서, 상태 차트는 상태 머신을 강화한 것입니다. 특히 상태 머신이 커지면서 발생하는 "상태 폭발(state explosion)" 문제를 해결하죠. 이 사이트의 목표는 상태 차트가 정확히 무엇이고, 어떻게 유용한지 설명하는 것입니다.

흥미롭게도, 여러분은 이미 상태 머신을 코딩하고 있습니다. 다만 그것이 코드 곳곳에 숨어 있을 뿐이죠.

## 상태 차트의 장점과 주의점

상태 차트는 다양한 이점을 제공합니다. 하지만 사용할 때 주의해야 할 단점들도 있고, 상태 차트에 대한 흔한 반박들도 존재합니다.

다행히 위에서 언급한 이점들을 고려하면, 상태 차트 도입은 전반적으로 긍정적인 선택입니다.

## 표준화와 라이브러리

W3C 위원회가 2005년부터 2015년까지 무려 10년 이상 SCXML(Statechart XML)이라는 표준을 만들었다는 사실을 알고 계신가요? 이 표준은 상태 차트의 의미론을 정의하고 엣지 케이스를 어떻게 다룰지 명시합니다.

SCXML로 작성된 상태 차트를 읽고, 작성하고, 실행할 수 있는 도구들이 다양한 언어로 존재합니다. SCXML과 동일한 모델을 지원하되 다른 문법을 사용하는 파생 도구들도 있죠.

각 플랫폼을 위한 상태 차트 라이브러리들도 SCXML의 의미론을 다양한 수준으로 지원합니다. 이러한 라이브러리들을 사용하면 엣지 케이스를 적절히 처리할 수 있고, 진입/퇴출 액션들이 올바른 순서로 실행되도록 보장받을 수 있습니다.

## 코드와 시각화의 단일 소스

문서와 실제 코드를 분리하여 상태 차트로 동작을 모델링하는 것도 좋지만, JSON이나 XML 같은 형식으로 상태 차트를 정의한 후 런타임에 직접 사용할 수도 있습니다. 이 방식의 핵심은 컴포넌트의 동작을 정의하는 단일 소스를 만드는 것입니다.

그러면 이 정의는 런타임 코드를 구동하면서 동시에 정확한 상태 차트 다이어그램을 자동으로 생성하는 데에도 사용될 수 있죠. 코드에 상태 차트 정의가 있다면, 그것을 가져다가 시각화 생성을 자동화하기만 하면 됩니다. 물론 정의가 JSON이나 XML 같은 별도 파일에 있으면 더 간단합니다.

상태 차트를 실제로 사용하는 방법은 관련 페이지에서 자세히 설명합니다.

## 커뮤니티와 참고자료

상태 차트에 대해 이야기하고 싶다면 Gitter(gitter.im)에 방문해보세요. 로그인 없이도 채팅을 볼 수 있으며, 상태 차트를 이해하고 활용하는 데 도움을 줄 수 있는 개발자 커뮤니티가 있습니다. 질답 형식을 선호한다면 GitHub Discussions을 통해 질문할 수 있고, 저희가 최선을 다해 답변해드리겠습니다.

많은 개발자들이 상태 차트에 관한 책을 쓰거나 발표를 진행했습니다. 이들은 우리 리소스 페이지에 정리되어 있습니다. 혹시 작성한 콘텐츠가 있다면 GitHub Discussions에서 공유해주세요.

이 문서 구조에 자연스럽게 맞지 않는 몇몇 페이지들도 있는데, 그들을 여기서 특별히 소개합니다.
