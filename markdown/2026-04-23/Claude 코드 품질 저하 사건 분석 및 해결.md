## 핵심 개념

> [!abstract]
> Anthropic은 지난 한 달간 일부 사용자들의 Claude 응답 품질 저하 보고를 조사했으며, Claude Code, Claude Agent SDK, Claude Cowork에 영향을 미친 세 가지 별개의 이슈를 발견하고 2024년 4월 20일에 모두 해결했습니다. API 레이어는 영향을 받지 않았습니다.

## 상세 내용

- Claude Code의 기본 reasoning effort가 3월 4일 high에서 medium으로 변경되었으나, 사용자 피드백에 따라 4월 7일 xhigh(Opus 4.7)와 high(기타 모델)로 복구되었습니다. 이 변경은 UI 동결 지연 문제를 해결하려는 의도였으나 지능도 함께 저하되는 부작용이 발생했습니다.
- 3월 26일에 배포된 prompt caching 최적화에서 1시간 이상 유휴 세션의 reasoning을 제거하는 기능이 버그로 인해 매 턴마다 실행되어 Claude가 건망증 있고 반복적으로 동작하는 것처럼 보였으며, 4월 10일에 수정되었습니다.
- 4월 16일에 추가된 verbosity 감소 시스템 프롬프트가 다른 프롬프트 변경과 함께 코딩 품질을 해쳤으며, 4월 20일에 되돌려졌습니다. 이는 Sonnet 4.6, Opus 4.6, Opus 4.7에 영향을 미쳤습니다.
- 각 변경사항이 서로 다른 트래픽 슬라이스와 일정에 영향을 미쳐 전체적으로 광범위하고 일관성 없는 저하로 보였으며, 초기 단계에서는 일반적인 사용자 피드백 변동과 구분하기 어려웠습니다.
- Anthropic은 모든 구독자의 사용량 제한을 4월 23일부터 초기화했으며, 향후 유사 문제 예방을 위한 개선 방안을 약속했습니다.

> [!tip] 왜 중요한가
> Claude Code 사용자들이 겪은 품질 저하 원인을 명확히 이해할 수 있으며, Anthropic의 문제 대응 및 개선 체계를 평가할 수 있는 중요한 사례입니다.

## 전문 번역

# Claude의 응답 품질 저하 원인 분석 및 해결 방안

지난 한 달간 저희는 일부 사용자들이 Claude의 응답 품질이 떨어졌다는 보고를 받고 조사했습니다. 조사 결과 Claude Code, Claude Agent SDK, Claude Cowork에 영향을 미친 세 가지 별도의 변경사항을 발견했고, API는 영향을 받지 않았습니다. 이 세 가지 문제는 모두 4월 20일(v2.1.116)을 기준으로 해결되었습니다.

이 글에서는 저희가 무엇을 발견했고, 어떻게 수정했으며, 앞으로 이런 일이 재발하지 않도록 어떻게 개선할 것인지 설명하겠습니다.

## 문제 인식과 대응

품질 저하에 관한 보고는 매우 심각하게 받아들입니다. 저희는 모델을 의도적으로 성능을 낮추는 일은 절대 하지 않으며, API와 추론 계층이 영향을 받지 않았음을 즉시 확인할 수 있었습니다.

조사를 통해 다음 세 가지 서로 다른 문제를 확인했습니다.

### 1. Claude Code의 기본 추론 수준 변경

3월 4일에 Claude Code의 기본 추론 수준을 high에서 medium으로 낮췄습니다. 일부 사용자들이 high 모드에서 UI가 멈춘 것처럼 보일 정도로 긴 지연 시간을 경험했기 때문이었는데요, 이는 잘못된 선택이었습니다.

사용자 피드백을 받은 후 4월 7일에 이 변경사항을 되돌렸습니다. 사용자들은 단순한 작업일 때는 낮은 수준으로 선택하되, 기본적으로는 더 높은 지능 수준을 원한다는 의사를 명확히 했거든요. 이 문제는 Sonnet 4.6과 Opus 4.6에 영향을 미쳤습니다.

### 2. 세션 사고 내용(thinking) 캐시 버그

3월 26일에 1시간 이상 비활성 상태였던 세션의 기존 사고 내용을 지우는 변경사항을 배포했습니다. 이를 통해 사용자가 세션을 재개할 때 지연 시간을 줄이려는 의도였는데요, 버그로 인해 이 작업이 세션 시작 시 한 번만이 아니라 이후 매 턴마다 계속 발생했습니다. 그 결과 Claude가 건망증이 있고 반복적인 답변을 하는 것처럼 보였습니다.

4월 10일에 이를 수정했습니다. 이 문제는 Sonnet 4.6과 Opus 4.6에 영향을 미쳤습니다.

### 3. 말투 간결화 시스템 프롬프트

4월 16일에 응답의 장황함을 줄이기 위한 시스템 프롬프트 지침을 추가했습니다. 다른 프롬프트 변경사항과 함께 작용하면서 코딩 품질이 떨어졌고, 4월 20일에 되돌렸습니다. 이 문제는 Sonnet 4.6, Opus 4.6, Opus 4.7에 영향을 미쳤습니다.

## 문제 감지의 어려움

각 변경사항이 서로 다른 사용자 그룹과 다른 시간표에 영향을 미쳤기 때문에, 종합적으로는 광범위하고 불규칙한 성능 저하처럼 보였습니다. 3월 초부터 보고를 받기 시작했지만, 처음에는 일반적인 사용자 피드백의 변동성과 구분하기가 어려웠습니다. 더구나 내부 사용 현황과 평가 지표에서도 식별된 문제들이 재현되지 않았습니다.

## 사용 한도 리셋

이것은 Claude Code 사용자들이 기대할 수 있는 경험이 아니었습니다. 4월 23일 기준으로 모든 구독자의 사용 한도를 초기화했습니다.

## Claude Code의 기본 추론 수준 변경 상세 분석

2월에 Claude Code에서 Opus 4.6을 출시했을 때, 저희는 기본 추론 수준을 high로 설정했습니다.

곧이어 사용자들로부터 고수준 모드에서 Opus 4.6이 너무 오래 생각하는 경향이 있다는 피드백을 받았습니다. 이로 인해 UI가 멈춘 것처럼 보이고 과도한 지연 시간과 토큰 사용량이 발생했습니다.

일반적으로 모델이 더 오래 생각할수록 출력 결과가 더 좋습니다. 이 트레이드오프를 조정하기 위해 Claude Code에서는 노력 수준(effort level)을 제공합니다. 더 많은 생각 시간 대 낮은 지연 시간과 적은 사용량 중에서 선택할 수 있다는 뜻이죠.

모델의 노력 수준을 조정할 때, 저희는 이 트레이드오프를 고려해서 테스트 타임 컴퓨팅 곡선을 따라 사용자에게 최고의 선택지를 제공하는 지점들을 선택합니다. 그리고 제품 계층에서는 이 곡선 중 어느 지점을 기본값으로 설정할지 결정해서, Messages API에 effort 파라미터로 보냅니다. 다른 옵션들은 /effort를 통해 제공됩니다.

내부 평가와 테스트에서 medium 수준은 대부분의 작업에서 약간 낮은 지능 수준으로 훨씬 빠른 응답 시간을 제공했습니다. 또한 생각 시간이 매우 길어지는 문제도 없었고, 사용자의 사용 한도를 최대한 활용할 수 있었습니다. 그 결과 저희는 medium을 기본값으로 변경하고 제품 내 대화창을 통해 그 근거를 설명했습니다.

배포 직후 사용자들은 Claude Code가 덜 지능적으로 느껴진다고 보고했습니다. 저희는 현재 설정된 노력 수준을 더 명확하게 표시하기 위해 여러 가지 디자인 반복을 거쳤습니다(시작 시 공지, 인라인 노력 선택기, ultrathink 복구). 하지만 대부분의 사용자들은 여전히 medium 수준으로 기본값을 유지했습니다.

더 많은 고객 피드백을 받은 후 4월 7일에 이 결정을 뒤집었습니다. 이제 모든 사용자는 Opus 4.7의 경우 xhigh 수준으로, 다른 모든 모델의 경우 high 수준으로 기본 설정됩니다.

## 캐싱 최적화로 인한 사고 내용 손실

Claude가 작업을 추론할 때, 해당 추론 내용은 보통 대화 기록에 보관됩니다. 이렇게 하면 이후 턴마다 Claude는 자신이 어떤 편집과 도구 호출을 왜 했는지 이유를 볼 수 있습니다.

3월 26일에 이 기능을 효율화하려는 개선사항을 배포했습니다. 저희는 프롬프트 캐싱을 사용해서 연속되는 API 호출을 사용자에게 더 저렴하고 빠르게 제공합니다. Claude가 API 요청을 할 때 입력 토큰을 캐시에 쓰고, 일정 기간 비활성 상태가 지나면 프롬프트가 캐시에서 제거되어 다른 프롬프트를 위한 공간을 확보합니다. 캐시 활용률은 저희가 신중하게 관리하는 것입니다.

설계 의도는 간단했습니다. 세션이 1시간 이상 비활성 상태였다면, 그 세션을 재개할 때의 비용을 줄이기 위해 기존 사고 내용을 삭제할 수 있다는 생각이었거든요. 어차피 요청이 캐시 미스가 될 것이므로, 불필요한 부분을 정리할 수 있다는 계산이었습니다.

그런데 버그로 인해 이 작업이 세션 시작 시 한 번만이 아니라 이후 매 턴마다 반복되었습니다. 그 결과 Claude가 이전 맥락을 계속 잊고 반복적인 답변을 하는 것처럼 보였던 것입니다. 4월 10일에 이를 수정했습니다.
