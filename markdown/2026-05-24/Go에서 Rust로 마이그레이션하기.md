## 핵심 개념

> [!abstract]
> Go 백엔드 서비스에서 Rust로 전환할 때의 차이점, 패턴 매핑, 점진적 마이그레이션 전략을 다룬 종합 가이드이다. Go의 성공을 인정하면서도 Rust의 정확성 보장과 런타임 트레이드오프를 객관적으로 비교한다.

## 상세 내용

- Go와 Rust의 도구체인, 타입시스템, 메모리 안전성이 크게 다르며 각각의 강점이 있음
- Rust의 borrow checker는 정확성을 보장하지만 학습 곡선이 가파르며, Go는 단순성과 배포 편의성이 장점
- 개별 마이크로서비스 단위로 점진적 마이그레이션이 실무에서 가장 현실적인 접근

> [!tip] 왜 중요한가
> 언어 전환 결정 시 성능뿐 아니라 개발자 생산성, 런타임 비용, 정확성 보장을 종합적으로 고려해야 합니다.

## 전문 번역

# Go에서 Rust로 마이그레이션하기

마이그레이션 가이드를 작성하면서 느낀 점이 있는데, Go에서 Rust로의 전환은 다른 마이그레이션들과 조금 다르다는 거예요.

흔히 "Rust가 더 빠른가?"나 "Rust도 타입 시스템이 있나?"라는 질문이 나오는데, 사실 Go도 대부분의 필요한 기능을 충분히 제공하거든요. 진짜 논점은 정확성 보장, 런타임 트레이드오프, 그리고 개발자 경험의 차이에 있습니다.

미리 말씀드릴 점이 있는데, 이 가이드는 백엔드에 무거운 초점을 두고 있어요. Go가 가장 강한 영역이 바로 백엔드 서비스 개발이니까요. 작고 정적인 바이너리를 만들 수 있고, 네트워킹에 특화된 표준 라이브러리, HTTP 서버부터 gRPC, 데이터베이스까지 생태계가 잘 갖춰져 있죠. 저한테 연락하는 팀들도 대부분 이 영역에서 Rust로의 전환을 고려하고 있어서, 실용적인 비교라고 생각합니다.

CLI 도구를 만들거나, 임베디드 펌웨어를 작성하거나, 게임 엔진을 개발 중이라면 이 글의 일부만 도움이 될 거예요. 솔직히 그쪽이 목표라면 다른 자료를 찾는 게 나을 것 같습니다.

## 이 글에서 배울 수 있는 것

- Go와 Rust가 어디서 겹치고 어디서 갈라지는지
- Go의 패턴이 Rust에서 어떻게 매핑되는지
- 빌림 체커(borrow checker)의 가치
- 언제 Go를 유지해야 하고 언제 Rust 마이그레이션이 가치 있는지
- Go 서비스를 점진적으로 마이그레이션하는 방법

## 내 입장 밝히기

솔직하게 말씀드리자면, 저는 Go를 좋아하지 않습니다. 매우 성공했지만 설계가 좋지 못한 언어라고 생각해요. 쉬움과 단순함을 헷갈리고 있고, 핵심 설계 결정들(모든 곳의 nil, 규칙으로서의 에러 처리, 제네릭 부재)이 저는 동의할 수 없는 방향을 향하고 있거든요.

그렇긴 해도 현실이 중요하죠! Go는 실제로 일하는 개발자들 중 17-19% 정도를 차지하고 있어요(JetBrains 개발자 생태계 조사). Rust는 꾸준히 성장 중이지만 아직 더 작은 점유율입니다.

Go가 많은 사람들에게 실제로 작동하고 있고, 그걸 무시하는 가이드는 도움이 안 됩니다. 그래서 이 글에서는 최대한 객관적으로 접근하려고 노력했어요. 하지만 제 선입견을 알고 있으면 좋을 겁니다.

또 하나 고백할 점: 저는 Rust 컨설팅 회사를 운영하고 있습니다. 당연히 편향되어 있죠. Rust를 쓰는 사람이 많아지면 제 사업에도 좋으니까요. 그래도 두 언어 모두에서 업무 경험이 있고 Go 서비스를 프로덕션에 배포했던 경험이 있습니다.

이 가이드는 Go에서 Rust로 넘어가려는 Go 개발자들을 위해 솔직하고 객관적인 비교를 제공하려고 만들었어요. 반대 관점도 보고 싶다면 Blain Smith의 "Just Fucking Use Go"를 추천합니다. 두 입장을 동시에 이해하는 게 어느 한쪽만 아는 것보다 훨씬 유용하거든요.

## 가장 중요한 명령어들부터 살펴보기

Go 개발자라면 이미 업계에서 가장 깔끔한 툴체인 중 하나를 쓰고 있어요.

과거에 Go가 시작한 흐름인 "배터리 포함" 툴체인 트렌드 덕분에, 빌드, 테스트, 포매팅, 린팅, 의존성 관리를 하나의 일관된 인터페이스로 할 수 있게 되었어요. Rust도 이 모델을 따랐는데, 저는 정말 좋은 선택이라고 생각합니다. 두 생태계의 가장 좋아하는 부분 중 하나에요.

Cargo는 사실 더 많은 기능을 내장하고 있어요:

| Go 도구 | Rust 동등물 | 설명 |
|--------|-----------|------|
| go.mod / go.sum | Cargo.toml / Cargo.lock | 프로젝트 설정 및 의존성 매니페스트 |
| go get / go mod tidy | cargo add / cargo update | 의존성 추가 및 해석 |
| go build | cargo build | 프로젝트 컴파일 |
| go run . | cargo run | 빌드 후 실행 |
| go test ./... | cargo test | 툴체인에 내장된 테스트 |
| go vet ./... | cargo clippy | 린터. Clippy는 vet보다 훨씬 의견이 강함 |
| gofmt / goimports | cargo fmt | 자동 포매터, 설정 필요 없음 |
| golangci-lint run | cargo clippy -- -D warnings | 엄격한 린트 모드 |
| go install ./cmd/foo | cargo install --path . | 바이너리 설치 |
| go doc | cargo doc --open | API 문서 생성 및 보기 |
| pprof | cargo flamegraph / samply | CPU 프로파일링 |
| govulncheck | cargo audit | 권고 데이터베이스에 대한 취약점 스캔 |

큰 차이점은 Go에서는 보통 서드파티 도구(golangci-lint, mockgen, air, goreleaser)에 의존해야 한다는 거예요. 반면 Rust는 1차 생태계가 처음부터 더 많이 포함하고 있습니다.

외부 크레이트를 필요로 하는 것들(예: cargo watch, cargo nextest)도 한 줄의 명령어로 설치되고 마치 기본 기능처럼 느껴져요. `cargo install cargo-nextest`를 입력하면 바로 `cargo nextest`를 쓸 수 있거든요.

두 커뮤니티 모두 포매터에 대해 같은 통찰에 도달했어요: 설령 완벽하지 않더라도 하나의 표준 스타일이 불필요한 논쟁을 줄이는 것보다 훨씬 가치 있다는 거죠.

> Gofmt의 스타일은 아무도 최고로 손꼽지 않는다. 하지만 gofmt는 모두가 최고로 손꼽는다.
> — Rob Pike, Go Proverbs

Rustfmt도 마찬가지예요. 모든 디테일을 좋아하는 사람은 없지만, 코드 리뷰에서 스타일 논쟁이 없다는 것의 가치가 자신의 포매팅 선호도를 관철하는 것보다 훨씬 크거든요.
