## 핵심 개념

> [!abstract]
> Jujutsu 버전 관리 시스템을 이용하여 개발 중 복잡한 커밋 정리 작업을 효율적으로 처리하는 방법을 소개합니다. 저자는 '지저분한 상태에서 먼저 이상적인 커밋 구조를 만든 후, 마지막에 변경사항을 올바른 커밋에 배치하는' 워크플로우를 제안합니다.

## 상세 내용

- Jujutsu의 `jj new`, `jj squash` 명령어를 활용하여 개발 중 임시 커밋들을 정리하는 전략 제시
- 기존 `jj absorb`와 `jj squash -i`의 한계를 극복하기 위해 모든 변경사항을 하나의 커밋으로 모은 후 정렬하는 방식 제안

> [!tip] 왜 중요한가
> 대규모 기능 개발 시 커밋 규칙을 엄격히 유지하면서도 개발 유연성을 잃지 않는 실질적인 워크플로우를 제공합니다.

## 전문 번역

# Jujutsu로 깔끔한 커밋 히스토리 만들기

Jujutsu 버전 관리 시스템을 기본 수준으로 이해하고 있다는 가정 하에 작성했습니다. 아직 Jujutsu를 써본 적 없다면 이 글의 주요 아이디어는 이해할 수 있겠지만, 나중에 Steve의 Jujutsu 튜토리얼을 읽어보길 추천합니다.

## 좋은 커밋은 정말 어렵다

큰 기능을 개발할 때 좋은 커밋을 작성하기란 참 어렵습니다. 제 말하는 "좋은 커밋"이란 이런 식을 말합니다.

```
- 타입 정의
- DB 함수 추가
- 서버 CRUD 로직
- 클라이언트 API
- 클라이언트 UI
```

리뷰어가 풀 리퀘스트를 작은 단위로 차근차근 살펴볼 수 있고, 각 커밋이 기능의 한 가지 측면에만 집중되어 있거든요.

그런데 저는 실제로는 이렇게 합니다.

```
- 타입 정의
- DB 함수 추가
- WIP 테스트 코드
- 클라이언트 API와 UI
- DB 함수 수정
- UI 버그 수정
- CRUD 리팩터링
- 또 다른 UI 버그 수정
```

이후의 커밋들이 이전 커밋의 작업을 덮어씌우면서 전체 스토리가 엉망이 되는 거죠.

## 기존 도구들의 한계

Jujutsu는 커밋 간에 쉽게 이동하고 변경 사항을 빠르게 반복할 수 있게 해줍니다. 하지만 여전히 노력이 많이 필요하고, 자연스레 회피하게 되죠.

`jj absorb`나 `jj squash -i` 같은 도구들이 어느 정도 도움은 됩니다. 다만 각각 문제가 있습니다.

**absorb**는 가장 최근에 해당 파일을 건드린 커밋으로 변경 사항을 자동 할당합니다. 그런데 실제로는 다른 커밋이 그 변경 사항을 소유해야 하는 경우가 있거든요.

**squash -i**는 경계가 매우 깔끔하지 않으면 병합 충돌 지옥에 빠질 수 있습니다.

## 새로운 방식: 커밋 먼저 설계하기

"깃 엄격함 피로"라는 문제를 해결할 방법을 생각해봤습니다.

시각적으로 표현해보겠습니다. 빨간색이 타입 정의의 변경사항, 파란색이 UI 변경사항이라고 하면:

```
혼란스러운 상태: 첫 번째 커밋이 빨간색과 파란색이 섞여 있고,
빨간색도 여러 곳에서 건드려져 있다.
```

이를 해결하려면 먼저 이상적인 커밋 히스토리를 만들어봅시다.

```bash
jj new -B messy-first -m 'red'
```

그 다음 나머지를 진행합니다. (이 시점에서 `jj new -A red -m 'blue'`로 전환합니다)

그 다음 모든 변경 사항이 포함된 커밋들을 하나로 합칩니다.

```bash
jj squash --from messy-first..messy-last --into messy-first
```

그리고 `jj squash -i --from messy-first --into red`를 사용해서 빨간색 변경사항만 선택해서 빨간색 박스에 넣습니다.

이런 식으로 계속 진행하면:

결국 모든 변경사항이 올바른 위치에 들어가고, "모든 것이 들어가는" 커밋은 비워집니다.

큰 기능을 개발할 때 저는 이 워크플로우가 개발 주기 동안 엄격한 규칙을 유지해야 하는 것보다 훨씬 쉽다고 느낍니다. 임시 디버깅 상태가 포함된 즉흥적인 커밋을 만들고, 마지막에 한 번에 정리하는 게 훨씬 간단하거든요.

## 추가 설명

이 방법의 이름을 딱 지으러 말하면... "세탁물 큰 더미처럼 커밋하기"일까요?

이 방법은 `jj split`과는 다릅니다. `split`은 빨간색에 들어가야 할 덩어리를 놓친다면 다시 split해야 하고 squash해야 합니다. 이 기법은 처음부터 끝까지 걱정 없이 가장 간단한 덩어리들을 먼저 정렬할 수 있게 해줍니다.

끝에서 모두 정렬하는 게 진행 중에 `jj squash -i`를 사용하는 것보다 (종종) 나은 이유는, 모든 것을 담는 커밋의 최종 상태가 충돌이 없다는 보장이 있기 때문입니다. 새로운 "빨강과 초록 수정" 커밋을 만들어서 이걸 빨강과 초록 커밋에 대화형으로 squash하면, 그 파일들을 건드린다면 파란색 커밋을 깨뜨릴 수 있거든요.

이 기법의 단점은 모든 커밋이 컴파일되리라는 보장이 없다는 겁니다. 이건 결정적인 문제일 수도 있습니다.
