## 핵심 개념

> [!abstract]
> GitHub의 신뢰성, 보안, 성능 문제가 심화되고 있으며, Microsoft가 AI 에이전트 기능을 과도하게 밀어붙이면서 기본적인 안정성을 무시하고 있다고 비판한다. 공식 성명과 달리 실제 업데이트 로그에서 안정성과 성능 개선은 전혀 없고 Copilot만 59번 언급되는 등 우선순위가 왜곡되어 있다.

## 상세 내용

- GitHub의 실제 가동 시간은 99.8%가 아니라 훨씬 낮으며, 리포지토리 생성과 PR 활동 증가로 인한 시스템 병목 현상이 심화되고 있다.
- Microsoft는 사용자에게 AI/에이전트 기능을 강제하기 위해 저가 정책과 UI 조작을 지속하고 있으며, 공식 성명에서 안정성을 최우선이라 했으나 실제 개발 우선순위는 AI 기능에만 집중되어 있다.

> [!tip] 왜 중요한가
> 개발자들이 의존하는 주요 플랫폼의 신뢰성 저하와 불일치하는 개발 방향성은 생태계 전체에 부정적 영향을 미칠 수 있다.

## 현재 상황

GitHub이 또 다운되었습니다. The Verge 같은 매체에서도 GitHub의 신뢰성, 보안, 성능 문제를 다루곤 했지만, 그들은 겨우 빙산의 일각만 건드린 것입니다.

GitHub은 단순한 코드 저장소가 아닙니다. 소프트웨어 개발의 거의 동의어에 가까운 서비스거든요. 2022년 어떤 채용담당자와 대화할 때, GitHub 계정이 없다는 이유만으로 저를 "진정한 프로그래머가 아니다"라고 평가한 적이 있습니다. 이것이 GitHub이 소프트웨어 업계에서 차지하는 위상을 보여줍니다.

GitHub의 인프라 문제는 AWS부터 Google Search까지 대형 테크 기업의 서비스 전반에 걸쳐 나타나는 광범위한 쇠퇴의 신호입니다. 저는 전문적으로 고성능, 고가용성, 고용량의 분산 시스템을 다루고 있어서, 일반 기술 기자보다는 이 문제를 조금 더 명확히 볼 수 있습니다.

---

## 공식 발표와 현실의 괴리

GitHub의 공식 상태 페이지는 99.8% 정도의 가동 시간을 주장합니다. 하지만 최근에 GitHub을 사용해본 사람이라면 실제 숫자는 그 정도가 아니라는 걸 압니다. Missing GitHub Status Page 같은 사이트들이 실제 상황을 더 정확히 보여주죠.

Microsoft는 최근 "GitHub 가용성 업데이트"라는 보도자료를 내놓았습니다. 비판을 피하려는 의도가 명확합니다. 그 안에서 Microsoft는 이렇게 주장했습니다:

> 주요 원인은 소프트웨어 개발 방식의 급격한 변화입니다. 2025년 12월 하반부 이후, 에이전트 기반 개발 워크플로우가 급속도로 증가했습니다. 저장소 생성, 풀 리퀘스트 활동, API 사용량, 자동화, 대규모 저장소 작업 등 거의 모든 지표가 빠르게 증가하고 있습니다.

---

## 자들이 자초한 문제

여기서 흥미로운 점이 있습니다. Microsoft는 "에이전트 기반 개발 워크플로우가 급속도로 증가했습니다"라고 수동태로 표현했는데, 이는 거짓입니다.

GitHub과 Microsoft는 모든 곳에 AI와 에이전트를 밀어붙이고 있습니다. 평균적인 저장소 페이지만 해도 상단 우측에만 Copilot을 실행할 수 있는 버튼이 4개나 있습니다. 4개요. 그리고 저는 이 글을 쓰면서 세 번째 에이전트 버튼을 발견했고, 에이전트를 명시적으로 비활성화하지 않은 저장소에는 네 번째 버튼까지 나타납니다. 이건 병적인 수준입니다.

더 문제적인 것은, Microsoft가 수년 동안 이러한 도구 사용 비용을 보조해서 채택을 유도했다는 점입니다. 사실상 자신에게 분산 서비스 거부 공격을 퍼붓고 있는 셈이죠.

---

## "우선순위는 명확합니다"라는 거짓말

Microsoft는 공개적으로 이렇게 선언했습니다: "우리의 우선순위는 명확합니다: 가용성 우선, 그 다음 용량, 그리고 새로운 기능들입니다."

이는 거짓입니다. GitHub과 Microsoft 조직 전반에서 실제 우선순위는 기본적인 신뢰성보다 화려한 AI 기능을 훨씬 더 높게 두고 있습니다.

GitHub의 공개 변경 로그를 보면 답이 나옵니다. 최근 30일간의 패치 노트에서:
- "copilot": 59회
- "agent": 8회
- "performance": 0회
- "reliability": 0회

변경 로그에는 카테고리별 필터 기능이 있는데, Copilot은 독립적인 카테고리로 존재합니다. 반면 성능과 신뢰성은 카테고리 자체가 없습니다.

이건 GitHub만의 문제가 아닙니다. Microsoft 전체 조직에 만연되어 있습니다. 같은 회사가 소유한 Visual Studio Code도 마찬가지 패턴을 보여줍니다.

---

## 데이터베이스 엔지니어들의 딜레마

사실, GitHub의 데이터베이스 엔지니어들의 일은 결코 쉽지 않습니다. 풀 리퀘스트 하나가 Git 저장소, 병합 가능성 검사, 브랜치 보호, GitHub Actions, 검색, 알림, 권한, 웹훅, API, 백그라운드 작업, 캐시, 데이터베이스 등 여러 시스템을 동시에 건드리기 때문입니다.

대규모에서는 작은 비효율이 복합되는데, 큐가 깊어지고, 캐시 미스가 데이터베이스 부하로 변하고, 인덱스가 밀리고, 재시도가 트래픽을 증폭시키고, 느린 한 시스템이 여러 제품 경험을 방해합니다.

하지만 이건 "작은 비효율"의 문제가 아닙니다. GitHub에는 엄청난 규모의 비효율이 있습니다. 그리고 그것들은 대부분 의도적입니다.

GitHub과 Microsoft가 사용자를 AI와 에이전트로 몰아붙이면서, 인프라가 이를 감당할 준비가 되지 않은 상태에서 부하만 계속 증가했거든요. 이것이 진짜 문제입니다.
