## 핵심 개념

> [!abstract]
> IBM은 엔터프라이즈용으로 특화된 오픈소스 언어 모델 Granite 4.1 계열을 출시했으며, 3B, 8B, 30B 세 가지 크기의 모든 모델이 Apache 2.0 라이선스로 제공됩니다. 15조 개의 토큰으로 훈련되었으며, 특히 8B 모델이 이전 세대의 32B MoE(9B 활성) 모델보다 ArenaHard, BFCL V3, GSM8K 등 주요 벤치마크에서 우수한 성능을 보여줍니다. 이는 매개변수 확장보다 데이터 품질에 집중한 결과입니다.

## 상세 내용

- 8B 모델의 우수한 벤치마크 성능: ArenaHard에서 69.0(32B MoE는 더 낮음), BFCL V3 68.3(32B MoE는 64.7), GSM8K 92.5, AlpacaEval, MMLU-Pro, BBH, EvalPlus, MBPP 등 전반적으로 더 큰 모델을 압도
- 아키텍처 철학: 3B, 8B, 30B 모두 동일한 디코더 전용 밀집 트랜스포머 설계로 MoE 라우팅, 스파스 레이어, 확장된 추론 체인 없음 - 입력이 예측 가능하게 일정한 토큰만 처리되어 비용과 레이턴시 예측이 용이
- 5단계 훈련 파이프라인: Phase 1은 CommonCrawl 59%, 코드 20%, 수학 7%; Phase 2에서 수학 35%, 코드 30%로 조정; Phase 3-4에서 체인-오브-쓰트 추론 궤적과 지시 데이터 혼합; Phase 5에서 컨텍스트 윈도우를 512K 토큰으로 확장
- 데이터 품질 필터링 시스템: LLM-as-Judge가 모든 어시스턴트 응답을 지시 준수, 정확성, 완전성, 간결성, 자연스러움, 보정 6가지 차원으로 평가하고 임계값 미만은 자동 제거, 환각, 거짓 전제, 오류 계산은 스코어 관계없이 자동 거부
- 컨텍스트 기반 평가: 판별기가 전체 컨텍스트에서 응답을 평가하므로 RAG 설정에서 검색된 문서에 근거하지 않으면 환각으로 판정, 도구 호출 시나리오에서는 출력을 허용된 도구와 매개변수 스키마에 대해 검증
- 규칙 기반 파이프라인: 길이, 포매팅, 스키마 검증, 데이터셋 전체 중복 제거를 체계적으로 확인하며 모든 과정이 로깅되고 감사 가능

> [!tip] 왜 중요한가
> 8B 모델이 32B 모델과 경쟁력 있는 성능을 보여주므로 엔터프라이즈에서 훨씬 낮은 계산 비용으로 고품질 AI를 배포할 수 있으며, 데이터 품질 중심의 훈련 철학은 업계의 스케일-위주 접근법에 대한 중요한 반례입니다.

## 전문 번역

# Granite 4.1: IBM의 8B 모델이 4배 큰 모델들과 경쟁하다

IBM이 최근 엔터프라이즈 환경을 위해 특화된 오픈소스 언어 모델 Granite 4.1을 출시했습니다. 3가지 사이즈, Apache 2.0 라이선스, 15조 개의 토큰으로 학습했는데요. 이 과정에서 데이터 파이프라인에 대한 집착이 정말 눈에 띄거든요.

벤치마크 결과 중 하나는 그들이 어떻게 만들었는지 이해하기 전까지는 뭔가 이상해 보입니다.

8B 모델 말이에요. 기본적인 밀집 구조(Dense Architecture), MoE 같은 트릭도 없고, 긴 추론 과정도 없습니다. 그런데 이전 세대인 Granite 4.0-H-Small과 거의 모든 벤치마크에서 같거나 더 좋은 성능을 냈어요. 저 모델은 320억 개의 파라미터 중 90억 개가 활성화되는데, 이번 모델은 80억 개입니다.

이건 정말 인상적이거나, 아니면 예전 모델이 너무 과하게 만들어진 거거나. 아마 둘 다일 겁니다.

## 어? 이게 뭐지? 싶게 만드는 결과들

ArenaHard는 GPT-4가 500개의 까다로운 실제 프롬프트를 얼마나 잘 처리하는지 평가하는 벤치마크인데, 실제 채팅 품질을 꽤 잘 반영하거든요. 8B 인스트럭트 모델이 여기서 69.0점을 얻었습니다. 이전 세대 32B MoE 모델(활성화 파라미터 9B)은 더 낮은 점수였어요.

BFCL V3(도구 호출 벤치마크의 표준)에서도요. 8B가 68.3점, 32B MoE가 64.7점입니다. GSM8K(초등학교 수준의 수학 문제)는 92.5점. AlpacaEval, MMLU-Pro, BBH, EvalPlus, MBPP 등 거의 모든 벤치마크에서 같은 패턴이 반복됩니다.

더 작고 간단한 모델이 계속 이기는 거예요.

## IBM이 정말 많이 성장했다는 뜻

사실 이건 IBM이 세대 간에 얼마나 크게 개선됐는지를 보여줍니다. 4.0-H-Small이 못 만들어진 모델은 아니었어요. 그때는 최고의 선택지였거든요. 4.1 8B가 이 정도 성능을 내는 건, 파라미터를 늘리는 대신 데이터 품질에 집착한 기간을 거친 결과입니다. 이게 Granite 4.1 전체를 관통하는 핵심 이야기예요.

## 3가지 사이즈, 하나의 집착: 실제 만드는 방식

Granite 4.1은 3B, 8B, 30B 3가지 사이즈로 나옵니다. 셋 다 같은 디코더 전용 밀집 트랜스포머 아키텍처, 같은 학습 파이프라인, 같은 데이터 전략을 씁니다. 차이는 크기뿐이에요. MoE 라우팅도 없고, 스파스 레이어도 없고, 토큰 수를 부풀리는 긴 추론 과정도 없습니다. 들어온 것이 나갈 것이고, 매번 예측 가능합니다.

긴 추론 과정에 의존하는 모델은 비용을 예측하기 어렵고 지연 시간을 예산화하기 어렵거든요. Granite 4.1은 설계 단계에서 이 모든 걸 빼버렸습니다. 하지만 이 아키텍처가 진짜 이야기는 아닙니다. 진짜 이야기는 15조 개의 토큰과 그걸 얼마나 신경 써서 다뤘는지예요.

IBM은 5단계의 학습 단계를 거쳤는데, 각 단계마다 다른 데이터 조합, 다른 학습률 스케줄, 다른 목표를 갖고 진행했습니다:

- **1단계**: CommonCrawl 59%, 코드 20%, 수학 7%
- **2단계**: 수학이 35%, 코드가 30%로 올라갑니다
- **3, 4단계**: 사고 연쇄(Chain-of-Thought) 추론 궤적과 명령어 데이터를 최고 품질의 웹 콘텐츠와 섞습니다
- **5단계**: 컨텍스트 윈도우를 확장해서 8B와 30B는 결국 512K 토큰까지 갑니다

대부분의 팀은 데이터 구성을 정하고 계속 쓰는데, IBM은 명확한 목적 아래 4번이나 바꿨어요.

## 안 좋은 데이터가 피해를 주기 전에 걸러내는 필터

IBM의 데이터 품질 파이프라인은 따로 설명할 만한 가치가 있습니다.

사전학습 후 기본 모델을 실제로 명령을 따르는 모델로 만들어야 하는데, 이때 좋은 동작 예시로 미세조정합니다. 그런데 여기서 문제가 생겨요. 나쁜 예시는 무시되지 않습니다. 학습되어 버립니다. 환각(Hallucination), 명령을 무시하는 응답, 자신감 있지만 틀린 계산... 모델은 이 모든 걸 신호로 취급해요.

그래서 IBM은 미세조정을 하기 전에 필터링 시스템을 만들었습니다. LLM-as-Judge가 모든 모델 응답을 6가지 차원으로 평가했어요:

- 명령어 준수(Instruction Following)
- 정확성(Correctness)
- 완전성(Completeness)
- 간결성(Conciseness)
- 자연스러움(Naturalness)
- 보정(Calibration)

각 응답이 점수를 받았고, 임계값 아래는 버려졌습니다. 단, 일부는 점수와 관계없이 자동으로 거부됐는데요. 환각, 거짓 전제, 계산 오류가 그것들입니다. 이건 부분점수가 없어요.

이 판사는 프롬프트나 사용자 입력을 따로 보지 않습니다. 모델이 접근할 수 있었던 전체 컨텍스트를 고려해서 응답을 평가했어요. RAG 환경에서라면, 응답이 검색된 문서에 기반하지 않으면 환각으로 처리했습니다. 도구 호출(Tool-Calling) 상황에서는 출력을 허용된 도구와 그들의 파라미터 스키마와 비교했어요.

이 위에 규칙 기반 파이프라인이 또 한 겹 있었습니다. 길이, 형식, 스키마 검증, 전체 데이터셋 중복 제거를 체크했어요. 모든 것이 기록되고 감사 가능했습니다.

이 과정을 거쳐 나온 결과가 Granite 4.1입니다.
