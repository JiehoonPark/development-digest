## 전문 번역

# Project Glasswing: Mythos Preview가 우리에게 보여준 것

지난 몇 개월간 우리는 자체 인프라에서 보안에 특화된 여러 대규모언어모델(LLM)을 테스트해왔습니다. 이 모델들은 우리 시스템의 잠재적 취약점을 찾는 데 도움이 될 뿐 아니라, 최신 모델로 공격자들이 무엇을 할 수 있을지 보여줍니다.

이 중에서 가장 주목받은 건 Anthropic의 Mythos Preview입니다. 몇 주 전 우리는 Project Glasswing의 일환으로 Mythos Preview를 사용할 기회를 얻었습니다. 우리는 50개 이상의 자체 저장소를 대상으로 테스트를 진행했는데요. 이 글에서는 우리가 발견한 것, 모델이 잘해낸 것과 부족했던 것, 그리고 대규모로 활용하기 위해 어떻게 개선되어야 하는지 공유하겠습니다.

## Mythos Preview의 변화

먼저 명확히 하자면, Mythos Preview는 정말로 큰 도약입니다. 우리는 한동안 모델들을 코드에 돌려봤는데, 이전 범용 최신 모델들로 가능했던 것과 Mythos Preview가 할 수 있는 것의 차이는 단순한 개선 수준이 아닙니다.

이건 완전히 다른 종류의 도구가 다른 종류의 일을 하는 거거든요. 그래서 Mythos Preview를 이전 모델들과 직접 비교하기는 어렵습니다. 대신 Mythos Preview가 실제로 무엇을 할 수 있는지 설명하는 게 더 유용한데, 우리가 테스트하면서 두 가지 기능이 특히 눈에 띄었습니다.

### 1. 익스플로잇 체인 구성

실제 공격은 보통 하나의 버그로는 이루어지지 않습니다. 여러 개의 작은 공격 기법을 조합해서 작동하는 익스플로잇을 만들어내죠. 예를 들어 use-after-free 버그를 임의의 읽기/쓰기 기능으로 변환하고, 제어 흐름을 탈취한 뒤, ROP 체인을 통해 시스템을 완전히 제어하는 식입니다.

Mythos Preview는 이런 여러 기법들을 인식하고, 어떻게 조합해야 작동하는 증명을 만들 수 있을지 추론할 수 있습니다. 이 과정에서 보여주는 추론 능력은 자동화된 스캐너의 결과라기보다는 시니어 연구원의 작업물처럼 보입니다.

### 2. 증명 생성

버그를 찾는 것과 그 버그가 실제로 익스플로잇 가능함을 증명하는 것은 다른 일입니다. Mythos Preview는 둘 다 할 수 있어요. 의심되는 버그를 트리거하는 코드를 작성하고, 격리된 환경에서 컴파일한 뒤 실행합니다. 프로그램이 예상대로 동작하면 그것이 증명이 됩니다. 그렇지 않으면 모델이 실패를 분석하고, 가설을 조정한 뒤 다시 시도합니다.

이 반복 과정이 중요한 이유는, 증명되지 않은 버그는 그냥 추측일 뿐이기 때문입니다. Mythos Preview는 이 간격을 스스로 채워냅니다.

## 다른 모델들과의 차이점

우리가 같은 테스트 환경에서 다른 최신 모델들을 돌려봤을 때, 그들도 같은 기저 버그 중 많은 것을 찾아냈고, 추론 부분에서도 우리 기대치를 넘어서기도 했습니다. 하지만 부족했던 부분은 바로 이 조각들을 이어붙이는 단계였어요.

모델이 흥미로운 버그를 찾고, 그것이 왜 중요한지 설명하는 텍스트를 작성한 뒤 멈춰버리는 식입니다. 실제 체인은 완성되지 않고, 그것이 실제로 익스플로잇 가능한지에 대한 질문은 열려있는 거죠.

Mythos Preview가 달라진 점은 전통적으로는 미처 해결하지 못한 채 백로그에만 남아있을 낮은 심각도 버그들을, 하나의 더 심각한 익스플로잇으로 연결할 수 있다는 겁니다.

## 정당한 취약점 연구에서의 모델 거부

Project Glasswing의 일환으로 Anthropic이 제공한 Mythos Preview는 일반 공개 모델들(Opus 4.7이나 GPT-5.5 같은)에 있는 추가 보안 장치들이 없었습니다.

그럼에도 불구하고 이 모델은 특정 요청에 대해 거부 반응을 보입니다. 사이버 보안 기능 덕분에 취약점 사냥에는 유용하지만, 정당한 보안 연구 요청에 대해서도 자발적인 보호 메커니즘이 활성화되는 거죠.

문제는 이런 자발적 거부가 일관성이 없다는 겁니다. 같은 작업이라도 다른 방식으로 표현하거나 다른 맥락에서 제시하면 완전히 다른 결과가 나옵니다.

### 구체적인 사례들

모델이 처음엔 어떤 프로젝트의 취약점 연구를 거부했는데, 프로젝트 환경에 전혀 관련 없는 다른 변경을 했더니 같은 코드에 대해 같은 연구를 수행하기로 동의한 경우가 있었습니다. 분석 대상이 된 코드는 아무것도 바뀌지 않았는데 말입니다.

또 다른 경우엔 모델이 코드베이스의 심각한 메모리 버그 여러 개를 찾아내고 확인한 후, 그 악용 방법을 시연하는 코드 작성을 거부했습니다. 같은 요청을 다르게 표현하면 다른 답변을 얻을 수 있었고, 모델의 확률적 특성 때문에 같은 요청도 실행 횟수에 따라 다른 결과를 낼 수 있었어요.

의미상 동일한 작업도 제시 방식과 타이밍에 따라 정반대의 결과가 나올 수 있다는 뜻입니다.

이건 중요한 문제인데요. 모델의 자발적 거부/보호 메커니즘이 실제로는 존재하지만, 완전한 안전장치로 기능하기엔 일관성이 부족하기 때문입니다.
