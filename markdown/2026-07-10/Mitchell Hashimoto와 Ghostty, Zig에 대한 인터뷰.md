## 핵심 개념

> [!abstract]
> Vagrant, Terraform 등을 만든 Mitchell Hashimoto가 새로운 터미널 에뮬레이터 Ghostty와 Zig 언어에 대해 이야기한다. 그는 CLI 애플리케이션 개발 경력을 바탕으로 GPU 프로그래밍, 데스크톱 시스템 프로그래밍, 그리고 Zig 학습을 목표로 Ghostty를 만들었다고 밝혔다.

## 상세 내용

- Ghostty는 빠르고 기능 풍부하며 크로스플랫폼을 지원하는 터미널 에뮬레이터로, 기존 터미널의 한계를 개선하기 위해 설계됨
- PTY의 대역내 신호(이스케이프 시퀀스 포함 바이트 스트림)는 근본적인 문제이며, 구조화된 데이터를 지원하는 더 나은 API 필요
- 텍스트 기반 애플리케이션은 브라우저나 데스크톱과 다른 장점이 있으며, UNIX 철학의 조합 가능성을 강화하면 더 나은 자동화와 스크립팅이 가능

> [!tip] 왜 중요한가
> 터미널 도구 개발자는 Ghostty와 같은 현대적 터미널 아키텍처의 설계 철학을 이해하고, 새로운 프로토콜과 API 설계 시 기존 생태계의 한계를 고려해야 한다.

## 전문 번역

# 미첼 해시모토 인터뷰: 터미널, Zig, 그리고 오픈소스

**2026년 7월**

Vagrant, Packer, Consul, Terraform, Vault, Nomad, Waypoint를 만들었고, 지금은 Ghostty와 Vouch를 개발 중인 미첼 해시모토. 이번 인터뷰에서는 터미널, Zig, 그리고 오픈소스에 대해 이야기 나눴습니다.

## 왜 자주 인터뷰 요청이 들어올까요?

사람들이 저에게 관심 갖는 이유는 다양해요. 소프트웨어 엔지니어에서 창업자로 넘어가는 과정을 궁금해하는 사람도 많고, HashiCorp에서의 경험이나 지금의 Ghostty 같은 프로덕트 이야기를 듣고 싶어하는 사람도 있죠.

이번이 특별한 이유라면, 어느 쪽도 특정 목표나 팔아야 할 무언가가 없다는 점입니다. 순수하게 대화를 나누는 거거든요.

## 터미널에 빠진 이유, Ghostty는 왜?

약 15년간 CLI 애플리케이션을 만들어왔어요. (요즘의 TUI와는 달리 순수 CLI요) 그 과정에서 자연스럽게 색상 처리, 커서 이동 같은 것들을 배웠죠.

HashiCorp를 떠나면서 기술 스킬을 다시 갈고닦고 싶었습니다. 특히 세 가지에 집중하고 싶었거든요.

- AI 이전의 GPU 프로그래밍
- 데스크톱과 단일 노드 시스템 프로그래밍 (분산 시스템만 다루면서 캐시 지역성이나 벡터 연산을 신경 쓸 일이 없었거든요)
- Zig로 뭔가 만들어보기

15년간 CLI를 만들면서도, 정작 터미널 에뮬레이터가 어떻게 동작하는지는 몰랐어요. 개념으로는 알지만, 진짜 깊이 이해하고 싶었습니다. 그래야 GPU, 데스크톱, Zig로 작업할 수 있었거든요.

처음 목표는 단순했습니다. Vim과 컴파일러를 실행할 수 있는 터미널을 만들고, 자체 빌드가 가능하게 한 다음 버리는 거였어요.

하지만 터미널 생태계를 공부하다 보니 깨달았어요. 제가 원하는 조건을 모두 충족하는 게 없더라고요. 빠르고, 풍부한 기능을 가지면서도, 크로스플랫폼 지원이 네이티브로 되어 있는 것 말이에요.

몇몇 친구들한테 Discord에서 공유했는데, 일상적으로 쓰기 좋다며 다른 사람들과도 나눠도 되냐고 물었어요. 지금의 Ghostty Discord는 원래 친구들끼리의 대화방이었던 거죠. 공개하고 싶지 않았던 이유는 제 이름값이 불필요한 주목을 끌까봐서였어요. 그래서 오래 비공개 베타를 진행했습니다.

## 터미널을 어디까지 밀어붙일 수 있을까?

저는 터미널을 극단까지 밀어붙이는 것에 반대합니다. 물론 터미널도 애플리케이션 플랫폼이고, 브라우저처럼 OS 위의 다른 플랫폼과 같은 역할을 할 수 있어요. 비디오, 마이크 접근, 반응형 레이아웃 같은 모든 기능을 넣을 수도 있죠.

하지만 각각의 플랫폼이 잘하는 게 따로 있다고 봅니다. 브라우저는 브라우저대로, 데스크톱은 데스크톱대로, 텍스트 기반(고정폭 그리드) 애플리케이션은 그것만의 장점이 있어요. 이런 텍스트 기반 애플리케이션들은 빠르게 만들어지고, 쉽게 상호작용하며, 보안 모델이 명확해야 한다고 생각합니다.

이 생태계에는 엄청난 기회가 있고, 이를 지원할 더 나은 프로토콜을 만들고 싶어요.

### 터미널 앱의 조합 가능성

터미널 기반 애플리케이션은 다른 패러다임보다 조합성이 훨씬 뛰어납니다. TUI는 덜하지만, 대부분의 CLI 도구들은 stdin과 stdout 외에도 함수처럼 사용할 수 있는 메커니즘을 가지고 있어요. UNIX의 "한 가지 일을 잘하자"는 철학의 극단이 바로 그것입니다.

Neovim과 AI 도구들을 보면 점점 더 많은 커맨드라인 플래그를 제공하고 있어요. 터미널 애플리케이션이 더 나아진다는 것은 자동화와 스크립팅도 더 나아진다는 뜻입니다.

저는 터미널을 애플리케이션을 위한 특별한 공간으로 만들고 싶어요.

### 구조화된 데이터의 필요성

PTY의 대역내 신호 방식(구조화되지 않은 바이트 스트림에 escape sequence를 섞는 방식)이 큰 문제입니다. Nushell 생태계는 다른 레이어로 이를 해결하려 하지만, 근본적인 개선이 필요해요.

Microsoft 생태계를 좋아하지 않는 사람도 많지만, PowerShell은 구조화된 데이터 처리에 있어 많은 것을 올바르게 했어요.

## 레거시가 아닌 새로운 터미널 API들

제 나침반은 현재 존재하는 주요 애플리케이션 플랫폼들입니다. 브라우저, Emacs, Apple 생태계, Microsoft 생태계, Android, 게임 콘솔 플랫폼들 말이에요.

각 생태계는 강점과 약점이 있지만, 프레임워크가 어떻게 작동하는지 살펴보면 패턴이 보여요. 웹은 DOM과 JS API, Apple은 AppKit와 Cocoa, SwiftUI, Windows는 Win32와 WinUI, Linux는 GTK와 Qt입니다.

누군가 "클립보드 데이터에 더 나은 접근 방식이 필요해"라고 말할 때, 저는 각 플랫폼의 클립보드 매니저 문서를 들춰봐요. 이미지, 여러 MIME 타입처럼 데스크톱에서 수십 년간 다루던 것들 말입니다.

자신의 이해만으로 무언가를 만들어야 할 이유가 없어요. 수십 년간 쌓인 사전학습이 있으니까요. 저는 이런 방식으로 접근하려고 해요. 아직까지는 커스텀 프로토콜을 도입하지 않았습니다.

### 두 가지 프로토콜이 눈에 띕니다

현재 터미널은 메인 스크린과 알트 스크린(primary와 secondary라고도 부르는) 두 개만 가지고 있고, 각각 다른 성질을 가져요. 메인 스크린은 셸처럼 스크롤백이 있고, 알트 스크린은 Neovim이나 대부분의 TUI처럼 화면 전체를 차지하면서 스크롤백을 잃습니다.

저는 무한한 수의 스크린을 배경에서 생성하고 채울 수 있는 n-screen API를 도입하고 싶어요. 그리고 스크린을 겹쳐서 표현할 수 있게 말이죠.
