## 핵심 개념

> [!abstract]
> Rust와 C/C++에서 메모리 안전 관련 CVE가 보도되는 방식에 근본적인 차이가 있다. C/C++에서는 API 오용으로 인한 미정의 동작이 라이브러리의 책임으로 간주되지 않지만, Rust의 타입 시스템은 이러한 계약 위반을 컴파일 단계에서 방지한다.

## 상세 내용

- C/C++는 제한된 타입 시스템으로 인해 API 계약을 명확히 지정하기 어려워 오용을 라이브러리 책임으로 보지 않음
- Rust는 강력한 타입 시스템으로 많은 메모리 안전 문제를 컴파일 타임에 방지하여 실제로 더 안전함
- 같은 수의 CVE가 존재해도 Rust는 `unsafe` 키워드 필요, C/C++는 일반적인 코드에서도 발생 가능

> [!tip] 왜 중요한가
> 개발자들이 CVE 개수만으로 언어의 안전성을 비교할 때 메모리 안전 문제의 근본적인 차이를 이해해야 한다.

## 전문 번역

# CVE와 메모리 안전성: Rust vs C/C++의 본질적 차이

CVE 데이터베이스는 소프트웨어의 보안 취약점을 분류하고 보고하는 데 사용됩니다. 취약점의 종류는 다양한데, 단순한 프로그램 로직 버그에서 비롯되기도 하지만 가장 심각한 것들은 메모리 안전성 문제에서 비롯됩니다. 이런 메모리 문제는 악용하기 쉬운 보안 취약점으로 이어질 수 있거든요.

이 글에서는 메모리 안전성 관련 CVE가 특히 라이브러리에서 어떻게 보고되는지, 그리고 Rust와 C/C++에서 어떻게 다르게 다루어지는지에 대해 살펴보겠습니다.

온라인에서 Rust와 C/C++ 소프트웨어의 CVE 개수를 비교하는 논의를 종종 보게 됩니다. 그런데 이런 비교에서 "Rust도 여전히 CVE가 있으니 진짜 메모리 안전하지 않다" 또는 "Rust를 굳이 도입할 필요가 없다"는 주장이 뒤따르곤 합니다. C나 C++에서 온 프로그래머들을 Rust로 가르칠 때도 비슷한 의문을 자주 만나게 되지요.

누구나 이런 비교를 할 자유가 있겠지만, 메모리 안전성과 관련된 잠재적 취약점을 Rust와 C/C++에서 다루는 방식에는 중요한 차이가 있습니다. 특히 Rust가 어떻게 동작하는지 모르면 이 차이가 명확하지 않을 수 있어서, 이 글에서 설명해보려고 합니다.

## Rust도 메모리 안전 문제가 있나요?

먼저 명확히 해야 할 점이 있습니다. Rust에서도 메모리 안전 버그와 정의되지 않은 동작(undefined behavior, UB)을 일으킬 수 있습니다. 대부분의 경우 `unsafe` 키워드가 필요하지만, "Rust 프로그램은 절대 UB를 겪을 수 없다"고 주장하는 사람은 틀렸습니다.

또한 메모리 안전성과 무관한 일반적인 취약점도 Rust에서 충분히 발생할 수 있습니다. 예를 들어, 관리자 대시보드 접근 권한을 확인하지 않는 실수는 어떤 언어에서나 일어날 수 있으니까요.

그렇다면 뭐가 다른 걸까요? Rust가 실제로 C/C++보다 메모리 안전한 이유의 핵심과 관련이 있는데, 이를 C로 작성된 curl 라이브러리로 설명해보겠습니다.

## curl 라이브러리로 본 취약점 이야기

libcurl은 세계에서 가장 널리 사용되고 유지보수가 잘 되는 오픈소스 라이브러리 중 하나입니다. 주 개발자인 Daniel Stenberg는 시대의 가장 활동적인 오픈소스 유지보수자 중 한 명이고, 지난 30년간 많은 사람들과 함께 이 라이브러리를 꾸준히 개선해왔습니다.

최근 LLM이 발견한 CVE들의 폭주에도 불구하고, curl 팀은 잠재적 보안 취약점으로부터 curl을 지키는 일을 훌륭히 해내고 있으며, curl이 매우 견고한 소프트웨어라는 점에 자부심을 가지고 있습니다.

그럼 한번 시험해볼까요? libcurl 문서를 열어서 첫 번째로 눈에 띄는 함수 `curl_getenv`를 찾아봤습니다. 이 함수는 서로 다른 운영체제에서도 환경 변수 값을 가져올 수 있도록 포터블한 추상화를 제공하는 간단한 함수입니다.

curl이 안전하고 견고하다면, 이 함수에는 UB나 메모리 안전 문제가 없겠지요?

그럼 다음 C 프로그램은 어떨까요?

```c
#include <curl/curl.h>

int main(void) {
    curl_getenv(NULL);
}
```

이 5줄짜리 프로그램은 간단하기 그지없습니다. `curl_getenv` 함수를 NULL 포인터로 호출할 뿐입니다. 경고 없이 컴파일되지만, 실행하면 어떻게 될까요?

```
$ gcc test.c -otest -lcurl -Wall -Wextra
$ ./test
Segmentation fault (core dumped)
```

메모리 안전 버그가 발생합니다. 그래서 이게 잠재적 취약점이 될 수 있는 거죠.

물론 이 프로그램은 의도적으로 단순하게 만들었지만, 그 게 요점입니다. 실제로는 더 큰 프로그램에서 이런 상황이 실수로 쉽게 발생할 수 있습니다.

## 그럼 curl이 안전하지 않은 건가?

아니요, 당연히 이걸 curl의 취약점으로 보고하면 안 됩니다. 누가 봐도 그럴 거거든요. 하지만 우리는 어떻게 그걸 알 수 있을까요? 여기가 흥미로운 부분입니다.

다음과 같이 호출하는 비슷한 프로그램을 생각해봅시다: `curl_getenv("FOO")`. 만약 이 프로그램도 세그멘테이션 폴트가 난다면? curl 유지보수자들은 확실히 이 문제를 알아야 할 것 같고, 정말 심각한 이슈라고 생각할 거예요. 하지만 동시에 그들은 첫 번째 프로그램을 취약점으로 보고받으면 당연히 짜증낼 겁니다. 그런데 두 프로그램의 차이는 정말 미미합니다.

뭐가 다른 걸까요? 실제로는 제 첫 번째 예제 같은 UB를 "잘못된 사용법(wrong usage)"이라고 부릅니다. 이것을 라이브러리나 API의 문제가 아니라, 내 애플리케이션 코드의 문제로 간주하는 거죠. 왜 이렇게 하냐면:

**첫째**, C는 타입 시스템이 제한적이라 API의 계약(invariants, preconditions, postconditions 등)을 정확히 명시하기 어렵습니다. 라이브러리 개발자들은 모든 잘못된 사용법을 다 설명하기가 실용적이지 않거든요.

**둘째**, API를 사용하는 쪽이 그 계약을 이해하고 지켜야 한다는 암묵적인 기대가 있습니다. NULL을 넘기지 말아야 한다는 걸 프로그래머가 알아야 한다는 뜻이죠.

결국 C에서는 라이브러리가 모든 잘못된 입력에 대해 방어할 책임이 없고, 사용자가 올바르게 사용할 책임이 있다는 가정 위에서 동작합니다.

## Rust는 왜 다를까?

Rust는 여기서 다릅니다. 타입 시스템이 훨씬 강력하거든요.

비슷한 함수를 Rust로 만들어본다고 생각해봅시다:

```rust
pub fn get_env(key: &str) -> Option<String> {
    // ...
}
```

이 함수 시그니처만 봐도 뭐가 명확해질까요? `key` 매개변수는 `&str`이어야 한다는 것, 즉 유효한 문자열 참조여야 한다는 게 명확합니다. NULL 포인터는 애초에 전달할 수 없습니다. 컴파일이 안 되거든요.

Rust의 타입 시스템은 API의 계약을 강제합니다. C처럼 "프로그래머가 알아서 지켜야 할" 암묵적인 계약이 아니라, 컴파일러가 보장하는 명시적인 계약이 되는 겁니다.

## 결론: CVE를 비교할 때 놓치기 쉬운 점

C와 C++에서는 많은 취약점이 "잘못된 사용법" 범주에 남아 있습니다. 라이브러리가 방어하기는 어렵고, 사용자가 조심해야 하는 것들이죠. 따라서 같은 상황이 발생해도 CVE로 등록되지 않을 수 있습니다.

반면 Rust에서는 이런 많은 상황들이 컴파일 단계에서 아예 불가능해집니다. 따라서 Rust 라이브러리에서 발견되는 메모리 관련 취약점은 상대적으로 더 진짜 문제인 경우가 많습니다.

**단순히 CVE 개수를 세어서 언어의 안전성을 비교하는 건 불공평합니다.** 같은 취약점도 한 언어에서는 "사용자의 책임"으로 간주되어 CVE가 아닐 수 있지만, 다른 언어에서는 라이브러리의 책임으로 여겨져 CVE가 될 수 있거든요.

Rust를 도입하는 진정한 가치는 이런 "사용자의 책임"으로 남아있던 많은 실수들을 컴파일 단계에서 미리 잡아낼 수 있다는 데 있습니다.
