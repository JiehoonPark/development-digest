## 핵심 개념

> [!abstract]
> OpenCode를 활용해 Git 연동 AI 코딩 환경을 홈랩에 구축하고, GitOps 워크플로우와 연계해 안전하게 인프라를 관리하는 방식을 소개한다. AI가 변경사항을 작성하면 사용자가 PR을 검토해 승인하는 구조로 인증된 코드만 배포된다.

## 상세 내용

- OpenCode 웹 UI를 VM 서버로 실행해 여러 기기에서 지속적 코딩 세션 관리
- AI는 쓰기 권한이 없고 feature branch에만 푸시하며, 사용자가 PR 검토 후 배포하는 구조로 보안 확보
- Docker compose 스택을 Arcane GitOps로 관리해 git 기반 배포 자동화

> [!tip] 왜 중요한가
> AI 도구의 능력을 활용하면서도 인증 및 감시 계층을 유지하는 안전한 홈 인프라 자동화 패턴을 제시한다.

## 전문 번역

# OpenCode와 GitOps로 홈랩 관리 자동화하기

홈랩 관리를 좀 더 편하게 하려고 OpenCode 웹 UI를 Git 접근 권한과 함께 설정했습니다. OpenCode가 변경사항을 Git에 푸시하면 저는 PR을 승인하고, GitOps가 배포를 담당하는 방식이죠. 가장 좋은 점은 OpenCode가 서버로 실행되면서 코딩 세션이 여러 기기에서 동기화된다는 겁니다.

관리 중인 서비스는 대략 12개 정도의 docker compose 스택으로 이루어져 있는데요. 최근에 Arcane으로 옮겨서 GitOps로 관리하고 배포할 수 있게 했습니다. 다음 단계는 당연히 AI 도구로 서비스 유지보수를 돕는 것이었어요.

## AI를 활용한 컨테이너 업데이트

가장 먼저 활용하기로 한 건 컨테이너 업데이트에 AI를 쓰는 것입니다. 기존엔 각 서비스의 릴리스 노트를 일일이 찾아보고, breaking change가 있는지 확인하고, 업데이트를 실행한 뒤, 각 서비스를 수동으로 확인하느라 몇 시간을 써야 했거든요. 이제는 AI 요약으로 몇 분 안에 릴리스 노트를 읽을 수 있어서 버전 업그레이드가 훨씬 쉽고 안전해졌습니다.

덤으로 AI를 이용해 대부분의 컨테이너에 healthcheck도 추가했어요. 문제를 더 빨리 발견할 수 있게 말이죠.

## OpenCode 선택 이유

주로 Claude Code를 썼는데, 최근 AI 제공업체들이 토큰 제한으로 사용자 가치를 짜내려는 추세라 다른 옵션을 살펴보기로 했습니다. 특정 벤더에 종속되지 않으면서 주요 플러그인들이 지원하는 것을 찾았어요. 여러 옵션을 시도해본 결과 OpenCode가 가장 마음에 들었습니다.

그런데 알고 보니 OpenCode가 내장 웹서버와 웹 UI를 제공하더군요. 이게 좋은 아이디어를 줬습니다.

## 개발 환경 구성

TrueNAS 호스트에 기본 개발 도구가 들어간 간단한 VM을 만들고, OpenCode 웹서버를 systemd 서비스로 등록했습니다. 내장 터미널, 파일 브라우저, git diff, git worktree 지원 등으로 훌륭한 환경이 되었어요. 특히 모바일 웹 UI의 Q&A 팝업이 정말 잘 만들어졌더라고요.

OpenCode에 Git 서버의 전용 사용자와 SSH 키를 설정했습니다. 프로젝트를 클론하고 브랜치를 푸시할 순 있지만, 배포 브랜치로는 직접 푸시할 수 없게 말이에요.

## 보안과 승인 프로세스

이 워크플로우는 AI를 PR 리뷰 뒤에 두는 겁니다. OpenCode가 변경사항을 만들면 제가 PR로 직접 병합하는 방식이죠. 약간 귀엽긴 하지만, 가장 중요한 건 검토되지 않은 코드가 배포되는 걸 막는다는 것입니다.

VM은 인터넷과 Git 서버에 접근할 수 있지만, 실제 서비스에는 접근할 수 없습니다. 피해 범위가 작으므로, OpenCode가 빌드 도구나 테스트 의존성을 설치할 때 필요하면 VM에 대한 root 권한을 줘도 안심할 수 있어요.

프로덕션 개발자 플랫폼으로 확장할 수도 있을 것 같습니다. 미리 도구가 설치된 임시 컨테이너, 접근 제어, 감사 로그 같은 것들이죠. 하지만 지금은 복잡하지 않으면서도 필요한 모든 걸 해주고 있습니다.

## 실제 워크플로우

제 기본 워크플로우는 이렇습니다:

1. OpenCode에서 기능이나 개선사항을 계획합니다 (명세, 구현 계획, 자체 리뷰)
2. 가능하면 변경사항을 테스트하거나 검증합니다
3. 마음에 안 드는 부분은 OpenCode와 다시 반복합니다
4. OpenCode가 feature 브랜치로 변경사항을 푸시합니다
5. 해당 브랜치에 대해 PR을 엽니다
6. 만족하면 PR을 병합합니다
7. 그 이후는 GitOps가 담당합니다
   - Docker 서비스 변경은 Arcane
   - Home Assistant 설정 변경은 GitOps 플러그인
   - 블로그 변경은 Cloudflare Pages worker

## Arcane으로의 마이그레이션

TrueNAS에서 실행하던 docker compose 스택들을 Arcane GitOps 프로젝트로 옮겼습니다. Git 기반 스토리지를 갖기 위함이었는데, OpenCode와 함께 사용하니 정말 잘 맞더라고요.

예를 들어, 모든 컨테이너의 네트워킹을 휴대폰에서 업데이트할 수 있게 된 거예요. 기존엔 모든 compose 스택을 돌며 네트워크 연결성을 추적하는 데 몇 시간이 걸렸거든요. 이제는 OpenCode에게 코드베이스와 목표를 가리키고, 결과 PR 변경사항을 확인하고 병합하면 끝입니다.

## 아직 부족한 부분

CI 피드백이 주요 문제입니다. GitHub라면 코딩 에이전트에게 Actions 로그를 보여줘서 실패한 테스트, 린터 에러, 스택 트레이스, IaC 플랜 변경을 진단하게 할 수 있거든요. 이렇게 하면 단위 테스트로 커버되지 않는 변경사항에 대해 빠른 피드백 루프를 유지할 수 있습니다.

다만 Forgejo에서는 이게 어렵습니다. Forgejo Actions가 공개 API를 통해 작업 로그를 노출하지 않거든요. 문서화되지 않은 API도 있지만, 그걸 기반으로 구축하고 싶지는 않습니다.

이런 설정 덕분에 AI가 변경하는 서비스에 직접 접근 권한을 주지 않으면서도 어느 기기에서나 홈 인프라를 수정할 수 있습니다. 컴퓨터에서 변경사항을 시작하고, 휴대폰에서 PR을 검토하고, GitOps가 배포를 처리하는 방식이죠.
