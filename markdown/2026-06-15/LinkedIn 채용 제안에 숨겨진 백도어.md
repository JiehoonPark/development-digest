## 핵심 개념

> [!abstract]
> 개발자가 LinkedIn을 통해 받은 채용 제안 관련 GitHub 저장소에 npm install 실행 시 악성 코드를 다운로드하는 백도어가 숨겨져 있었다. 저장소의 test 파일에 원격 서버에서 명령을 받아 실행하는 코드가 위장되어 있었으며, 실제 개발자의 정보를 도용해 커밋 이력을 조작했다.

## 상세 내용

- npm prepare 스크립트가 자동 실행되는 특성을 악용해 npm install 직후 백도어 실행
- 원격 서버와 통신해 임의 코드를 실행하는 페이로드가 주석 처리된 테스트 코드 사이에 위장됨
- AI 에이전트를 읽기 전용 샌드박스에서 실행해 수동 코드 리뷰보다 빠르게 위협 탐지

> [!tip] 왜 중요한가
> 채용 관련 코드 리뷰 요청 시 주의가 필요하며, 의심스러운 상황에서는 샌드박스 환경에서 검토하는 것이 효과적임을 보여준다.

## 전문 번역

# 린크드인 메시지로 온 악의적인 코드 리뷰 요청을 피한 이야기

지난주 암호화폐 스타트업의 채용담당자에게서 린크드인 메시지를 받았어요. 며칠간 몇 가지 메시지를 주고받다가 그들이 리드 엔지니어가 필요한 망가진 개념증명 프로토타입이 있다고 했고, 공개 GitHub 저장소를 검토해달라고 요청했습니다. 특히 "deprecated Node modules 이슈를 확인해봐달라"고 했거든요.

기존 코드베이스 검토 요청 자체는 흔한 일이지만, 뭔가 이상한 느낌이 들어서 조금 의심스러운 마음으로 접근하기로 했어요.

## 안전하게 코드 검토하기

저장소를 직접 클론해서 의존성을 설치하는 대신, Hetzner에서 일회용 VPS를 하나 띄웠습니다. 그곳에 저장소를 클론한 뒤, AI 에이전트(Pi)를 읽기 전용 모드로 설정하고 파일 읽기 도구만 활성화했어요.

에이전트에게 코드베이스를 검토해달라고 했고, 의심스러운 부분이 있으면 표시해달라고 요청했습니다. 그러자 거의 즉시 **app/test/index.js** 파일에서 멈춰버렸어요.

## 악의적인 코드 발견

저장소는 React 프론트엔드와 Node 백엔드로 이루어진 것 같았는데, 함정은 app/test/index.js에 숨어있었습니다. 약 250줄 길이의 이 파일은 일반 테스트 스위트처럼 위장되어 있었어요.

파일 내부에서는 URL이 여러 조각으로 나뉜 뒤 조합되었습니다:

```javascript
https://rest-icon-handler.store/icons/77
```

주석 처리된 테스트 코드의 벽 사이에 숨겨진 실제 코드는 서버에서 보낸 모든 것을 당신 머신에서 실행합니다.

## 실행 흐름: npm install부터 시작

여기가 중요한데요. app/index.js 자체가 `const test = require('./test')`를 실행하기 때문에 app/test/index.js가 로드되고 실행됩니다.

package.json은 app/index.js를 시작 스크립트로 연결하고 있었어요:

```json
"prepare": "npm run app:pre",
"app:pre": "node app/index.js"
```

**prepare 스크립트가 핵심입니다.** npm은 `npm install` 후에 prepare를 자동으로 실행하거든요. 즉, 의존성을 설치하는 것만으로도 백도어가 실행되는 셈이었던 거예요.

"deprecated Node modules 이슈를 확인해봐달라"는 지시는 저를 `npm install`을 실행하도록 유도하기 위한 미끼였던 것입니다.

## 피싱의 흔적

저는 페이로드가 실행되는 것을 샌드박스에서 지켜본 뒤, 서버가 두 번째 단계로 뭘 보내는지 확인할 수도 있었어요. 하지만 멈췄습니다. 서버가 보내는 어떤 것이든 실행하는 저장소라는 증거만으로도 충분했거든요.

흥미롭게도 저장소의 모든 커밋(총 39개)은 실제 개발자의 이름과 이메일로 되어있었습니다. 그 개발자는 일반적인 린크드인 프로필, 개인 웹사이트, 오래된 GitHub 히스토리까지 가지고 있었어요.

저는 마치 그 코드베이스를 물려받은 척하면서 몇 가지 구현 질문이 있다고 메시지를 보냈습니다. 그의 반응을 보고 싶었거든요.

그는 그 회사에서 일한 적이 없다고 말했습니다. 이전에도 GitHub에서 신원 도용을 당한 적이 있었고, 그때도 저장소가 삭제되었던 거예요. 이번에도 그 저장소와는 관련이 없다며, 마찬가지로 신고 중이라고 했습니다.

## 신원 도용의 흔적

채용담당자의 프로필을 조사해보니, 실제 문화 저널리스트였어요. 꽤 유명한 기자더군요. 기술 관련 경력은 전혀 없었는데, 제가 설치가 안 된다고 장난을 친 순간 갑자기 npm과 Node 버전에 대한 전문가로 변신했습니다. 꽤 웃겼어요.

기술 배경이 전혀 없는 채용담당자가 갑자기 Node 버전에 대해 토론하며 `npm install`을 강력하게 추진했거든요.

## 배운 점

이런 공격은 해킹 뉴스에서 읽었지만, 직접 겪으니 여전히 놀랐어요. 처음 메시지들부터 뭔가 이상한 느낌이 들었지만, 더 피곤하거나 바쁜 날이었다면 생각해볼 겨를도 없이 `npm install`을 실행했을 수도 있었습니다.

린크드인에서 저장소 검토를 요청하는 메시지를 받으면, 약간의 의심스러운 마음과 좋은 보안 습관은 절대 손해가 아닙니다.

또 하나 배운 점은 읽기 전용 에이전트로 코드를 검토하는 게 제가 직접 읽는 것보다 훨씬 더 효과적이었다는 거예요. 백도어는 서툰 초급 코드처럼 위장되어 있었지만, 에이전트는 몇 초 만에 찾아냈거든요.

저는 GitHub에 저장소를, LinkedIn에 그 채용담당자를 신고했습니다. 지금까지 아무런 변화가 없고 코드는 여전히 올라와 있네요.
