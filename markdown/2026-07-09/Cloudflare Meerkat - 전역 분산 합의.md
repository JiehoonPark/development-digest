## 핵심 개념

> [!abstract]
> Cloudflare는 330개 이상의 글로벌 데이터센터에서 강한 일관성을 보장하면서 고가용성을 제공하는 새로운 분산 합의 서비스 Meerkat을 개발했습니다. 기존 Raft 알고리즘의 한계를 극복하기 위해 QuePaxa 알고리즘을 기반으로 구현되었으며, 모든 복제본이 언제든 쓰기를 수행할 수 있어 예측 불가능한 네트워크 환경에 더 적합합니다.

## 상세 내용

- Meerkat은 QuePaxa 알고리즘 기반으로 모든 복제본이 동시에 쓰기 가능하며, 타임아웃으로 인한 시스템 중단이 없습니다.
- Raft와 달리 리더 장애로 인한 가용성 문제를 해결하여 광역 네트워크 환경에 최적화되었습니다.
- 제어 평면 상태 관리와 분산 잠금 시스템 등의 애플리케이션을 Meerkat 위에 구축할 수 있습니다.

> [!tip] 왜 중요한가
> Linearizability를 보장하면서 글로벌 규모의 고가용성 시스템을 구축하려는 개발자들에게 새로운 솔루션을 제공합니다.

## 전문 번역

# Meerkat 소개: 글로벌 합의(Consensus)에 대한 도전

**James Larisch, Bob Halley, João Pedro Leite | 2026년 7월 8일 | 약 12분**

## 문제: 전 지구적 규모의 일관성 있는 상태 관리

Cloudflare의 많은 내부 서비스들은 전 세계 330개 이상의 데이터센터에서 동일한 제어 평면(control-plane) 상태를 읽고 수정해야 합니다. 여러 읽기 작업이 항상 일관된 상태를 보도록 보장해야 하고, 일부 데이터센터나 네트워크 링크에 장애가 발생해도 시스템이 쓰기 작업을 계속 처리할 수 있어야 한다는 뜻이죠.

하지만 Cloudflare의 네트워크는 전 인터넷을 걸쳐 운영되기 때문에 상황이 복잡합니다. 서버와 데이터센터는 언제든 내려갈 수 있고, 네트워크 큐는 가득 찰 수 있으며, 통신선과 케이블은 끊길 수 있거든요. 이런 악조건 속에서 강한 일관성(strong consistency)을 보장하면서 전 지구적 규모로 이용 가능한 데이터 시스템을 운영하기는 매우 어렵습니다. 이런 환경에서는 분산 시스템 레플리카들이 데이터를 안정적으로 동기화하기 힘들기 때문입니다.

## 기존 합의 알고리즘의 한계

악조건 속에서 데이터를 안전하게 동기화하는 한 가지 방법은 합의 알고리즘(consensus algorithm)을 사용하는 것입니다. 이를 통해 머신들의 집합이 동일한 일련의 값(예: 키-값 저장소의 읽기/쓰기 작업)에 합의할 수 있죠. 다만 대다수 머신이 살아있고 통신 가능하다는 조건이 있어야 합니다.

문제는 Raft 같은 기존의 널리 사용되는 합의 알고리즘들이 Cloudflare 같은 광역 네트워크 환경에서는 잘 작동하지 않는다는 점입니다. 이 알고리즘들은 리더(leader) 패턴에 의존하기 때문이죠.

Raft에서는 리더만 쓰기 작업을 수행할 수 있습니다. 리더가 크래시나 네트워크 성능 저하로 인해 장애를 겪으면, 다른 레플리카가 타임아웃을 기다려 새로운 리더를 선출할 때까지 시스템은 사용 불가능한 상태가 됩니다. 그런데 광역 네트워크 환경에서는 지연 시간이 예측 불가능하기 때문에 타임아웃 값을 설정하기가 정말 어렵습니다.

실제로 Cloudflare에서도 합의 기반 시스템의 리더가 사용 불가능해지면서 여러 차례 장애를 겪었습니다.

## Meerkat과 QuePaxa

바로 이런 문제를 해결하기 위해 Cloudflare 연구팀은 지난 1년간 새로운 분산 합의 서비스인 **Meerkat**을 개발해왔습니다. Meerkat은 2023년 EPFL 연구원들이 발표한 **QuePaxa**라는 합의 알고리즘으로 구동됩니다.

QuePaxa가 Raft와 다른 점은 여러 곳입니다:

- 모든 레플리카가 항상 쓰기 작업을 수행할 수 있습니다
- 타임아웃으로 인해 시스템 진행이 멈추지 않습니다

이런 특성 덕분에 QuePaxa는 Cloudflare의 네트워크 환경에 훨씬 잘 맞습니다.

Meerkat 위에는 트랜잭셔널 키-값 저장소나 리싱(leasing) 시스템 같은 애플리케이션들을 얹을 수 있습니다. 우리가 알기로는 이것이 QuePaxa의 첫 번째 글로벌 규모 상용 배포가 될 것입니다.

## 현재 상태와 향후 계획

Meerkat은 아직 개발 중인 실험적 합의 서비스입니다. 현재로서는 제어 평면의 작은 부분(예: 복제 데이터베이스의 리더십 정보)을 관리하도록 설계되고 있으며, 가까운 시일 내에는 내부용으로만 유지될 예정입니다.

이 글은 Meerkat을 소개하고 앞으로 나올 Meerkat 관련 블로그 글들의 기초를 마련하는 것을 목표로 합니다.

# 글로벌 제어 평면 데이터 시스템에 필요한 것들

## 제어 평면 데이터란?

Cloudflare의 많은 서비스들은 전 세계에 분산된 여러 머신에서 제어 평면 데이터를 읽고 쓰는 작업을 합니다. 제어 평면 데이터는 각 서비스가 올바르게 동작하도록 돕는 정보들이죠.

구체적인 예를 들어보겠습니다. **배치 정보**(placement information)는 제어 평면 데이터의 한 예입니다. 특정 리소스(예: AI 모델 인스턴스)가 어디에 저장되어 있는지를 나타내죠. **리더십 정보**(leadership information)도 제어 평면 데이터입니다. 데이터베이스에 쓰기 작업을 수행할 수 있는 머신이 어느 것인지를 나타냅니다.

제어 평면 데이터는 두 가지 조건을 만족해야 합니다:
- 강한 일관성(strong consistency)을 보장해야 함
- 특정 종류의 장애가 발생해도 접근 가능해야 함

## 강한 일관성(Linearizability)의 중요성

분산 데이터 시스템의 일관성 수준은 시스템이 동시 읽기/쓰기를 받을 때 어떤 동작을 허용하는지를 정의합니다.

예를 들어, 여러 노드에 `x = 6`이라는 값을 저장하는 분산 키-값 저장소를 생각해봅시다. 다음과 같은 쓰기 작업들을 최선 시도(best-effort) 방식으로 다른 노드에 제출한다고 가정합니다:

```
x = x + 1
x = x / 2
```

시스템의 일관성 수준에 따라 클라이언트가 읽을 수 있는 `x`의 값은 달라집니다.

**약한 일관성**(weak consistency)에서는 쓰기 작업의 순서가 바뀔 수 있습니다. **더 강한 일관성 모델**에서는 쓰기 순서는 유지되지만 읽기 작업이 재정렬될 수 있죠. 그리고 **가장 강한 일관성 수준**에서는 작업들이 실시간으로 발생한 순서 그대로 처리됩니다. 이 성질을 **선형화 가능성(linearizability)**이라고 부릅니다.

## Cloudflare가 선택한 방식

Cloudflare의 많은 서비스들은 선형화 가능성을 원합니다. 약한 일관성과 달리, 선형화 가능성은 프로그래머들이 데이터 시스템의 모든 이상한 동작을 일일이 생각할 필요가 없게 해줍니다. 대신 단일 스레드 머신의 로컬 메모리를 다루듯이 분산 시스템을 생각할 수 있거든요. 즉, 쓰기 작업 이후의 모든 읽기는 그 쓰기 결과를 봅니다.

약한 일관성의 위험성에 대해 더 알고 싶다면 Marc Brooker의 글을 참고하면 좋습니다. 참고로 Meerkat의 키-값 저장소도 선형화 가능성을 제공합니다.
