## 핵심 개념

> [!abstract]
> GitHub의 중앙화 의존성을 벗어나기 위해 Tangled라는 연합형 코드 협업 플랫폼을 제안합니다. 이는 git을 코드 전송에, AT 프로토콜을 이벤트 통신에 사용하여 여러 서버 간 협업을 가능하게 합니다.

## 상세 내용

- GitHub의 중앙화 문제를 해결하기 위한 연합형 아키텍처 제안 - 여러 git 서버('knots')에서 협업하고 크로스 서버 포크/PR 가능
- email + git, GitHub, ForgeFed, Tangled 등 코드 협업의 진화 과정 분석 - git은 코드 전송, 통신 프로토콜만 변화
- AT 프로토콜을 활용한 인증된 이벤트 전송 - 이슈, PR, 팔로우, 별표 등 소셜 기능 지원

> [!tip] 왜 중요한가
> 오픈소스 생태계가 단일 벤더에 의존하는 위험을 줄이고, 자체 서버에서도 GitHub과 같은 협업 경험을 가능하게 합니다.

## 전문 번역

# GitHub 중앙화의 한계, 분산형 코드 협업 플랫폼 Tangled 소개

지난 몇 주간 GitHub이 여러 문제를 겪고 있습니다. 이유가 무엇이든 간에, 전 세계 오픈소스의 90%가 단 하나의 플랫폼에 의존한다는 건 결코 바람직하지 않죠. 중앙화된 시스템은 결국 무너지기 마련이고, 시간을 견디는 건 이메일, Git, IRC 같은 프로토콜들입니다. 저희가 개발 중인 Tangled는 정확히 이런 공백을 채우려는 프로젝트입니다.

## 코드 협업의 진화

코드 협업은 항상 두 개의 프로토콜 위에 구축되어 왔습니다. 하나는 코드를 주고받는 용도고, 다른 하나는 커뮤니케이션 용도죠.

**이메일 시대:**
- Git (코드 전송) + 이메일 (통신)

**GitHub 시대:**
- Git (코드 전송) + GitHub 웹사이트 (통신)

**ForgeFed 프로젝트:**
- Git (코드 전송) + ActivityPub (통신, 예정)

**Tangled:**
- Git (코드 전송) + AT 프로토콜 (통신)

## Tangled의 작동 원리

Tangled는 Git 서버들(우리는 이를 "knots"라고 부릅니다) 사이에서 이벤트를 연동시킵니다. 어느 서버에든 저장소를 만들고 협업할 수 있으며, 서로 다른 서버 간에 포크도 가능합니다. 심지어 자신의 서버에 있는 저장소에 푸시한 뒤, 완전히 다른 서버에 호스팅된 저장소에 풀 리퀘스트를 열 수도 있죠. 어떻게 보면 자신만의 cgit 인스턴스를 운영하면서 이메일로 패치를 주고받던 방식과 유사합니다.

## AT 프로토콜의 역할

Tangled는 AT 프로토콜을 사용해서 이슈나 풀 리퀘스트 같은 코드 관련 이벤트를 인증된 상태로 전송합니다. 이뿐 아니라 약간의 소셜 기능도 포함하죠:

- 이벤트 타임라인
- 팔로우
- 스타
- 보증(vouches) - 곧 출시 예정

협업자 초대와 SSH 공개키 공유에도 AT를 활용하지만, 나머지는 그저 좋은 old Git일 뿐입니다.

## 마치며

오픈소스는 GitHub 같은 단일 생태계로부터 벗어나야 합니다. 동시에 코드 협업은 여전히 즐겁고 사회적이어야 한다고 생각합니다.
