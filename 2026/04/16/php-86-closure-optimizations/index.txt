13:T123a,<h2>PHP 클로저 최적화 RFC: 성능과 하위호환성의 균형</h2>
<p>PHP의 클로저 성능을 개선하기 위한 새로운 최적화 방안이 제시되었습니다. 이번 RFC는 두 가지 최적화 기법을 제안하는데, 일부 하위호환성 관련 변경사항을 동반합니다. 핵심은 이러한 변경들이 얻게 될 성능 향상만큼 가치 있는지 검토하는 것입니다.</p>
<h3>첫 번째 최적화: Static 클로저 자동 추론</h3>
<h4>문제 상황</h4>
<p>생각해보세요. 클로저가 <code>$this</code>를 전혀 사용하지 않는데도 암묵적으로 캡처하고 있다면요?</p>
<pre><code class="language-php">class Foo {
    public function __construct() {
        $closure = function() {
            return 42;
        };
    }
}</code></pre>
<p>위 코드에서 클로저는 불필요하게 <code>$this</code>를 캡처합니다. 결과적으로 Foo 인스턴스가 클로저와 순환 참조(reference cycle)를 형성하게 됩니다. 인스턴스가 클로저를 유지하고, 클로저가 인스턴스를 유지하는 것이죠.</p>
<p>이 순환 참조를 정리하려면 PHP의 가비지 컬렉터가 실행되어야 합니다. 그런데 실제로는 가비지 컬렉터가 자주 동작하지 않아서, 요청이 끝날 때까지 이러한 순환 구조가 메모리에 남아있을 수 있습니다. 더 큰 문제는, 이런 불필요한 순환 참조들이 쌓이면 가비지 컬렉터 자체가 더 자주 실행되어야 하고, 결과적으로 성능이 저하된다는 점입니다.</p>
<h4>해결책: 자동 추론</h4>
<p>이 최적화는 다음 조건을 만족하는 클로저에 자동으로 <code>static</code> 수정자를 적용하려고 시도합니다:</p>
<ul>
<li><code>$this</code>를 사용하지 않음</li>
<li>슈퍼글로벌 변수를 참조하지 않음</li>
<li>정적 프로퍼티에 접근하지 않음</li>
<li>정적 메서드를 호출하지 않음</li>
<li><code>use(&amp;$var)</code> 형태의 참조 캡처를 하지 않음</li>
</ul>
<p>규칙이 다소 복잡해 보이지만, 실제로는 꽤 효과적입니다. Symfony Demo 프로젝트에서 테스트해본 결과, 명시적으로 <code>static</code>으로 표기된 87개의 클로저 중 약 78%(68개)를 자동으로 추론할 수 있었습니다.</p>
<p>명시적으로 <code>static</code>을 붙이는 것이 가장 좋겠지만, 이 최적화는 두 가지 상황에서 도움이 됩니다:</p>
<ul>
<li>코드의 시각적 깔끔함을 선호하는 팀들</li>
<li>클로저의 성능 영향을 몰랐던 개발자들</li>
</ul>
<h3>두 번째 최적화: Stateless 클로저 캐싱</h3>
<h4>문제 상황</h4>
<p>같은 클로저를 반복해서 생성하는 코드를 생각해봅시다:</p>
<pre><code class="language-php">for ($i = 0; $i &lt; 10000000; $i++) {
    $closure = function() {
        return &quot;hello&quot;;
    };
}</code></pre>
<p>이 코드는 1천만 개의 클로저 인스턴스를 생성합니다. 그런데 실제로는 모두 동일한 클로저입니다.</p>
<h4>해결책: 클로저 캐싱</h4>
<p>Static이면서 변수를 캡처하지 않고 정적 변수도 선언하지 않는 클로저(즉, stateless 클로저)는 처음 생성된 후 재사용을 위해 캐시됩니다.</p>
<p>이 최적화 덕분에 위의 합성 벤치마크는 약 <strong>80% 성능 향상</strong>을 기록했습니다. 더 실용적인 예제인 Laravel 템플릿에서는 3,637개의 클로저 인스턴스 중 2,384개를 건너뛸 수 있어, 전체적으로 <strong>약 3% 성능 향상</strong>을 달성했습니다.</p>
<h3>하위호환성 고려사항</h3>
<p>세 가지 BC(하위호환성) 문제가 있습니다.</p>
<p>주목할 점은 <code>Closure::bind()</code>와 <code>Closure::bindTo()</code> 메서드입니다. 일반적으로 이 메서드들은 static 클로저에 객체를 바인딩하려 할 때 예외를 던집니다.</p>
<p>이번 RFC에서는 이 메서드들에 객체를 전달하는 것을 명시적으로 허용합니다. 다만 자동으로 추론된 static 클로저에서만 해당 객체를 무시합니다. 명시적으로 <code>static</code>으로 선언된 클로저는 여전히 기존 동작을 유지합니다.</p>
<p>이렇게 구분한 이유는 하위호환성 때문입니다. 예를 들어, 정적 메서드 호출을 제거하는 식의 관련 없어 보이는 코드 변경이 클로저를 갑자기 <code>static</code>으로 추론하게 만들 수 있으니까요.</p>
