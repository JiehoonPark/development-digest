14:T1ac1,<h2>RSC는 왜 번들러와 통합되어야 할까?</h2>
<p><strong>주의: 이 글은 기술에 깊게 파고드는 내용입니다.</strong></p>
<p>React Server Components(RSC)는 서버와 클라이언트 애플리케이션을 하나의 프로그램으로 표현하는 프로그래밍 패러다임입니다. 두 가지 런타임을 거쳐가면서 말이죠.</p>
<p>RSC의 구현은 크게 두 부분으로 이루어져 있습니다.</p>
<ul>
<li>React 트리를 직렬화하는 부분 (React 저장소의 packages/react-server)</li>
<li>React 트리를 역직렬화하는 부분 (React 저장소의 packages/react-client)</li>
</ul>
<p>react-server와 react-client 패키지는 React 저장소의 내부 패키지입니다. 물론 완전히 오픈소스지만, npm에 원본 형태로 배포되지는 않아요. 가장 중요한 요소인 모듈 시스템 통합이 빠져있기 때문입니다.</p>
<p>많은 (역)직렬화 도구들과 달리, RSC는 단순히 데이터를 보내는 게 아니라 <strong>코드를 보낸다</strong>는 점이 핵심입니다.</p>
<p>예를 들어 이런 트리가 있다고 해봅시다.</p>
<pre><code class="language-jsx">&lt;p&gt;Hello, world&lt;/p&gt;</code></pre>
<p>이 <code>&lt;p&gt;</code> 태그를 JSON으로 변환하는 건 간단합니다.</p>
<pre><code class="language-json">{
  &quot;type&quot;: &quot;p&quot;,
  &quot;props&quot;: {
    &quot;children&quot;: &quot;Hello world&quot;
  }
}</code></pre>
<p>그런데 이번엔 <code>&lt;Counter&gt;</code> 태그라면 어떻게 할까요?</p>
<pre><code class="language-jsx">import { Counter } from './client';
&lt;Counter initialCount={10} /&gt;</code></pre>
<pre><code class="language-jsx">'use client';
import { useState, useEffect } from 'react';

export function Counter({ initialCount }) {
  const [count, setCount] = useState(initialCount);
  // ...
}</code></pre>
<p>모듈을 어떻게 직렬화하죠?</p>
<h3>모듈 직렬화하기</h3>
<p>우리가 원하는 건 단순한 스냅샷이 아닙니다. 반대편에서 실제로 동작하는 <code>&lt;Counter&gt;</code>를 복원하고 싶어요. 즉, <strong>상호작용을 위한 전체 로직</strong>이 필요합니다.</p>
<p>가장 단순한 방법은 Counter 코드를 JSON에 문자열로 직접 포함시키는 거예요.</p>
<pre><code class="language-json">{
  &quot;type&quot;: `
    import { useState, useEffect } from 'react';
    export function Counter({ initialCount }) {
      const [count, setCount] = useState(initialCount);
      // ...
    }
  `,
  &quot;props&quot;: {
    &quot;initialCount&quot;: 10
  }
}</code></pre>
<p>그런데 이건 좋은 방법이 아닙니다. 코드를 문자열로 보내서 eval로 실행하고 싶지 않을 뿐더러, 같은 컴포넌트 코드를 여러 번 보내는 것도 낭비거든요.</p>
<p>더 나은 방식은 그 코드가 이미 앱의 정적 JS 에셋으로 제공되고 있다고 가정하는 겁니다. JSON에서 참조하기만 하면 되는 거죠. <code>&lt;script&gt;</code> 태그처럼 말이에요.</p>
<pre><code class="language-json">{
  &quot;type&quot;: &quot;/src/client.js#Counter&quot;,
  &quot;props&quot;: {
    &quot;initialCount&quot;: 10
  }
}</code></pre>
<p>클라이언트에서는 <code>&lt;script&gt;</code> 태그를 동적으로 생성해서 로드할 수 있습니다.</p>
<h3>번들러가 필요한 이유</h3>
<p>그런데 생각해보세요. 한 파일이 다른 여러 파일을 import하고 있을 때, 클라이언트는 전체 import 트리를 미리 알지 못합니다. 소스 파일들을 하나하나 네트워크로 로드하면 폭포수 같은 문제가 생길 거예요.</p>
<p>우리는 클라이언트 애플리케이션으로 20년을 개발하면서 이런 문제를 어떻게 해결하는지 이미 알고 있습니다. <strong>번들링</strong>이죠.</p>
<h3>RSC와 번들러의 통합</h3>
<p>이 때문에 RSC는 번들러와 통합됩니다. 번들러가 필수는 아닙니다—번들러 없이도 동작하는 RSC ESM 개념 증명이 있거든요. 하지만 번들러 없이 천진하게 구현하면 얼마나 비효율적인지를 알기에, 실제로는 번들러를 사용하는 게 현실적입니다.</p>
<p>실제 RSC 통합은 번들러 전용입니다. Parcel, Webpack, (그리고 앞으로) Vite에 대한 바인딩이 React 저장소에 있으며, 모듈을 어떻게 보내고 로드할지 정의합니다.</p>
<p><strong>빌드 시점에서:</strong></p>
<ul>
<li>'use client' 마크가 붙은 파일을 찾고, 그 진입점들에 대한 번들 청크를 생성합니다. Astro Islands 같은 개념이죠.</li>
</ul>
<p><strong>서버에서:</strong></p>
<ul>
<li>이 바인딩은 React에게 모듈을 클라이언트로 어떻게 보낼지 알려줍니다. 예를 들어 번들러가 'chunk123.js#Counter' 같은 형태로 모듈을 참조할 수 있게요.</li>
</ul>
<p><strong>클라이언트에서:</strong></p>
<ul>
<li>바인딩은 React에게 번들러 런타임에 그 모듈을 어떻게 로드할지 알려줍니다. 예를 들어 Parcel 바인딩은 Parcel 전용 함수를 호출합니다.</li>
</ul>
<p>이 세 가지 덕분에 React Server는 모듈을 만났을 때 어떻게 직렬화할지 알고, React Client는 어떻게 역직렬화할지 알게 됩니다.</p>
<h3>실제 사용법</h3>
<p>React Server에서 트리를 직렬화하는 API는 번들러 바인딩을 통해 노출됩니다.</p>
<pre><code class="language-jsx">import { serialize } from 'react-server-dom-yourbundler'; // 번들러 전용 패키지

const reactTree = &lt;Counter initialCount={10} /&gt;;
const outputString = serialize(reactTree); // 위의 JSON 같은 형태</code></pre>
<p>이 outputString을 디스크에 저장하거나, 네트워크로 전송하거나, 캐시하거나, 어떻게든 할 수 있습니다. 그리고 나중에 React Client에 넘겨주면 돼요.</p>
<pre><code class="language-jsx">import { deserialize } from 'react-server-dom-yourbundler/client'; // 번들러 전용 패키지

const outputString = // ... 네트워크로 받거나, 디스크에서 읽거나...
const reactTree = deserialize(outputString); // &lt;Counter initialCount={10} /&gt;</code></pre>
<p>모든 게 잘 작동했다면, 마치 클라이언트에서 직접 <code>&lt;Counter initialCount={10} /&gt;</code>를 작성한 것처럼 일반적인 JSX가 됩니다. 그 이후론 평범한 JSX 트리처럼 뭐든 할 수 있어요—렌더링, 상태에 저장, HTML로 변환 등등.</p>
