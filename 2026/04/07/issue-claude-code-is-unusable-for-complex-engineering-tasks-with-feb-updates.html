이슈: Claude Code가 2월 업데이트 이후 복잡한 엔지니어링 작업에 사용 불가

Issue: Claude Code is unusable for complex engineering tasks with Feb updates

📅 2026-04-07 📰 Hacker News (Top) ⬆ 819

요약

사용자들의 로그 분석 결과, 2월 중순부터 시작된 Extended Thinking 콘텐츠 감소(redact-thinking-2026-02-12)가 Claude Code의 복잡한 엔지니어링 작업 품질 저하와 정확히 일치한다는 것을 발견했다. Extended Thinking이 단순한 옵션이 아닌 구조적으로 필요한 기능임을 데이터로 입증하고 있다.

핵심 포인트

  • 3월 8일 품질 저하 보고가 정확히 Thinking 블록 감소량이 50%를 넘은 날짜와 일치
  • Thinking 깊이 감소로 인해 Read:Edit 비율이 6.6에서 2.0으로 70% 감소, 코드 읽기 없이 수정하는 edit-first 패턴으로 변화
  • Stop hook 위반, 불완전한 수정, 소유권 회피 행동 등 정량적 품질 지표가 모두 악화(예: 불만 지시 사항 5.8% → 9.8%)

왜 중요한가

Claude Code 사용자는 현재 버전의 신뢰성 한계를 인식하고 작업 복잡도 조절이 필요하며, Anthropic은 Extended Thinking 할당 정책 검토가 긴급하다.

📄 전문 번역

Claude의 복잡한 엔지니어링 작업 성능 저하 분석

문제 요약

지난 몇 개월간 로그 데이터를 분석한 결과, 2월 이후 Claude의 성능이 눈에 띄게 하락했습니다. 특히 복잡한 엔지니어링 작업에서 지시를 무시하거나, 부정확한 수정을 제안하고, 반대 동작을 수행하는 현상이 반복되고 있는데요. 저희는 이 문제가 단순한 일시적 버그가 아닌, 근본적인 구조적 변화와 연관이 있다고 판단합니다.

Extended Thinking의 중요성

분석 결과를 한 문장으로 요약하자면, Extended Thinking 토큰의 감소가 모델의 성능 저하와 정확히 일치한다는 것입니다.

17,871개의 thinking 블록과 234,760개의 도구 호출을 분석한 데이터에 따르면, Extended Thinking은 단순한 부가 기능이 아니라 복잡한 멀티스텝 작업에 필수적인 요소입니다. Thinking 토큰이 부족해지면서 모델은 먼저 생각하고 수정하는 방식(research-first)에서 곧바로 수정하는 방식(edit-first)으로 전환되었고, 이것이 사용자들이 보고한 품질 저하로 이어진 것 같습니다.

1. Thinking 감소와 품질 저하의 타이밍이 일치합니다

Thinking 콘텐츠 감소 정책(redact-thinking-2026-02-12)이 적용된 시점을 추적하면:

기간Thinking 표시Thinking 감소
1월 30일 - 3월 4일100%0%
3월 5일98.5%1.5%
3월 7일75.3%24.7%
3월 8일41.6%58.4%
3월 10-11일<1%>99%

흥미롭게도 사용자들이 성능 저하를 보고한 날짜가 정확히 3월 8일입니다. 이날 감소된 thinking 블록이 전체의 50%를 넘어섰거든요.

2. 정책 변화 이전부터 이미 Thinking 깊이가 감소하고 있었습니다

Thinking 블록의 서명(signature) 필드와 실제 내용 길이 사이에는 0.971의 높은 상관관계가 있습니다. 이를 통해 감소된 thinking도 추정할 수 있었는데요.

기간예상 Thinking 길이기준 대비 변화
1월 30일 - 2월 8일 (기준)~2,200자—
2월 말~720자-67%
3월 1-5일~560자-75%
3월 12일 이후 (완전 감소)~600자-73%

감소 정책이 공식 시행되기 전부터 이미 thinking 깊이가 67% 떨어져 있었습니다. 정책 변화는 이 감소를 사용자가 볼 수 없게 만든 것뿐입니다.

3. 측정 가능한 성능 저하 지표들

18,000개 이상의 사용자 프롬프트를 독립적으로 분석했을 때:

소유권 회피 행동6건13건+117%
사용자 답답함 표시5.8%9.8%+68%
세션당 프롬프트 수35.927.9-22%
추론 루프 반복 세션07새로 발생

특히 주목할 점은 불완전한 수정, 지시 무시, 확인만 반복하는 패턴이 3월 8일 이후로 갑자기 나타난다는 것입니다. 이전에는 이런 일이 전혀 없었거든요.

4. 도구 사용 패턴의 극적인 변화

파일 읽기와 파일 편집의 비율을 보면 가장 명확한 변화가 드러납니다.

기간읽기:편집 비율변화
1월 30일 - 2월 12일 (양호)6.6 : 1—
2월 13일 - 3월 7일 (전환기)2.8 : 1-57%
3월 8일 - 3월 23일 (저하)2.0 : 1-70%

성능이 좋을 때 모델은 대상 파일 → 관련 파일 → 코드베이스 전체 검색 → 헤더와 테스트 확인 → 정확한 편집의 순서를 따랐습니다. 하지만 지금은 해당 파일만 훑어보고 바로 편집을 시도하곤 합니다.

결론

이 분석은 몇 가지 중요한 점을 시사합니다.

첫째, Extended Thinking 토큰이 충분할 때와 부족할 때 모델의 동작 방식이 근본적으로 다릅니다. Thinking이 풍부하면 꼼꼼한 사전 조사를 거쳐 신중하게 수정하지만, 부족하면 성급하게 수정을 시도합니다.

둘째, 이 문제는 특정 사용자나 상황만의 문제가 아닙니다. 저희는 고복잡도 엔지니어링 작업을 하는 전형적인 파워 유저인데, 같은 문제를 겪는 팀이나 개인들이 분명 있을 것입니다.

복잡한 작업을 해야 하는 개발자라면 이 데이터가 Claude의 성능 기대치를 설정하는 데 도움이 되길 바랍니다. 그리고 Anthropic에서는 파워 유저를 위한 thinking 토큰 할당 정책을 다시 검토해주기를 간절히 부탁드립니다.
