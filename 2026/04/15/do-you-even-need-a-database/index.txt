27:T1ba2,<h2>파일로 충분할 때도 있다</h2>
<p>데이터베이스를 자세히 들여다보면 결국 파일일 뿐이에요. SQLite는 디스크 위의 하나의 파일이고, PostgreSQL은 프로세스가 관리하는 여러 파일의 집합입니다. 지금까지 사용한 모든 데이터베이스는 코드에서 <code>open()</code>을 호출할 때처럼 파일 시스템을 읽고 씁니다.</p>
<p>그래서 실제 질문은 &quot;파일을 써야 하는가&quot;가 아닙니다. 어차피 파일은 쓰고 있거든요. 진짜 질문은 &quot;데이터베이스의 파일을 쓸 것인가, 아니면 직접 만든 파일을 쓸 것인가&quot;입니다. 특히 초기 단계의 서비스라면 직접 파일을 관리하는 게 더 나을 수도 있어요.</p>
<p>당연히 우리도 데이터베이스를 좋아합니다. 우리가 만드는 DB Pro도 Mac, Windows, Linux용 데이터베이스 클라이언트니까요. 그렇지만 &quot;정말 데이터베이스가 필요한가?&quot;라는 질문에 솔직하게 답하자면, 그건 스케일에 달려있습니다. 대부분의 서비스는 생각보다 작거든요. 실제로 우리가 테스트해봤습니다. Go, Bun, Rust로 같은 HTTP 서버를 두 가지 저장 전략으로 만들어서 wrk로 부하 테스트를 돌렸어요. 결과가 흥미로웠습니다.</p>
<h3>테스트 환경</h3>
<p>세 개의 평문 파일을 사용했어요: users.jsonl, products.jsonl, orders.jsonl. 형식은 JSONL(newline-delimited JSON)이라서 한 줄에 한 레코드씩 들어가고, 쓸 때는 뒤에 추가합니다. 각 파일은 한 종류의 데이터만 담고 있어요.</p>
<pre><code class="language-json">{&quot;id&quot;:&quot;a3f1...&quot;,&quot;name&quot;:&quot;Alice Chen&quot;,&quot;email&quot;:&quot;alice@example.com&quot;,&quot;created_at&quot;:&quot;2026-04-15T...&quot;}
{&quot;id&quot;:&quot;b7d2...&quot;,&quot;name&quot;:&quot;Bob Torres&quot;,&quot;email&quot;:&quot;bob@example.com&quot;,&quot;created_at&quot;:&quot;2026-04-15T...&quot;}</code></pre>
<p>HTTP 엔드포인트는 두 개입니다: POST /users는 사용자 생성, GET /users/:id는 ID로 조회. 우리는 GET 요청의 성능을 벤치마크했어요. 읽기 작업에서 두 전략의 차이가 두드러지거든요.</p>
<h3>방법 1: 요청할 때마다 파일 읽기</h3>
<p>가장 단순한 접근법입니다. 사용자 abc-123을 찾으라는 요청이 들어오면, 파일을 열어서 한 줄씩 읽고 각 줄을 JSON으로 파싱해서 ID를 비교합니다. 일치하는 게 나오면 반환하는 거죠.</p>

<h3>방법 2: 메모리에 로드하기</h3>
<p>시작할 때 파일을 한 번만 읽어서 모든 레코드를 ID를 키로 하는 해시맵에 저장합니다. 새로운 데이터를 쓸 때는 맵과 파일 둘 다에 기록해요. 조회는 단순한 맵 검색이 됩니다.</p>
<p>이 구조에서 파일은 영구적인 저장소 역할을 하고, 맵이 인덱스가 되는 거죠. 프로세스가 재시작되면 파일에서 다시 로드하면 돼요.</p>
