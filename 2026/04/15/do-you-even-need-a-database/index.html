데이터베이스가 정말 필요한가?

Do you even need a database?

📅 2026-04-15📰 Hacker News (Top)
backend

요약

데이터베이스 대신 파일 기반 저장소를 사용하는 경우와 성능을 비교한 글로, 초기 단계 애플리케이션에서는 직접 파일을 관리하는 것이 충분할 수 있음을 보여줍니다. Go, Bun, Rust로 구현한 벤치마크 결과를 통해 파일 스캔과 메모리 인덱싱 방식의 성능을 실증적으로 분석했습니다.

핵심 포인트

  • 모든 데이터베이스는 결국 파일 시스템 위에서 동작하며, 작은 규모 애플리케이션은 직접 파일 관리 가능
  • 메모리 해시맵 인덱싱으로 파일 기반 저장소의 성능을 크게 향상 가능
  • Go, Bun, Rust로 구현한 벤치마크를 통해 정말 필요한 경우만 데이터베이스 도입 권장

왜 중요한가

💡
과다 설계를 피하고 프로젝트 규모에 맞는 적절한 저장소 기술을 선택하는 데 실증적 기준을 제공합니다.

📄 전문 번역

파일로 충분할 때도 있다

데이터베이스를 자세히 들여다보면 결국 파일일 뿐이에요. SQLite는 디스크 위의 하나의 파일이고, PostgreSQL은 프로세스가 관리하는 여러 파일의 집합입니다. 지금까지 사용한 모든 데이터베이스는 코드에서 open()을 호출할 때처럼 파일 시스템을 읽고 씁니다.

그래서 실제 질문은 "파일을 써야 하는가"가 아닙니다. 어차피 파일은 쓰고 있거든요. 진짜 질문은 "데이터베이스의 파일을 쓸 것인가, 아니면 직접 만든 파일을 쓸 것인가"입니다. 특히 초기 단계의 서비스라면 직접 파일을 관리하는 게 더 나을 수도 있어요.

당연히 우리도 데이터베이스를 좋아합니다. 우리가 만드는 DB Pro도 Mac, Windows, Linux용 데이터베이스 클라이언트니까요. 그렇지만 "정말 데이터베이스가 필요한가?"라는 질문에 솔직하게 답하자면, 그건 스케일에 달려있습니다. 대부분의 서비스는 생각보다 작거든요. 실제로 우리가 테스트해봤습니다. Go, Bun, Rust로 같은 HTTP 서버를 두 가지 저장 전략으로 만들어서 wrk로 부하 테스트를 돌렸어요. 결과가 흥미로웠습니다.

테스트 환경

세 개의 평문 파일을 사용했어요: users.jsonl, products.jsonl, orders.jsonl. 형식은 JSONL(newline-delimited JSON)이라서 한 줄에 한 레코드씩 들어가고, 쓸 때는 뒤에 추가합니다. 각 파일은 한 종류의 데이터만 담고 있어요.

{"id":"a3f1...","name":"Alice Chen","email":"alice@example.com","created_at":"2026-04-15T..."}
{"id":"b7d2...","name":"Bob Torres","email":"bob@example.com","created_at":"2026-04-15T..."}

HTTP 엔드포인트는 두 개입니다: POST /users는 사용자 생성, GET /users/:id는 ID로 조회. 우리는 GET 요청의 성능을 벤치마크했어요. 읽기 작업에서 두 전략의 차이가 두드러지거든요.

방법 1: 요청할 때마다 파일 읽기

가장 단순한 접근법입니다. 사용자 abc-123을 찾으라는 요청이 들어오면, 파일을 열어서 한 줄씩 읽고 각 줄을 JSON으로 파싱해서 ID를 비교합니다. 일치하는 게 나오면 반환하는 거죠.

func findUser(path, id string) (*User, error) {
	f, err := os.Open(path)
	if err != nil {
		return nil, err
	}
	defer f.Close()
	scanner := bufio.NewScanner(f)
	buf := make([]byte, 1024*1024)
	scanner.Buffer(buf, len(buf))
	for scanner.Scan() {
		var u User
		if json.Unmarshal(scanner.Bytes(), &u) == nil && u.ID == id {
			return &u, nil
		}
	}
	return nil, scanner.Err()
}
async function findUser(id: string): Promise<User | null> {
	if (!existsSync(USERS_FILE)) return null;
	const stream = createReadStream(USERS_FILE, { encoding: "utf8" });
	const rl = createInterface({ input: stream, crlfDelay: Infinity });
	for await (const line of rl) {
		if (!line.trim()) continue;
		const user = JSON.parse(line) as User;
		if (user.id === id) {
			stream.destroy();
			return user;
		}
	}
	return null;
}
fn find_user(path: &str, id: &str) -> Option<User> {
	let f = File::open(path).ok()?;
	for line in BufReader::new(f).lines().flatten() {
		if let Ok(user) = serde_json::from_str::<User>(&line) {
			if user.id == id {
				return Some(user);
			}
		}
	}
	None
}

이 방식은 O(n) 복잡도를 가집니다. 모든 요청이 파일 전체를 처음부터 끝까지 읽고, 평균적으로는 절반 정도를 스캔한 후에야 원하는 레코드를 찾아요. 파일이 커질수록 모든 요청이 느려집니다.

방법 2: 메모리에 로드하기

시작할 때 파일을 한 번만 읽어서 모든 레코드를 ID를 키로 하는 해시맵에 저장합니다. 새로운 데이터를 쓸 때는 맵과 파일 둘 다에 기록해요. 조회는 단순한 맵 검색이 됩니다.

이 구조에서 파일은 영구적인 저장소 역할을 하고, 맵이 인덱스가 되는 거죠. 프로세스가 재시작되면 파일에서 다시 로드하면 돼요.

type Store struct {
	mu    sync.RWMutex
	users map[string]User
	f     *os.File
}

func NewStore(path string) (*Store, error) {
	f, err := os.OpenFile(path, os.O_CREATE|os.O_RDWR|os.O_APPEND, 0644)
	if err != nil {
		return nil, err
	}
	users := make(map[string]User)
	scanner := bufio.NewScanner(f)
	buf := make([]byte, 1024*1024)
	scanner.Buffer(buf, len(buf))
	for scanner.Scan() {
		var u User
		if json.Unmarshal(scanner.Bytes(), &u) == nil {
			users[u.ID] = u
		}
	}
	return &Store{users: users, f: f}, scanner.Err()
}

func (s *Store) Get(id string) (User, bool) {
	s.mu.RLock()
	defer s.mu.RUnlock()
	return s.users[id]
}

func (s *Store) Create(name, email string) (User, error) {
	u := User{ID: uuid.NewString(), Name: name, Email: email, CreatedAt: time.Now().UTC()}
	b, _ := json.Marshal(u)
	s.mu.Lock()
	defer s.mu.Unlock()
	fmt.Fprintf(s.f, "%s\n", b)
	s.users[u.ID] = u
	return u, nil
}
function loadUsers(): Map<string, User> {
	const map = new Map<string, User>();
	if (!existsSync(USERS_FILE)) return map;
	for (const line of readFileSync(USERS_FILE, "utf8").split("\n")) {
		if (!line.trim()) continue;
		const user = JSON.parse(line) as User;
		map.set(user.id, user);
	}
	return map;
}

const users = loadUsers();

function createUser(name: string, email: string): User {
	const user = { id: randomUUID(), name, email, created_at: new Date().toISOString() };
	appendFileSync(USERS_FILE, JSON.stringify(user) + "\n");
	users.set(user.id, user);
	return user;
}
struct Store {
	users: RwLock<HashMap<String, User>>,
	file: Mutex<File>,
}

impl Store {
	fn load(path: &str) -> Arc<Self> {
		let mut map = HashMap::new();
		if let Ok(f) = File::open(path) {
			for line in BufReader::new(f).lines().flatten() {
				if let Ok(u) = serde_json::from_str::<User>(&line) {
					map.insert(u.id.clone(), u);
				}
			}
		}
		let file = OpenOptions::new().create(true).append(true).open(path).unwrap();
		Arc::new(Store {
			users: RwLock::new(map),
			file: Mutex::new(file),
		})
	}
}

이제 조회는 O(1)이에요. 맵 조회는 상수 시간에 이루어지니까 데이터가 얼마나 많든 성능은 일정합니다. 당연히 데이터베이스만큼 정교한 기능은 없지만, 충분히 빠르고 관리도 간단합니다.
