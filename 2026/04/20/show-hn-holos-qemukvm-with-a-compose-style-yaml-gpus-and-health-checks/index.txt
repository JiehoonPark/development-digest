15:T1dcf,<h2>Holos: KVM 기반 멀티 VM 스택 관리 도구</h2>
<p>Docker Compose처럼 YAML 파일 하나로 여러 VM을 정의하고 관리할 수 있는 도구, Holos를 소개합니다. libvirt나 XML 설정 없이 간단하게 시작할 수 있습니다.</p>
<h3>핵심 특징</h3>
<p>Holos의 기본 단위는 컨테이너가 아니라 <strong>VM</strong>입니다. 각 워크로드는 독립적인 커널 경계, qcow2 오버레이, cloud-init 설정을 갖게 됩니다.</p>
<h3>빠른 시작</h3>
<p><code>holos.yaml</code> 파일을 작성해보겠습니다.</p>
<pre><code class="language-yaml">name: my-stack
services:
  db:
    image: ubuntu:noble
    vm:
      vcpu: 2
      memory_mb: 1024
    cloud_init:
      packages:
        - postgresql
      runcmd:
        - systemctl enable postgresql
        - systemctl start postgresql
  web:
    image: ubuntu:noble
    replicas: 2
    depends_on:
      - db
    ports:
      - &quot;8080:80&quot;
    volumes:
      - ./www:/srv/www:ro
    cloud_init:
      packages:
        - nginx
      write_files:
        - path: /etc/nginx/sites-enabled/default
          content: |
            server {
              listen 80;
              location / { proxy_pass http://db:5432; }
            }
      runcmd:
        - systemctl restart nginx</code></pre>
<p>이제 한 줄만 실행하면 됩니다.</p>
<pre><code class="language-bash">holos up</code></pre>
<p>끝입니다. Nginx VM 2개와 PostgreSQL VM 1개가 같은 호스트 위에서 실행되며, 서로 이름으로 통신할 수 있습니다.</p>
<h3>주요 명령어</h3>
<div class="table-wrapper"><table><thead><tr><th>명령어</th><th>설명</th></tr></thead><tbody><tr><td><code>holos up [-f holos.yaml]</code></td><td>모든 서비스 시작</td></tr><tr><td><code>holos down [-f holos.yaml]</code></td><td>모든 서비스 중지 및 제거</td></tr><tr><td><code>holos ps</code></td><td>실행 중인 프로젝트 목록</td></tr><tr><td><code>holos start/stop [-f holos.yaml] [svc]</code></td><td>특정 서비스 시작/중지</td></tr><tr><td><code>holos console [-f holos.yaml] &lt;inst&gt;</code></td><td>인스턴스 시리얼 콘솔 접속</td></tr><tr><td><code>holos exec [-f holos.yaml] &lt;inst&gt; [cmd...]</code></td><td>인스턴스에서 명령 실행</td></tr><tr><td><code>holos logs [-f holos.yaml] &lt;svc&gt;</code></td><td>서비스 로그 조회</td></tr><tr><td><code>holos validate [-f holos.yaml]</code></td><td>설정 파일 검증</td></tr><tr><td><code>holos pull &lt;image&gt;</code></td><td>클라우드 이미지 다운로드</td></tr><tr><td><code>holos images</code></td><td>사용 가능한 이미지 목록</td></tr><tr><td><code>holos devices [--gpu]</code></td><td>PCI 장치 및 IOMMU 그룹 목록</td></tr><tr><td><code>holos install/uninstall [-f holos.yaml]</code></td><td>systemd 유닛 생성/제거</td></tr><tr><td><code>holos import [vm...]</code></td><td>virsh 정의 VM을 holos.yaml로 변환</td></tr></tbody></table></div>
<h3>설정 파일 형식</h3>
<p>Holos의 설정 파일은 docker-compose와 유사하게 설계되었습니다.</p>
<p><strong>주요 항목:</strong></p>
<ul>
<li><code>services</code> - 각 서비스는 자체 이미지, 리소스, cloud-init 설정을 가진 VM</li>
<li><code>depends_on</code> - 서비스는 의존성 순서대로 시작</li>
<li><code>ports</code> - &quot;host:guest&quot; 형식, 레플리카 간 자동 증가</li>
<li><code>volumes</code> - &quot;./source:/target:ro&quot; 형식의 바인드 마운트 또는 named volume</li>
<li><code>replicas</code> - N개 인스턴스 실행</li>
<li><code>cloud_init</code> - packages, write_files, runcmd 등 표준 cloud-init 설정</li>
<li><code>stop_grace_period</code> - ACPI 종료 대기 시간 (예: &quot;30s&quot;, &quot;2m&quot;, 기본값 30s)</li>
<li><code>healthcheck</code> - test, interval, retries, start_period, timeout으로 의존성 제어</li>
</ul>
<h3>우아한 종료(Graceful Shutdown)</h3>
<p><code>holos stop</code>이나 <code>holos down</code>을 실행하면, 먼저 QMP를 통해 <code>system_powerdown</code>을 게스트에 전송합니다(전원 버튼을 누르는 것과 동일). 그 다음 <code>stop_grace_period</code>만큼 대기합니다.</p>
<p>이 시간 내에 QEMU가 종료되지 않으면, docker-compose처럼 SIGTERM 후 SIGKILL로 강제 종료합니다.</p>
<pre><code class="language-yaml">services:
  db:
    image: ubuntu:noble
    stop_grace_period: 60s  # DB 버퍼 플러시 시간 확보</code></pre>
<h3>데이터 볼륨</h3>
<p>top-level <code>volumes</code> 블록으로 선언한 named volume은 <code>state_dir/volumes/&lt;project&gt;/&lt;name&gt;.qcow2</code> 위치에 저장되고, 각 인스턴스의 작업 디렉토리에 심링크됩니다. <code>holos down</code>을 실행해도 데이터는 유지됩니다(심링크만 제거됨).</p>
<pre><code class="language-yaml">name: demo
services:
  db:
    image: ubuntu:noble
    volumes:
      - pgdata:/var/lib/postgresql
volumes:
  pgdata:
    size: 20G</code></pre>
<p>볼륨은 virtio-blk 장치로 마운트되며 <code>serial=vol-&lt;name&gt;</code>이 설정되어 있어서, 게스트 내부에서 <code>/dev/disk/by-id/virtio-vol-pgdata</code>로 접근할 수 있습니다.</p>
<p>Cloud-init는 첫 부팅 시 자동으로 <code>mkfs.ext4</code> 및 <code>/etc/fstab</code> 설정을 수행하므로 별도의 수동 구성이 필요 없습니다.</p>
<h3>헬스 체크와 의존성 관리</h3>
<p><code>healthcheck</code>를 설정한 서비스는 의존하는 다른 서비스가 시작되기 전에 반드시 헬스 체크를 통과해야 합니다. 프로브는 SSH를 통해 실행됩니다(<code>holos exec</code>와 동일한 키 사용).</p>
<pre><code class="language-yaml">services:
  db:
    image: postgres-cloud.qcow2
    healthcheck:
      test: [&quot;pg_isready&quot;, &quot;-U&quot;, &quot;postgres&quot;]
      interval: 2s
      retries: 30
      start_period: 10s
      timeout: 3s
  api:
    image: api.qcow2
    depends_on: [db]  # db의 헬스 체크 통과까지 대기</code></pre>
<p><strong>팁:</strong></p>
<ul>
<li><code>test</code>는 리스트(exec 형식) 또는 문자열(<code>sh -c</code>로 래핑됨)로 작성 가능</li>
<li><code>HOLOS_HEALTH_BYPASS=1</code> 환경변수를 설정하면 실제 프로브를 건너뛸 수 있습니다(인게스트 SSHD가 없는 CI 환경에 유용)</li>
</ul>
<h3>원격 접속(holos exec)</h3>
<p><code>holos up</code>을 실행하면 프로젝트별 SSH 키 쌍이 자동으로 <code>state_dir/ssh/&lt;project&gt;/</code> 아래에 생성되고, 공개 키는 cloud-init를 통해 주입됩니다. 각 인스턴스마다 호스트 포트가 할당되어 게스트 포트 22로 포워딩됩니다.</p>
<pre><code class="language-bash">holos exec web-0                    # 인터랙티브 셸
holos exec db-0 -- pg_isready      # 일회성 명령 실행</code></pre>
<p><code>-u &lt;user&gt;</code> 옵션으로 로그인 사용자를 지정할 수 있습니다(기본값: cloud_init.user 또는 ubuntu).</p>
<h3>재부팅 후 자동 복구</h3>
<p>호스트 재부팅 후에도 프로젝트를 자동으로 시작할 수 있도록 systemd 유닛을 생성할 수 있습니다.</p>
<pre><code class="language-bash">holos install --enable              # 사용자 레벨, sudo 불필요
holos install --system --enable     # 시스템 레벨, 모든 로그인 전
holos install --dry-run             # 유닛 내용만 출력</code></pre>
