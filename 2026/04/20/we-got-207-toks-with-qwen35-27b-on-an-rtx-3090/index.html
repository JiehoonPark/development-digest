RTX 3090에서 Qwen3.5-27B로 207 tok/s 달성

We got 207 tok/s with Qwen3.5-27B on an RTX 3090

📅 2026-04-20📰 Hacker News (Top)
aibackend

요약

Luce-Org에서 개발한 최적화된 추론 엔진들이 RTX 3090 같은 기존 GPU에서 대규모 언어 모델의 추론 속도를 크게 향상시킵니다. Megakernel과 DFlash 기법을 통해 높은 처리량과 에너지 효율을 달성했습니다.

핵심 포인트

  • DFlash와 DDTree 기법으로 Qwen3.5-27B을 RTX 3090에서 207 tok/s로 처리 (자동회귀 대비 5.46배 빠름)
  • Megakernel을 통해 Qwen 3.5-0.8B의 모든 레이어를 단일 CUDA 디스패치로 처리하여 에너지 효율성 극대화
  • 하드웨어별 맞춤 최적화로 GPU의 전체 성능을 활용하는 기존 범용 프레임워크의 한계를 극복

왜 중요한가

💡
개발자는 고가의 신규 GPU 없이도 기존 하드웨어에서 대규모 언어 모델을 효율적으로 로컬 배포할 수 있게 되어 비용과 지연 시간을 절감할 수 있습니다.

📄 전문 번역

로컬 LLM 추론을 칩 맞춤형으로 다시 손질하다

각 칩별로 커널, 추론 최적화, 양자화를 손으로 직접 튜닝합니다. 더 좋은 하드웨어를 기다리지 않아요. 대신 소프트웨어를 다시 씁니다.

상자 안에 담긴 두 가지 프로젝트

각 프로젝트는 독립적인 릴리스이며 벤치마크와 논문 수준의 분석 자료를 함께 제공합니다.

01 · Qwen 3.5 0.8B RTX 3090용 메가커널

DeltaNet과 Attention을 섞은 LLM용 첫 메가커널입니다. Qwen 3.5-0.8B의 24개 레이어를 하나의 CUDA 디스패치로 처리하고, 2020년형 GPU에서 1.87 tok/J를 달성합니다. 이는 Apple의 최신 칩과 같은 에너지 효율을 보이면서 처리량은 2배 빠릅니다.

# 1. 클론 후 진입
git clone https://github.com/Luce-Org/lucebox-hub && cd lucebox-hub/megakernel

# 2. 설치 (Python 3.10+, CUDA 12+, PyTorch 2.0+ 필요)
pip install -e .

# 3. 벤치마크 실행 (prefill pp520 + decode tg128)
python final_bench.py
방식Prefill pp520Decode tg128tok/J

작동 원리: 82개 블록, 512개 스레드로 하나의 지속형 커널을 구성합니다. 레이어 사이에 CPU를 오갈 필요가 없고, HuggingFace에서 가중치를 직접 스트리밍합니다. 토큰당 ~100번의 커널 실행 대신 협력형 그리드 동기화를 사용해요. 전력 한계에 먼저 도달하므로, DVFS(동적 전압/주파수 스케일링)가 빡빡한 실행을 곧바로 전력 절감으로 변환합니다.

02 · DFlash DDTree Qwen 3.5 27B on RTX 3090

DFlash 추측 디코딩을 GGUF로 처음 포팅한 버전입니다. Qwen 3.5-27B을 RTX 3090 한 장에서 돌리는데, Q4_K_M 타겟 모델과 BF16 드래프트를 결합하고 DDTree 예산을 22로 설정합니다.

데모에서는 최대 207 tok/s를 달성합니다 (자동회귀 38.0 tok/s 대비 5.46배). HumanEval 10-프롬프트 벤치에선 평균 129.5 tok/s를 보입니다.

# 1. 서브모듈과 함께 클론
git clone --recurse-submodules https://github.com/Luce-Org/lucebox-hub && cd lucebox-hub/dflash

# 2. C++/CUDA 디코더 빌드 (약 3분, CUDA 12+, CMake 3.18+ 필요)
cmake -B build -S . -DCMAKE_CUDA_ARCHITECTURES=86 -DCMAKE_BUILD_TYPE=Release
cmake --build build --target test_dflash -j

# 3. 가중치 다운로드 (Q4_K_M 약 16GB + BF16 드래프트 3.46GB)
huggingface-cli download unsloth/Qwen3.5-27B-GGUF Qwen3.5-27B-Q4_K_M.gguf --local-dir models/
huggingface-cli download z-lab/Qwen3.5-27B-DFlash model.safetensors --local-dir models/draft/

# 4a. 스트리밍 생성
python3 scripts/run.py --prompt "def fibonacci(n):"

# 4b. 또는 벤치마크 재현 (약 15분)
python3 scripts/bench_llm.py

이 선택지들이 만들어진 이유: RTX 3090의 24GB 메모리에서 타겟 모델 + 드래프트 + DDTree 검증 상태 + KV 캐시를 모두 담으려면 Q4_K_M GGUF(약 16GB)가 최대 크기입니다. 이 제약이 ggml 위에 새로운 포트를 만들도록 강제했는데, GGUF 타겟을 지원하는 공개 DFlash 런타임이 없거든요.

우리가 만든 것과 만들지 않은 것: 알고리즘은 원래 논문에서 비롯됐습니다.

  • DFlash (z-lab, 2026): 타겟 숨겨진 상태로 조건화된 블록 확산 드래프트
  • DDTree (Ringel et al., 2026): 체인 검증과 같은 계산 예산으로 더 나은 성능의 트리 구조 검증

우리가 포팅하고 튜닝한 부분:

  • ggml 위의 C++/CUDA 디코더 엔진 (libllama 없음, Q4_K_M 타겟 경로)
  • 트리 인식 SSM 상태 롤백을 위한 3개 커스텀 CUDA 커널
  • RTX 3090용 DDTree 예산 최적화 (budget=22가 최적)
  • Q4_0 KV 캐시 + 슬라이딩 타겟_feat 링으로 128K 컨텍스트를 24GB에 담기 (~3% 손실)

왜 이게 필요한가

로컬 AI는 특권이 아니라 기본값이어야 합니다. 개인 데이터, 토큰당 요금 없음, 벤더 종속성 없음. 모델을 돌릴 수 있는 하드웨어는 이미 책상 위에 있어요. 그 칩을 잘 쓸 소프트웨어가 없을 뿐입니다.

지난 10년간 범용 프레임워크가 지배했던 이유는 칩별 커널 튜닝이 너무 비용이 컸기 때문입니다. 모든 것에 준수하지만 어디에도 뛰어나지 않은 하나의 스택. 실리콘 성능의 대부분이 활용되지 않는 거죠.

AI 지원 개발은 이 계산식을 뒤집었습니다. 분기가 걸리던 리라이트가 이제 릴리스 사이클에 맞습니다. Lucebox는 우리가 이런 최적화를 발행하는 곳입니다. 칩 하나, 모델 계열 하나씩요. MIT 라이선스 소스, 완전한 분석 자료, 재현 가능한 벤치마크와 함께요.

요구사항

  • NVIDIA GPU (Ampere+, sm_86+), CUDA 12+, PyTorch 2.0+
  • RTX 3090 (2020년형)에서 테스트됨
  • dflash: CMake 3.18+, --recurse-submodules로 Luce-Org/llama.cpp@luce-dflash 포크 필요
  • 선택사항: GPU 최적점 찾기 - sudo nvidia-smi -pl 220 (메가커널이 220W에서 최고 효율 달성)
