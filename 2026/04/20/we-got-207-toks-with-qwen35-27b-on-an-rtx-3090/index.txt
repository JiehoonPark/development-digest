28:T188e,<h2>로컬 LLM 추론을 칩 맞춤형으로 다시 손질하다</h2>
<p>각 칩별로 커널, 추론 최적화, 양자화를 손으로 직접 튜닝합니다. 더 좋은 하드웨어를 기다리지 않아요. 대신 소프트웨어를 다시 씁니다.</p>
<h3>상자 안에 담긴 두 가지 프로젝트</h3>
<p>각 프로젝트는 독립적인 릴리스이며 벤치마크와 논문 수준의 분석 자료를 함께 제공합니다.</p>
<h4>01 · Qwen 3.5 0.8B RTX 3090용 메가커널</h4>
<p>DeltaNet과 Attention을 섞은 LLM용 첫 메가커널입니다. Qwen 3.5-0.8B의 24개 레이어를 하나의 CUDA 디스패치로 처리하고, 2020년형 GPU에서 1.87 tok/J를 달성합니다. 이는 Apple의 최신 칩과 같은 에너지 효율을 보이면서 처리량은 2배 빠릅니다.</p>
<pre><code class="language-bash"># 1. 클론 후 진입
git clone https://github.com/Luce-Org/lucebox-hub &amp;&amp; cd lucebox-hub/megakernel

# 2. 설치 (Python 3.10+, CUDA 12+, PyTorch 2.0+ 필요)
pip install -e .

# 3. 벤치마크 실행 (prefill pp520 + decode tg128)
python final_bench.py</code></pre>
<div class="table-wrapper"><table><thead><tr><th>방식</th><th>Prefill pp520</th><th>Decode tg128</th><th>tok/J</th></tr></thead><tbody><tr><td>Megakernel @220W</td><td>37,800</td><td>413</td><td>1.87</td></tr><tr><td>llama.cpp BF16 @350W</td><td>11,247</td><td>267</td><td>0.76</td></tr><tr><td>PyTorch HF</td><td>7,578</td><td>108</td><td>n/a</td></tr></tbody></table></div>
<p><strong>작동 원리:</strong> 82개 블록, 512개 스레드로 하나의 지속형 커널을 구성합니다. 레이어 사이에 CPU를 오갈 필요가 없고, HuggingFace에서 가중치를 직접 스트리밍합니다. 토큰당 ~100번의 커널 실행 대신 협력형 그리드 동기화를 사용해요. 전력 한계에 먼저 도달하므로, DVFS(동적 전압/주파수 스케일링)가 빡빡한 실행을 곧바로 전력 절감으로 변환합니다.</p>
<h4>02 · DFlash DDTree Qwen 3.5 27B on RTX 3090</h4>
<p>DFlash 추측 디코딩을 GGUF로 처음 포팅한 버전입니다. Qwen 3.5-27B을 RTX 3090 한 장에서 돌리는데, Q4_K_M 타겟 모델과 BF16 드래프트를 결합하고 DDTree 예산을 22로 설정합니다.</p>
<p>데모에서는 최대 207 tok/s를 달성합니다 (자동회귀 38.0 tok/s 대비 5.46배). HumanEval 10-프롬프트 벤치에선 평균 129.5 tok/s를 보입니다.</p>
<pre><code class="language-bash"># 1. 서브모듈과 함께 클론
git clone --recurse-submodules https://github.com/Luce-Org/lucebox-hub &amp;&amp; cd lucebox-hub/dflash

# 2. C++/CUDA 디코더 빌드 (약 3분, CUDA 12+, CMake 3.18+ 필요)
cmake -B build -S . -DCMAKE_CUDA_ARCHITECTURES=86 -DCMAKE_BUILD_TYPE=Release
cmake --build build --target test_dflash -j

# 3. 가중치 다운로드 (Q4_K_M 약 16GB + BF16 드래프트 3.46GB)
huggingface-cli download unsloth/Qwen3.5-27B-GGUF Qwen3.5-27B-Q4_K_M.gguf --local-dir models/
huggingface-cli download z-lab/Qwen3.5-27B-DFlash model.safetensors --local-dir models/draft/

# 4a. 스트리밍 생성
python3 scripts/run.py --prompt &quot;def fibonacci(n):&quot;

# 4b. 또는 벤치마크 재현 (약 15분)
python3 scripts/bench_llm.py</code></pre>
<div class="table-wrapper"><table><thead><tr><th>벤치마크</th><th>자동회귀 (tok/s)</th><th>DFlash+DDTree (tok/s)</th><th>속도 향상</th></tr></thead><tbody><tr><td>HumanEval</td><td>37.8</td><td>129.5</td><td>3.43×</td></tr><tr><td>Math500</td><td>37.7</td><td>110.5</td><td>2.93×</td></tr><tr><td>GSM8K</td><td>37.7</td><td>96.2</td><td>2.55×</td></tr></tbody></table></div>
<p><strong>이 선택지들이 만들어진 이유:</strong> RTX 3090의 24GB 메모리에서 타겟 모델 + 드래프트 + DDTree 검증 상태 + KV 캐시를 모두 담으려면 Q4_K_M GGUF(약 16GB)가 최대 크기입니다. 이 제약이 ggml 위에 새로운 포트를 만들도록 강제했는데, GGUF 타겟을 지원하는 공개 DFlash 런타임이 없거든요.</p>
<p><strong>우리가 만든 것과 만들지 않은 것:</strong> 알고리즘은 원래 논문에서 비롯됐습니다.</p>
<ul>
<li><strong>DFlash</strong> (z-lab, 2026): 타겟 숨겨진 상태로 조건화된 블록 확산 드래프트</li>
<li><strong>DDTree</strong> (Ringel et al., 2026): 체인 검증과 같은 계산 예산으로 더 나은 성능의 트리 구조 검증</li>
</ul>
<p>우리가 포팅하고 튜닝한 부분:</p>
<ul>
<li>ggml 위의 C++/CUDA 디코더 엔진 (libllama 없음, Q4_K_M 타겟 경로)</li>
<li>트리 인식 SSM 상태 롤백을 위한 3개 커스텀 CUDA 커널</li>
<li>RTX 3090용 DDTree 예산 최적화 (budget=22가 최적)</li>
<li>Q4_0 KV 캐시 + 슬라이딩 타겟_feat 링으로 128K 컨텍스트를 24GB에 담기 (~3% 손실)</li>
</ul>
<h3>왜 이게 필요한가</h3>
<p>로컬 AI는 특권이 아니라 기본값이어야 합니다. 개인 데이터, 토큰당 요금 없음, 벤더 종속성 없음. 모델을 돌릴 수 있는 하드웨어는 이미 책상 위에 있어요. 그 칩을 잘 쓸 소프트웨어가 없을 뿐입니다.</p>
<p>지난 10년간 범용 프레임워크가 지배했던 이유는 칩별 커널 튜닝이 너무 비용이 컸기 때문입니다. 모든 것에 준수하지만 어디에도 뛰어나지 않은 하나의 스택. 실리콘 성능의 대부분이 활용되지 않는 거죠.</p>
<p>AI 지원 개발은 이 계산식을 뒤집었습니다. 분기가 걸리던 리라이트가 이제 릴리스 사이클에 맞습니다. Lucebox는 우리가 이런 최적화를 발행하는 곳입니다. 칩 하나, 모델 계열 하나씩요. MIT 라이선스 소스, 완전한 분석 자료, 재현 가능한 벤치마크와 함께요.</p>
<h3>요구사항</h3>
<ul>
