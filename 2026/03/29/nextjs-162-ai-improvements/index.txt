13:T1a78,<h2>Next.js 16.2: AI 개발을 위한 개선 사항들</h2>
<p>Next.js 16.2에서는 AI 어시스턴트 개발을 염두에 두고 여러 기능을 개선했습니다. 이제 AI 에이전트가 프로젝트를 더 잘 이해하고, 터미널에서 직접 문제를 디버깅하며, 실행 중인 앱을 검사할 수 있게 됐거든요. 브라우저를 열 필요도 없습니다.</p>
<p><strong>주요 개선 사항</strong></p>
<ul>
<li>AI 기반 create-next-app: 바로 사용할 수 있는 프로젝트 스캐폴딩</li>
<li>브라우저 로그 포워딩: 브라우저 에러를 터미널로 전달해 AI 디버깅 지원</li>
<li>개발 서버 락 파일: 두 개의 개발 서버가 동시에 시작되는 문제 해결</li>
<li>실험 중인 Agent DevTools: AI 에이전트에 React DevTools와 Next.js 진단 도구 접근 권한 제공</li>
</ul>
<h3>AI 기반 create-next-app</h3>
<p>이제 <code>create-next-app</code>으로 새 프로젝트를 만들면 기본적으로 <code>AGENTS.md</code> 파일이 포함됩니다. 이 파일을 통해 AI 코딩 에이전트가 프로젝트 시작 단계부터 버전에 맞는 Next.js 문서에 접근할 수 있어요.</p>
<p>우리가 <code>AGENTS.md</code>에 대해 연구한 결과가 흥미로웠는데요. 에이전트에게 번들로 포함된 문서를 제공했을 때 Next.js 평가에서 100% 통과율을 달성했습니다. 반면 기술 기반 접근법은 최대 79% 수준에 머물렀거든요. 핵심 인사이트는 이겁니다. 에이전트는 필요할 때만 검색하는 방식보다는 항상 접근 가능한 문서가 있을 때 훨씬 잘 작동한다는 점이에요.</p>
<p><code>AGENTS.md</code> 파일은 단순한 지시문입니다. 코드를 작성하기 전에 <code>node_modules/next/dist/docs/</code>에 번들로 포함된 문서를 먼저 읽도록 에이전트에게 알려주는 거죠. Next.js npm 패키지에 전체 문서가 마크다운 파일로 포함되어 있어서, 에이전트는 외부 데이터를 가져올 필요 없이 로컬에서 정확한 버전의 문서를 참조할 수 있습니다.</p>
<p><strong>기존 프로젝트에 추가하는 방법</strong></p>
<p>Next.js 16.2 이상이라면 문서가 이미 next 패키지에 포함되어 있습니다. 프로젝트 루트에 이 두 파일을 추가하세요.</p>
<pre><code class="language-markdown">&lt;!-- AGENTS.md --&gt;
&lt;!-- BEGIN:nextjs-agent-rules --&gt;
# Next.js: ALWAYS read docs before coding
Before any Next.js work, find and read the relevant doc in `node_modules/next/dist/docs/`. Your training data is outdated — the docs are the source of truth.
&lt;!-- END:nextjs-agent-rules --&gt;</code></pre>
<p>주석 마커는 Next.js가 관리하는 섹션을 구분합니다. 마커 바깥에는 프로젝트 특화 지시사항을 추가해도 됩니다. 나중에 업데이트할 때는 마커 사이의 내용만 교체될 테니까요.</p>
<p>Claude Code용 instruction 파일인 <code>CLAUDE.md</code>에는 <code>@</code> 지시어로 <code>AGENTS.md</code>를 추가 컨텍스트로 포함시킵니다.</p>
<pre><code class="language-markdown">&lt;!-- CLAUDE.md --&gt;
@AGENTS.md</code></pre>
<p>더 오래된 버전을 사용 중이라면 codemod로 자동 생성할 수 있습니다.</p>
<pre><code class="language-bash">npx @next/codemod@latest agents-md</code></pre>
<p>더 자세한 내용은 [AI 에이전트 설정 가이드](https://nextjs.org/docs/app/building-your-application/guides/agents)를 참고하세요.</p>
<h3>브라우저 로그 포워딩</h3>
<p>이제 Next.js는 개발 중에 브라우저 에러를 기본적으로 터미널로 전달합니다. 브라우저 콘솔을 열지 않고도 클라이언트 사이드 에러를 볼 수 있게 된 거예요. 주로 터미널을 통해 작동하는 AI 에이전트에게 특히 유용합니다.</p>
<p>기본적으로는 에러만 터미널로 전달되지만, <code>next.config.ts</code>의 <code>logging.browserToTerminal</code> 옵션으로 수준을 조절할 수 있습니다.</p>
<pre><code class="language-typescript">// next.config.ts
const nextConfig = {
  logging: {
    browserToTerminal: true,
    // 'error' — 에러만 (기본값)
    // 'warn' — 경고와 에러
    // true — 모든 콘솔 출력
    // false — 비활성화
  },
};

export default nextConfig;</code></pre>
<h3>개발 서버 락 파일</h3>
<p>Next.js는 이제 실행 중인 개발 서버의 PID, 포트, URL을 <code>.next/dev/lock</code> 파일에 저장합니다. 같은 프로젝트 디렉토리에서 두 번째 <code>next dev</code> 프로세스가 시작되면, Next.js가 락 파일을 읽고 실행 가능한 에러 메시지를 출력합니다.</p>
<pre><code>Error: Another next dev server is already running.
- Local: http://localhost:3000
- PID: 12345
- Dir: /path/to/project
- Log: .next/dev/logs/next-development.log

Run kill 12345 to stop it.</code></pre>
<p>AI 코딩 에이전트는 서버가 이미 실행 중인지 확인하지 않고 <code>next dev</code>를 시작하려는 경우가 자주 있는데요. 이렇게 구조화된 에러 메시지가 있으면 에이전트가 자동으로 기존 프로세스의 PID를 종료하거나 해당 URL에 연결할 수 있게 되어 수동 개입이 필요 없어집니다.</p>
<p>락 파일은 두 개의 <code>next build</code> 프로세스가 동시에 실행되는 것도 방지하므로, 빌드 산출물이 손상되는 문제도 예방할 수 있어요.</p>
<h3>실험 중인 Agent DevTools</h3>
<p>위의 기능들이 에이전트의 프로젝트 이해와 문제 디버깅을 돕는다면, <code>@vercel/next-browser</code>는 실행 중인 Next.js 애플리케이션을 검사하는 기능을 추가합니다.</p>
<p><code>next-browser</code>는 실험 단계의 CLI 도구입니다. 스크린샷, 네트워크 요청, 콘솔 로그 같은 브라우저 수준의 데이터뿐 아니라 React DevTools와 Next.js dev overlay에서 얻을 수 있는 컴포넌트 트리, props, hooks, Partial Prerendering(PPR) 셸, 에러 같은 프레임워크 특화 정보를 구조화된 텍스트 형태로 셸 커맨드를 통해 반환합니다.</p>
<p>LLM은 DevTools 패널을 읽을 수 없습니다. 하지만 <code>next-browser tree</code>를 실행하고 그 출력을 파싱한 후 다음에 뭘 검사할지 결정할 수 있죠. 각 커맨드는 지속되는 브라우저 세션에 대한 일회성 요청이므로, 에이전트는 브라우저 상태를 관리할 필요 없이 앱을 반복적으로 조회할 수 있습니다. 이제 브라우저는 에이전트가 접근할 수 없는 UI가 아니라 추론할 수 있는 대상이 되었어요.</p>
