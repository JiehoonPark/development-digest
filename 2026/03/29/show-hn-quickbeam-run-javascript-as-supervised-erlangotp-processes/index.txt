28:T23f8,<h2>QuickBEAM: BEAM 위에서 JavaScript를 실행하다</h2>
<p>QuickBEAM은 JavaScript 런타임인데, BEAM 위에서 직접 동작합니다. Web API는 OTP로 지원하고, 네이티브 DOM을 제공하며, TypeScript 도구도 내장되어 있죠.</p>
<p>가장 흥미로운 점은 JS 런타임 자체가 GenServer라는 겁니다. 감독 트리에 살아있으면서 메시지를 주고받고, Erlang/OTP 라이브러리를 호출할 수 있습니다. 모두 BEAM을 떠나지 않으면서 말이죠.</p>
<h3>설치하기</h3>
<p>먼저 의존성을 추가해줍니다.</p>
<pre><code class="language-elixir">def deps do
  [{:quickbeam, &quot;~&gt; 0.7.1&quot;}]
end</code></pre>
<p>Zig 0.15 이상이 필요합니다. Zigler가 자동으로 설치하거나, 시스템 Zig를 사용할 수 있습니다.</p>
<h3>시작해보기</h3>
<p>런타임을 띄우고 JavaScript 코드를 실행하는 건 간단합니다.</p>
<pre><code class="language-elixir">{:ok, rt} = QuickBEAM.start()
{:ok, 3} = QuickBEAM.eval(rt, &quot;1 + 2&quot;)
{:ok, &quot;HELLO&quot;} = QuickBEAM.eval(rt, &quot;'hello'.toUpperCase()&quot;)</code></pre>
<p>상태도 유지됩니다. 함수를 정의한 뒤 나중에 호출할 수 있어요.</p>
<pre><code class="language-elixir">QuickBEAM.eval(rt, &quot;function greet(name) { return 'hi ' + name }&quot;)
{:ok, &quot;hi world&quot;} = QuickBEAM.call(rt, &quot;greet&quot;, [&quot;world&quot;])

QuickBEAM.stop(rt)</code></pre>
<h3>BEAM과의 통합</h3>
<p>JavaScript에서 Elixir 함수를 호출하고 OTP 라이브러리에 접근할 수 있다는 게 정말 강력합니다.</p>
<pre><code class="language-elixir">{:ok, rt} = QuickBEAM.start(handlers: %{
  &quot;db.query&quot; =&gt; fn [sql] -&gt; MyRepo.query!(sql).rows end,
  &quot;cache.get&quot; =&gt; fn [key] -&gt; Cachex.get!(:app, key) end,
})</code></pre>
<p>이제 JavaScript 코드에서 이 핸들러들을 비동기로 호출할 수 있습니다.</p>
<pre><code class="language-elixir">{:ok, rows} = QuickBEAM.eval(rt, &quot;&quot;&quot;
  const rows = await Beam.call(&quot;db.query&quot;, &quot;SELECT * FROM users LIMIT 5&quot;);
  rows.map(r =&gt; r.name);
&quot;&quot;&quot;)</code></pre>
<p>JavaScript는 어떤 BEAM 프로세스에도 메시지를 보낼 수 있습니다.</p>
<pre><code class="language-javascript">// 런타임 자신의 PID 가져오기
const self = Beam.self();

// 다른 PID로 메시지 전송
Beam.send(somePid, {type: &quot;update&quot;, data: result});

// BEAM 메시지 수신
Beam.onMessage((msg) =&gt; {
  console.log(&quot;got:&quot;, msg);
});

// 프로세스 모니터링
const ref = Beam.monitor(pid, (reason) =&gt; {
  console.log(&quot;process died:&quot;, reason);
});

Beam.demonitor(ref);</code></pre>

<h3>감독 트리와 함께 사용하기</h3>
<p>QuickBEAM은 OTP 자식으로 동작하기 때문에 감독 트리에 바로 등록할 수 있습니다. 크래시 복구도 자동으로 되죠.</p>
<pre><code class="language-elixir">children = [
  {QuickBEAM,
    name: :renderer,
    id: :renderer,
    script: &quot;priv/js/app.js&quot;,
    handlers: %{
      &quot;db.query&quot; =&gt; fn [sql, params] -&gt; Repo.query!(sql, params).rows end,
    }},
  {QuickBEAM, name: :worker, id: :worker},
  # 고동시성 시나리오용 컨텍스트 풀
  {QuickBEAM.ContextPool, name: MyApp.JSPool, size: 4},
]

Supervisor.start_link(children, strategy: :one_for_one)

{:ok, html} = QuickBEAM.call(:renderer, &quot;render&quot;, [%{page: &quot;home&quot;}])</code></pre>
<p><code>:script</code> 옵션을 사용하면 시작 시 JavaScript 파일을 로드합니다. 런타임이 크래시되면 감독자가 이를 감지하고 새로운 컨텍스트로 재시작하며, 스크립트를 다시 평가합니다.</p>
<p>개별 Context 프로세스는 보통 동적으로 시작됩니다. 예를 들어 LiveView mount에서 연결 프로세스에 링크해서요.</p>
<h3>컨텍스트 풀 (Context Pool)</h3>
<p>수천 개의 동시 연결처럼 정말 높은 동시성이 필요한 상황에서는 개별 런타임보다 ContextPool을 사용하는 게 낫습니다. 많은 가벼운 JS 컨텍스트가 적은 수의 런타임 스레드를 공유하니까요.</p>
<pre><code class="language-elixir"># N개 런타임 스레드를 가진 풀 시작 (기본값: 스케줄러 개수)
{:ok, pool} = QuickBEAM.ContextPool.start_link(name: MyApp.JSPool, size: 4)

# 각 컨텍스트는 자신만의 JS 글로벌 스코프를 가진 GenServer
{:ok, ctx} = QuickBEAM.Context.start_link(pool: MyApp.JSPool)
{:ok, 3} = QuickBEAM.Context.eval(ctx, &quot;1 + 2&quot;)
{:ok, &quot;HELLO&quot;} = QuickBEAM.Context.eval(ctx, &quot;'hello'.toUpperCase()&quot;)

QuickBEAM.Context.stop(ctx)</code></pre>
<p>컨텍스트는 전체 API를 지원합니다. eval, call, Beam.call/callSync, DOM, 메시징, 브라우저/Node API, 핸들러, 감독 트리 모두요.</p>
<pre><code class="language-elixir"># Phoenix LiveView에서 사용 예
def mount(_params, _session, socket) do
  {:ok, ctx} = QuickBEAM.Context.start_link(
    pool: MyApp.JSPool,
    handlers: %{&quot;db.query&quot; =&gt; &amp;MyApp.query/1}
  )
  {::ok, assign(socket, js: ctx)}
end</code></pre>
