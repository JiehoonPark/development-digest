13:T1b1b,<h2>Next.js 16.2 출시 — 성능 개선과 개발 경험 업그레이드</h2>
<p>Next.js 16.2가 릴리스되었습니다. 이번 버전은 성능 개선, 더 나은 디버깅 환경, AI 에이전트 관련 개선사항, 그리고 200개 이상의 Turbopack 버그 수정을 담고 있습니다.</p>
<h3>주요 개선사항 한눈에</h3>
<ul>
<li><strong>개발 서버 시작 시간</strong>: 약 400% 단축</li>
<li><strong>렌더링 속도</strong>: 약 50% 개선</li>
<li><strong>기본 에러 페이지</strong>: 새롭게 디자인된 500 페이지</li>
<li><strong>서버 함수 로깅</strong>: 개발 터미널에서 실행 과정 확인 가능</li>
<li><strong>Hydration 차이 표시</strong>: 에러 오버레이에서 서버/클라이언트 차이를 명확히 표시</li>
<li><strong>프로덕션 디버깅</strong>: <code>next start --inspect</code>로 Node.js 디버거 연결 가능</li>
</ul>
<p>이번 릴리스의 핵심 내용은 두 가지 별도의 심화 글에서 다룹니다.</p>
<p><strong>Turbopack</strong>: 빌드 속도 개선, SRI 지원, postcss.config.ts, 트리 셰이킹 개선, Server Fast Refresh, 200개 이상의 버그 수정</p>
<p><strong>AI 개선사항</strong>: create-next-app에 AGENTS.md 추가, 브라우저 로그 포워딩, next-browser(실험 단계)</p>
<h3>지금 업그레이드하기</h3>
<p>자동 업그레이드 CLI를 사용하거나, 수동으로 진행하거나, 새 프로젝트를 시작할 수 있습니다.</p>
<pre><code class="language-terminal"># 자동 업그레이드
npx @next/codemod@canary upgrade latest

# 수동 업그레이드
npm install next@latest react@latest react-dom@latest

# 새 프로젝트 생성
npx create-next-app@latest</code></pre>
<hr>
<h3>개발 환경에서의 빠른 시작 속도</h3>
<p><code>next dev</code>를 실행했을 때 localhost:3000이 준비되는 시간을 대폭 단축했습니다. 같은 환경과 프로젝트 기준으로, 기본 앱 설정 시 Next.js 16.1 대비 약 87% 더 빠릅니다.</p>
<h3>더 빠른 렌더링</h3>
<p>React에 기여한 변경사항이 이번 버전에 반영되었는데요. Server Components의 payload 역직렬화가 최대 350% 빨라졌습니다.</p>
<p>기존 방식은 JSON.parse의 reviver 콜백을 사용했는데, 이렇게 하면 V8의 C++/JavaScript 경계를 JSON의 각 키-값 쌍마다 넘나들어야 합니다. 아무것도 하지 않는 reviver라도 JSON.parse 속도를 약 4배 느리게 만들 정도입니다.</p>
<p>새로운 접근법은 두 단계로 진행됩니다. 먼저 일반 JSON.parse()를 실행한 뒤, 순수 JavaScript로 재귀적 순회를 합니다. 이렇게 하면 경계 넘나드는 오버헤드를 없애면서, 변환이 필요 없는 단순한 문자열은 건너뛰는 최적화까지 추가할 수 있죠.</p>
<p>실제 Next.js 애플리케이션에서는 RSC payload 크기에 따라 HTML 렌더링이 25%에서 60% 빨라집니다.</p>

<h3>새롭게 디자인된 기본 에러 페이지</h3>
<p>프로덕션 환경에서 에러가 발생했을 때, 별도의 <code>global-error.tsx</code>나 <code>error.tsx</code>를 정의하지 않았다면 Next.js가 기본 폴백 페이지를 렌더링합니다. 이번 버전에서 이 페이지가 더 깔끔하고 현대적인 디자인으로 변경되었습니다.</p>
<h3>서버 함수 실행 로깅</h3>
<p>개발 중에 서버 함수가 실행될 때 터미널에 로그가 출력됩니다. 각 로그에는 함수명, 전달된 인자, 실행 시간, 정의된 파일 위치가 표시됩니다. 성능 문제를 빠르게 파악하거나 병목 지점을 찾을 때 매우 유용합니다.</p>
<h3>Hydration 차이를 명확하게 표시</h3>
<p>hydration 불일치가 발생하면 에러 오버레이가 어느 부분이 서버에서 왔고 어느 부분이 클라이언트에서 생겼는지 명확하게 표시합니다. <code>+ Client / - Server</code> 범례를 사용해서 한눈에 어디가 다른지 알 수 있습니다.</p>
<h3>프로덕션 환경에서 디버깅: --inspect for next start</h3>
<p>Next.js 16.1에서는 개발 중에 <code>next dev --inspect</code>로 Node.js 디버거를 연결할 수 있었습니다. 16.2에서는 이 기능이 <code>next start</code>로도 확장되어, 프로덕션 서버에 Node.js 디버거를 붙일 수 있게 되었습니다.</p>
<pre><code class="language-terminal">next start --inspect</code></pre>
<p>CPU나 메모리 사용량을 프로파일링하거나 프로덕션 환경의 문제를 디버깅할 때 유용합니다. Chrome DevTools를 활용한 디버깅 방법은 공식 문서를 참고하세요.</p>
<h3>next/link의 transitionTypes Prop</h3>
<p><code>&lt;Link&gt;</code> 컴포넌트가 이제 <code>transitionTypes</code> prop을 받을 수 있습니다. 이것은 네비게이션 시 적용할 View Transitions 타입을 지정하는 문자열 배열입니다. 각 타입은 네비게이션 Transition 중에 React.addTransitionType으로 전달되므로, 네비게이션 방향이나 맥락에 따라 다른 애니메이션을 트리거할 수 있습니다.</p>
<pre><code class="language-javascript">&lt;Link href=&quot;/about&quot; transitionTypes={['slide']}&gt;
  About
&lt;/Link&gt;</code></pre>
<p>이 기능은 App Router에서만 지원됩니다. Pages Router는 네비게이션에 React Transitions를 사용하지 않기 때문이죠. Pages Router의 <code>transitionTypes</code>는 조용히 무시되므로, 두 라우터를 모두 사용하는 공유 Link 컴포넌트에서도 문제없이 작동합니다.</p>
<p>View Transitions에 대해 더 알고 싶다면 공식 문서를 확인하세요.</p>
