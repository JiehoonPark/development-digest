16:T1c7d,<h2>노트북에서 397B 파라미터 모델 실행하기: Flash-MoE</h2>
<p>MacBook Pro 48GB RAM에서 Qwen3.5-397B-A17B(397억 개의 파라미터를 가진 Mixture-of-Experts 모델)를 초당 4.4개 이상의 토큰으로 추론할 수 있다면 어떨까요? Flash-MoE는 이를 실현했습니다. 도구 호출을 포함한 프로덕션 수준의 출력도 지원합니다.</p>
<p>C/Metal 기반의 순수 추론 엔진이거든요. Python도, 프레임워크도 없습니다. C, Objective-C, 그리고 손으로 최적화한 Metal 셰이더만 있으면 됩니다. 209GB 크기의 전체 모델이 SSD에서 커스텀 Metal 파이프라인을 통해 스트리밍되고 있습니다.</p>
<h3>성능 결과</h3>
<div class="table-wrapper"><table><thead><tr><th>구성</th><th>tok/s</th><th>품질</th><th>비고</th></tr></thead><tbody><tr><td>4-bit 전문가, FMA 커널</td><td>4.36</td><td>우수</td><td>현재 최고 성능. 전체 도구 호출 지원. 디스크 209GB</td></tr><tr><td>4-bit 전문가, 기본 커널</td><td>3.90</td><td>우수</td><td>FMA 커널 최적화 적용 전</td></tr><tr><td>2-bit 전문가, OS 신뢰</td><td>5.74</td><td>좋음*</td><td>디스크 120GB. *JSON/도구 호출 불안정</td></tr><tr><td>2-bit 피크 싱글 토큰</td><td>7.05</td><td>좋음*</td><td>웜 캐시 버스트. *도구 사용에 부적합</td></tr></tbody></table></div>
<p>2-bit 양자화는 JSON 출력에서 <code>\name\</code> 대신 <code>&quot;name&quot;</code>을 생성하지 못해 도구 호출이 불안정합니다. 프로덕션 환경에서는 4-bit 구성을 사용하세요.</p>
<h3>하드웨어 사양</h3>
<ul>
<li><strong>기기</strong>: MacBook Pro, Apple M3 Max</li>
<li><strong>칩</strong>: 16코어 CPU(12P + 4E), 40코어 GPU, 16코어 ANE</li>
<li><strong>메모리</strong>: 48GB 통합 메모리(대역폭 약 400GB/s)</li>
<li><strong>SSD</strong>: 1TB Apple Fabric, 순차 읽기 17.5GB/s(실측)</li>
<li><strong>OS</strong>: macOS 26.2(Darwin 25.2.0)</li>
</ul>
<h3>모델 구조</h3>
<p>60개의 트랜스포머 레이어로 구성되어 있습니다. 45개의 GatedDeltaNet(선형 어텐션) 레이어와 15개의 표준 풀 어텐션 레이어가 섞여 있죠. 각 레이어는 512개의 전문가를 가지고 있는데, 토큰당 K=4개만 활성화됩니다(공유 전문가 1개 추가). 숨겨진 차원은 4096입니다.</p>
<h3>핵심 기술들</h3>
<p><strong>SSD 전문가 스트리밍</strong></p>
<p>전문가 가중치(4-bit 기준 209GB)를 SSD에서 필요에 따라 parallel pread()와 GCD dispatch groups로 읽어 옵니다. 레이어당 활성 전문가 K=4개(각각 약 6.75MB)만 로드하면 되죠. OS 페이지 캐시가 캐싱을 자동으로 관리하므로 커스텀 캐시는 필요 없습니다. 이것이 바로 &quot;OS를 믿자&quot;는 원칙인데요. Apple의 &quot;LLM in a Flash&quot; 논문에서 영감을 얻었습니다.</p>
<p><strong>FMA 최적화 양자화 해제 커널</strong></p>
<p>4-bit 양자화 해제 행렬-벡터 곱셈의 내부 루프를 다시 정렬합니다. 기존 방식인 <code>(nibble <em> scale + bias) </em> x</code>를 <code>fma(nibble, scale<em>x, bias</em>x)</code>로 바꾸는 거죠. <code>scale<em>x</code>와 <code>bias</em>x</code>를 미리 계산해두면 GPU의 fused multiply-add 유닛이 양자화 해제와 곱셈을 한 번에 처리할 수 있습니다. 순진한 방식보다 12% 빠릅니다.</p>
<p><strong>Metal 컴퓨트 셰이더</strong></p>
<p>다음 작업들을 위해 직접 작성한 Metal 커널을 사용합니다:</p>
<ul>
<li>4-bit 및 2-bit 양자화 해제 행렬-벡터 곱셈(타일, SIMD 감소, 공유 입력 캐시, FMA 최적화)</li>
<li>Fused SwiGLU 활성화</li>
<li>RMS 정규화(2단계: 제곱합 감소 + 적용)</li>
<li>배치 GPU 어텐션(Q@K^T, softmax, scores@V) - 풀 어텐션 레이어용</li>
<li>GPU RoPE(Q deinterleave 및 K 정규화와 함께 fused)</li>
<li>MoE combine + residual + sigmoid gate(fused 커널)</li>
</ul>
<p><strong>지연된 GPU 전문가 계산</strong></p>
<p>CMD3(전문가 포워드 패스)를 기다리지 않고 제출합니다. GPU가 이를 실행하는 동안 CPU는 다음 레이어를 준비하고 있거든요. combine + residual + norm도 GPU에서 처리되어 다음 레이어의 어텐션 프로젝션으로 직접 공급됩니다.</p>
<p><strong>선형 어텐션을 위한 Accelerate BLAS</strong></p>
<p>GatedDeltaNet recurrence는 64-head × 128×128 상태 행렬 업데이트를 위해 <code>cblas_sscal</code>, <code>cblas_sgemv</code>, <code>cblas_sger</code>을 사용합니다. 스칼라 코드보다 64% 빠릅니다.</p>
<p><strong>OS를 믿자</strong></p>
<p>커스텀 전문가 캐시가 필요 없습니다. OS 페이지 캐시(약 35GB)가 표준 LRU를 통해 전문가 데이터 캐싱을 자동으로 관리합니다. 우리가 테스트한 모든 커스텀 캐싱 방식(Metal LRU, malloc 캐시, LZ4 압축 캐시)은 GPU 메모리 압박이나 오버헤드 때문에 더 느렸거든요. 페이지 캐시는 자연스럽게 약 71%의 히트율을 달성합니다.</p>
<h3>레이어당 파이프라인(4-bit 기준 평균 4.28ms)</h3>
<pre><code>CMD3(prev) → CMD1: attention projections + delta-net [1.22ms GPU]
→ CPU: flush results [0.01ms CPU]
→ CMD2: o_proj + norm + routing + shared [0.55ms GPU]
→ CPU: softmax + topK routing [0.003ms]
→ I/O: parallel pread K=4 experts [2.41ms SSD]
→ CMD3: expert forward + combine + norm [0.04ms encode, DEFERRED]</code></pre>
<h3>통합 메모리 제약</h3>
<p>Apple Silicon에서는 SSD DMA와 GPU 계산이 같은 메모리 컨트롤러를 공유하며 서로 겹칠 수 없습니다. GPU의 양자화 해제 커널은 약 418GiB/s에서 대역폭 포화 상태입니다. 백그라운드 SSD DMA가 조금이라도 생기면 메모리 컨트롤러 중재를 통해 GPU 지연이 불균형적으로 증가합니다. 따라서 GPU → SSD → GPU 순차 파이프라인이 하드웨어상 최적입니다.</p>
