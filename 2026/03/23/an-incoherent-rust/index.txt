13:T19cf,<h2>Rust 생태계의 근본적인 문제: 일관성(Coherence) 규칙</h2>
<p>이 글은 어떤 LLM의 도움도 받지 않고 작성되었습니다.</p>
<h3>생태계 발전의 정체</h3>
<p>Rust 생태계는 성장 방식에 근본적인 문제를 안고 있습니다.</p>
<p><code>serde</code> 같은 기초 라이브러리들이 <code>Serialize</code> 같은 핵심 트레이트를 정의하면, 생태계의 모든 크레이트가 자신의 타입에 대해 이 트레이트를 구현해야 합니다. 만약 어떤 크레이트가 자신의 타입에 대해 <code>serde</code>의 트레이트를 구현하지 않으면, 그 타입은 <code>serde</code>와 호환되지 않거든요. 다운스트림 크레이트가 다른 크레이트의 타입에 대해 <code>serde</code> 트레이트를 구현할 수 없기 때문입니다.</p>
<p>더 큰 문제는 <code>serde</code>의 대안(예를 들어 <code>nextserde</code>)이 나타났을 때 발생합니다. <code>serde</code> 지원을 추가한 모든 크레이트가 이제 <code>nextserde</code>도 지원해야 합니다. 모든 새로운 직렬화 라이브러리에 대응하는 건 크레이트 작성자에게 현실적이지 않은 작업이 되어버리죠.</p>
<p>크레이트를 사용하는 입장에서 새로운 직렬화 라이브러리를 쓰려면, 해당 크레이트들을 모두 포크하고 <code>nextserde</code> 지원을 직접 패치해야 합니다. 이렇게 되니 <code>serde</code> 같은 기초 라이브러리의 대안이 나오기 어렵고, 생태계에 퍼지는 것은 더욱 힘들어집니다.</p>
<p>결국 기존의 &quot;먼저 나온&quot; 크레이트들이 더 나은 대안이 있어도 생태계에 남아있을 수밖에 없습니다. 인위적으로 대체하기 어렵게 만들어지니까요.</p>
<p>이건 라이브러리나 개발자의 잘못이 아닙니다. 언어 자체가 일관성(coherence) 규칙과 고아 규칙(orphan rules)을 통해 생태계에 강제하는 문제거든요.</p>
<p>참고: Niko Matsakis의 글 [&quot;Coherence and crate-level where clauses&quot;](https://nikomatsakis.medium.com/coherence-and-crate-level-where-clauses-1a5b2b3cd4a7)에서 일관성이 Rust 생태계를 어떻게 해치는지 자세히 설명합니다.</p>
<h3>일관성과 고아 규칙</h3>
<p>일관성(Coherence) 검사는 같은 타입과 제네릭 인자에 대해 트레이트 구현이 최대 한 번만 나타나도록 보장합니다.</p>
<pre><code class="language-rust">trait Trait {}
trait Thingies {}
trait OtherThingies {}
impl&lt;T: Thingies&gt; Trait for T {}
impl&lt;T: OtherThingies&gt; Trait for T {}
error[E0119]: conflicting implementations of trait `Trait`
--&gt; src/lib.rs:7:1
|
6 | impl&lt;T: Thingies&gt; Trait for T {}
| ----------------------------- first implementation here
7 | impl&lt;T: OtherThingies&gt; Trait for T {}
| ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ conflicting implementation
For more information about this error, try `rustc --explain E0119`.
error: could not compile `playground` (lib) due to 1 previous error</code></pre>
<p>고아 규칙(Orphan rules)은 일관성을 구현하기 위한 검사입니다. 현재 크레이트에서 정의한 트레이트 또는 타입 중 하나라도 포함되어야만 트레이트를 구현할 수 있다는 규칙이에요. (실제로는 더 복잡하지만, 이 글의 범위 내에서는 충분합니다.)</p>
<pre><code class="language-rust">// crate a
pub trait Trait {}
pub struct Foo;
// crate b
use a::*;
impl Trait for Foo {}
error[E0117]: only traits defined in the current crate can be implemented for types defined outside of the crate
--&gt; src/lib.rs:8:1
|
8 | impl Trait for Foo {}
| ^^^^^^^^^^^^^^^---
| |
| `a::Foo` is not defined in the current crate
|
= note: impl doesn't have any local type before any uncovered type parameters
= note: for more information see https://doc.rust-lang.org/reference/items/implementations.html#orphan-rules
= note: define and implement a trait or new type instead</code></pre>
<p>겹치는 구현이 없어도 고아 규칙 때문에 이 코드는 거절됩니다.</p>
<p>자세한 내용은 [Rust Reference의 &quot;Trait implementation coherence&quot;](https://doc.rust-lang.org/reference/items/implementations.html#trait-implementation-coherence)를 보세요.</p>
<h3>왜 일관성이 필요한가</h3>
<h4>HashMap 문제</h4>
<pre><code class="language-rust">// crate a
#[derive(PartialEq, Eq)]
pub struct MyData(u8);
// crate b
impl Hash for MyData {
fn hash(&amp;self) {
self.0.hash();
}
}
pub fn make_hashset() -&gt; HashSet&lt;MyData&gt; {
// Uses the `Hash` impl defined in this crate to insert
[MyData(1), MyData(12)].into()
}
// crate c
impl Hash for MyData {
fn hash(&amp;self) {
// You probably don't want this to be your hash function...
0.hash();
}
}
pub fn check_hashset(set: HashSet&lt;MyData&gt;) {
// Uses the `Hash` impl defined in this crate to lookup
assert!(set.contains(MyData(1)));
assert!(set.contains(MyData(12)))
}
// crate d
c::check_hashset(b::make_hashset());</code></pre>
<p>이 예제에서 crate b로 만든 <code>HashSet</code>을 crate c의 함수에 넘깁니다. 그런데 crate b에서 사용한 <code>Hash</code> 구현과 crate c에서 사용한 <code>Hash</code> 구현이 다릅니다.</p>
<p><code>Hash</code> 구현이 다르니 <code>check_hashset</code>은 완전히 잘못된 결과를 낼 겁니다. 실제로 포함되어 있는 값들을 찾을 수 없게 되죠.</p>
<p>참고: Niko Matsakis의 글 [&quot;Coherence and crate-level where clauses&quot;](https://nikomatsakis.medium.com/coherence-and-crate-level-where-clauses-1a5b2b3cd4a7)에서 &quot;So wait, how does the orphan rule protect composition&quot; 섹션을 보세요.</p>
<h4>건전성(Soundness)</h4>
<p>현재 일관성은 타입 시스템의 건전성을 유지하기 위해 정말 중요합니다.</p>
<pre><code class="language-rust">trait Trait {
type Assoc;
}
// crate a
impl Trait for () {
type Assoc = *const u8;
}
pub fn make_assoc() -&gt; &lt;() as Trait&gt;::Assoc {
// `&lt;() as Trait&gt;::Assoc` is implemented as being `*const u8`
0x0 as *const u8
}
// crate b
impl Trait for () {
type Assoc = Box&lt;u8&gt;;
}
fn drop_assoc(a: &lt;() as Trait&gt;::Assoc) {
// `&lt;() as Trait&gt;::Assoc` is implemented as being `Box&lt;u8&gt;`
let a: Box&lt;u8&gt; = a;
// free'ing an allocation here
drop(a);
}</code></pre>
