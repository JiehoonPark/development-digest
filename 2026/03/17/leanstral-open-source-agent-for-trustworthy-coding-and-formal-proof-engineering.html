Leanstral: 신뢰할 수 있는 코드 및 형식 증명 엔지니어링을 위한 오픈소스 에이전트

Leanstral: Open-source agent for trustworthy coding and formal proof engineering

📅 2026-03-17 📰 Hacker News (Top) ⬆ 467

요약

Mistral은 Lean 4 증명 보조기를 위해 설계된 최초의 오픈소스 코드 에이전트 Leanstral을 발표했다. 6B 활성 매개변수로 구성된 고효율 희소 아키텍처(sparse architecture)를 사용하며, 형식 증명 작업에 특화되어 있다. Leanstral은 Apache 2.0 라이선스로 공개되었으며, pass@2에서 점수 26.3을 달성하여 Claude Sonnet을 2.6점 초과하면서 비용은 $549 대비 $36만 소요된다. FLTEval이라는 새로운 평가 스위트를 통해 실제 형식 저장소 작업에 기반한 평가를 제공한다.

핵심 포인트

  • 형식 증명 자동화의 필요성: AI 코드 생성 모델이 고도의 수학 연구와 미션 크리티컬 소프트웨어 같은 고위험 분야로 확대되면서, 인간의 수동 검증이 주요 병목이 되었다. Leanstral은 기계 생성 로직 디버깅 대신 인간이 원하는 것을 명확히 명시하는 방식으로 해결하려 한다.
  • 고효율 희소 아키텍처: 6B 활성 매개변수(6 billion active parameters)로 구성되어 매우 효율적이며, Lean 4 증명 보조기와의 완벽한 검증을 통해 병렬 추론이 가능하여 비용 효율적이다. 이는 훨씬 큰 경쟁 모델들보다 우수한 성능을 제공한다.
  • 오픈소스 공개 및 접근성: Apache 2.0 라이선스로 모델 가중치를 공개하고 Mistral Vibe 환경에서 에이전트 모드로 제공되며, 무료 API 엔드포인트도 제공된다. 기술 보고서와 새로운 평가 스위트 FLTEval도 함께 공개할 예정이다.
  • 오픈소스 모델 대비 성능: Leanstral-120B-A6B는 GLM5-744B-A40B(점수 16.6)와 Kimi-K2.5-1T-A32B(점수 20.1)를 상회하며, Qwen3.5-397B-A17B가 4회 시도로 25.4를 달성할 때 Leanstral은 2회 시도(pass@2)로 26.3을 달성하고 선형적으로 확장하여 pass@16에서 29.3을 기록한다.
  • Claude 제품군 대비 비용 효율성: Leanstral pass@2(점수 26.3)는 Claude Sonnet(점수 23.7, $549)을 2.6점 초과하면서 비용은 $36만 소요되며, pass@16(점수 31.9)은 Claude Opus(점수 39.6, $1,650)에 비해 8점 초과하면서 비용은 92배 저렴하다.
  • MCP 지원과 최적화: Mistral Vibe를 통해 임의의 Model Context Protocol(MCP)을 지원하며, 특히 자주 사용되는 lean-lsp-mcp를 통해 최대 성능을 달성하도록 특화 훈련되었다.
  • 실제 사례: StackExchange 질문에 대응하는 Lean 4.29.0-rc6 호환성 문제를 진단할 때, Leanstral은 단순 타입 별칭의 def 정의가 rw 태틱의 패턴 매칭을 막는 근본 원인을 파악하고 abbrev로 변경하는 해결책을 제시했다. 이는 단순한 시행착오가 아닌 정의상 동등성(definitional equality)의 이해에 기반한다.
  • 형식 저장소 기반 평가: FLTEval은 고립된 수학 문제 대신 FLT(Fermat's Last Theorem) 프로젝트로의 각 PR에서 형식 증명 완성과 새로운 수학 개념 정의를 평가하여 실제 증명 엔지니어링 시나리오에서의 유용성을 반영한다.

왜 중요한가

Leanstral은 형식 증명 작업의 인간 검증 부담을 획기적으로 줄여 고위험 분야의 코드와 증명 개발 속도를 높일 수 있으며, 오픈소스로 공개되어 접근 가능한 고효율 도구를 제공함으로써 형식 검증 기술의 민주화를 진전시킨다. 클로드 같은 고가의 모폐쇄 모델 대비 저렴한 비용의 경쟁력 있는 대안을 제시한다.

📄 전문 번역

AI 코딩 에이전트의 한계를 넘다: Leanstral 공개

AI 모델이 코드 생성에 뛰어난 능력을 보여주고 있습니다. 하지만 수학 연구나 미션 크리티컬 소프트웨어처럼 높은 신뢰도가 필요한 영역으로 나아갈수록 큰 병목이 생기는데요. 바로 인간의 검증입니다. 기계가 생성한 코드를 일일이 검토하고 검증하는 데 드는 시간과 전문 지식이 개발 속도를 가로막고 있거든요.

우리가 꿈꾸는 건 다릅니다. 코딩 에이전트가 단순히 작업을 수행하는 것을 넘어, 자신의 구현이 엄격한 스펙을 만족하는지 형식적으로 증명할 수 있는 세상이죠. 사람이 일일이 디버깅하지 않아도, 원하는 것을 명확히 명시하면 나머지는 에이전트가 처리하는 방식입니다. 오늘 우리는 그 비전을 향한 첫 발을 내딛습니다.

Leanstral 소개

이제 Lean 4를 위해 설계된 최초의 오픈소스 코딩 에이전트인 Leanstral을 공개합니다.

Lean 4는 증명 보조 시스템(proof assistant)으로, 완벽한 공간(perfectoid spaces) 같은 복잡한 수학 객체부터 Rust 코드의 안전성 같은 소프트웨어 스펙까지 표현할 수 있습니다. 기존의 대형 범용 모델을 그냥 감싸거나 수학 문제 하나에만 집중하는 시스템과는 달리, Leanstral은 6B의 활성 파라미터만으로 매우 효율적이며 실제 형식 검증 저장소에서 동작하도록 훈련되었습니다.

주요 특징

열려있고 접근 가능합니다

Apache 2.0 라이선스로 Leanstral 가중치를 공개합니다. Mistral vibe 에이전트 모드와 무료 API 엔드포인트로도 접근할 수 있고요. 훈련 방식을 다룬 기술 보고서와 함께, 수학 경시대회 문제를 넘어선 평가를 위한 새로운 벤치마크 FLTEval도 함께 릴리스합니다.

효율적이면서도 강력합니다

Leanstral은 극도로 희소한 아키텍처(sparse architecture)를 사용하며, 증명 공학 작업에 최적화되어 있습니다. Lean을 완벽한 검증자로 활용한 병렬 추론으로, 기존의 폐쇄형 경쟁 모델보다 뛰어난 성능을 훨씬 저렴하게 제공합니다.

MCP로 확장 가능합니다

Leanstral은 vibe를 통해 임의의 MCP를 지원하며, 자주 사용되는 lean-lsp-mcp와 최고 성능을 내도록 특별히 훈련되었습니다.

성능 평가

고립된 수학 문제가 아닌, 실제 증명 공학 시나리오의 유용성을 반영하기 위해 우리는 특별한 접근을 취했습니다. FLT 프로젝트로의 각 PR에서 형식 증명을 완성하고 새로운 수학 개념을 올바르게 정의하는 능력으로 Leanstral을 벤치마크했습니다.

주요 코딩 에이전트(Claude Opus 4.6, Sonnet 4.6, Haiku 4.5)와 오픈소스 모델들(Qwen3.5 397B-A17B, Kimi-K2.5 1T-A32B, GLM5 744B-A40B)을 비교했는데요.

오픈소스 모델과의 비교

Leanstral-120B-A6B는 훨씬 큰 오픈소스 경쟁자들 대비 눈에 띄는 효율성 우위를 보여줍니다. GLM5-744B나 Kimi-K2.5 같은 모델들은 아무리 크기가 크더라도 FLTEval 점수가 각각 16.6과 20.1에 정체되는데요. Leanstral은 단 한 번의 추론만으로 둘 다 앞서갑니다.

가장 강한 오픈소스 경쟁자인 Qwen3.5-397B-A17B도 25.4점에 도달하려면 4번의 추론(pass@4)이 필요합니다. 반면 Leanstral은 절반의 비용으로 26.3점을 달성해서 이미 능가하며, 계속해서 선형으로 성능이 올라가 같은 비용대에서 29.3점에 도달합니다.

Claude 제품군과의 비교

Leanstral은 Claude 시리즈의 뛰어난 비용 대체재가 됩니다.

  • Leanstral (pass@2): 26.3점, 비용 $36 vs Sonnet (23.7점, $549)
  • 성능은 2.6점 우수하면서 비용은 15분의 1 수준
  • Leanstral (pass@16): 31.9점, 비용 $290 vs Sonnet (23.7점, $549)
  • 성능은 8점 우수하면서 비용은 절반 이하

Claude Opus 4.6이 품질 면에서는 최고지만(39.6점), 가격은 $1,650으로 Leanstral의 92배입니다.

실제 활용 사례

Stack Exchange 질문 해결: 새 Lean 버전에서의 호환성 문제

새로운 Lean 버전이 나올 때마다 기존 코드를 마이그레이션하는 건 정말 골치 아픈 일입니다. 우리는 Proof Assistants Stack Exchange의 실제 질문을 Leanstral에 던졌는데요. 어떤 스크립트가 Lean 4.29.0-rc6에서 갑자기 컴파일되지 않는다는 문제였습니다. (당연히 이 버전은 훈련 데이터에 없습니다.)

원인은 rw (rewrite) 전술이 갑자기 패턴 매칭을 못 하기 시작한 것이었는데요. 특히 단순한 타입 앨리어스인 def T2 := List Bool로 정의된 타입을 매칭하지 못했거든요.

Leanstral은 무작정 추측하지 않았습니다. 테스트 코드를 작성해서 실패하는 환경을 재현했고, 정의적 동치성(definitional equality)을 철저히 진단했어요. 그리고 정확히 지적했습니다: def는 명시적 전개를 요구하는 엄격한 정의를 만드는데, 그래서 rw 전술이 매칭해야 할 기초 구조를 못 본다는 거죠.

해결책은 간단했습니다. def를 abbrev로 바꾸는 거였어요. abbrev는 즉시 정의적으로 동치인 투명한 앨리어스를 만들어서, rw 전술이 다시금 (L2 n).length 패턴을 완벽하게 매칭할 수 있게 해줍니다. Leanstral은 문제를 완전히 해결하고 사용자에게 그 이유까지 명확하게 설명해주었습니다.
