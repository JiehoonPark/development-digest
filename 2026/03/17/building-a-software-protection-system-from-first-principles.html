소프트웨어 보호 시스템을 처음부터 설계하기

building a software protection system from first principles

📅 2026-03-17 📰 Lobste.rs

요약

소프트웨어 라이선싱 시스템을 단계적으로 구축하며 각 접근 방식의 취약점을 분석합니다. 하드코딩된 시리얼에서 시작하여 난독화, 디지털 서명 등으로 진화하며, 위협 모델에 맞는 '충분한 수준'의 보호를 찾는 것의 중요성을 강조합니다.

핵심 포인트

  • 단순 문자열: strings 명령어로 즉시 노출, 난독화: 디컴파일로 여전히 취약
  • 비대칭 암호화(디지털 서명)을 통해 공개키로 검증 가능하면서도 위조 불가능한 라이선스 생성 가능
  • 공격 대상(캐주얼 사용자, 취미 역공학, 조직적 크래킹)과 수용 가능한 마찰도(사용자 불편) 정의하여 '충분한' 보호 수준 결정

왜 중요한가

완벽한 보호는 불가능하므로 비즈니스 요구사항에 맞는 현실적인 보호 수준을 단계적으로 구축하는 방법론을 제시합니다.

📄 전문 번역

처음부터 만드는 소프트웨어 보호 시스템

소프트웨어 보호 시스템을 기초부터 차근차근 만들어보겠습니다. 미리 완성된 형태를 설명하지 않고, 각 단계를 만들고 부수고를 반복하면서 실제로 쓸 수 있는 해결책에 도달하게 될 거예요.

라이선싱 시스템을 만들기 전에

먼저 당신의 제품에 대해 "충분히 좋은 수준"이 무엇인지 정의해야 합니다. 누로부터 보호하려고 하나요? 일반인의 우발적 공유, 취미로 역공학을 시도하는 사람들, 전문 크래킹 그룹, 아니면 모두 다요? 사용자 경험을 얼마나 해칠 수 있고, 운영 복잡도는 어디까지 감당할 수 있나요?

이 글을 체크리스트가 아닌 성장의 사다리로 생각해보세요. 각 단계에서 당신의 위협 모델에 맞으면 멈춰도 되고, 더 강한 보호가 필요하면 계속 올라가면 됩니다.

0단계: 간단한 애플리케이션부터 시작

먼저 유용한 기능을 하는 간단한 프로그램을 만들어봅시다. 우주의 의미를 출력하는 프로그램이죠.

#include <stdio.h>
#define ANSWER 42
int app_main(void) {
    printf("the meaning of life is %d\n", ANSWER);
    return 0;
}
int main(void) {
    return app_main();
}
$ cc -o app iter0.c
$ ./app
the meaning of life is 42

여기서 ANSWER는 우리가 보호해야 할 비밀입니다. 실제 애플리케이션이라면 독점 알고리즘이거나 프리미엄 콘텐츠의 복호화 키일 수 있겠네요.

우리가 풀어야 할 문제는 이거예요: 유료 고객만 이 프로그램을 실행하도록 하려면 어떻게 해야 할까?

1단계: 하드코딩된 시리얼 번호

가장 단순한 접근법은 유용한 기능을 실행하기 전에 비밀번호를 요청하는 것입니다.

#include <stdio.h>
#include <string.h>
#define ANSWER 42
#define SERIAL "XXXX-YYYY-ZZZZ"

int app_main(void) {
    printf("the meaning of life is %d\n", ANSWER);
    return 0;
}

int check_license(const char *input) {
    return strcmp(input, SERIAL) == 0;
}

int main(int argc, char *argv[]) {
    if (argc != 2) {
        fprintf(stderr, "usage: %s <serial>\n", argv[0]);
        return 1;
    }
    if (!check_license(argv[1])) {
        fprintf(stderr, "invalid license\n");
        return 1;
    }
    return app_main();
}
$ cc -o app iter1.c
$ ./app wrong
invalid license
$ ./app XXXX-YYYY-ZZZZ
the meaning of life is 42

잘 작동합니다! 이대로 배포할 수도 있지만, 얼마나 견고한지 살펴봅시다.

이것을 깨는 방법

하드코딩된 문자열의 문제는 바이너리에 노출되어 있다는 거예요.

$ strings app | grep -E '^[A-Z]{4}-[A-Z]{4}-[A-Z]{4}$'
XXXX-YYYY-ZZZZ

0.5초 만에 공격자가 유효한 시리얼을 얻었습니다.

2단계: 난독화된 시리얼 번호

평문 문자열이 너무 쉽게 발견되니까, 시리얼을 숨겨봅시다. XOR, 해싱 같은 난독화 기법을 쓸 수 있어요.

#include <stdio.h>
#include <string.h>
#include <stddef.h>
#define ANSWER 42

int app_main(void) {
    printf("the meaning of life is %d\n", ANSWER);
    return 0;
}

// "XXXX-YYYY-ZZZZ"를 0x42로 XOR한 결과
static const unsigned char obfuscated_serial[] = {
    0x1a, 0x1a, 0x1a, 0x1a, 0x6f, 0x1b, 0x1b, 0x1b,
    0x1b, 0x6f, 0x18, 0x18, 0x18, 0x18, 0x00
};

int check_license(const char *input) {
    size_t len = strlen(input);
    if (len != sizeof(obfuscated_serial) - 1) return 0;
    for (size_t i = 0; i < len; i++) {
        if ((input[i] ^ 0x42) != obfuscated_serial[i]) {
            return 0;
        }
    }
    return 1;
}

int main(int argc, char *argv[]) {
    if (argc != 2) {
        fprintf(stderr, "usage: %s <serial>\n", argv[0]);
        return 1;
    }
    if (!check_license(argv[1])) {
        fprintf(stderr, "invalid license\n");
        return 1;
    }
    return app_main();
}

이제 strings 명령어로는 시리얼이 보이지 않습니다.

$ cc -o app iter2.c
$ strings app | grep -E '^[A-Z]{4}-'
$

그런데 여기가 핵심인데요. 난독화를 푸는 로직이 바이너리에 남아있다는 거예요. 입력을 검증하기 위해 0x42로 XOR하고 있으니까, 공격자는 그 코드를 찾고 알고리즘을 이해한 다음 역설계하면 됩니다. 더 강한 난독화, 중첩 암호화, 코드 가상화, 안티-디스어셈블리 트릭 같은 기법을 쓸 수도 있지만, 결국 숨바꼭질 게임일 뿐이에요. 불가능하게 만드는 게 아니라 조금 번거롭게 만드는 것 뿐입니다.

첫 번째 깨달음을 얻었습니다: 난독화는 보안 대신 은폐에 의존하는 방식입니다. 충분한 시간과 동기가 있으면 역설계자는 항상 이깁니다. 왜냐하면 바이너리에는 검증을 수행하는 데 필요한 모든 것이 있기 때문이죠.

덧붙이자면, 현대의 역설계 도구들은 간단한 XOR 인코딩을 자동으로 탐지하는 경우가 많아서, 이 방식의 난독화는 실전에서 더욱 효과가 없습니다!

3단계: 디지털 서명

1단계와 2단계의 근본적인 문제는 비밀을 공유해야 한다는 거예요. 누군가 소프트웨어를 사고 시리얼을 온라인에 올리면, 이제 모든 사람이 쓸 수 있습니다.

우리가 필요한 건 다음 조건을 만족하는 라이선스입니다:

  • 누구나 검증할 수 있음 (애플리케이션이 검증 가능)
  • 제3자는 위조할 수 없음 (공급자가 관리)
  • 각 고객마다 고유함

이게 비대칭 암호화의 출장지입니다.

핵심 아이디어는 하나의 공유 비밀 대신 키 쌍을 사용하는 거예요. 우리가 보관하는 개인 키(private key)와 바이너리에 들어가는 공개 키(public key)를 쌍으로 만듭니다. 개인 키로는 데이터에 서명할 수 있고, 공개 키로는 그 서명을 검증할 수 있죠.

한 고객을 위해 라이선스를 발급할 때, 우리는 개인 키로 라이선스 데이터(예: 고객 ID, 만료 날짜)에 서명합니다. 고객이 프로그램을 실행할 때는 공개 키를 사용해서 그 서명이 우리가 만든 게 맞는지 검증합니다. 공격자가 자신만의 라이선스를 만들려면? 개인 키가 필요한데, 이건 바이너리에 없으니까 불가능합니다.

이게 실제로 작동하는 보호 메커니즘입니다!
