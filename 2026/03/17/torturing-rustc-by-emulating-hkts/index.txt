<h3>들어가며</h3>
<p>2026년 2월 28일, 나는 재미로 FP 스크립팅 언어를 만들어보려고 했어요. 먼저 기본 계산기용 AST enum을 만들었습니다.</p>
<pre><code class="language-rust">pub enum Ast {
    Binary {
        left: Box&lt;Ast&gt;,
        operator: Token,
        right: Box&lt;Ast&gt;,
    },
    Unary {
        operator: Token,
        right: Box&lt;Ast&gt;,
    },
    Value {
        value: Value,
    },
}</code></pre>
<p>그런데 필요할 때만 스팬(span) 정보를 포함하는 방식을 추가하고 싶었어요. 그래서 이렇게 만들었습니다.</p>
<pre><code class="language-rust">pub struct Spanned&lt;T&gt; {
    inner: T,
    span: std::range::Range,
}

pub struct Simple&lt;T&gt; {
    inner: T
}</code></pre>
<p>이 프로젝트가 제 것이니까 좀 장난스럽게 접근하고 싶었거든요. 이 구조를 마치 Haskell인 평행우주의 Java 엔터프라이즈 소프트웨어 엔지니어처럼 추상화했습니다.</p>
<pre><code class="language-rust">pub struct Wrapper&lt;T, M&gt; {
    inner: T,
    metadata: M,
}

type Spanned&lt;T&gt; = Wrapper&lt;T, std::range::Range&gt;;
type Simple&lt;T&gt; = Wrapper&lt;T, ()&gt;;</code></pre>
<p>이제 AST 구조체에 이걸 추가해볼까요.</p>
<pre><code class="language-rust">pub enum Ast&lt;W: ???&gt; {
    Binary {
        left: Box&lt;W&lt;Ast&gt;&gt;,

    },
    Unary {
        operator: Token,
        right: Box&lt;W&lt;Ast&gt;&gt;,
    },

    },
}</code></pre>
<p>어? 뭔가 빠뜨렸네요...</p>
<h3>Rust는 HKT가 없다 (사실 조금은 있다)</h3>
<p>고계 타입(HKT, Higher Kinded Types)은 제네릭이 제네릭을 가질 수 있다는 개념입니다. 예를 들어 <code>struct Foo&lt;T&gt;(T&lt;i32&gt;);</code> 같은 코드가 있다면, T는 하나의 제네릭 타입을 받아들일 수 있는 구체적인 타입이어야 해요. 이 경우 T는 arity가 1인 타입 생성자인데, i32를 인자로 받아서 T&lt;i32&gt;라는 새로운 타입을 만들어냅니다.</p>
<p>일단 한 발 물러나서 생각해봅시다.</p>
<p>타입 시스템은 자기만의 우주에 존재합니다. 우리가 사는 우주와 비슷한 구조를 가지고 있죠. 우리 우주에 값이 있다면, 그 위의 우주에는 타입이 있어요. 우리 우주에 함수가 있다면, 위의 우주에는 타입 생성자가 있습니다. 즉 T는 타입을 받아서 새로운 타입을 반환하는 &quot;함수&quot; 같은 거예요.</p>
<p>이제 이해가 되나요? 그런데 &quot;kinded&quot;라는 게 뭐죠? &quot;우주&quot;는 또 뭐고요?</p>
<p>음... 타입에 대해 생각해봅시다. 제네릭 타입 말이에요.</p>
<p>많은 사람이 착각하지만, Vec은 타입이 아닙니다.</p>
<p>뭐라고요?</p>
<p>이렇게 보여드리겠습니다.</p>
<pre><code class="language-rust">fn accepts_a_type&lt;T&gt;() {}

accepts_a_type::&lt;Vec&gt;();</code></pre>
<pre><code>error[E0107]: missing generics for struct `Vec`
  --&gt; src/main.rs:4:22
   |
 4 | accepts_a_type::&lt;Vec&gt;();
   | ^^^ expected at least 1 generic argument</code></pre>
<p>논리를 따라가봅시다.</p>
<ul>
<li><code>accepts_a_type</code>은 타입을 받습니다. 어떤 타입이든 상관없이요.</li>
<li>Vec을 전달하려고 합니다.</li>
<li>컴파일에 실패합니다.</li>
<li>따라서 Vec은 타입이 아닙니다.</li>
</ul>
<p>증명 완료.</p>
<p>정말 견고한 논리네요.</p>
<p>그럼 Vec은 뭘까요? Vec 자체로는 타입이 아닙니다. <code>Vec&lt;i32&gt;</code>는 어떨까요? 그건 타입이 맞습니다. 확인해보세요.</p>
<pre><code class="language-rust">accepts_a_type::&lt;Vec&lt;i32&gt;&gt;();</code></pre>
<pre><code>Compiling playground v0.0.1 (/playground)
Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.65s</code></pre>
<p>짜잔! 컴파일로 증명되었습니다.</p>
<p>그래서 <code>Vec&lt;i32&gt;</code>는 타입이에요. <code>i32</code>도 타입이고요. 하지만 Vec은 아닙니다. 정확히 뭘까요?</p>
<p>타입 시스템 용어로는 Vec을 타입 생성자(type constructor)라고 부릅니다. 이걸 <code>fn(type) -&gt; type</code>이라고 생각할 수 있는데, 타입을 받아서(예: i32) 새로운 타입을 뱉어냅니다(Vec&lt;i32&gt;). 비교표를 한번 봅시다.</p>
<div class="table-wrapper"><table><thead><tr><th></th><th>값</th><th>타입</th></tr></thead><tbody><tr><td>함수</td><td><code>fn(bool) -&gt; bool</code> (예: 단항 not <code>!</code>)</td><td><code>fn(type) -&gt; type</code> (예: Vec)</td></tr><tr><td>인자</td><td>bool (예: true)</td><td>타입 (예: i32)</td></tr><tr><td>결과값</td><td>bool (예: false)</td><td>타입 (예: Vec&lt;i32&gt;)</td></tr></tbody></table></div>
<p>아, 그래서 <code>!</code>의 타입은 사실상 <code>bool -&gt; bool</code>이고, <code>!true</code>의 타입은 <code>bool</code>이라는 뜻이군요.</p>
<p><code>bool</code>과 <code>i32</code>는 사람들이 기본 타입이라고 부르는 것들인데, 언어 자체에 내재된 원시 타입이라서 다른 타입으로 구성할 수 없습니다.</p>
