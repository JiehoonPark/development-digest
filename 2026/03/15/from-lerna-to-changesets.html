From Lerna to Changesets

📅 2026-03-15 📰 DEV Community

요약

Pedalboard 모노레포 프로젝트에서 릴리스 관리를 위해 Lerna에서 Changesets로 마이그레이션하는 과정을 설명한다. Lerna는 현재 패키지 변경 감지, Conventional Commits 기반 버전 범프, git 태깅, npm 발행만 수행하고 있으며, Changesets는 개발자가 의도적으로 changeset 파일을 작성하도록 해 버전 범프의 명확성을 높인다. Changesets 설정 시 baseBranch와 access 속성을 프로젝트에 맞게 조정해야 하며, 불필요한 스크립트도 정리할 수 있다.

핵심 포인트

  • Lerna의 역할이 릴리스 관리로 축소되었으며, 현재는 변경 감지 → 버전 범프(Conventional Commits 기반) → git 태깅 → npm 발행만 수행
  • Changesets는 Conventional Commits 자동 감지 대신 개발자가 직접 changeset 파일을 작성하도록 강제해, 의도하지 않은 버전 범프(예: breaking change가 fix: 커밋으로 들어가는 경우) 방지
  • pnpm dlx @changesets/cli init으로 초기화 후 baseBranch(main → master), access(restricted → public) 두 가지 필수 설정 조정 필요
  • Changesets는 versioning과 publishing을 분리된 명령어로 제공하며, commit: false 설정으로 git 자동 커밋을 비활성화 가능
  • 마이그레이션 시 사용되지 않는 레거시 스크립트(publish:lerna:skip-git 등) 정리 기회

왜 중요한가

모노레포 릴리스 관리 도구 선택 시 명시성과 실패 방지가 중요하며, Changesets의 의도적 접근이 버전 관리의 신뢰성을 높일 수 있기 때문

📄 전문 번역

당신이 Pedalboard 모노레포 이야기를 따라오셨다면, Lerna와 나의 관계가 복잡하다는 것을 아실 겁니다. 저는 원래 "No BS monorepo" 시리즈에서 모노레포가 지속적으로 필요로 하는 두 가지 일(버전 관리와 배포)을 처리하기 위해 Lerna를 선택했습니다. 처음에는 잘 작동했고, 그 다음에는 유지보수가 중단되었다가, 불사조처럼 다시 살아났고, 여기 우리는 여전히 그것을 사용하고 있습니다. 하지만 최근에 Changesets를 살펴보면서 드디어 전환할 시점인지 궁금해하고 있습니다.

Lerna가 Pedalboard에서 실제로 하는 일 (현재 시점)

이것에 대해 솔직해질 가치가 있습니다. Lerna의 역할이 이 프로젝트에서 한동안 축소되어 왔기 때문입니다.

Lerna에 남은 일은 lerna.json의 두 블록에 담겨 있습니다:

{
"npmClient": "pnpm",
"useNx": false,
"command": {
"publish": {
"ignoreChanges": ["ignored-file", "*.md"],
"message": "chore(release): publish %s"
},
"version": {
"message": "chore(release): version",
"allowBranch": "master",
"conventionalCommits": true
}
},
"packages": ["packages/*"],
"version": "independent"
}

이게 전부입니다. Lerna의 전체 업무 목록은:

  • 마지막 릴리스 이후 변경된 패키지 감지
  • conventional commit 메시지를 기반으로 버전 범프(patch/minor/major)
  • git에서 릴리스 태깅
  • lerna publish --yes --no-verify-access를 통해 npm에 배포

root package.json에서도 이를 확인할 수 있습니다. Lerna에 대한 유일한 언급은 단 하나의 스크립트입니다:

"publish:lerna": "lerna publish --yes --no-verify-access"

완성된 모노레포 도구가 매우 구체적이면서도 매우 중요한 한 가지 일로 축소되었습니다: 릴리스 관리입니다.

저는 이러한 철학적 진화를 "One Ring To Rule Them all" 모노레포 매니저 재고찰에서 다루었습니다. 이는 하나의 도구가 모든 것을 할 필요는 없으며, 각 도구는 자신의 영역을 소유해야 한다는 개념입니다. Lerna는 릴리스 영역을 소유합니다. 하지만 그것이 그 영역에 최선의 도구일까요?

Changesets의 등장

Changesets(기술적으로는 @changesets/cli)는 동일한 문제에 대해 다른 접근 방식을 취합니다. conventional commit 메시지를 읽고 변경사항을 자동으로 감지하는 대신, 개발자에게 의도적이 되도록 요구합니다: 변경을 하면 changeset을 실행하고 변경사항과 이것이 필요로 하는 semver 범프(patch, minor, major)에 대한 간단한 설명을 작성합니다. 이렇게 하면 .changeset/ 디렉토리에 작은 마크다운 파일이 생성됩니다.

릴리스할 시간이 되면, changeset version을 실행하여 해당 파일들을 소비하고 적절한 패키지 버전을 범프한 다음, changeset publish를 실행하여 npm에 푸시합니다.

저에게 매력적인 점은 명시성입니다. Lerna의 conventional commits 접근 방식에서는 버전 범프가 commit 메시지에서 추론됩니다. 즉, 이는 당신의 commit 규칙만큼만 신뢰할 수 있습니다. 누군가 breaking change를 fix: 접두사로 몰래 넣으면(우리 모두 이런 경험이 있습니다. 거짓말하지 마세요), Lerna는 행복하게 patch를 배포할 것입니다. Changesets는 범프 의도를 PR 자체의 일부로 만듦으로써 이를 우회합니다.

Changesets 초기화

충분한 이론입니다. 초기화는 한 명령어입니다:

pnpm dlx @changesets/cli init

이것은 README.md와 함께 .changeset/ 디렉토리를 생성하고, 더 중요하게는 config.json을 생성합니다:

{
"$schema": "https://unpkg.com/@changesets/config@3.1.3/schema.json",
"changelog": "@changesets/cli/changelog",
"commit": false,
"fixed": [],
"linked": [],
"access": "restricted",
"baseBranch": "main",
"updateInternalDependencies": "patch",
"ignore": []
}

시작점으로는 나쁘지 않지만, 두 가지가 즉시 수정되어야 합니다.

바로 마주치는 두 가지 설정 문제

baseBranch: "main" - Pedalboard의 브랜치는 main이 아니라 master입니다. Changesets는 마지막 릴리스 이후 변경사항을 결정하기 위해 이를 사용합니다. 잘못된 base는 changeset status가 잘못된 것과 비교하는 것을 의미합니다:

"baseBranch": "master"

access: "restricted" - 이것은 npm 접근 수준을 제어합니다. "restricted"는 private 패키지를 의미하며, 이는 합리적인 안전 기본값이지만, Pedalboard의 모든 패키지는 npm에서 public입니다:

"access": "public"

단 두 줄입니다. 릴리스를 푸시했다가 아무것도 배포되지 않은 이유를 궁금해하기 전에 기꺼이 발견하게 된 두 가지입니다.

나머지 기본값들은 괜찮습니다: "commit": false는 Changesets가 버전 범프를 자동 커밋하지 않도록 한다는 의미입니다(CI가 처리합니다). 그리고 "updateInternalDependencies": "patch"는 패키지가 범프될 때, 이에 의존하는 모든 내부 패키지도 patch 범프를 받는다는 의미입니다.

데드 스크립트를 정리하기 좋은 시간

Lerna가 무엇을 하는지 살펴보던 중 package.json에 publish:lerna:skip-git 스크립트가 있는 것을 발견했습니다:

"publish:lerna:skip-git": "lerna publish --yes --no-verify-access --no-git-tag-version --no-push --loglevel=silly"

플래그들이 이야기를 합니다. --no-git-tag-version --no-push는 git 히스토리를 오염시키지 않으면서 로컬에서 배포를 테스트할 수 있도록 하기 위한 것이었습니다. 디버그 탈출구였습니다.

저장소 전체와 모든 블로그 포스트에서 빠르게 검색한 결과는 확실합니다: 아무도 이것을 호출한 적이 없습니다. GitHub Actions 워크플로우도 아니고, 다른 스크립트도 아니고, 단 하나의 포스트도 아닙니다. 그냥 거기에 있었을 뿐입니다.
