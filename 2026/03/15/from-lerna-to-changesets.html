Lerna에서 Changesets로의 마이그레이션

From Lerna to Changesets

📅 2026-03-15 📰 DEV Community

요약

Pedalboard 모노레포의 저자가 Lerna에서 Changesets로 마이그레이션하는 과정을 설명합니다. Lerna는 원래 다목적 모노레포 관리자였으나 현재는 릴리스 관리(버전 관리 및 npm 배포)에만 사용되고 있습니다. 반면 Changesets는 개발자가 각 PR에서 의도적으로 변경사항을 명시하도록 하여 conventional commit 방식의 자동 버전 감지보다 더 정확한 의존성 버전 관리를 제공합니다.

핵심 포인트

  • Lerna는 현재 conventional commit 메시지를 읽어 패치/마이너/메이저 버전을 자동으로 감지하는 단순한 릴리스 관리자로만 기능하고 있습니다.
  • Changesets는 pnpm dlx @changesets/cli init으로 초기화하며, .changeset/ 디렉토리에 마크다운 파일로 변경사항을 기록하는 명시적 방식을 사용합니다.
  • baseBranch 설정(master vs main)과 access 설정(public vs restricted)을 올바르게 구성해야 릴리스 감지 및 npm 배포가 정상 작동합니다.
  • Changesets는 'changeset version'과 'changeset publish'를 별도 명령으로 분리하여, 버전 관리와 배포를 더 명확하게 제어할 수 있습니다.
  • commit: false 설정으로 Changesets는 버전 범프를 자동 커밋하지 않으며, 이는 CI 워크플로우가 커밋을 처리하도록 설계되었음을 반영합니다.

왜 중요한가

모노레포 릴리스 관리에서 자동 감지의 불확실성(커밋 규칙 위반)을 제거하고 개발자의 명시적 의도 표기로 더 정확한 의존성 버전 관리를 달성할 수 있습니다.

📄 전문 번역

당신이 내 Pedalboard 모노레포 이야기를 따라왔다면, Lerna와 나의 관계가 복잡하다는 것을 알 것이다. 원래 "No BS 모노레포" 시리즈에서 모노레포가 끊임없이 필요로 하는 두 가지, 즉 버전 관리와 배포를 다루기 위해 Lerna를 선택했다. 그것이 작동했고, 그다음 유지보수가 중단되었으며, 그러다가 불사조처럼 부활했고, 지금도 여전히 사용 중이다. 하지만 최근에 Changesets를 살펴보면서 드디어 전환할 시간이 되지 않았나 하는 생각이 든다.

Pedalboard에서 Lerna가 실제로 하는 일 (지금 현재)

이것을 솔직하게 말하는 것이 중요한데, Pedalboard 프로젝트에서 Lerna의 역할이 한동안 좁혀져 왔기 때문이다.

Lerna에 남아 있는 것은 lerna.json의 두 블록에 담겨 있다:

{
  "npmClient": "pnpm",
  "useNx": false,
  "command": {
    "publish": {
      "ignoreChanges": ["ignored-file", "*.md"],
      "message": "chore(release): publish %s"
    },
    "version": {
      "message": "chore(release): version",
      "allowBranch": "master",
      "conventionalCommits": true
    }
  },
  "packages": ["packages/*"],
  "version": "independent"
}

그게 전부다. 현 시점에서 Lerna의 전부:

  • 마지막 릴리스 이후 어떤 패키지가 변경되었는지 감지
  • Conventional Commit 메시지를 기반으로 버전 범프 (patch/minor/major)
  • git에서 릴리스 태그 지정
  • lerna publish --yes --no-verify-access를 통해 npm에 배포

root package.json이 이를 확인시켜준다 - lerna에 대한 언급은 단 하나의 스크립트뿐이다:

"publish:lerna": "lerna publish --yes --no-verify-access"

완전히 성숙한 모노레포 도구가 매우 구체적이면서도 매우 중요한 일로 축소되었다: 릴리스 관리.

나는 이러한 철학적 진화에 대해 "모노레포 관리자의 '절대 반지'를 재고하기"에서 썼다. 한 가지 도구가 모든 것을 해서는 안 되고, 각 도구가 자신의 영역을 담당해야 한다는 개념이다. Lerna는 릴리스 영역을 담당한다. 하지만 그것이 그 영역의 최고의 도구일까?

Changesets 소개

Changesets (정확히는 @changesets/cli)는 같은 문제에 다른 접근 방식을 취한다. Conventional Commit 메시지를 읽고 변경된 것을 자동으로 감지하는 대신, 개발자에게 의도를 갖도록 요구한다: 변경을 할 때마다 changeset을 실행하고 무엇이 변경되었는지, 그리고 어떤 종류의 semver 범프가 필요한지 (patch, minor, major) 간단히 설명한다. 그러면 .changeset/ 디렉토리에 작은 마크다운 파일이 생성된다.

릴리스할 시간이 되면, changeset version을 실행하여 해당 파일들을 소비하고 적절한 패키지 버전을 범프한 후, changeset publish로 npm에 푸시한다.

나에게 매력적인 점은 명확성이다. Lerna의 conventional commits 접근 방식에서는 버전 범프가 커밋 메시지에서 추론되는데, 이는 커밋 규칙만큼만 신뢰할 수 있다는 의미다. 누군가 breaking change를 fix: 접두사로 슬쩍 포함시키면 (우리 모두 그런 적이 있다, 거짓말하지 말자), Lerna는 기꺼이 patch를 배포한다. Changesets는 범프 의도를 PR 자체의 일부로 만들어서 이를 우회한다.

Changesets 초기화하기

이론은 충분하다. 초기화는 한 줄의 명령어다:

pnpm dlx @changesets/cli init

이것은 README.md가 있는 .changeset/ 디렉토리를 생성하고, 더 중요하게는 config.json을 생성한다:

{
  "$schema": "https://unpkg.com/@changesets/config@3.1.3/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}

시작점으로는 나쁘지 않지만 두 가지를 즉시 수정해야 한다.

두 가지 설정 문제, 바로 나타남

baseBranch: "main" - Pedalboard의 브랜치는 master이지, main이 아니다. Changesets는 이것을 사용해서 마지막 릴리스 이후 무엇이 변경되었는지 판단한다. 잘못된 base는 changeset status가 잘못된 것과 비교하게 된다:

"baseBranch": "master"

access: "restricted" - 이것은 npm 접근 수준을 제어한다. "restricted"는 비공개 패키지를 의미하며, 이것은 합리적인 안전한 기본값이지만, Pedalboard의 모든 패키지는 npm에서 공개다:

"access": "public"

두 줄이다. 릴리스를 푸시한 후 아무것도 배포되지 않은 이유를 궁금해하기 전에 주목해서 정말 다행이다.

다른 기본값들은 괜찮다: "commit": false는 Changesets가 버전 범프를 자동으로 커밋하지 않음을 의미하며 (CI가 처리), "updateInternalDependencies": "patch"는 패키지가 범프될 때 이를 의존하는 모든 내부 패키지도 patch 범프를 받는다는 의미다.

죽은 스크립트를 정리할 좋은 기회

Lerna가 무엇을 하는지 살펴보다가 package.json에 publish:lerna:skip-git 스크립트가 앉아 있는 것을 발견했다:

"publish:lerna:skip-git": "lerna publish --yes --no-verify-access --no-git-tag-version --no-push --loglevel=silly"

플래그들이 이야기를 들려준다 - --no-git-tag-version --no-push는 git 히스토리를 오염시키지 않으면서 로컬에서 배포를 테스트하게 해주려고 있었다. 디버그 탈출구였다.

저장소 전체와 모든 블로그 포스트에 대한 빠른 검색으로 확인했다: 아무도 이것을 호출한 적이 없었다. GitHub Actions 워크플로우도, 다른 어떤 스크립트도, 단 하나의 포스트도 아니었다. 그냥 그 자리에 앉아만 있었다.

나는 Changesets 동등물을 만들어야 하는지 고민했지만, Changesets는 이미 버전 관리와 배포를 두 개의 서로 다른 명령어로 분리하고 있으며, config에서 "commit": false로 인해 changeset version은 git에 전혀 손대지 않는다. 만약 내가 태그 없이 배포해야 한다면, changeset publish로...
