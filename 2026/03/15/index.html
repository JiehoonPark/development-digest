🛠 프론트엔드 기술

Lerna에서 Changesets로의 마이그레이션

Pedalboard 모노레포의 저자가 Lerna에서 Changesets로 마이그레이션하는 과정을 설명합니다. Lerna는 원래 다목적 모노레포 관리자였으나 현재는 릴리스 관리(버전 관리 및 npm 배포)에만 사용되고 있습니다. 반면 Changesets는 개발자가 각 PR에서 의도적으로 변경사항을 명시하도록 하여 conventional commit 방식의 자동 버전 감지보다 더 정확한 의존성 버전 관리를 제공합니다.

  • Lerna는 현재 conventional commit 메시지를 읽어 패치/마이너/메이저 버전을 자동으로 감지하는 단순한 릴리스 관리자로만 기능하고 있습니다.
  • Changesets는 pnpm dlx @changesets/cli init으로 초기화하며, .changeset/ 디렉토리에 마크다운 파일로 변경사항을 기록하는 명시적 방식을 사용합니다.
  • baseBranch 설정(master vs main)과 access 설정(public vs restricted)을 올바르게 구성해야 릴리스 감지 및 npm 배포가 정상 작동합니다.
  • Changesets는 'changeset version'과 'changeset publish'를 별도 명령으로 분리하여, 버전 관리와 배포를 더 명확하게 제어할 수 있습니다.
  • commit: false 설정으로 Changesets는 버전 범프를 자동 커밋하지 않으며, 이는 CI 워크플로우가 커밋을 처리하도록 설계되었음을 반영합니다.

💡 모노레포 릴리스 관리에서 자동 감지의 불확실성(커밋 규칙 위반)을 제거하고 개발자의 명시적 의도 표기로 더 정확한 의존성 버전 관리를 달성할 수 있습니다.

💡 알면 좋은 정보

통합 API가 프론트엔드를 실패하게 만드는 이유 (그리고 메타데이터가 해결책인 이유)

기존 통합 API는 백엔드 간 데이터 동기화에만 최적화되어 있어 동적 UI 생성 시 필드 정보를 손실하거나 평탄화하는 문제가 발생합니다. 이를 해결하기 위해 데이터뿐 아니라 스키마, 필드 유형, 사용자 권한 등을 포함한 메타데이터를 제공하는 '통합 메타데이터 API'가 필요합니다. 이를 통해 React나 Vue 같은 단일 컴포넌트로 여러 SaaS 플랫폼(Salesforce, HubSpot 등)의 폼을 동적으로 생성할 수 있습니다.

  • 기존 통합 API는 ETL 도구로 설계되어 사용자 정의 필드를 제거하거나 원본 데이터 블롭으로 묻어버려 동적 UI 구성에 부적합합니다.
  • 통합 메타데이터 API는 데이터 자체가 아닌 스키마와 규칙(필드 타입, 선택지, 읽기 전용 여부)을 제공하여 런타임에 UI를 자동 생성할 수 있습니다.
  • 헤드리스 CMS 개념처럼, 헤드리스 인테그레이션은 콘텐츠와 프레젠테이션을 분리하여 메타데이터 기반의 유연한 UI 구축을 가능하게 합니다.
  • 고객 맞춤 필드나 권한 기반 UI 변경 같은 실시간 상호작용이 필요한 인앱 통합에서 메타데이터 API의 필요성이 극대화됩니다.

💡 SaaS 인테그레이션을 구축하는 프론트엔드 개발자에게 메타데이터 기반 접근법은 다양한 플랫폼별로 반복되는 커스텀 매핑을 제거하고 확장 가능한 동적 UI 솔루션을 제공합니다.
