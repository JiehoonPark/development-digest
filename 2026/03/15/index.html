🛠 프론트엔드 기술

How a Browser Renders a Webpage — And Where React, CSR, and SSR Fit In

브라우저가 URL 입력부터 웹페이지 렌더링까지 거치는 과정을 상세히 설명하는 글입니다. DNS 조회, HTTP 요청, HTML 파싱, DOM/CSSOM 생성, 레이아웃 계산, 페인트 등의 단계를 거쳐 최종적으로 화면에 표시되며, React의 CSR(클라이언트 사이드 렌더링)과 Next.js의 SSR(서버 사이드 렌더링), 그리고 하이드레이션 과정을 비교 분석합니다.

  • 브라우저 렌더링 과정: 캐시 확인 → DNS 조회 → TCP 연결 → HTTP 요청 → HTML 파싱 → DOM/CSSOM 생성 → 렌더 트리 → 레이아웃 → 페인트 → 컴포지팅
  • JavaScript는 HTML 파싱을 중단시킬 수 있으므로, defer와 async 속성으로 렌더링 블로킹을 방지해야 합니다.
  • CSR은 빈 HTML과 JavaScript를 받아 브라우저에서 UI를 렌더링하므로 초기 로딩이 느리지만 상호작용이 빠르고, SSR은 서버에서 완성된 HTML을 보내 초기 로딩이 빠르고 SEO에 유리합니다.
  • SSR 후 JavaScript가 로드되면 기존 HTML에 이벤트 핸들러를 연결하는 하이드레이션 과정이 일어납니다.
  • Next.js 같은 현대 프레임워크는 CSR과 SSR을 결합하여 초기 로딩 성능과 상호작용성을 모두 확보합니다.

💡 렌더링 과정을 이해하면 성능 최적화, 렌더링 블로킹 방지, 적절한 렌더링 전략 선택이 가능하여 사용자 경험을 크게 개선할 수 있습니다.

From Lerna to Changesets

모노레포 프로젝트인 Pedalboard에서 Lerna를 사용한 릴리스 관리를 Changesets로 마이그레이션하는 과정을 설명합니다. Lerna는 현재 버전 관리와 npm 배포만 담당하고 있으며, Changesets는 개발자가 의도적으로 변경사항과 semver 버전 범프 종류를 명시하는 더 명확한 접근 방식을 제시합니다.

  • Lerna는 기존에 유지보수 중단되었다가 다시 활성화되었으며, 현재 Pedalboard에서는 릴리스 관리 전용으로 축소되어 있습니다.
  • Changesets는 conventional commit 대신 개발자가 .changeset/ 디렉토리에 마크다운 파일을 작성하여 변경사항을 명시하므로, 커밋 규율에 의존하지 않습니다.
  • Changesets 초기화 후 baseBranch와 access 설정을 조정해야 하며, commit: false 옵션으로 CI가 버전 관리를 하도록 설정할 수 있습니다.
  • 사용하지 않는 디버그 스크립트(publish:lerna:skip-git)를 정리하는 것이 좋습니다.

💡 모노레포에서 버전 관리와 배포 도구를 선택할 때, 명확성과 의도성 측면에서 Changesets의 장점을 이해하면 릴리스 프로세스의 안정성과 신뢰도를 높일 수 있습니다.
