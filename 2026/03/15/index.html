🛠 프론트엔드 기술

How a Browser Renders a Webpage — And Where React, CSR, and SSR Fit In

브라우저가 URL 입력 후 웹페이지를 렌더링하는 전체 과정을 설명하는 기사로, DNS 조회부터 캐시 확인, HTML/CSS 파싱, DOM 생성, 레이아웃 계산, 페인트, 컴포지팅까지의 단계를 다룬다. React의 CSR(클라이언트 사이드 렌더링)과 SSR(서버 사이드 렌더링)이 이 과정에서 어떻게 작동하는지, 그리고 JavaScript가 렌더링을 어떻게 차단할 수 있는지를 구체적으로 설명한다.

  • 브라우저 렌더링 9단계: 캐시 확인 → DNS 조회 → TCP 연결 → HTTP 요청 → HTML 파싱(DOM 생성) → CSS 파싱(CSSOM 생성) → 렌더 트리 생성 → 레이아웃(리플로우) → 페인트 → 컴포지팅
  • JavaScript 실행 방식에 따른 성능 차이: defer 속성은 HTML 파싱 후 실행, async 속성은 다운로드 직후 즉시 실행하여 페인팅을 차단할 수 있음
  • CSR은 빈 HTML을 보내고 브라우저에서 React가 UI를 동적으로 생성하므로 초기에 빈 화면이 표시되지만, SSR은 서버에서 완성된 HTML을 보내므로 콘텐츠가 즉시 표시됨
  • Hydration 개념: SSR로 렌더링된 HTML이 로드된 후 JavaScript가 다운로드되어 이벤트 핸들러를 기존 HTML에 연결하는 과정
  • CSR은 높은 상호작용성이 필요한 애플리케이션에 적합하고, SSR은 초기 로드 성능과 SEO 최적화에 유리하며, Next.js 같은 현대 프레임워크는 두 방식을 결합하여 최적의 경험 제공

💡 브라우저 렌더링 메커니즘을 이해하면 성능 최적화, 렌더링 차단 문제 해결, 적절한 렌더링 전략(CSR vs SSR) 선택이 가능하여 사용자 경험을 크게 개선할 수 있다.
