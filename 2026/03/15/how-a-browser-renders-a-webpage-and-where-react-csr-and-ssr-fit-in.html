How a Browser Renders a Webpage — And Where React, CSR, and SSR Fit In

📅 2026-03-15 📰 DEV Community

요약

브라우저가 URL 입력 후 웹페이지를 렌더링하는 전체 과정을 설명하는 기사로, DNS 조회부터 캐시 확인, HTML/CSS 파싱, DOM 생성, 레이아웃 계산, 페인트, 컴포지팅까지의 단계를 다룬다. React의 CSR(클라이언트 사이드 렌더링)과 SSR(서버 사이드 렌더링)이 이 과정에서 어떻게 작동하는지, 그리고 JavaScript가 렌더링을 어떻게 차단할 수 있는지를 구체적으로 설명한다.

핵심 포인트

  • 브라우저 렌더링 9단계: 캐시 확인 → DNS 조회 → TCP 연결 → HTTP 요청 → HTML 파싱(DOM 생성) → CSS 파싱(CSSOM 생성) → 렌더 트리 생성 → 레이아웃(리플로우) → 페인트 → 컴포지팅
  • JavaScript 실행 방식에 따른 성능 차이: defer 속성은 HTML 파싱 후 실행, async 속성은 다운로드 직후 즉시 실행하여 페인팅을 차단할 수 있음
  • CSR은 빈 HTML을 보내고 브라우저에서 React가 UI를 동적으로 생성하므로 초기에 빈 화면이 표시되지만, SSR은 서버에서 완성된 HTML을 보내므로 콘텐츠가 즉시 표시됨
  • Hydration 개념: SSR로 렌더링된 HTML이 로드된 후 JavaScript가 다운로드되어 이벤트 핸들러를 기존 HTML에 연결하는 과정
  • CSR은 높은 상호작용성이 필요한 애플리케이션에 적합하고, SSR은 초기 로드 성능과 SEO 최적화에 유리하며, Next.js 같은 현대 프레임워크는 두 방식을 결합하여 최적의 경험 제공

왜 중요한가

브라우저 렌더링 메커니즘을 이해하면 성능 최적화, 렌더링 차단 문제 해결, 적절한 렌더링 전략(CSR vs SSR) 선택이 가능하여 사용자 경험을 크게 개선할 수 있다.

📄 전문 번역

브라우저가 웹페이지를 렌더링하는 방식 이해하기

브라우저 주소창에 URL을 입력하고 Enter 키를 누르면, 웹페이지가 화면에 나타나기 전에 많은 일들이 백그라운드에서 일어납니다. 어떤 웹사이트는 즉시 로드되고 어떤 사이트는 먼저 빈 화면을 보여주는 이유를 이해하려면 이 과정을 알아야 합니다.

이 글에서는 다음을 다룹니다:

  • 브라우저가 요청을 처리하는 방식
  • UI가 렌더링되는 방식
  • React, Client-Side Rendering (CSR), Server-Side Rendering (SSR)의 역할

1. 요청 처리 과정

사용자가 다음과 같은 URL을 입력할 때:

https://example.com

브라우저는 여러 단계를 거칩니다.

Step 1: 캐시 확인

브라우저는 먼저 웹사이트 리소스가 로컬 캐시에 저장되어 있는지 확인합니다.

Step 2: DNS 조회

캐시에 없으면, 브라우저는 Domain Name System에 도메인 이름을 IP 주소로 변환해달라고 요청합니다.

예시:

example.com → 93.184.216.34

Step 3: 연결 수립

브라우저는 Transmission Control Protocol을 사용하여 서버와의 연결을 엽니다.

웹사이트가 HTTPS를 사용하면 Transport Layer Security를 통해 보안 연결을 수립합니다.

Step 4: HTTP 요청 전송

브라우저는 다음과 같은 요청을 전송합니다:

GET /index.html HTTP/1.1
Host: example.com

2. 서버 응답

서버는 다음과 같은 리소스를 다시 보냅니다:

  • HTML
  • CSS
  • JavaScript
  • 이미지
  • 폰트

이러한 리소스들은 종종 청크(스트림) 단위로 전송되므로, 브라우저는 전체 파일 다운로드가 끝나기 전에 처리를 시작합니다.

3. HTML 파싱 및 DOM 생성

브라우저가 HTML을 받으면 즉시 파싱을 시작합니다.

브라우저는 HTML 요소를 Document Object Model이라는 트리 구조로 변환합니다.

예시 HTML:

<body>
<h1>Products</h1>
<p>Item 1</p>
</body>

DOM 구조:

body
├── h1
└── p

이 구조 덕분에 JavaScript가 요소에 동적으로 접근하고 수정할 수 있습니다.

4. CSS 파싱 및 CSSOM

브라우저가 CSS 파일이나 <style> 태그를 마주치면, CSS Object Model이라는 또 다른 구조를 생성합니다.

이는 선택자를 스타일에 매핑합니다.

예시:

p {
  color: blue;
}

브라우저는 스타일 규칙의 트리를 구성하고 DOM 요소에 적용합니다.

5. 렌더 트리 생성

브라우저는 다음을 결합합니다:

  • DOM
  • CSSOM

이를 통해 렌더 트리를 생성합니다.

이 트리는 눈에 보이는 요소만 포함합니다.

제외되는 요소들:

  • <meta>
  • <head>
  • display: none이 적용된 요소

6. 레이아웃 (Reflow)

다음으로 브라우저는 각 요소의 위치와 크기를 계산합니다.

다음을 결정합니다:

  • 너비와 높이
  • X, Y 축의 위치
  • 페이지에서 요소들이 흘러가는 방식

이 단계는 UI가 기기 뷰포트에 올바르게 맞는지 확인합니다.

7. 페인트 (Paint)

레이아웃이 계산되면, 브라우저는 다음과 같은 시각적 세부 사항을 채웁니다:

  • 색상
  • 테두리
  • 텍스트
  • 그림자
  • 이미지

이 단계에서 페이지가 시각적으로 완성됩니다.

8. 컴포지팅 (Compositing)

일부 요소는 별도의 레이어에서 렌더링됩니다:

  • sticky 헤더
  • 팝업
  • 애니메이션
  • 변환된 요소

이 레이어들은 GPU로 전송되어 효율적으로 렌더링됩니다.

이 최종 단계를 컴포지팅이라고 합니다.

9. JavaScript가 작동하는 방식

브라우저가 스크립트를 만나면:

<script src="app.js"></script>

HTML 파싱이 중지되고, 브라우저는:

  • JavaScript를 다운로드
  • 실행
  • HTML 파싱 재개

이 때문에 JavaScript가 페이지 렌더링을 차단할 수 있습니다.

차단을 피하기 위해 개발자들은 다음을 사용합니다:

<script defer src="app.js"></script>
<script async src="analytics.js"></script>
  • async는 다운로드 후 즉시 실행됩니다
  • defer는 HTML 파싱이 끝난 후 실행됩니다

10. Client-Side Rendering (CSR)

React로 만들어진 많은 현대 웹 앱은 Client-Side Rendering을 사용합니다.

CSR에서는 서버가 최소한의 HTML을 전송합니다:

<body>
<div id="root"></div>
<script src="main.js"></script>
</body>

브라우저는 처음에 다음을 봅니다:

  • 빈 페이지

그 다음 JavaScript가 실행되고 React가 UI를 구성합니다.

예시 React 코드:

ReactDOM.createRoot(root).render(<App />)

React는 DOM 요소를 동적으로 생성합니다.

사용자 경험:

페이지 로드
↓
빈 화면
↓
JavaScript 로드
↓
UI 나타남

11. Server-Side Rendering (SSR)

Next.js와 같은 프레임워크는 Server-Side Rendering을 사용합니다.

빈 HTML을 보내는 대신, 서버가 React 컴포넌트를 렌더링하고 완전히 구성된 HTML을 전송합니다.

예시 응답:

<div id="root">
<h1>Products</h1>
<p>Item 1</p>
</div>

이제 브라우저가 즉시 콘텐츠를 표시합니다.

12. 하이드레이션 (Hydration)

SSR HTML이 로드된 후에도 JavaScript가 다운로드됩니다.

React는 기존 HTML에 이벤트 핸들러를 부착합니다.

이 과정을 하이드레이션이라고 합니다.

하이드레이션 전:

  • 버튼이 보이지만 클릭 불가능

하이드레이션 후:

  • 버튼이 상호작용 가능해집니다

CSR vs SSR 비교

| 기능 | CSR | SSR |

|------|-----|-----|

| 초기 HTML | 비어 있음 | 완전히 렌더링됨 |

| 첫 로드 | 빈 화면 | 콘텐츠가 즉시 나타남 |

| SEO | 제한적 | 더 나음 |

| 렌더링 위치 | 브라우저 | 서버 |

결론

브라우저가 웹페이지를 렌더링하는 방식을 이해하면 프론트엔드 개발자는 다음을 할 수 있습니다:

  • 성능 최적화
  • 렌더링 차단 문제 회피
  • 사용자 경험 개선
  • 올바른 렌더링 전략 선택

CSR은 높은 상호작용성이 필요한 애플리케이션에 훌륭하고, SSR은 초기 로드 성능과 SEO를 개선합니다.

Next.js와 같은 현대 프레임워크는 두 가지 방식을 결합하여 최고의 사용자 경험을 제공합니다.
