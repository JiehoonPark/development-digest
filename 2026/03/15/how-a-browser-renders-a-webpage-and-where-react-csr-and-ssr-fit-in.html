How a Browser Renders a Webpage — And Where React, CSR, and SSR Fit In

📅 2026-03-15 📰 DEV Community

요약

브라우저가 URL을 입력받아 웹페이지를 렌더링하기까지의 전체 과정을 설명하며, React의 CSR(클라이언트 사이드 렌더링)과 SSR(서버 사이드 렌더링)이 이 과정에서 어떻게 작동하는지를 다룬다. DNS 조회, HTTP 요청, HTML 파싱, DOM/CSSOM 생성, 렌더 트리 구성, 레이아웃, 페인트, 컴포지팅의 8단계를 거쳐 페이지가 표시되며, JavaScript는 HTML 파싱을 차단할 수 있으므로 defer/async 속성으로 최적화해야 한다.

핵심 포인트

  • 브라우저 렌더링은 캐시 확인 → DNS 조회 → TCP 연결 → HTTP 요청 → HTML 파싱 → DOM/CSSOM 생성 → 렌더 트리 구성 → 레이아웃 → 페인트 → 컴포지팅 단계를 거침
  • JavaScript는 실행 시 HTML 파싱을 중단하므로, defer는 파싱 후 실행, async는 다운로드 후 즉시 실행하도록 사용해야 렌더링 차단 방지 가능
  • CSR은 서버가 빈 HTML만 전송하고 JavaScript가 브라우저에서 UI를 동적으로 생성하므로 초기 로드는 느리지만 상호작용성이 높고, SSR은 서버가 완성된 HTML을 전송해 초기 로드가 빠르고 SEO에 유리
  • Hydration은 SSR 후 JavaScript가 기존 HTML에 이벤트 핸들러를 연결하는 과정으로, 이 과정 전까지 버튼은 보이지만 동작하지 않음
  • Next.js 같은 현대 프레임워크는 CSR과 SSR을 결합해 초기 로드 성능과 상호작용성을 모두 확보

왜 중요한가

프론트엔드 개발자가 렌더링 성능을 최적화하고 렌더 차단 문제를 해결하며, CSR과 SSR 중 올바른 전략을 선택할 수 있기 때문

📄 전문 번역

브라우저가 웹페이지를 렌더링하는 방식 이해하기

브라우저에서 URL을 입력하고 Enter를 누르면, 웹페이지가 화면에 나타나기 전에 백그라운드에서 많은 일들이 일어납니다. 어떤 웹사이트는 즉시 로드되고 어떤 웹사이트는 먼저 빈 화면을 보여주는 이유를 이해하기 위해, 이 과정을 이해하는 것은 프론트엔드 개발자에게 매우 중요합니다.

이 글에서 우리는 다음을 살펴볼 것입니다:

  • 브라우저가 요청을 어떻게 처리하는지
  • UI가 어떻게 렌더링되는지
  • React, Client-Side Rendering (CSR), Server-Side Rendering (SSR)이 어떤 역할을 하는지

1. 요청 프로세스

사용자가 다음과 같은 URL을 입력할 때:

https://example.com

브라우저는 여러 단계를 거칩니다.

Step 1: 캐시 확인

브라우저는 먼저 웹사이트 리소스가 로컬 캐시에 이미 저장되어 있는지 확인합니다.

Step 2: DNS 조회

캐시에 없으면, 브라우저는 Domain Name System에 도메인 이름을 IP 주소로 변환해달라고 요청합니다.

예시:

example.com → 93.184.216.34

Step 3: 연결 설정

브라우저는 Transmission Control Protocol을 사용하여 서버와 연결을 엽니다.

웹사이트가 HTTPS를 사용하면, Transport Layer Security를 사용하여 보안 연결을 설정합니다.

Step 4: HTTP 요청 전송

브라우저는 다음과 같은 요청을 전송합니다:

GET /index.html HTTP/1.1
Host: example.com

2. 서버 응답

서버는 다음과 같은 리소스를 다시 보냅니다:

  • HTML
  • CSS
  • JavaScript
  • Images
  • Fonts

이러한 리소스들은 종종 청크(스트림) 형태로 전송되므로, 브라우저는 전체 파일 다운로드가 완료되기 전에 이들을 처리하기 시작합니다.

3. HTML 파싱 및 DOM 생성

브라우저가 HTML을 받으면, 즉시 파싱을 시작합니다.

브라우저는 HTML 요소를 Document Object Model이라고 불리는 트리 구조로 변환합니다.

예시 HTML:

<body>
<h1>Products</h1>
<p>Item 1</p>
</body>

DOM 구조:

body
├── h1
└── p

이 구조를 통해 JavaScript는 요소에 동적으로 접근하고 수정할 수 있습니다.

4. CSS 파싱 및 CSSOM

브라우저가 CSS 파일이나 <style> 태그를 만나면, CSS Object Model이라고 불리는 또 다른 구조를 생성합니다.

이는 선택자를 그들의 스타일에 매핑합니다.

예시:

p {
color: blue;
}

브라우저는 스타일 규칙 트리를 만들고 이를 DOM 요소에 적용합니다.

5. 렌더 트리 생성

브라우저는 다음을 결합합니다:

  • DOM
  • CSSOM

렌더 트리를 생성하기 위해 말입니다.

이 트리는 보이는 요소들만 포함합니다.

제외되는 요소들은:

  • <meta>
  • <head>
  • display: none이 적용된 요소들

6. 레이아웃 (Reflow)

다음으로, 브라우저는 각 요소의 위치와 크기를 계산합니다.

이는 다음을 결정합니다:

  • 너비와 높이
  • X와 Y축 상의 위치
  • 페이지에서 요소들이 흐르는 방식

이 단계는 UI가 기기의 뷰포트에 올바르게 맞도록 보장합니다.

7. 페인트

레이아웃이 계산되면, 브라우저는 다음과 같은 시각적 세부사항을 채웁니다:

  • 색상
  • 테두리
  • 텍스트
  • 그림자
  • 이미지

이 단계에서 페이지는 시각적으로 완성됩니다.

8. 컴포지팅

일부 요소들은 별도의 레이어에서 렌더링됩니다. 예를 들어:

  • sticky 헤더
  • 팝업
  • 애니메이션
  • 변환된 요소들

이러한 레이어들은 GPU로 보내져서 효율적으로 렌더링됩니다.

이 최종 단계를 컴포지팅이라고 합니다.

9. JavaScript가 어디에 맞는지

브라우저가 스크립트를 만날 때:

<script src="app.js"></script>

HTML 파싱이 일시 중지되고, 브라우저는:

  • JavaScript를 다운로드합니다
  • 이를 실행합니다
  • HTML 파싱을 재개합니다

이것이 JavaScript가 페이지 렌더링을 막을 수 있는 이유입니다.

블로킹을 피하기 위해 개발자들은 다음을 사용합니다:

<script defer src="app.js"></script>
<script async src="analytics.js"></script>
  • async는 다운로드 후 즉시 실행됩니다
  • defer는 HTML 파싱이 완료된 후에 실행됩니다

10. Client-Side Rendering (CSR)

React로 만든 많은 현대 웹 앱은 Client-Side Rendering을 사용합니다.

CSR에서, 서버는 최소한의 HTML을 전송합니다:

<body>
<div id="root"></div>
<script src="main.js"></script>
</body>

브라우저는 초기에 다음을 봅니다:

  • 빈 페이지

그러면 JavaScript가 실행되고 React가 UI를 만듭니다.

예시 React 코드:

ReactDOM.createRoot(root).render(<App />)

React는 DOM 요소를 동적으로 생성합니다.

사용자 경험:

페이지 로드
↓
빈 화면
↓
JavaScript 로드
↓
UI 나타남

11. Server-Side Rendering (SSR)

Next.js와 같은 프레임워크는 Server-Side Rendering을 사용합니다.

빈 HTML을 보내는 대신, 서버가 React 컴포넌트를 렌더링하고 완전히 구성된 HTML을 보냅니다.

예시 응답:

<div id="root">
<h1>Products</h1>
<p>Item 1</p>
</div>

이제 브라우저는 즉시 콘텐츠를 표시합니다.

12. Hydration

SSR HTML이 로드된 후, JavaScript는 여전히 다운로드됩니다.

React는 기존 HTML에 이벤트 핸들러를 붙입니다.

이 과정을 Hydration이라고 합니다.

Hydration 전:

  • 버튼이 보이지만 클릭 불가능

Hydration 후:

  • 버튼이 상호작용 가능해짐

CSR vs SSR 비교

| 특성 | CSR | SSR |

|------|-----|-----|

| 초기 HTML | 비어있음 | 완전히 렌더링됨 |

| 첫 로드 | 빈 화면 | 콘텐츠가 즉시 나타남 |

| SEO | 제한적 | 더 나음 |

| 렌더링 위치 | 브라우저 | 서버 |

결론

브라우저가 웹페이지를 렌더링하는 방식을 이해하면 프론트엔드 개발자는 다음을 할 수 있습니다:

  • 성능 최적화
  • 렌더링을 막는 문제 회피
  • 사용자 경험 개선
  • 올바른 렌더링 전략 선택

CSR은 매우 상호작용적인 애플리케이션에 좋고, SSR은 초기 로드 성능과 SEO를 개선합니다.

Next.js와 같은 현대 프레임워크는 최고의 사용자 경험을 제공하기 위해 두 접근 방식을 결합합니다.
