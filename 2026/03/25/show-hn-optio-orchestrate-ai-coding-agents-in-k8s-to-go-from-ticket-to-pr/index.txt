26:T1f12,<h2>Optio: AI 코딩 에이전트를 위한 자동화된 CI/CD</h2>
<p>Optio는 코딩 작업을 자동으로 완료하고 PR을 머지하는 시스템입니다. 인간의 개입 없이 말이죠. 작업을 한 번만 제출하면(수동 입력, GitHub Issue, 또는 Linear 티켓으로), 나머지는 Optio가 알아서 처리합니다. 격리된 환경을 준비하고, AI 에이전트를 실행하고, PR을 열고, CI를 모니터링하고, 코드 리뷰를 촉발하고, 실패를 자동으로 수정하고, 모든 게 통과하면 병합하는 식으로요.</p>
<p>Optio의 핵심은 <strong>피드백 루프</strong>입니다. CI가 실패하면 에이전트가 자동으로 재개되는데, 실패 컨텍스트를 함께 받게 됩니다. 리뷰어가 변경을 요청하면, 에이전트가 리뷰 코멘트를 읽고 수정본을 올립니다. 모든 게 통과하고 승인받으면 PR은 squash-merge되고 이슈가 닫힙니다. 당신이 해야 할 일은 작업 설명뿐이고, Optio가 완성까지 책임집니다.</p>
<h3>대시보드와 상세 보기</h3>
<p><strong>대시보드</strong>에서는 실시간으로 실행 중인 에이전트, Pod 상태, 비용, 최근 활동을 한눈에 볼 수 있습니다.</p>
<p><strong>작업 상세 보기</strong>는 에이전트 출력을 라이브 스트리밍하고, 파이프라인 진행 상황, PR 추적, 비용 분석을 보여줍니다.</p>
<h3>어떻게 동작하는가</h3>
<pre><code>작업 생성              에이전트 실행           루프 종료
───────────────────  ──────────────────────  ──────────────────────
GitHub Issue    →  Repo pod 준비      →  CI 실패?
수동 입력           git worktree 생성      → 에이전트 재개 (실패 컨텍스트 포함)
Linear 티켓        Claude Code / Codex 실행  리뷰 변경 요청?
                                          → 에이전트 재개 (피드백 포함)
                  PR 오픈 → CI 통과 + 승인?
                           → Squash-merge + 이슈 닫기</code></pre>
<p><strong>Intake(입수)</strong>: 작업은 웹 UI, GitHub Issues(원클릭 할당), 또는 Linear 티켓에서 옵니다.</p>
<p><strong>Provisioning(준비)</strong>: Optio는 저장소용 Kubernetes pod를 찾거나 생성하고, 격리를 위해 git worktree를 만듭니다.</p>
<p><strong>Execution(실행)</strong>: AI 에이전트(Claude Code 또는 OpenAI Codex)가 설정된 프롬프트, 모델, 옵션으로 돌아갑니다.</p>
<p><strong>PR 라이프사이클</strong>: Optio는 30초마다 PR을 폴링해서 CI 상태, 리뷰 상태, 머지 준비 상황을 확인합니다.</p>
<p><strong>Feedback loop(피드백 루프)</strong>: CI 실패, 머지 충돌, 리뷰 피드백이 발생하면 자동으로 에이전트가 컨텍스트와 함께 재개됩니다.</p>
<p><strong>Completion(완성)</strong>: PR은 squash-merge되고, 연결된 이슈는 닫히며, 비용이 기록됩니다.</p>
<h3>핵심 기능들</h3>
<h4>자동화된 피드백 루프</h4>
<p>이게 Optio를 다른 솔루션과 구별하는 가장 중요한 특징입니다. 에이전트를 실행하고 떠나는 게 아니라, PR을 계속 모니터링하면서 완성까지 주도합니다:</p>
<ul>
<li><strong>CI 실패 시 자동 재개</strong> — 실패한 체크의 이름과 함께 에이전트가 다시 큐에 들어갑니다</li>
<li><strong>머지 충돌 시 자동 재개</strong> — 에이전트에게 리베이스하고 force-push하도록 지시합니다</li>
<li><strong>리뷰 피드백 시 자동 재개</strong> — 리뷰어 코멘트가 재개 프롬프트로 전달됩니다</li>
<li><strong>자동 머지</strong> — CI를 통과하고 리뷰가 승인되면 PR을 squash-merge합니다</li>
<li><strong>자동 이슈 종료</strong> — 연결된 GitHub Issues는 머지된 PR 링크와 함께 닫힙니다</li>
</ul>
<h4>Pod-per-repo 아키텍처</h4>
<p>각 저장소마다 하나의 장기 실행 Kubernetes pod가 있습니다. Pod는 저장소를 한 번 클론한 후 계속 살아있게 됩니다. 각 작업은 자신만의 git worktree를 받으므로, 여러 작업이 동시에 실행되어도 서로 간섭하지 않습니다.</p>
<ul>
<li><strong>다중 pod 스케일링</strong> — 저장소는 단일 pod를 넘어 확장 가능합니다(maxPodInstances는 최대 20)</li>
<li><strong>영구 볼륨</strong> — 설치된 도구와 캐시는 pod 재시작 후에도 유지됩니다</li>
<li><strong>유휴 정리</strong> — pod는 10분간 비활성화된 후 회수됩니다(설정 가능)</li>
<li><strong>헬스 모니터링</strong> — 충돌하거나 OOM으로 죽은 pod는 자동 감지되어 재시작됩니다</li>
<li><strong>Worktree 라이프사이클</strong> — 재시도를 위한 유예 기간을 포함하여 자동으로 정리됩니다</li>
</ul>
<h4>코드 리뷰 에이전트</h4>
<p>Optio는 원본 코딩 작업의 하위 작업으로 자동으로 리뷰 에이전트를 시작할 수 있습니다:</p>
<ul>
<li>CI 통과, PR 오픈, 또는 수동으로 촉발됩니다</li>
<li>별도의 리뷰 전용 프롬프트와 모델로 실행됩니다(리뷰는 더 저렴한 모델 사용 가능)</li>
<li>차단 방식 — 부모 작업은 리뷰 완료 전까지 머지를 기다립니다</li>
<li>저장소별로 설정 가능하며 커스텀 리뷰 프롬프트 템플릿을 지원합니다</li>
</ul>
<h4>저장소별 에이전트 튜닝</h4>
<p>각 저장소는 개별적으로 설정할 수 있습니다:</p>
<ul>
<li><strong>Claude 모델</strong> — Opus 또는 Sonnet, 컨텍스트 윈도우(200k 또는 1M), thinking 온/오프, effort level</li>
<li><strong>컨테이너 이미지</strong> — 저장소 내용에서 자동 감지(Node, Python, Go, Rust, Full) 또는 커스텀 Dockerfile</li>

<li><strong>추가 패키지와 설정 명령</strong> — pod 시작 시 실행되는 apt 패키지와 셸 명령</li>
<li><strong>.optio/setup.sh</strong> — 클론 후 실행되는 저장소 레벨 설정 스크립트</li>
