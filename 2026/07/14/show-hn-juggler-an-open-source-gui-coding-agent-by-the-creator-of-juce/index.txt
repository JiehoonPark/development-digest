15:T26dc,<p>AI 코딩 에이전트가 이미 차고 넘치는 시장에 또 하나가 등장했습니다. 다만 이번엔 좀 다른 지점을 노립니다. 오디오 프로그래밍 프레임워크 JUCE의 개발자가 만든 <strong>Juggler</strong>는 &quot;LLM이 내 코드베이스에 무슨 짓을 하는지 손으로 직접 들여다보고 통제하고 싶은 사람들&quot;을 위한 도구를 표방합니다. 콘솔에 텍스트가 주르륵 흐르는 챗봇형 UI 대신, Finder의 컬럼 뷰를 닮은 제대로 된 GUI 워크벤치를 제공하는 것이 핵심입니다.</p>
<h3>Juggler가 다른 에이전트와 다른 점</h3>
<p>대부분의 코딩 에이전트는 단일 선형 트랜스크립트를 보여줍니다. 운이 좋으면 되감기 정도는 가능하죠. Juggler는 애초에 대화 구조 자체를 <strong>트리</strong>로 취급합니다. 대화의 어느 지점에서든 서브 스레드로 분기할 수 있고, 그 서브 스레드가 다시 분기할 수도 있습니다. 이 구조를 탐색·검사·편집하는 것을 UI가 직접 지원합니다.</p>
<p>이때 사용하는 UI 패턴이 Miller 컬럼입니다. macOS Finder의 컬럼 뷰를 써본 적이 있다면 바로 감이 올 텐데요, 왼쪽에 루트가 있고 항목을 선택하면 그 속성과 자식 항목들이 오른쪽으로 펼쳐지는 방식입니다. 툴 호출, 승인 여부, 스레드 구조, 아이템 속성, LLM에 전달되는 원본 컨텍스트까지 — 접혀 있는 채팅창 안에 파묻히지 않고 이 컬럼 뷰 위에 그대로 드러납니다.</p>
<p>내부적으로 세션은 단순한 대화 로그가 아니라 <strong>Yjs 문서</strong>로 관리됩니다. 그래서 서브 스레드 생성, 드릴다운, 백트래킹, 비교, 편집 같은 조작이 문서 편집하듯 자연스럽게 이루어집니다.</p>
<h3>아키텍처: 전부 플러그인, 전부 멀티 클라이언트</h3>
<p>Juggler의 코어 앱이 하는 일은 문서 관리와 오케스트레이션뿐입니다. 대화를 구성하는 거의 모든 객체는 JavaScript 확장(extension)으로 정의됩니다.</p>
<ul>
<li><strong>Context items</strong> — <code>read-file</code>, <code>replace-text</code>, <code>bash</code> 등 대화에 등장하는 모든 아이템 타입이, LLM과 어떻게 통신하고 UI에 어떻게 표시될지를 스스로 정의합니다.</li>
<li><strong>Strategies</strong> — plan, research 같은 고수준 LLM 루프도, 사용자가 직접 만든 실험적인 오케스트레이션 로직도 모두 플러그인입니다.</li>
<li><strong>Commands</strong> — <code>/clear</code>, <code>/compact</code> 같은 슬래시 커맨드 역시 세션 문서를 조작하는 플러그인일 뿐입니다.</li>
</ul>
<p>read/write/bash 같은 기본 툴조차 확장으로 정의되어 있어서 검사하거나, 포크하거나, 통째로 갈아끼울 수 있습니다. 터미널 뒤에 숨어 돌아가는 헤드리스 파이썬 스크립트로는 표현하기 어려운, 자체 UI와 시각화가 필요한 오케스트레이션 아이디어를 위한 플랫폼을 지향합니다.</p>
<p>또 하나 눈에 띄는 설계는 <strong>멀티 클라이언트 구조</strong>입니다. Juggler는 겉보기엔 네이티브 데스크톱 앱이지만, 실제로는 로컬 웹서버가 실시간 협업 세션을 서빙하는 구조입니다. 데스크톱 앱은 그 세션에 붙는 클라이언트 중 하나일 뿐이고, 브라우저 탭도 또 다른 클라이언트가 될 수 있으며, 다른 머신에서도 접속할 수 있습니다. 즉 코드가 실제로 존재하는 로컬 워크스테이션, 개발 서버, 서버 팜 등에서 서버를 돌려두고, 원하는 곳에서 뷰만 붙여서 볼 수 있다는 뜻입니다. 데스크톱 앱과 휴대폰 브라우저가 동시에 같은 라이브 세션을 보고 조작하는 것도 가능합니다.</p>
<h3>설치와 실행</h3>
<p>배포 패키지에는 항상 두 가지 구성 요소가 함께 담겨 있습니다.</p>
<ul>
<li><strong>Juggler app</strong> — 네이티브 데스크톱 앱</li>
<li><strong>juggler</strong> — 헤드리스 커맨드라인 서버. 장기 실행/원격/네트워크 접근 세션용으로, 창은 없지만 터미널에서 <code>w</code>를 입력하면 데스크톱 앱을 열 수 있고, 출력된 브라우저 URL로도 접속할 수 있습니다.</li>
</ul>
<p>플랫폼별 설치 방법:</p>
<ul>
<li><strong>macOS</strong> — <code>.dmg</code>를 열어 Applications로 드래그. 앱과 서버가 번들로 묶여 있어 서버가 자동으로 시작됩니다. Gatekeeper가 처음 실행을 막으면 앱을 우클릭 → Open → Open, 혹은 시스템 설정에서 허용하면 됩니다.</li>
<li><strong>Windows</strong> — <code>Juggler-&lt;version&gt;-setup.exe</code> 실행. 데스크톱 앱과 대응하는 <code>juggler.exe</code> 서버가 같은 디렉터리에 함께 설치되어 (PATH 등록도 가능) 두 컴포넌트가 버전 불일치를 일으키지 않습니다.</li>
<li><strong>Linux</strong> — <code>juggler</code> 서버 바이너리를 받아 터미널에서 실행하고, 브라우저나 데스크톱 앱으로 접속합니다.</li>
</ul>
<p>헤드리스 세션은 다음 한 줄로 충분합니다.</p>
<pre><code>juggler # prints everything you need to connect from a browser</code></pre>
<p>기본적으로 웹 UI URL과 QR코드를 출력해줍니다. 서버는 기본값이 <strong>localhost 전용</strong>이라 외부 기기에서 접근할 수 없습니다. 같은 네트워크의 다른 기기가 연결하려면 터미널에서 <code>p</code>를 누르거나 <code>--public</code> 옵션으로 실행하면 되는데, LAN 접근에는 비밀번호가 없으므로 신뢰할 수 있는 네트워크에서만 켜라고 문서는 경고합니다. 참고로 이 저장소에서 빌드한 버전은 로컬+LAN 범위까지만 지원하며, juggler.studio에서 배포하는 공식 바이너리에는 인터넷 너머로 접속하는 WAN 접근 모드가 추가로 포함되어 있습니다(해당 부분은 이 저장소에 포함되지 않으며 LICENSING.md에 별도 명시).</p>
<h3>모델 지원 및 소스 빌드</h3>
<p>연결 가능한 모델/제공자는 Claude Code(CLI 또는 API), OpenAI(codex plan 또는 API), Gemini, Ollama, OpenRouter, Z.AI, Deepseek 등입니다. 새 프로바이더 추가도 어렵지 않아서, 원하는 LLM에게 PR을 만들어달라고 시키면 될 정도라고 소개하고 있습니다.</p>
<p>소스 빌드는 다음과 같습니다.</p>
<pre><code>git clone --recurse-submodules https://github.com/juggler-ai/juggler.git
cd juggler &amp;&amp; make build</code></pre>
<p>이미 <code>--recurse-submodules</code> 없이 클론했다면:</p>
<pre><code>git submodule update --init --recursive</code></pre>
<p>Windows 바이너리는 어떤 호스트에서도 <code>make build-windows</code>로 크로스 컴파일이 가능하지만, Linux 데스크톱 앱은 네이티브 빌드가 필요합니다.</p>
<h3>기술 스택</h3>
<p>Juggler는 군더더기 없는 네이티브 앱을 지향합니다.</p>
<ul>
<li>백엔드: <strong>Go</strong> + <strong>Wails</strong>(윈도잉)</li>
<li>UI: Go 백엔드가 서빙하는 <strong>HTML/JS</strong></li>
<li>세션 문서 저장·동기화: <strong>Yjs</strong></li>
<li>확장: <strong>JavaScript</strong></li>
<li>Electron 미사용</li>
</ul>
<p>프런트엔드는 TypeScript가 아니라 <strong>타입 체크된 JavaScript</strong>로 작성됩니다. 타입 정보는 JSDoc에 담기고, CI에서 엄격한 정적 린팅으로 검증합니다. 소스와 배포물 사이에 별도의 빌드 단계가 없다는 점도 특징입니다.</p>
<h3>라이선스</h3>
<p>애플리케이션 코드는 <strong>AGPL v3.0 이상</strong>으로 배포됩니다. 반면 확장 SDK(<code>web/sdk/</code>)와 번들 확장(<code>web/extensions/</code>)은 <strong>Apache-2.0</strong>이라서, 클로즈드소스 확장을 포함해 카피레프트 의무 없이 확장을 만들 수 있습니다. AGPL 부분은 자유롭게 사용·수정·재배포할 수 있지만, 수정본을 배포하거나 서비스로 호스팅하려면 그 버전 역시 AGPLv3로 공개해야 합니다. 클로즈드소스로 활용하고 싶다면 저작자에게 연락해 상업용 라이선스를 논의해야 합니다.</p>
<h3>정리</h3>
<ul>
<li>Juggler는 콘솔 기반 챗봇형 에이전트가 아니라, Finder 스타일 Miller 컬럼 뷰를 갖춘 <strong>제대로 된 GUI 코딩 에이전트</strong>입니다. 툴 호출, 승인, 스레드 구조, 아이템 속성, 원본 컨텍스트가 모두 화면에 드러납니다.</li>
<li>대화 세션을 선형 트랜스크립트가 아닌 <strong>Yjs 기반 트리 문서</strong>로 다뤄, 서브 스레드 분기·드릴다운·백트래킹·비교·편집이 자유롭습니다.</li>
<li>컨텍스트 아이템, 전략(strategy), 슬래시 커맨드까지 전부 <strong>JavaScript 확장</strong>으로 정의되어 검사·포크·교체가 가능하고, read/write/bash 같은 기본 툴조차 예외가 아닙니다.</li>
