via React Blog
🔥오늘의 화제frontend

React Server Function 엔드포인트의 페이로드 디코딩 결함으로 인해 인증 없이 원격 코드 실행이 가능한 CVSS 10.0 등급의 취약점(CVE-2025-55182)이 발견되었습니다. react-server-dom-webpack, react-server-dom-parcel, react-server-dom-turbopack의 19.0~19.2.0 버전에서 발생하며, Next.js·React Router·Waku·rwsdk 등 RSC 지원 프레임워크 전반이 영향을 받습니다. React 팀은 각 릴리스 라인별 패치 버전을 공개했으며, 이후 DoS 및 소스 코드 노출 취약점도 추가로 확인되어 지속적인 업데이트가 안내되고 있습니다.

💡
RSC 기반 프레임워크(Next.js 등)를 프로덕션에서 운영 중인 프론트엔드 개발자라면 CVSS 10.0의 RCE 취약점에 즉시 대응하지 않을 경우 서비스 전체가 심각한 보안 위협에 노출될 수 있습니다.

AI와 코딩할 때 완벽한 스펙을 만드는 데 집착하는 대신, 실제로 동작하는 프로토타입을 만들어 논의의 피델리티를 높이는 방법을 소개합니다. Matt Pocock은 자신의 Wayfinder 스킬 안에 프로토타입 스킬을 두어, '어떻게 보여야 하는가/어떻게 동작해야 하는가'가 핵심 질문일 때 값싼 일회용 코드를 만들어 반응하고 반복하는 워크플로우를 제안합니다. TLDraw 기반 다이어그램 앱의 검색 기능 프로토타이핑 사례를 통해 실제 적용 과정을 보여주며, 이 방식이 UI뿐 아니라 백엔드 로직과 상태 머신 검증에도 유효함을 설명합니다.

💡

line-height, 숫자와 퍼센트는 이렇게 다르게 동작합니다
via Kevin Powell (CSS)
▶영상frontend

line-height에 unitless 숫자(1.5)와 퍼센트(150%)를 쓰는 것이 겉보기엔 비슷해 보이지만, 상속되는 방식은 완전히 다릅니다. unitless 숫자는 값 자체가 상속되어 각 요소가 자신의 font-size로 매번 재계산하지만, 퍼센트는 선언 지점에서 고정된 px 값으로 계산되어 그대로 상속됩니다. 개발자 도구로 계산값을 비교해보면 폰트 크기가 다른 요소에서 이 차이가 명확히 드러납니다. 특히 fluid font-size를 쓰는 프로젝트에서는 이 차이가 실제 레이아웃 문제로 이어질 수 있습니다.

💡
line-height 단위 선택은 사소해 보이지만 반응형·fluid 타이포그래피 환경에서 실제 레이아웃 버그를 만들 수 있어, CSS 기본기를 다시 점검해야 하는 실무 포인트입니다.
