15:T4662,<p>AI 코딩 에이전트를 프로덕션에 도입하다 보면 &quot;이 도구가 실제로 API에 얼마나 많은 토큰을 보내는가&quot;라는 질문에 부딪히게 됩니다. 특히 EU AI Act처럼 시스템 동작에 대한 로깅과 이해를 요구하는 규제 환경에서는, 이 질문에 감으로 답할 수 없습니다. 이 글에서는 Claude Code와 OpenCode 두 에이전트 하네스를 동일한 모델, 동일한 머신, 동일한 태스크에서 돌리고, API 경계에서 오가는 모든 페이로드를 실측한 결과를 정리합니다.</p>
<h3>왜 이걸 측정해야 하는가</h3>
<p>에이전트 하네스가 보내는 토큰 오버헤드는 곧 비용이자 지연 시간이며, 무엇보다 컨텍스트 예산입니다. 하네스가 매 요청마다 실어 보내는 토큰 하나하나는 실제 코드 작업에 쓸 수 없는 컨텍스트입니다. 게다가 이 기본 페이로드는 세션의 모든 턴마다 다시 전송되거나(또는 캐시에서 다시 읽히거나) 합니다.</p>
<p>프로덕션에서 에이전트형 AI를 운영한다면, &quot;내 에이전트가 실제로 무엇을 보내는가&quot;는 데이터로 답해야 할 질문이지 소문으로 넘길 문제가 아닙니다.</p>
<h3>측정 방법</h3>
<p>각 하네스와 모델 엔드포인트 사이에 로깅 프록시를 끼워 넣어 측정했습니다.</p>
<pre><code>harness (Claude Code / OpenCode)
→ logging proxy (captures request payloads + response usage)
→ model endpoint</code></pre>
<p>프록시는 요청마다 두 가지를 기록합니다. 하나는 하네스가 실제로 보낸 JSON 페이로드 원문(시스템 블록, 툴 스키마, 메시지)이고, 다른 하나는 API가 돌려준 usage 블록(입력 토큰, 캐시 쓰기, 캐시 읽기, 출력 토큰)입니다. 페이로드 캡처는 하네스가 실제로 무엇을 보내는지에 대한 그라운드 트루스이고, usage 블록은 실제로 과금된 내용에 대한 그라운드 트루스입니다.</p>
<p>테스트 조건은 다음과 같습니다.</p>
<ul>
<li><strong>하네스</strong>: Claude Code 2.1.207과 OpenCode 1.17.18, 둘 다 2026년 7월 기준 claude-sonnet-4-5에 고정. 이후 축소된 매트릭스(기본 오버헤드, 캐시 태스크, 멀티스텝 태스크)를 claude-fable-5로 다시 실행했고, 결과가 달라진 부분은 본문에 별도 표기했습니다.</li>
<li><strong>베이스라인 격리</strong>: MCP 서버 없음, 사용자 설정 없음, 메모리 없음의 신선한 설정 디렉터리, 인스트럭션 파일이 없는 빈 워크스페이스, 권한 확인 우회. 이후 배율(multiplier) 레인에서 변수를 하나씩 추가했습니다.</li>
<li><strong>태스크</strong>: T1은 &quot;Reply with exactly: OK&quot;로 고정 오버헤드를 격리(하네스당 3회 실행)합니다. T2는 미리 준비된 파일을 읽고 요약합니다. T3는 FizzBuzz와 체커 스크립트를 대상으로 작성-실행-테스트-수정 루프를 돌립니다.</li>
<li><strong>툴 제거 변형</strong>: Claude Code는 <code>--tools &quot;&quot;</code>, OpenCode는 <code>&quot;tools&quot;: {&quot;*&quot;: false}</code>로 설정해 시스템 프롬프트와 툴 스키마의 무게를 분리했습니다.</li>
</ul>
<p>한 가지 밝혀둘 점은, 트래픽이 로컬 LLM 게이트웨이를 거치면서 자체 엔벨로프로 요청을 감싸는데, 이 상수 오버헤드를 순수 캘리브레이션 요청으로 약 6,200 토큰으로 측정해 아래의 모든 과금 수치에서 제외했다는 것입니다. 반면 페이로드 레벨 수치는 게이트웨이가 손댈 수 없는 캡처된 요청 본문에서 나온 것이라 정확합니다. 컴포넌트 추정을 위한 문자-토큰 변환은 각 하네스별로 실측한 문자당 4.1~4.4 토큰 비율을 사용했으며, 이는 일반적인 휴리스틱이 아니라 콜드 캐시 앵커(과금된 쓰기가 전체 페이로드와 일치하는 지점)에서 도출한 값입니다.</p>
<h3>Part I. 기본 오버헤드(The floor)</h3>
<h4>&quot;OK&quot;라고만 답하는 데도 드는 고정 비용</h4>
<p>22자짜리 태스크에서 첫 요청에 각 하네스가 무엇을 실어 보냈는지 정리하면 다음과 같습니다.</p>
<div class="table-wrapper"><table><thead><tr><th>구성 요소</th><th>Claude Code</th><th>OpenCode</th></tr></thead><tbody><tr><td>시스템 프롬프트</td><td>27,344자, 3개 블록</td><td>9,324자, 1개 블록</td></tr><tr><td>툴 스키마</td><td>27개 툴, 99,778자</td><td>10개 툴, 20,856자</td></tr><tr><td>첫 메시지 스캐폴딩</td><td><code>&lt;system-reminder&gt;</code> 블록 7,997자</td><td>없음</td></tr><tr><td>실제 프롬프트</td><td>22자</td><td>22자</td></tr><tr><td>첫 턴 페이로드(보정치)</td><td>약 32,800 토큰</td><td>약 6,900 토큰</td></tr></tbody></table></div>
<p>OpenCode의 요청은 거의 최소한에 가깝습니다. &quot;You are OpenCode, the best coding agent on the planet&quot;으로 시작하는 시스템 블록 하나, 전형적인 코딩 툴 10개, 그리고 사용자 프롬프트가 유일한 사용자 콘텐츠입니다.</p>
<p>반면 Claude Code의 요청은 플랫폼 부트스트랩에 가깝습니다. 27개 툴에는 코딩 핵심 기능 외에 백그라운드 에이전트와 오케스트레이션 전체 스위트가 포함되어 있는데, CronCreate와 Monitor, Task 계열, 워크트리 관리, 푸시 알림까지 들어갑니다. 프롬프트가 도달하기도 전에 첫 사용자 메시지에는 위임 가능한 에이전트 종류 목록, 사용 가능한 스킬 카탈로그, 사용자 컨텍스트라는 세 개의 인젝션 블록이 딸려옵니다.</p>
<p>두 하네스 모두 툴 스키마가 오버헤드의 지배적인 항목입니다. Claude Code의 약 33,000 토큰 중 약 24,000 토큰이 툴 정의이고, OpenCode의 약 6,900 토큰 중에서는 약 4,800 토큰이 여기에 해당합니다.</p>
<h4>툴을 다 빼고 순수 하네스만 남기면</h4>
<p>툴을 제거하면 시스템 프롬프트 자체의 무게만 남습니다. Claude Code는 26,891자, 약 6.5k 토큰이고, OpenCode는 8,811자, 약 2.0k 토큰입니다.</p>
<p>두 하네스 모두 툴을 비활성화하면 프롬프트가 약간 줄어듭니다. 하지만 툴을 완전히 제거해도 Claude Code의 인스트럭션 세트는 OpenCode의 3배가 넘습니다. 이 잔여분은 톤 규칙, 안전 가이드, 태스크 관리 지침, 환경 설명 등의 행동 규범(behavioural doctrine)입니다.</p>
<h4>툴 하나짜리 태스크</h4>
<p>T2는 각 하네스에 파일을 읽고 요약하도록 요청한 태스크입니다. 둘 다 정답을 냈습니다.</p>
<p>Claude Code는 HTTP 요청 6회, 누적 과금 입력 토큰 약 199,000을 사용했습니다. OpenCode는 요청 4회, 약 41,000 토큰에 세션 제목을 위한 Haiku 사이드콜 1회가 추가됐습니다.</p>
<p>이 토큰 대부분은 캐시 읽기로, 일반 입력가의 10분의 1로 과금됩니다. 하지만 캐시 할인과 무관하게 페이로드 크기에 비례하는 세 가지가 있습니다. 첫 턴 캐시 쓰기, 턴마다의 읽기, 그리고 어떤 캐시 할인으로도 줄지 않는 컨텍스트 윈도우 소비입니다.</p>
<p>33k 토큰짜리 베이스라인이라는 것은, 코드 한 줄이 대화에 들어가기도 전에 이미 200k 윈도우의 6분의 1 지점에서 시작한다는 뜻입니다.</p>
<h4>멀티스텝 태스크에서는 격차가 좁혀진다</h4>
<p>작성-실행-테스트-수정 루프인 T3는 앞선 베이스라인이 만든 기대를 뒤집었습니다.</p>
<div class="table-wrapper"><table><thead><tr><th>지표</th><th>Claude Code</th><th>OpenCode</th></tr></thead><tbody><tr><td>모델 요청 수</td><td>3</td><td>9 (+1 제목 호출)</td></tr><tr><td>툴 호출 방식</td><td>한 라운드트립에 병렬 배치</td><td>턴마다 툴 호출 1회</td></tr><tr><td>누적 과금 입력</td><td>약 121,000 토큰</td><td>약 132,000 토큰</td></tr></tbody></table></div>
<p>Claude Code는 파일 쓰기 2회와 스크립트 실행 2회로 이루어진 작업 전체를 하나의 병렬 툴 라운드트립으로 묶었습니다. OpenCode는 턴마다 정확히 툴 호출 하나씩, 총 9턴을 거쳤습니다.</p>
<p>베이스라인이 요청마다 다시 전송되는 구조이다 보니, 요청 수가 곧 베이스라인의 배수가 됩니다. OpenCode는 약 7k 베이스라인을 9번 지불했고, Claude Code는 약 33k 베이스라인을 3번 지불했으며, 결과적으로 총합이 수렴했습니다.</p>
<p>전체 태스크의 입력 토큰은 대략 &quot;베이스라인 × 요청 수 + 대화 증가분&quot;으로 근사됩니다. 베이스라인이 크지만 공격적으로 배치하는 하네스와, 베이스라인이 작지만 순차 처리하는 하네스는 결국 비슷한 지점에 도달할 수 있습니다.</p>
<p>페이로드에서 두 가지 구조적 특징도 드러났습니다. Claude Code는 대화가 진행될수록 <code>&lt;system-reminder&gt;</code> 블록을 추가로 주입하는데, 첫 턴에 3개였다가 첫 툴 라운드트립 시점에는 4개로 늘어나 스캐폴딩이 턴 수에 비례해 커집니다. 반면 OpenCode의 턴당 한계 페이로드(턴당 약 400~2,200자)는 순수한 대화 콘텐츠뿐입니다.</p>
<h4>최신 모델이면 그림이 달라질까</h4>
<p>Sonnet 특유의 현상인지 확인하기 위해 Claude Fable 5로 기본 오버헤드를 재측정했습니다. 격차는 줄었지만, 예상 못한 이유였습니다.</p>
<p>Claude Code의 시스템 프롬프트는 모델에 따라 달라집니다. Sonnet에는 27,787자의 인스트럭션을 보냈지만 Fable에는 10,526자만 보냈고, 툴 스키마도 99,778자에서 82,283자로 줄었습니다. 같은 27개 툴인데 규범 텍스트가 훨씬 적었던 것입니다.</p>
<p>반면 OpenCode의 페이로드는 두 모델에서 바이트 단위로 동일했습니다.</p>
<p>결과적으로 Fable 기준 기본 오버헤드 격차는 페이로드 기준 약 3.3배로, Sonnet의 4.7배보다 좁혀졌습니다. 여전히 훨씬 더 많이 먹지만, 배율 자체는 모델에 따라 달라진다는 점이 확인됩니다.</p>
<h3>Part II. 배율 요인들(The multipliers)</h3>
<p>기본 오버헤드는 세션이 가볍게 시작해서 짧게 끝나는 경우를 설명할 뿐입니다. 실제 세션은 둘 다 아닌 경우가 대부분입니다. 여기서는 실제 사용 시 쌓이는 각 레이어를 측정했습니다.</p>
<h4>배율 1. 인스트럭션 파일</h4>
<p>프로덕션 저장소에서 가져온 실제 72KB짜리 AGENTS.md를 워크스페이스에 넣고 T1을 다시 실행했습니다.</p>
<p>효과는 대칭적이면서도 컸습니다. 두 하네스 모두 요청당 20,000 토큰을 조금 넘게 더 소모했습니다. OpenCode의 과금 총합은 13,152에서 33,336으로, Claude Code는 39,005에서 59,243으로 늘어났습니다.</p>
<p>비대칭은 메커니즘 쪽에 있었고, 실험 도중 실제로 발목을 잡았습니다. Claude Code 2.1.207은 AGENTS.md를 완전히 무시했고, CLAUDE.md로 이름을 바꿔야만 파일을 읽어들여 첫 사용자 메시지에 주입했습니다. OpenCode는 두 파일명 중 어느 것이든 읽어서 시스템 프롬프트에 주입합니다.</p>
<p>여기서 실무적으로 얻을 교훈은 두 가지입니다. 사용 중인 하네스가 실제로 어떤 파일명을 인식하는지 확인해야 합니다. 무시된 인스트럭션 파일은 조용히 실패하기 때문입니다. 그리고 무거운 인스트럭션 파일은 가벼운 하네스의 베이스라인을 거의 4배로 불려버릴 수 있으며, 이 부담은 해당 저장소의 모든 세션, 모든 요청에 실립니다.</p>
<h4>배율 2. MCP 서버</h4>
<p>공개된, 자격 증명이 필요 없는 MCP 서버를 1개, 5개 구성으로 붙여봤습니다.</p>
<p>스키마 자체는 두 하네스에서 동일하므로 부담도 거의 동일합니다. 작은 서버 하나당 요청마다 약 1,000~1,400 토큰이 추가됩니다. 서버 5개를 붙였을 때 Claude Code는 페이로드 기준 4,900 토큰, OpenCode는 과금 기준 6,967 토큰이 늘었고, 툴 개수는 각각 27개→69개, 10개→52개로 증가했습니다.</p>
<p>작은 공개 서버는 그나마 온건한 경우입니다. 풍부한 API를 가진 프로덕션 서버는 이보다 몇 배 큰 스키마를 실어 보내는데, 이는 아래에서 다룰 &quot;everything&quot; 측정에서 그대로 드러납니다.</p>
<p>운영상 짚고 넘어갈 점 하나: Claude Code는 print 모드에서 프로젝트 스코프의 <code>.mcp.json</code>을 명시적으로 <code>--mcp-config</code> 플래그를 넘기기 전까지는 조용히 무시했습니다. 서버가 붙었다고 가정하지 말고, 경계에서 직접 확인해야 합니다.</p>
<h4>배율 3. 프레임워크 템플릿</h4>
<p>BMAD 같은 스토리 기반 워크플로우 프레임워크는 슬래시 커맨드 하나를 페르소나, 프로토콜, 체크리스트로 이루어진 큰 프롬프트 템플릿으로 확장합니다.</p>
<p>8,405자짜리 대표 템플릿을 두 하네스에서 동일한 T3 스토리의 프롬프트로 실행했습니다. 템플릿 자체는 약 2,100 토큰에 불과하지만, 이것이 대화 히스토리에 들어가고 나면 이후 모든 요청에서 계속 함께 실려갑니다. 요청 9회짜리 세션이면 이 템플릿을 9번 다시 보내는 셈입니다.</p>
<p>프레임워크 부담은 &quot;템플릿 크기 × 요청 수&quot;로 계산되며, 앞서 다룬 모든 배율 위에 그대로 누적됩니다.</p>
<h4>배율 4. 서브에이전트</h4>
<p>각 하네스에 작업을 두 개의 병렬 서브에이전트로 분배하도록 요청했습니다. 여기서 총합이 가장 가파르게 치솟았습니다.</p>
<p>Claude Code는 세 가지 뚜렷한 요청 클래스에 걸쳐 총 9회의 모델 요청으로 작업을 완료했습니다. 약 33k 베이스라인을 가진 메인 세션 하나와, 각각 3,554자짜리 에이전트 시스템 프롬프트와 27개 툴 중 24개를 실어 나르는 서브에이전트 호출 5회입니다.</p>
<p>누적 과금 입력은 513,000 토큰에 달했는데, 같은 작업을 직접 수행했을 때의 121,000 토큰과 비교하면 4.2배입니다. 서브에이전트 하나하나가 자신만의 부트스트랩 비용을 지불하고, 그 결과물을 부모 세션이 다시 트랜스크립트로 흡수하기 때문에 벌어지는 일입니다.</p>
<p>OpenCode는 이 지점에서 훨씬 가벼운 설계를 보였습니다. 서브에이전트 요청은 1,379자짜리 시스템 프롬프트와 툴 5개로 축소된 프로필을 실어 날랐습니다. (다만 OpenCode의 서브에이전트 실행은 게이트웨이를 통해 완결되지 못했고, 원문에서는 이 설계상의 차이를 보고하는 선에서 끝맺고 있습니다.)</p>
<h3>정리</h3>
<ul>
<li><strong>기본 오버헤드부터 격차가 큽니다.</strong> 한 줄 응답을 요청해도 Claude Code는 프롬프트 도착 전에 시스템 프롬프트, 툴 스키마, 인젝션된 스캐폴딩만으로 약 33,000 토큰을 소비했고, OpenCode는 약 7,000 토큰이었습니다. 이 격차의 대부분은 툴 스키마(Claude Code 약 24,000 vs OpenCode 약 4,800)에서 나옵니다.</li>
<li><strong>격차는 모델에 따라 달라집니다.</strong> Sonnet 4.5에서는 약 4.7배였던 격차가 Fable 5에서는 약 3.3배로 좁혀졌는데, Claude Code가 신모델에는 더 짧은 시스템 프롬프트를 보내기 때문입니다. OpenCode의 페이로드는 모델과 무관하게 동일했습니다.</li>
<li><strong>캐시 효율도 크게 차이납니다.</strong> OpenCode의 요청 프리픽스는 실행마다 바이트 단위로 동일해 세션당 한 번만 캐시에 쓰면 됐지만, Claude Code는 세션 중간에도 수만 토큰의 프롬프트 캐시를 반복해서 다시 쓰며, 같은 작업에서 OpenCode보다 최대 54배 많은 캐시 토큰을 썼습니다. 캐시 쓰기는 프리미엄 요금이 붙기 때문에 이는 곧바로 청구 대시보드 수치로 이어집니다.</li>
<li><strong>설정 파일과 MCP 서버가 부담을 더 키웁니다.</strong> 72KB짜리 인스트럭션 파일(AGENTS.md/CLAUDE.md)만으로 요청당 평균 20,000 토큰이 추가되고, MCP 서버 5개를 붙이면 5,000~7,000 토큰이 더해집니다. 실제 작업 환경에서는 사용자가 아무것도 입력하기 전에 이미 75,000~85,000 토큰 깊이에 도달합니다.</li>
<li><strong>서브에이전트로 분산하면 비용이 급증합니다.</strong> 직접 수행하면 121,000 토큰이었던 작업이 서브에이전트 2개로 분산하면 513,000 토큰(4.2배)까지 늘었는데, 서브에이전트마다 자체 부트스트랩 비용을 지불하고 부모 세션이 그 결과를 다시 흡수하기 때문입니다.</li>
<li><strong>다만 멀티스텝 작업에서는 예외가 나타났습니다.</strong> Claude Code가 툴 호출을 하나의 요청으로 배치하는 반면 OpenCode는 매 턴 재요청하기 때문에, 요청 수가 늘어나는 작업에서는 오히려 Claude Code의 전체 토큰 총합이 더 낮게 나온 경우도 있었습니다. 즉 세션이 어떻게 전개되느냐에 따라 승자가 바뀔 수 있습니다.</li>
</ul>
