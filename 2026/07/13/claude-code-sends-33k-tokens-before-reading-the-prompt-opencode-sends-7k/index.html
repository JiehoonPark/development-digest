Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k

📅 2026-07-13📰 Hacker News (Top)
aifrontend

AI 코딩 에이전트를 프로덕션에 투입하는 팀이라면 한 번쯤 궁금했을 질문이 있습니다. "내 에이전트는 실제로 얼마나 많은 토큰을 보내고 있을까?" Claude Code와 OpenCode를 같은 모델, 같은 머신, 같은 태스크에 놓고 API 경계에서 오가는 모든 요청과 응답을 직접 캡처해 비교한 실험이 있어 소개합니다. 결과는 단순한 "어느 쪽이 싸다"는 이야기가 아니라, 하네스 설계가 토큰 비용·지연시간·컨텍스트 예산에 어떻게 구조적으로 영향을 미치는지를 보여줍니다.

왜 이걸 측정해야 하는가

하네스가 보내는 페이로드의 모든 토큰은 곧 비용이자 지연시간이며, 무엇보다 코드에 쓸 수 없는 컨텍스트 예산의 손실입니다. 게다가 이 베이스라인은 캐시로 읽히든 새로 쓰이든 매 턴마다 반복해서 전송됩니다.

에이전틱 AI를 프로덕션에서 운영한다면, 특히 EU AI Act Article 12가 시스템 동작에 대한 로깅과 이해를 요구하는 상황이라면, "내 에이전트가 실제로 무엇을 보내는가"는 감으로 답할 게 아니라 데이터로 답해야 할 질문입니다.

측정 방법

각 하네스와 모델 엔드포인트 사이에 로깅 프록시를 끼워 넣었습니다.

harness (Claude Code / OpenCode)
→ logging proxy (요청 페이로드 + 응답 usage 캡처)
→ model endpoint

프록시는 요청당 두 가지를 기록합니다. 하나는 하네스가 실제로 내보낸 JSON 페이로드(시스템 블록, 툴 스키마, 메시지)이고, 다른 하나는 API가 반환한 usage 블록(입력 토큰, 캐시 쓰기, 캐시 읽기, 출력 토큰)입니다. 페이로드 캡처는 하네스가 무엇을 보내는지에 대한 그라운드 트루스이고, usage 블록은 실제로 과금된 값에 대한 그라운드 트루스입니다.

테스트 조건은 다음과 같습니다.

  • 하네스: Claude Code 2.1.207과 OpenCode 1.17.18, 둘 다 claude-sonnet-4-5에 고정(2026년 7월 기준). 이후 핵심 매트릭스(플로어, 캐시 태스크, 멀티스텝 태스크)만 claude-fable-5로 재실행했고, 결과가 달라진 부분은 본문에 별도로 명시합니다.
  • 베이스라인 격리: 신규 config 디렉토리, MCP 서버 없음, 유저 설정 없음, 메모리 없음, 인스트럭션 파일이 없는 빈 워크스페이스, 권한 우회. 이후 멀티플라이어 실험은 변수를 하나씩 추가합니다.
  • 태스크: T1은 "Reply with exactly: OK"로 고정 오버헤드만 격리(하네스당 3회 실행). T2는 시드 파일을 읽고 요약. T3는 FizzBuzz 작성-실행-테스트-수정 루프에 체커 스크립트를 결합.
  • 툴 없음 변형: Claude Code는 --tools "", OpenCode는 "tools": {"*": false}로 시스템 프롬프트와 툴 스키마의 무게를 분리해서 측정.

측정의 정직성을 위해 밝히자면, 트래픽은 로컬 LLM 게이트웨이를 거치며 자체 envelope를 씌우는데, 이 오버헤드는 순수 캘리브레이션 요청 기준 약 6,200토큰으로 측정되어 이하 모든 과금 수치에서 차감했습니다. 페이로드 레벨 수치는 게이트웨이가 손댈 수 없는 캡처된 요청 본문에서 나온 것이라 정확합니다. 컴포넌트 추정에 쓴 문자-토큰 변환은 일반적인 휴리스틱이 아니라, 콜드 캐시 기준점(과금된 캐시 쓰기가 전체 페이로드와 일치하는 지점)에서 도출한 각 하네스 고유의 문자당 토큰 비율(4.1~4.4 char/token)을 사용했습니다.

1부: 바닥선(Floor) — "OK"만 답할 때도 드는 비용

22자짜리 태스크에 각 하네스가 첫 요청에서 무엇을 함께 보냈는지 정리하면 다음과 같습니다.

구성 요소Claude CodeOpenCode
시스템 프롬프트27,344자, 3블록9,324자, 1블록
툴 스키마27개 툴, 99,778자10개 툴, 20,856자

첫 턴 페이로드(과금 기준)약 32,800토큰약 6,900토큰

OpenCode의 요청은 거의 최소치에 가깝습니다. "You are OpenCode, the best coding agent on the planet"으로 시작하는 시스템 블록 하나, 전통적인 코딩 툴 10개, 그리고 프롬프트가 유일한 유저 콘텐츠입니다.

반면 Claude Code의 요청은 사실상 플랫폼 부트스트랩입니다. 27개 툴에는 코딩 코어 외에도 백그라운드 에이전트 및 오케스트레이션 스위트 전체가 포함됩니다. CronCreate, Monitor, Task 계열, worktree 관리, 푸시 알림까지 들어갑니다. 게다가 프롬프트보다 앞서 첫 유저 메시지에 세 가지 주입 블록—위임 가능한 에이전트 타입 카탈로그, 사용 가능한 스킬 카탈로그, 유저 컨텍스트—이 실려 있습니다.

양쪽 모두에서 툴 스키마가 지배적인 항목입니다. Claude Code의 약 33,000토큰 중 대략 24,000토큰이 툴 정의이고, OpenCode는 약 6,900토큰 중 대략 4,800토큰이 툴 정의입니다.

툴을 다 빼면 시스템 프롬프트만 남는다

툴을 제거하면 시스템 프롬프트 자체의 무게가 드러납니다. Claude Code는 26,891자(약 6.5k토큰), OpenCode는 8,811자(약 2.0k토큰)입니다. 툴을 비활성화하면 양쪽 다 프롬프트가 약간 줄어드는데, 그럼에도 Claude Code의 인스트럭션 세트는 툴이 하나도 없는 상태에서도 OpenCode의 3배가 넘습니다. 남는 것은 톤 규칙, 안전 가이드, 태스크 관리 지침, 환경 설명 같은 "행동 강령"입니다.

툴 하나짜리 태스크

T2는 파일을 읽고 요약하는 태스크였고, 둘 다 정확한 요약을 냈습니다. Claude Code는 HTTP 요청 6회에 누적 과금 입력 토큰 약 199,000, OpenCode는 요청 4회에 약 41,000(+세션 제목 지정을 위한 Haiku 사이드콜 1회)이었습니다.

이 토큰 대부분은 캐시 읽기라 정가의 10분의 1로 과금됩니다. 그러나 규모와 무관하게 세 가지는 그대로 남습니다. 첫 턴의 캐시 쓰기, 매 턴의 캐시 읽기, 그리고 어떤 캐시 할인으로도 줄지 않는 컨텍스트 윈도우 소비입니다. 33k 토큰짜리 베이스라인이라면, 코드가 대화에 한 줄도 들어가기 전에 이미 200k 윈도우의 6분의 1을 써버린 셈입니다.

멀티스텝 태스크에서는 격차가 좁혀진다

T3(작성-실행-테스트-수정 루프)는 예상을 뒤집었습니다.

지표Claude CodeOpenCode
모델 요청 수3(+제목 콜 1회)9
툴 호출 방식한 라운드트립에 병렬 배치턴마다 툴 호출 1회
누적 과금 입력약 121,000토큰약 132,000토큰

Claude Code는 파일 작성 2회, 스크립트 실행 2회로 이루어진 전체 작업을 하나의 병렬 툴 라운드트립에 묶었습니다. OpenCode는 턴마다 툴 호출을 정확히 하나씩 해서 9번의 턴을 거쳤습니다.

베이스라인이 매 요청마다 재전송되는 구조이다 보니, 요청 횟수가 곧 베이스라인의 배수가 됩니다. OpenCode는 자신의 ~7k 베이스라인을 9번 지불했고, Claude Code는 ~33k 베이스라인을 3번 지불해서 총합이 비슷해졌습니다.

전체 작업의 입력 토큰은 대략 "베이스라인 × 요청 수 + 대화 성장분"으로 근사됩니다. 베이스라인이 크지만 공격적으로 배치하는 하네스와, 베이스라인이 작지만 직렬로 처리하는 하네스가 결국 같은 지점에 도달할 수 있다는 뜻입니다.

페이로드에서 흥미로운 구조적 차이도 발견됐습니다. Claude Code는 대화가 진행될수록 <system-reminder> 블록을 추가로 주입합니다. 첫 턴에 3개였던 것이 첫 툴 라운드트립 시점엔 4개가 되며, 스캐폴딩 자체가 턴 수에 비례해 커집니다. 반면 OpenCode의 턴당 증분 페이로드(약 400~2,200자)는 순수한 대화 내용뿐입니다.

최신 모델이면 격차가 줄어들까?

이 격차가 Sonnet 특유의 현상인지 확인하기 위해 Claude Fable 5로 플로어 테스트를 재실행했습니다. 격차는 줄었는데, 예상 밖의 이유였습니다.

Claude Code의 시스템 프롬프트는 모델에 따라 달라집니다. Sonnet에는 27,787자의 인스트럭션을 보냈지만 Fable에는 10,526자만 보냈고, 툴 스키마도 99,778자에서 82,283자로 줄었습니다. 27개 툴 개수는 동일한데 행동 강령의 분량 자체가 크게 줄어든 것입니다.

반면 OpenCode의 페이로드는 두 모델에서 바이트 단위로 완전히 동일했습니다.

결과적으로 Fable에서의 플로어 격차는 페이로드 기준 약 3.3배로, Sonnet의 4.7배보다 좁혀집니다. 여전히 훨씬 더 무겁지만, 그 배율은 모델에 따라 달라진다는 점이 확인됐습니다.

2부: 멀티플라이어 — 실제 세션에서 쌓이는 비용들

플로어 수치는 세션이 짧고 가볍게 유지될 때의 이야기입니다. 실제 세션은 둘 다 아닙니다. 실사용에서 겹겹이 쌓이는 각 레이어를 하나씩 측정했습니다.

멀티플라이어 1: 인스트럭션 파일

프로덕션 레포지토리에서 가져온 실제 72KB AGENTS.md 파일을 워크스페이스에 넣고 T1을 재실행했습니다.

효과는 대칭적이면서도 큽니다. 두 하네스 모두 요청당 20,000토큰이 넘게 늘었습니다. OpenCode의 과금 총량은 13,152에서 33,336으로, Claude Code는 39,005에서 59,243으로 증가했습니다.

비대칭은 메커니즘에 있었고, 실험 도중 이 부분에서 실제로 발목을 잡혔습니다. Claude Code 2.1.207은 AGENTS.md를 아예 무시했고, CLAUDE.md로 이름을 바꿔야만 파일을 인식해 첫 유저 메시지에 주입했습니다. OpenCode는 두 파일명 모두 읽어서 시스템 프롬프트에 주입합니다.

여기서 실무적으로 따라오는 결론이 두 가지 있습니다. 첫째, 자신의 하네스가 실제로 어떤 파일명을 인식하는지 확인해야 합니다. 무시된 인스트럭션 파일은 티가 나지 않기 때문입니다. 둘째, 무거운 인스트럭션 파일은 가벼운 하네스의 베이스라인을 거의 4배로 불릴 수 있으며, 이는 해당 레포지토리의 모든 세션, 모든 요청에 실려 갑니다.

멀티플라이어 2: MCP 서버

공개된 자격증명 불필요 MCP 서버를 1개, 5개 구성으로 붙여봤습니다.

스키마 자체는 두 하네스에서 거의 동일하므로 세금도 거의 동일합니다. 소형 서버 하나당 요청당 대략 1,000~1,400토큰입니다. 서버 5개를 붙였을 때 Claude Code는 페이로드 기준 4,900토큰, OpenCode는 과금 기준 6,967토큰이 늘었고, 툴 개수는 각각 27→69개, 10→52개로 늘었습니다.

여기서 쓴 소형 공개 서버는 가벼운 케이스에 불과합니다. 풍부한 API를 가진 프로덕션 서버는 스키마가 몇 배 더 클 수 있는데, 이는 바로 아래 "everything" 측정에서 확인됩니다.

운영상 짚고 넘어갈 점 하나. Claude Code는 print 모드에서 프로젝트 스코프의 .mcp.json을 명시적으로 --mcp-config 플래그를 넘기지 않으면 조용히 무시했습니다. 서버가 붙어 있다고 가정하지 말고 경계에서 직접 검증해야 합니다.

멀티플라이어 3: 프레임워크 템플릿

BMAD 같은 스토리 기반 워크플로 프레임워크는 슬래시 커맨드를 페르소나, 프로토콜, 체크리스트로 구성된 대형 프롬프트 템플릿으로 확장합니다.

8,405자짜리 대표 템플릿을 동일한 T3 스토리의 프롬프트로 사용해 두 하네스에서 실행했습니다. 템플릿 자체는 약 2,100토큰에 불과하지만, 이것이 대화 히스토리에 편입되고 나면 세션 내 이후 모든 요청에서 다시 함께 실려 갑니다. 요청 9번짜리 세션이라면 9번 다시 전송되는 셈입니다.

즉 프레임워크 세금은 "템플릿 크기 × 요청 수"로 계산되며, 이는 앞선 모든 오버헤드 위에 그대로 누적됩니다.

멀티플라이어 4: 서브에이전트

작업을 두 개의 병렬 서브에이전트로 분산시켜봤습니다. 총량이 가장 가파르게 치솟는 지점입니다.

Claude Code는 이 작업을 세 가지 서로 다른 요청 클래스에 걸친 9번의 모델 요청으로 완료했습니다. 전체 ~33k 베이스라인을 가진 메인 세션, 그리고 각각 3,554자짜리 에이전트 시스템 프롬프트와 27개 중 24개 툴을 실어 나르는 서브에이전트 호출 5회입니다.

누적 과금 입력은 513,000토큰까지 치솟았는데, 같은 작업을 직접 수행했을 때(121,000토큰)와 비교하면 4.2배입니다. 서브에이전트를 두 개만 파생시켰을 뿐인데도 이 정도인 이유는, 모든 서브에이전트가 각자의 부트스트랩 비용을 지불하고, 그 결과 트랜스크립트를 다시 부모가 통째로 흡수하기 때문입니다.

OpenCode의 설계는 이 지점에서 눈에 띄게 가볍습니다. 서브에이전트 요청은 축소된 프로필(1,379자짜리 시스템 프롬프트, 툴 5개)로 구성됩니다. 다만 OpenCode의 서브에이전트 경로는 게이트웨이를 통과하며 깔끔하게 완료되지 않아, 이 부분은 설계상의 차이만 보고합니다.

정리

  • 플로어부터 다르다: 아무 작업도 하지 않고 "OK"만 답할 때도 Claude Code는 약 33,000토큰(Sonnet 기준), OpenCode는 약 6,900토큰을 소비합니다. 대부분은 툴 스키마(각각 약 24,000토큰 대 4,800토큰)에서 발생합니다.
  • 모델에 따라 배율이 달라진다: Claude Fable 5에서는 Claude Code가 더 작은 시스템 프롬프트를 보내면서 격차가 4.7배에서 3.3배로 좁혀졌습니다. OpenCode의 페이로드는 모델과 무관하게 고정적입니다.
  • 캐시 효율성도 차이가 크다: OpenCode는 요청 프리픽스가 세션 내내 바이트 단위로 동일해 캐시를 한 번만 써서 계속 저렴하게 읽습니다. Claude Code는 세션 도중 프롬프트 캐시를 반복해서 다시 쓰며, 같은 태스크에서 최대 54배 더 많은 캐시 토큰을 기록했습니다. 캐시 쓰기는 프리미엄 요금이 붙습니다.
  • 인스트럭션 파일과 MCP 서버는 누구에게나 세금을 매긴다: 72KB 인스트럭션 파일 하나로 요청당 약 20,000토큰이 추가되고, MCP 서버 5개는 5,000~7,000토큰을 더합니다. 실전 구성이라면 유저가 한 글자도 치기 전에 이미 75,000~85,000토큰이 소모됩니다.
  • 서브에이전트 팬아웃은 비용을 급격히 불린다: 직접 하면 121,000토큰이던 작업이 서브에이전트 두 개로 분산하자 513,000토큰(4.2배)이 됐습니다. 서브에이전트마다 별도의 부트스트랩 비용이 들고, 그 트랜스크립트를 부모가 다시 흡수하기 때문입니다.
  • 멀티스텝 태스크에서는 역전도 가능하다: Claude Code가 툴 호출을 하나의 라운드트립으로 배치하는 반면 OpenCode는 턴마다 하나씩 호출하다 보니, 요청 횟수가 많아지는 태스크에서는 작은 베이스라인을 가진 쪽이 오히려 총량에서 밀릴 수 있습니다. "베이스라인 × 요청 수"라는 공식을 기억해야 합니다.

실무적으로는 하네스를 고를 때 단순히 "토큰당 가격"이 아니라 세션당 요청 패턴, 캐시 재작성 빈도, 인스트럭션 파일 인식 여부, MCP·서브에이전트 사용 여부를 함께 따져야 실제 비용을 예측할 수 있습니다. 특히 규제 대응 차원에서 에이전트 동작을 로깅해야 하는 조직이라면, API 경계에서 페이로드와 usage 블록을 직접 캡처하는 것이 감이 아닌 데이터로 답하는 유일한 방법입니다.
