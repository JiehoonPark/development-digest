Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k

📅 2026-07-13📰 Hacker News (Top)
ai

코딩 에이전트를 프로덕션에 붙여본 사람이라면 한 번쯤 "이 도구는 왜 이렇게 토큰을 많이 먹지?"라는 의문을 가져봤을 겁니다. Systima 팀은 이 궁금증을 실제로 파헤쳐, Claude Code와 OpenCode를 동일한 모델·동일한 머신·동일한 작업 위에 올려놓고 API 경계에서 오가는 모든 요청과 응답을 낱낱이 캡처했습니다. 결과는 예상보다 훨씬 큰 격차였는데요, 프롬프트를 입력하기도 전에 이미 수만 토큰이 소모되고 있었습니다.

측정 방법: API 경계에서 실제로 무슨 일이 벌어지는가

측정 방식은 단순합니다. 각 하네스와 모델 엔드포인트 사이에 로깅 프록시를 끼워 넣었습니다.

harness (Claude Code / OpenCode)
→ logging proxy (captures request payloads + response usage)
→ model endpoint

이 프록시는 요청당 두 가지를 기록합니다. 하나는 하네스가 실제로 내보낸 JSON 페이로드(시스템 블록, 도구 스키마, 메시지)이고, 다른 하나는 API가 반환한 usage 블록(입력 토큰, 캐시 쓰기, 캐시 읽기, 출력 토큰)입니다. 전자는 "무엇을 보냈는가"의 그라운드 트루스이고, 후자는 "실제로 얼마가 과금됐는가"의 그라운드 트루스입니다.

테스트 조건은 다음과 같습니다.

  • 하네스: Claude Code 2.1.207과 OpenCode 1.17.18, 둘 다 claude-sonnet-4-5(2026년 7월 기준)에 고정. 이후 floor·캐시·멀티스텝 태스크로 구성된 축소 매트릭스를 claude-fable-5에 다시 고정해 재실행했고, 모델 변화로 결과가 달라진 부분은 본문에 별도로 표시했습니다.
  • 베이스라인 격리: MCP 서버나 사용자 설정, 메모리가 없는 깨끗한 설정 디렉터리, instruction 파일이 없는 빈 워크스페이스, 권한 확인은 우회. 이후 변수를 하나씩 추가하며 배수(multiplier) 조건을 테스트했습니다.
  • 태스크: T1은 "정확히 OK라고만 답하라"는 지시로 고정 오버헤드만 분리(하네스당 3회 실행). T2는 미리 준비된 파일을 읽고 요약. T3는 FizzBuzz 작성-실행-테스트-수정 루프에 체커 스크립트를 붙인 작업입니다.
  • 품질 체크: 별도의 10개 레인 벤치마크에서 하네스당 5회씩, 시드 고정된 해시 검증 테스트 스위트로 통과율과 토큰 비용을 함께 측정했습니다.
  • 제로 툴 변형: Claude Code는 --tools "", OpenCode는 "tools": {"*": false}로 설정해 시스템 프롬프트와 도구 스키마의 비중을 분리했습니다.

한 가지 밝혀둘 점이 있습니다. 이 연구의 트래픽은 Claude Max 구독으로 서드파티 도구를 구동할 수 있게 해주는 로컬 게이트웨이 Meridian을 거칩니다. 이 게이트웨이는 모든 요청을 SDK 고유의 envelope로 감싸는데, 이 상수값은 순수 calibration 요청으로 측정한 결과 Sonnet 경로에서 약 6,200토큰, Fable 경로에서 약 3,500토큰이었고 아래 모든 계측 수치에서 이 값을 뺐습니다. 페이로드 레벨 수치는 캡처된 요청 본문 그 자체에서 나온 것이라 게이트웨이의 영향을 받지 않으며 정확한 값입니다. 문자-토큰 환산에는 일반적인 휴리스틱이 아니라, 콜드 캐시 상황에서 계측된 캐시 쓰기 값이 전체 페이로드와 일치하는 지점을 기준으로 각 하네스별로 실측한 문자당 4.1~4.4 토큰 비율을 사용했습니다.

기본 오버헤드: "OK" 한마디에 드는 비용

작업 내용은 22자짜리 문장이었습니다. 이 짧은 프롬프트와 함께 각 하네스가 첫 요청에서 무엇을 보냈는지 살펴보겠습니다.

OpenCode의 요청은 거의 미니멀한 수준입니다. "You are OpenCode, the best coding agent on the planet"으로 시작하는 시스템 블록 하나, 전통적인 코딩 도구 10개, 그리고 사용자 프롬프트가 콘텐츠의 전부입니다.

반면 Claude Code의 요청은 사실상 플랫폼 부트스트랩입니다. 27개 도구에는 코딩 핵심 기능뿐 아니라 백그라운드 에이전트와 오케스트레이션 스위트 전체가 포함됩니다. CronCreate와 Monitor부터 Task 계열, worktree 관리, 푸시 알림까지 다양합니다. 사용자가 프롬프트를 입력하기도 전에 첫 메시지에는 이미 위임용 에이전트 타입 카탈로그, 사용 가능한 스킬 카탈로그, 사용자 컨텍스트라는 세 가지 리마인더 블록이 주입되어 있습니다.

두 하네스 모두 도구 스키마가 지배적인 항목입니다. Claude Code의 약 33,000토큰 중 대략 24,000토큰이 도구 정의이고, OpenCode의 약 6,900토큰 중에는 약 4,800토큰이 도구 정의입니다.

도구 없이 순수 하네스만 비교하면

도구를 완전히 제거하면 시스템 프롬프트 자체의 무게를 볼 수 있습니다. Claude Code는 26,891자, 약 6.5k 토큰입니다. OpenCode는 8,811자, 약 2.0k 토큰입니다.

두 하네스 모두 도구를 비활성화하면 프롬프트가 약간 줄어듭니다. 그런데 도구가 전혀 없는 상태에서도 Claude Code의 명령어 세트는 OpenCode보다 3배 이상 큽니다. 이 차이는 순수하게 톤 규칙, 안전 가이드, 태스크 관리 지침, 환경 설명 같은 "행동 교리(behavioural doctrine)"에서 나옵니다.

파일 하나 요약하는 작업

T2는 각 하네스에게 파일을 읽고 요약하도록 시켰습니다. 둘 다 정확한 요약을 만들어냈습니다.

Claude Code는 6번의 HTTP 요청과 약 199,000 누적 계측 입력 토큰을 사용했습니다. OpenCode는 4번의 요청과 약 41,000 토큰, 그리고 세션 제목을 붙이기 위한 Haiku 사이드 콜 1회를 사용했습니다.

이 토큰의 대부분은 정가의 10분의 1 가격으로 과금되는 캐시 읽기입니다. 그러나 페이로드 크기와 무관하게 확장되는 세 가지가 있습니다. 첫 턴 캐시 쓰기, 매 턴의 읽기, 그리고 어떤 캐시 할인으로도 줄일 수 없는 컨텍스트 윈도우 소모량입니다. 33k 토큰짜리 베이스라인이라는 건, 코드가 대화에 등장하기도 전에 이미 200k 윈도우의 6분의 1 지점에서 매 턴이 시작된다는 뜻입니다.

여러 단계 작업에서는 격차가 좁혀진다

T3(작성-실행-테스트-수정 루프)는 앞선 베이스라인이 만든 예상을 뒤집었습니다.

지표Claude CodeOpenCode
모델 요청 수39 (+제목 호출 1회)
도구 호출 방식하나의 라운드트립에 병렬 배치턴마다 도구 호출 1회
누적 계측 입력 토큰약 121,000 토큰약 132,000 토큰

Claude Code는 파일 쓰기 2회, 스크립트 실행 2회를 포함한 전체 작업을 단 한 번의 병렬 도구 라운드트립으로 처리했습니다. OpenCode는 턴마다 정확히 한 번씩 도구를 호출해 9번의 턴을 사용했습니다.

베이스라인이 매 요청마다 다시 전송되는 구조이기 때문에, 요청 수는 곧 베이스라인의 배수가 됩니다. OpenCode는 약 7k짜리 베이스라인을 9번 지불했고, Claude Code는 약 33k짜리 베이스라인을 3번 지불했으며, 결과적으로 총합이 비슷해진 겁니다.

즉 작업 전체의 입력 토큰량은 대략 "베이스라인 × 요청 수 + 대화 성장분"으로 근사할 수 있습니다. 베이스라인이 크지만 공격적으로 배치 처리하는 하네스와, 베이스라인이 작지만 순차 처리하는 하네스는 결국 비슷한 지점에 도달할 수 있습니다.

페이로드에서는 두 가지 구조적 특징도 드러났습니다.

  • Claude Code는 대화가 진행될수록 <system-reminder> 블록을 추가로 주입합니다. 첫 턴에 3개였던 것이 첫 도구 라운드트립 시점에는 4개로 늘어나, 스캐폴딩이 턴 수와 함께 증가합니다.
  • OpenCode의 턴당 한계 페이로드는 약 400~2,200자로, 이는 순수하게 대화 콘텐츠일 뿐입니다.

캐시 효율성의 큰 차이

OpenCode의 요청 프리픽스는 캡처된 모든 실행에서 바이트 단위로 동일했습니다. 즉 세션당 한 번만 페이로드를 캐시에 기록하는 비용을 지불하고, 이후에는 푼돈 수준으로 다시 읽어올 수 있었습니다.

반면 Claude Code는 세션 도중 수만 개의 프롬프트 캐시 토큰을 계속해서 다시 써 내려갔습니다. 실행할 때마다 반복되었고, 동일한 작업에서 OpenCode보다 최대 54배 더 많은 캐시 토큰을 기록했습니다. 캐시 쓰기는 프리미엄 요금으로 과금되기 때문에, 이것이 Claude Code 사용 시 사용량 대시보드가 치솟는 이유였습니다.

최신 모델에서는 격차가 어떻게 달라지나

이 격차가 Sonnet 특유의 현상인지 확인하기 위해 baseline 측정을 Claude Fable 5로 다시 실행했는데, 예상과 달리 격차가 줄어들었습니다.

Claude Code의 시스템 프롬프트는 모델에 따라 달라집니다. Sonnet에는 27,787자의 지침을 보냈지만 Fable에는 10,526자만 보냈고, 도구 스키마도 99,778자에서 82,283자로 줄었습니다. 도구 27개는 동일했지만 교리 부분이 훨씬 가벼워진 겁니다.

반면 OpenCode의 페이로드는 두 모델에서 바이트 단위로 완전히 동일했습니다.

결과적으로 floor 단계의 격차는 Sonnet에서 페이로드 기준 약 4.7배였던 것이 Fable에서는 약 3.3배로 줄었습니다. 여전히 훨씬 더 많이 먹지만, 그 비율은 모델에 따라 달라진다는 뜻입니다.

멀티스텝 작업에서의 "격차 수렴" 현상도 모델을 바꾸자 사라졌습니다. Fable에서 Claude Code는 3번이 아니라 6번의 요청을 사용했고, 그중에는 85,686토큰짜리 세션 중간 캐시 재작성이 포함되어 약 298,000토큰을 소비했습니다. OpenCode는 133,000토큰이었습니다. 즉 배치 처리의 이점은 하네스의 고정된 특성이 아니라 모델의 행동 방식에 달려 있었던 겁니다.

이 실험 구간에서는 연구 전체를 통틀어 OpenCode가 유일하게 세션 중간에 캐시를 다시 쓴 사례도 나타났는데, 약 6,000토큰 규모의 단일 이벤트였고 그 외에는 모두 바이트 단위로 안정적이었습니다.

한 가지 덧붙이자면, Fable 조건에서는 응답이 claude-fable-5와 claude-opus-4-8로 번갈아가며 왔는데, 이는 두 티어가 인프라를 공유한다는 것과 일치합니다. Fable 구간의 표본 수는 작아서(floor와 캐시 태스크는 2회, 멀티스텝 태스크는 1회) 완전한 2차 연구라기보다는 방향성 확인 정도로 받아들이는 게 맞습니다.

프로덕션 환경의 추가 비용: instruction 파일과 MCP 서버

지금까지가 "짧고 깨끗한 세션"을 설명하는 floor였다면, 실제 세션은 훨씬 길고 산만합니다. 이를 반영하기 위해 실제 프로덕션 조건에 가까운 배수(multiplier) 변수들을 하나씩 추가해봤습니다.

배수 1. Instruction 파일

프로덕션 저장소에서 가져온 실제 72KB짜리 AGENTS.md 파일을 워크스페이스에 넣고 T1을 다시 실행했습니다.

효과는 대칭적이면서도 컸습니다. 두 하네스 모두 요청당 20,000토큰이 넘게 늘어났습니다. OpenCode의 계측 총량은 13,152에서 33,336으로, Claude Code는 39,005에서 59,243으로 증가했습니다.

비대칭은 오히려 작동 방식에 있었고, 실험 도중 실제로 발목을 잡았습니다. Claude Code 2.1.207은 AGENTS.md를 아예 무시했고, 파일명을 CLAUDE.md로 바꿔야만 이를 첫 사용자 메시지에 주입했습니다. 반면 OpenCode는 두 파일명 모두 인식해 시스템 프롬프트에 주입합니다.

여기서 실무적인 시사점 두 가지가 나옵니다. 첫째, 하네스가 실제로 어떤 파일명을 인식하는지 확인해야 합니다. 무시된 instruction 파일은 조용히 무시될 뿐 오류를 내지 않기 때문입니다. 둘째, 무거운 instruction 파일 하나가 가벼운 하네스의 베이스라인을 거의 4배 가까이 부풀릴 수 있으며, 이는 해당 저장소의 모든 세션에서 매 요청마다 반복해서 실린다는 점입니다.

배수 2. MCP 서버

자격 증명이 필요 없는 공개 MCP 서버를 1개, 5개 구성으로 붙여봤습니다.

스키마는 두 하네스에서 거의 동일하기 때문에 비용도 거의 동일했습니다. 소규모 서버 하나당 대략 1,000~1,400토큰 수준이었습니다. 5개 서버를 붙였을 때 Claude Code는 페이로드 기준 4,900토큰이, OpenCode는 계측 기준 6,967토큰이 늘어났고, 도구 개수는 각각 27개→69개, 10개→52개로 증가했습니다.

작고 공개된 서버는 그나마 가벼운 경우입니다. 풍부한 API를 가진 프로덕션 MCP 서버는 이보다 훨씬 더 큰 스키마를 갖고 있을 것으로 예상됩니다.

한 가지 운영상 주의할 점도 있습니다. Claude Code는 print 모드에서 프로젝트 범위의 .mcp.json을 명시적으로 --mcp-config 플래그를 전달하기 전까지는 조용히 무시했습니다. 서버가 붙어 있다고 가정하지 말고, 반드시 API 경계에서 실제로 붙었는지 확인해야 합니다.

정리

이 연구는 Claude Code와 OpenCode를 동일 모델·동일 조건에서 정면으로 비교해, "에이전트 하네스가 프롬프트도 입력하기 전에 얼마나 많은 토큰을 이미 소모하고 있는가"를 실측한 결과입니다. 핵심을 정리하면 다음과 같습니다.

  • 고정 오버헤드: "OK"라고만 답하는 극단적으로 단순한 작업에서도 Claude Code는 약 33,000토큰, OpenCode는 약 6,900토큰을 프롬프트 전에 이미 소모합니다. 도구가 전혀 없어도 Claude Code의 시스템 프롬프트만으로 OpenCode보다 3배 이상 큽니다.
  • 캐시 효율성: OpenCode는 요청 프리픽스가 세션 내내 바이트 단위로 동일해 캐시를 한 번 쓰고 계속 싸게 읽지만, Claude Code는 세션 중간에 반복적으로 캐시를 재작성하며 동일 작업에서 최대 54배 더 많은(프리미엄 과금되는) 캐시 쓰기 토큰을 발생시킵니다.
  • 설정 요소의 누적 비용: 실제 72KB짜리 instruction 파일은 요청당 약 20,000토큰을 추가하고, MCP 서버 5개는 5,000~7,000토큰을 추가합니다. 실제 프로덕션 환경에서는 사용자가 한 글자도 입력하기 전에 이미 75,000~85,000토큰이 소모됩니다.
  • 서브에이전트와 배치 처리의 함정과 반전: 직접 처리하면 121,000토큰인 작업이 두 개의 서브에이전트로 분산되면 513,000토큰까지 뛰는 반면, 특정 멀티스텝 작업에서는 Claude Code가 도구 호출을 하나의 라운드트립으로 병렬 배치해 총 토큰량이 OpenCode보다 오히려 낮게 나오기도 했습니다. 다만 이 배치 우위는 모델에 종속적이어서, 최신 모델로 바꾸자 같은 작업이 두 배 많은 요청과 약 298,000 대 133,000토큰이라는 정반대 결과로 뒤집혔습니다.

실무적으로는, 에이전트 하네스를 선택하거나 프로덕션에 배포할 때 "베이스라인 토큰이 얼마인가"만 볼 게 아니라 캐시 안정성, instruction 파일 인식 방식, MCP 스키마 크기, 그리고 실제 세션 길이에서의 배수 효과까지 API 경계에서 직접 측정해봐야 한다는 점을 이 연구가 잘 보여줍니다. 특히 EU AI Act처럼 시스템 동작에 대한 로깅과 이해를 요구하는 규제 환경에서는 "내 에이전트가 실제로 무엇을 보내는가"라는 질문에 데이터로 답할 수 있어야 합니다.
