13:T2f96,<h2>Wayfinder 실전 시연: 스펙을 만들기 전에 방향부터 찾는 법</h2>
<p>이번 라이브 스트림에서는 Matt Pocock이 자신이 만든 새로운 워크플로우 도구 Wayfinder를 실제로 사용하는 과정을 처음부터 끝까지 보여줍니다. Wayfinder는 기존에 쓰던 &quot;Grill Me&quot; 스킬의 진화형으로, 크고 모호한 작업을 하나의 스펙으로 정리하기 전 단계, 즉 방향을 못 잡고 있는 상태(fog)를 해소하는 데 초점을 맞춘 도구입니다. 이 글에서는 실제 프로젝트(Course Video Manager)에 TikTok 크리에이터 기능을 추가하는 과정을 예시로, Wayfinder가 어떻게 작동하고 왜 필요한지를 정리합니다.</p>
<h3>Wayfinder란 무엇인가</h3>
<p>Wayfinder는 한마디로 &quot;스테로이드 맞은 Grill Me&quot;입니다. 기존 Grill Me 스킬이 하나의 작업에 대해 질문을 던지며 요구사항을 명확히 하는 도구였다면, Wayfinder는 훨씬 더 큰 덩어리의 작업 — 여러 개의 결정이 얽혀 있고 아직 방향조차 잡히지 않은 작업 — 을 대상으로 합니다.</p>
<p>핵심 아이디어는 이렇습니다: Wayfinder의 역할은 <strong>문제를 해결하는 것이 아니라, 지금 있는 곳에서 목적지까지 가는 길을 찾는 것</strong>입니다. 지도를 그리는 작업(map)에 가깝습니다. 먼저 코드베이스를 탐색해서 무엇이 있고 무엇이 가능한지 파악한 다음, 사용자에게 질문을 던지기 시작합니다. 그리고 열려 있는 의사결정들을 좁혀가며 최종적으로는 실행 가능한 스펙(spec)을 만들어내는 것이 목표입니다. 실제 구현 코드를 작성하는 건 Wayfinder의 역할이 아닙니다.</p>
<h3>실전 시연: TikTok 크리에이터 기능 추가하기</h3>
<p>시연은 다음과 같은 다소 모호한 요구사항으로 시작됩니다.</p>
<p>&gt; &quot;기존 비디오 앱(Course Video Manager)을 확장해서 TikTok 크리에이터 기능을 추가하고 싶다. 기존 비디오를 재사용하되, 비디오 플레이어 로직은 재활용하면서도 모양은 살짝 다르게 만들어야 한다. 그리고 TikTok API가 뭘 제공하는지 조사해서, CVM에서도 업로드하고 YouTube에서도 업로드할 수 있게 하고 싶다.&quot;</p>
<p>이 정도로 큰 그림만 던진 상태에서 Wayfinder 에이전트(Opus 4.5를 기반으로 실행, 화면상 자막에는 4.8로 표기)를 실행합니다. Grilling(질문을 통한 요구사항 명확화)에는 Opus 정도면 충분하고, 굳이 더 비싼 모델(Fable 등)을 쓸 필요가 없다는 점도 언급됩니다.</p>
<p>에이전트는 먼저 탐색을 마친 뒤 다음과 같은 질문들을 순서대로 던집니다.</p>
<h4>1. 목적지가 무엇인가?</h4>
<p>에이전트는 가장 먼저 &quot;이 작업의 destination(목적지)이 뭔지&quot;부터 확인합니다. 즉, 지금 하려는 건 구현 코드를 만드는 게 아니라 <strong>구현 에이전트에게 넘길 수 있는, 열린 의사결정이 모두 잠긴(locked) 스펙을 만드는 것</strong>이 맞는지 확인하는 것입니다. Matt는 이를 정확히 확인해주며 &quot;우리는 스펙을 만들고 싶은 거고, 구현을 시작하고 싶은 게 아니다&quot;라고 명시합니다.</p>
<h4>2. TikTok 비디오란 기존 비디오와 무엇이 다른가?</h4>
<p>schema.ts에는 이미 <code>videos</code> 테이블이 존재하고, 여기에는 id, lessonId, pitchId 등의 필드가 있습니다. 에이전트는 &quot;기존 비디오를 재사용한다&quot;는 말이 정확히 무엇을 의미하는지 파고듭니다 — 완전히 새로운 엔티티를 만드는 건지, 기존 비디오에 새 속성만 추가하는 건지를 구분해야 한다는 것이죠.</p>
<p>여기서 에이전트가 처음 잘못된 추천(기존 비디오를 &quot;선택&quot;하는 방식)을 내놓았을 때, Matt는 바로 이를 정정합니다. 결론은 &quot;기존 videos 테이블을 그대로 쓰되, TikTok 여부를 나타내는 별도 속성(attribute)을 추가한다&quot;는 방향이었습니다. 다만 이 속성을 어떻게 UI에 노출할지, 새로운 최상위 비디오 목록으로 만들지 여부는 여전히 열린 질문으로 남았습니다.</p>
<h4>3. TikTok 비디오는 Pitch와 관계가 있는가?</h4>
<p>CVM에는 <code>pitch</code>라는 개념이 있는데, 이는 비디오를 만들기 전 기획/패키징 단계를 의미합니다. 아직 만들지 않은 비디오들의 목록이 pitch 목록인 셈입니다.</p>
<p>Matt는 고민 끝에 &quot;TikTok 비디오는 pitch와 무관한 독립적인 흐름이어야 한다&quot;고 결론 냅니다. Pitch는 더 긴 형식의, 제작 비용이 크고 기획이 필요한 비디오를 위한 것이고, TikTok 비디오는 &quot;아이디어가 있으면 바로 만든다&quot;는 정도의 낮은 마찰(low friction)을 가져야 하기 때문입니다. 즉, pitch 연결은 필수가 아니라는 결론이었습니다.</p>
<h4>4. 어떤 플랫폼에 배포할 것인가?</h4>
<p>Matt는 TikTok과 YouTube Shorts를 우선순위로 꼽았습니다. Instagram은 팔로워가 없어 의미가 적다고 판단했지만, 일단 데이터베이스에 데이터가 들어가고 나면 여러 플랫폼으로 배포하는 건 매우 저렴한 작업이 되기 때문에 추가 리서치는 가치가 있다고 언급합니다.</p>
<h4>5. &quot;실제 콘텐츠로 만든다&quot;는 것은 무엇을 의미하는가?</h4>
<p>여기서 중요한 기존 자산이 등장합니다. Matt의 컴퓨터에는 <code>repos</code> 아래에 <code>total-typescript-monorepo</code>(현재는 deprecated 상태)라는 저장소가 있고, 그 안에 세로형 비디오에 자막을 자동으로 입히는 Remotion 기반 파이프라인이 이미 존재합니다. 이전 워크플로우에서 OBS로 녹화한 가장 최근 비디오를 가져와 자막을 붙이는 CLI로 쓰던 것입니다.</p>
<p>에이전트는 이 Remotion 프로젝트를 &quot;prior art(기존 자산)&quot;로 파악하고, 이를 스펙의 핵심 범위(in-scope pillar)에 포함시킬 것을 제안합니다. 다만 이 monorepo에서 필요한 부분만 별도 저장소로 추출해야 할지는 여전히 미결정 상태로 남습니다.</p>
<h3>왜 이렇게 모호하게 진행하는가</h3>
<p>시연 중 Matt는 스스로 &quot;지금 얼마나 모호하게(vague) 이야기하고 있는지 보라&quot;고 짚습니다. 구체적인 답을 정확히 내놓기보다는 마치 회의에서 대화하듯 진행하는 것이 의도된 방식입니다. 아직 방향(way)을 모르는 상태이고, 이 안개(fog)를 걷어내는 것이 바로 Wayfinder가 하는 일이기 때문입니다.</p>
<h3>전체 워크플로우 속에서의 위치</h3>
<p>Matt는 자신의 전체 개발 워크플로우 다이어그램을 보여주며 Wayfinder의 위치를 설명합니다.</p>
<ul>
<li><strong>Grill with docs</strong> 또는 <strong>Wayfinder</strong> (둘 중 편한 방식 선택, 단 Wayfinder는 여러 세션이 필요할 만큼 큰 작업에 적합)</li>
<li>→ <strong>to-spec</strong> (스펙으로 전환)</li>
<li>→ <strong>to-tickets</strong> (티켓 분해)</li>
<li>→ <strong>implement</strong> (구현)</li>
<li>→ <strong>code review</strong> (코드 리뷰)</li>
</ul>
<p>특히 리서치 티켓 작업에 대해서는 최근 서브에이전트 기반 접근으로 방식을 바꿨다고 언급합니다. 리서치 자체가 상대적으로 단순한 작업이기 때문입니다.</p>
<h3>스펙 확정과 티켓 생성</h3>
<p>논의를 거쳐 에이전트는 다음과 같은 destination을 제안합니다.</p>
<ul>
<li>CVM 내 TikTok 크리에이터 기능에 대한 의사결정이 완료된 스펙</li>
<li>Remotion 기존 자산을 활용한 자막·세로형 렌더링 파이프라인</li>
<li>TikTok 및 YouTube Shorts 게시</li>
<li><strong>실행(구현)은 범위 밖</strong></li>
</ul>
<p>이를 9개의 티켓으로 분해하겠다는 제안에 대해 Matt는 세세히 검토하지 않고 &quot;좋아 보인다&quot;고 승인한 뒤, 실제 GitHub 이슈 생성을 진행시킵니다. 평소라면 한 번 더 검토했겠지만 라이브 스트림이라는 상황상 생략했다고 밝힙니다.</p>
<h3>모델·하네스·환경: 에이전트를 보는 3단 구조</h3>
<p>시연 중간에 Matt는 AI 에이전트를 바라보는 자신만의 프레임을 소개합니다.</p>
<ul>
<li><strong>모델(Model)</strong>: 중심에 있는 것 — Opus, GPT-5.1 등</li>
<li><strong>하네스(Harness)</strong>: 모델을 감싸는 것 — Claude Code, Codex, ChatGPT 등. 모델 + 하네스를 합쳐 흔히 &quot;에이전트&quot;라고 부릅니다.</li>
<li><strong>환경(Environment)</strong>: 에이전트가 그 안에서 동작하는 대상 — 파일 시스템, 코드베이스 등</li>
</ul>
<p>많은 사람들이 &quot;더 좋은 모델로 바꾸면 다 해결된다&quot;고 생각하지만, 이는 전체 그림의 일부에 불과합니다. 모델을 개선하는 것도 방법이지만, <strong>비용이 들지 않는 개선</strong>으로 하네스를 개선하거나 환경을 개선하는 방법도 있습니다. Matt는 이 세 가지에 각각 얼마나 비중을 둬야 하냐는 질문에 &quot;50/50&quot; 정도라고 답합니다 (모델 개선과, 하네스·환경 개선을 합친 비중이 비슷하다는 취지로 언급됨).</p>
<h3>시청자 Q&amp;A에서 나온 실무 팁</h3>
<ul>
<li><strong>모델 선택</strong>: Grilling(요구사항 명확화) 단계에는 Opus 정도면 충분하며, 반드시 최상위 모델(Fable 등)을 쓸 필요는 없습니다.</li>
<li><strong>버그를 찾는 데 좋은 스킬</strong>: 프론트엔드든 백엔드든, 결국 필요한 건 코드 리뷰 스킬입니다. 스펙을 두고 구현 결과를 비교 검토하는 &quot;2차 패스(second pass)&quot; 하나만으로도 버그를 잘 잡아낼 수 있습니다.</li>
<li><strong>토큰 사용량</strong>: 일반적인 작업에서는 150k 토큰 이하, 이른바 &quot;smart zone&quot; 안에 머무르려고 합니다.</li>
<li><strong>모델이 작업에서 벗어나지 않게 하는 법</strong>: 플래닝 단계에서는 결국 사람이 책임지고 통제해야 합니다. 많은 사람들이 이 단계에서 지나치게 수동적인데, 에이전트는 주니어이고 나는 리드라는 마인드로 적극적으로 몰아가야 합니다.</li>
<li><strong>코드를 읽는 법을 배우고 싶다면</strong>: Matt의 스킬 저장소에 있는 &quot;teach&quot; 스킬을 활용하면 됩니다.</li>
</ul>
<h3>정리</h3>
<ul>
<li><strong>Wayfinder는 스펙 작성 이전 단계</strong>를 위한 도구입니다. 크고 모호한 요구사항을 실행 가능한 스펙으로 좁혀가는 과정에서, 무엇을 만들지 결정하는 게 목적이지 실제로 만드는 게 목적이 아닙니다.</li>
<li>진행 방식은 <strong>에이전트가 코드베이스를 먼저 탐색</strong>한 뒤, 목적지(destination), 기존 개념과의 관계, 스코프, 기존 자산(prior art) 활용 여부 등을 순차적으로 질문하며 결정을 좁혀가는 식입니다.</li>
<li>사용자는 처음부터 구체적인 답을 낼 필요가 없습니다. 회의에서 대화하듯 모호하게 시작해도 되고, 에이전트의 질문에 답하면서 점차 구체화하면 됩니다. 다만 <strong>에이전트가 틀린 방향으로 갈 때는 즉시 개입해서 바로잡아야</strong> 합니다.</li>
<li>전체 워크플로우는 <strong>grill/wayfinder → to-spec → to-tickets → implement → code review</strong>로 이어지며, Wayfinder는 이 중 가장 앞단, 큰 작업을 여러 세션에 걸쳐 다뤄야 할 때 사용하는 단계입니다.</li>
<li>에이전트 성능을 개선하고 싶다면 모델 교체만이 답이 아닙니다. <strong>하네스(도구/워크플로우)와 환경(코드베이스, 파일 구조)을 개선하는 것도 비용 없이 큰 효과</strong>를 낼 수 있는 방법입니다.</li>
