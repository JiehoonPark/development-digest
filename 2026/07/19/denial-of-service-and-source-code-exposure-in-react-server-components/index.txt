15:T2c0d,<p>React Server Components를 노리는 원격 코드 실행 취약점(React2Shell)이 지난주 공개되고 패치까지 배포된 직후, 보안 연구자들이 그 패치 자체를 우회하려는 시도 과정에서 두 가지 추가 취약점을 발견해 공개했습니다. 다행히 이번에 발견된 취약점들은 원격 코드 실행(RCE)으로 이어지지는 않으며, 지난주 배포된 React2Shell 패치는 여전히 RCE 익스플로잇을 막아내고 있습니다. 다만 서비스 거부(DoS)와 소스 코드 노출이라는 새로운 위험이 확인된 만큼, React 팀은 즉각적인 업데이트를 강력히 권고하고 있습니다.</p>
<h3>새로 공개된 취약점 개요</h3>
<p>이번에 공개된 취약점은 다음과 같습니다.</p>
<ul>
<li><strong>서비스 거부(DoS) - 심각도 높음</strong>: CVE-2025-55184, CVE-2025-67779, CVE-2026-23864 (CVSS 7.5)</li>
<li><strong>소스 코드 노출 - 심각도 중간</strong>: CVE-2025-55183 (CVSS 5.3)</li>
</ul>
<p>여기서 주의할 점은, 앞서 배포된 패치 자체에 취약점이 남아 있었다는 사실입니다. 즉 이전 취약점 대응으로 이미 19.0.3, 19.1.4, 19.2.3으로 업데이트했더라도 이는 불완전한 패치이므로, <strong>다시 한 번 업데이트해야</strong> 합니다. 세부 내용은 패치 롤아웃이 완료된 뒤 추가로 공개될 예정입니다.</p>
<h3>즉시 조치가 필요한 대상</h3>
<p>이번 취약점들은 지난번 CVE-2025-55182와 동일한 패키지·버전에 존재합니다. 영향을 받는 버전은 다음과 같습니다.</p>
<ul>
<li>19.0.0, 19.0.1, 19.0.2, 19.0.3</li>
<li>19.1.0, 19.1.1, 19.1.2, 19.1.3</li>
<li>19.2.0, 19.2.1, 19.2.2, 19.2.3</li>
</ul>
<p>영향을 받는 패키지는 아래 세 가지입니다.</p>
<ul>
<li><code>react-server-dom-webpack</code></li>
<li><code>react-server-dom-parcel</code></li>
<li><code>react-server-dom-turbopack</code></li>
</ul>
<p>수정 사항은 19.0.4, 19.1.5, 19.2.4 버전으로 백포트되었습니다. 위 패키지를 사용 중이라면 즉시 이 수정 버전으로 업그레이드해야 합니다.</p>
<p>이전과 마찬가지로, 앱의 React 코드가 서버를 사용하지 않는다면 영향을 받지 않습니다. React Server Components를 지원하는 프레임워크나 번들러, 번들러 플러그인을 사용하지 않는다면 역시 해당되지 않습니다.</p>
<p>React 팀은 이런 후속 취약점 발견이 드문 일이 아니라고 설명합니다. 심각한 CVE가 공개되면 연구자들이 인접한 코드 경로를 파고들어 초기 완화책을 우회할 수 있는 변형 익스플로잇을 찾아내는 것은 업계 전반에서 흔히 나타나는 패턴이라는 겁니다. 예를 들어 Log4Shell 사태 이후에도 커뮤니티가 원래 패치를 검증하는 과정에서 추가 CVE들이 보고된 바 있습니다. 추가 공개가 다소 답답하게 느껴질 수 있지만, 일반적으로는 대응 사이클이 건강하게 돌아가고 있다는 신호로 볼 수 있습니다.</p>
<h3>영향을 받는 프레임워크와 번들러</h3>
<p>일부 React 프레임워크와 번들러가 취약한 React 패키지에 의존하거나, 피어 의존성으로 포함하고 있었습니다. 영향을 받는 프레임워크·번들러는 다음과 같습니다.</p>
<ul>
<li>Next.js</li>
<li>React Router</li>
<li>Waku</li>
<li>@parcel/rsc</li>
<li>@vite/rsc-plugin</li>
<li>rwsdk</li>
</ul>
<p>업그레이드 절차는 이전 공지 글의 안내를 참고하면 됩니다.</p>
<h3>호스팅 제공업체의 완화 조치</h3>
<p>이전과 마찬가지로 React 팀은 여러 호스팅 제공업체와 협력해 임시 완화 조치를 적용했습니다. 다만 이런 임시 조치에 의존해서는 안 되며, 앱을 안전하게 지키려면 반드시 직접 업데이트를 진행해야 합니다.</p>
<h3>React Native 사용자</h3>
<p>모노레포를 사용하지 않고 <code>react-dom</code>도 사용하지 않는 React Native 사용자라면, <code>package.json</code>에 React 버전이 고정되어 있을 것이므로 별도 조치가 필요 없습니다.</p>
<p>모노레포 환경에서 React Native를 사용 중이라면, 아래 패키지가 설치되어 있는 경우에 한해 해당 패키지만 업데이트하면 됩니다.</p>
<ul>
<li><code>react-server-dom-webpack</code></li>
<li><code>react-server-dom-parcel</code></li>
<li><code>react-server-dom-turbopack</code></li>
</ul>
<p>이 조치만으로 보안 권고 사항을 완화할 수 있으며, <code>react</code>와 <code>react-dom</code>까지 업데이트할 필요는 없으므로 React Native에서 흔히 발생하는 버전 불일치 오류는 발생하지 않습니다.</p>
<h3>취약점 상세</h3>
<p><strong>심각도 높음: 다중 서비스 거부 (CVE-2026-23864, CVSS 7.5, 2026년 1월 26일 공개)</strong></p>
<p>보안 연구자들이 React Server Components에 여전히 남아있던 추가 DoS 취약점을 발견했습니다. 이 취약점은 Server Function 엔드포인트로 특별히 조작된 HTTP 요청을 전송함으로써 트리거되며, 취약한 코드 경로와 애플리케이션 구성·코드에 따라 서버 크래시, 메모리 부족 예외, 과도한 CPU 사용으로 이어질 수 있습니다. 1월 26일 배포된 패치가 이 DoS 취약점들을 완화합니다.</p>
<p>원래 CVE-2025-55184의 DoS를 해결하기 위한 최초 패치는 불완전했으며, 이로 인해 이전 버전들이 여전히 취약한 상태였습니다. 19.0.4, 19.1.5, 19.2.4 버전은 안전합니다.</p>
<p><strong>심각도 높음: 서비스 거부 (CVE-2025-55184, CVE-2025-67779, CVSS 7.5)</strong></p>
<p>보안 연구자들은 Server Functions 엔드포인트로 조작된 HTTP 요청을 전송하면, React가 이를 역직렬화하는 과정에서 무한 루프가 발생해 서버 프로세스가 멈추고 CPU를 소진하게 만들 수 있다는 사실을 발견했습니다. 앱이 React Server Function 엔드포인트를 직접 구현하지 않았더라도, React Server Components를 지원하기만 하면 취약할 수 있습니다. 이는 공격자가 사용자의 제품 접근을 차단하고, 서버 환경 성능에 영향을 줄 수 있는 취약점 벡터를 만듭니다. 배포된 패치는 무한 루프 발생을 막아 이를 완화합니다.</p>
<p><strong>심각도 중간: 소스 코드 노출 (CVE-2025-55183, CVSS 5.3)</strong></p>
<p>한 보안 연구자는 취약한 Server Function에 조작된 HTTP 요청을 보내면 해당 Server Function의 소스 코드가 안전하지 않게 반환될 수 있음을 발견했습니다. 이 공격이 성립하려면, 명시적으로든 암묵적으로든 문자열화된 인자를 노출하는 Server Function이 존재해야 합니다. 예를 들어 다음과 같은 코드가 있다고 해보겠습니다.</p>
<pre><code class="language-js">'use server';

  const conn = db.createConnection('SECRET KEY');
  const user = await conn.createUser(name);
  return { id: user.id, message: `Hello, ${name}!` }

<p>이런 경우 공격자는 아래와 같이 함수의 내부 소스 코드까지 유출시킬 수 있습니다.</p>
<pre><code>0:{&quot;a&quot;:&quot;$@1&quot;,&quot;f&quot;:&quot;&quot;,&quot;b&quot;:&quot;Wy43RxUKdxmr5iuBzJ1pN&quot;}

<p>배포된 패치는 Server Function 소스 코드가 문자열화되는 것을 막습니다.</p>
<p>다만 이 취약점으로 노출될 수 있는 범위는 소스 코드에 하드코딩된 비밀 값에 한정됩니다. <code>process.env.SECRET</code>처럼 런타임에 주입되는 비밀 값은 영향을 받지 않습니다. 또한 노출되는 코드 범위는 해당 Server Function 내부 코드로 제한되지만, 번들러의 인라이닝 정도에 따라 다른 함수까지 포함될 수 있으므로 반드시 프로덕션 번들을 기준으로 검증해야 합니다.</p>

<h3>정리</h3>
<ul>
<li>지난주 공개된 React2Shell RCE 취약점에 대한 패치는 여전히 유효하지만, 그 패치를 검증하는 과정에서 DoS 취약점 3건(CVE-2025-55184, CVE-2025-67779, CVE-2026-23864)과 소스 코드 노출 취약점 1건(CVE-2025-55183)이 새로 발견됐습니다.</li>
<li>이전에 19.0.3, 19.1.4, 19.2.3으로 업데이트했더라도 이는 불완전한 패치이므로, 반드시 <strong>19.0.4, 19.1.5, 19.2.4</strong>로 다시 업데이트해야 합니다.</li>
<li>영향을 받는 패키지는 <code>react-server-dom-webpack</code>, <code>react-server-dom-parcel</code>, <code>react-server-dom-turbopack</code>이며, Next.js, React Router, Waku, @parcel/rsc, @vite/rsc-plugin, rwsdk 등을 사용 중이라면 프레임워크 차원의 업데이트도 확인해야 합니다.</li>
<li>DoS 취약점은 조작된 HTTP 요청으로 무한 루프를 유발해 서버를 마비시킬 수 있고, 소스 코드 노출 취약점은 Server Function 내부에 하드코딩된 비밀 값을 유출시킬 수 있습니다. 단, <code>process.env</code> 같은 런타임 비밀 값은 영향을 받지 않습니다.</li>
<li>Server Components를 사용하지 않거나 서버 없이 React를 사용하는 앱은 이번 취약점들의 영향을 받지 않으므로, 자신의 스택이 RSC를 지원하는 프레임워크·번들러를 쓰고 있는지 먼저 확인한 뒤 필요한 경우 즉시 업그레이드하는 것이 중요합니다.</li>
