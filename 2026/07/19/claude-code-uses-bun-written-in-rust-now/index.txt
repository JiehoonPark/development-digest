14:T1005,<h2>Claude Code가 Rust로 재작성된 Bun을 쓰고 있다</h2>
<p>Bun의 창시자 Jarred Sumner가 &quot;Bun을 Rust로 재작성했다(Rewriting Bun in Rust)&quot;는 글에서 흥미로운 언급을 남겼습니다. Claude Code v2.1.181(6월 17일 출시) 이후 버전부터 Rust로 포팅된 Bun을 사용하고 있다는 내용인데요. Simon Willison이 이 주장을 직접 검증해본 과정을 정리했습니다.</p>
<h3>Jarred Sumner의 주장</h3>
<p>Jarred Sumner는 Bun을 Rust로 재작성한 이야기를 다루면서, 이미 Claude Code가 이 Rust 버전 Bun을 실제 프로덕션에서 쓰고 있다고 밝혔습니다. 그의 표현을 그대로 옮기면 다음과 같습니다.</p>
<p>&gt; Claude Code v2.1.181(6월 17일 출시) 이후 버전은 Rust로 포팅된 Bun을 사용합니다. Linux에서 시작 속도가 10% 빨라졌지만, 그 외에는 거의 아무도 눈치채지 못했습니다. Boring is good(눈에 띄지 않는 게 좋은 겁니다).</p>
<h3>직접 확인해보기</h3>
<p>Simon Willison은 자신의 Claude Code 설치 환경에서 이 주장이 사실인지 직접 확인해봤습니다. 다음 두 가지 명령어를 통해 설득력 있는 증거를 찾았다고 합니다.</p>
<p>첫 번째 명령어를 실행하자 다음과 같은 결과가 나왔습니다.</p>
<pre><code>Bun v1.4.0 (macOS arm64)</code></pre>
<p>여기서 눈여겨볼 점은 버전 번호입니다. GitHub에 공개된 Bun의 최신 릴리스는 이 글 작성 시점 기준 5월 12일에 나온 v1.3.14였습니다. 그런데 Claude Code 안에 포함된 Bun은 v1.4.0이었죠. 아직 정식 출시되지 않은 버전이 Claude Code 내부에는 프리뷰 형태로 이미 탑재되어 있었다는 뜻입니다.</p>
<p>(업데이트: 이후 이 Rust 버전은 Bun canary 채널로 정식 공개되었습니다. <code>bun upgrade --canary</code> 명령을 실행하면 이 버전을 설치할 수 있습니다.)</p>
<p>두 번째 명령어는 563개의 파일명 목록을 출력했는데, 이 역시 Rust로 작성된 Bun의 내부 구성을 짐작할 수 있는 단서였습니다.</p>
<p>이 두 가지 결과를 종합해보면, Rust로 작성된 Bun이 실제로 수백만 대의 디바이스에서 프로덕션 환경으로 이미 돌아가고 있다는 걸 확인할 수 있었습니다. Jarred Sumner의 말대로 &quot;Boring is good&quot;이라는 표현이 딱 들어맞는 상황입니다.</p>
<h3>추가로 확인된 증거들</h3>
<p>Ajan Raj가 흥미로운 방법 하나를 추가로 공유했습니다. 이를 통해 Claude Code의 <code>package.json</code>에서 Bun 버전이 1.4.0으로 업데이트된 커밋을 직접 찾을 수 있었는데요. 이 커밋은 5월 17일에 작성된 것으로 확인됐습니다. 흥미롭게도 이 버전 번호는 이후로 전혀 바뀌지 않았고, 아직 canary 이외의 정식 태그된 릴리스로는 나오지 않은 상태입니다.</p>
<h3>정리</h3>
<ul>
<li>Jarred Sumner는 Claude Code v2.1.181(6월 17일 출시) 이후 버전이 Rust로 포팅된 Bun을 사용 중이며, Linux 기준 시작 속도가 10% 빨라졌다고 밝혔습니다.</li>
<li>Simon Willison은 자신의 Claude Code 설치본에서 Bun 버전을 확인하는 명령을 실행해 v1.4.0(macOS arm64)이라는 결과를 얻었고, 이는 당시 공식 릴리스(v1.3.14)보다 앞선 버전이었습니다.</li>
<li>563개 파일 목록 출력 등 추가 증거를 통해 Rust 버전 Bun이 수백만 디바이스에서 실제로 프로덕션 운영 중임을 확인했습니다.</li>
<li>이 Rust 버전은 이후 Bun canary 채널로 공식 배포되었으며, <code>bun upgrade --canary</code>로 설치할 수 있습니다. <code>package.json</code>의 버전 변경 커밋은 5월 17일자로 확인됐습니다.</li>
<li>대규모로 쓰이는 핵심 런타임을 언어 자체를 바꿔 재작성하면서도 사용자가 거의 체감하지 못할 정도로 안정적으로 전환했다는 점에서, &quot;눈에 띄지 않는 마이그레이션&quot;이 잘 이뤄진 사례로 볼 수 있습니다.</li>
